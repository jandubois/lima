## Executive Summary

PR #4985 adds a `Windows` OS type, two arch-specific `autounattend.xml` files (amd64 and arm64), a developer ISO-generation tool, and a new `generateWindowsISO()` branch in `GenerateISO9660`. The structural choices are sound: dispatching on `*instConfig.OS == limatype.WINDOWS` at the cidata seam keeps the Linux/Darwin/FreeBSD paths untouched, Joliet labeling is correctly applied so `autounattend.xml` survives ISO9660 8.3 truncation, and using NVMe on arm64 avoids needing `virtio-win.iso`.

The most consequential finding is that the two `autounattend.xml` files contain **zero Go template directives** — they pass through `textutil.ExecuteTemplate` unchanged, so the host's SSH public keys, the configured `User.Name`, the instance name (`Hostname`), and the timezone never reach the guest. The PR description claims "OpenSSH server is installed and started during OOBE so Lima's host-agent can connect"; with `ssh-rsa PLACEHOLDER` written to `administrators_authorized_keys` and the local account hardcoded to `lima`, that claim does not hold today. A separate PowerShell bug in the same `FirstLogonCommands` block uses backtick escapes (`` `r`n ``) inside a single-quoted string, which PowerShell does not interpret — the literal characters land in `sshd_config`. And the same line's unanchored `-replace '#?AuthorizedKeysFile.*'` regex rewrites every `AuthorizedKeysFile` line in `sshd_config`, including the line inside the default `Match Group administrators` block.

Beyond the answer files, three integration gaps will surface as soon as Lima tries to drive the path: `limayaml.Validate` accepts `os: Windows` but `pkg/instance/start.go`'s `Prepare` switch has no Windows case, so it falls through to `default` and calls `usrlocal.GuestAgentBinary("Windows", arch)`, which then returns `fs.ErrNotExist` because the Makefile only builds Linux and Darwin guest agents; the QEMU driver still exposes virtio disk, virtio-scsi cidata, and virtio-net while the answer files assume SATA/AHCI (amd64) or NVMe (arm64), so WinPE would not see the disk during install; and `arch` is silently mapped to `"amd64"` for anything that is not exactly `AARCH64`, so a user writing `arch: riscv64, os: Windows` passes validation and gets an amd64 answer file with no warning. Finally, the Windows path silently drops `mounts`, `provision`, and `bootScripts` rather than rejecting them at validation time.

**Structure of findings:** 0 critical, 7 important, 10 suggestions, 3 design concerns (1 in-scope, 2 future). Strong agreement across the cloud trio on the substitution/placeholder and arch-fallback issues; the PowerShell escape bug (Claude I2), the `Prepare`/guest-agent gap (Codex I1), and the QEMU device-graph mismatch (Codex I2) are unique to a single agent each. The local trio (Qwen, Gemma) added no findings the cloud trio had not already raised, and Qwen failed its first pass entirely (see Agent Performance Retro).

## Important Issues

I1. **Templates contain no Go template directives — every per-instance value is lost** — `pkg/cidata/cidata.TEMPLATE.d.Windows.amd64/autounattend.xml`, `pkg/cidata/cidata.TEMPLATE.d.Windows.arm64/autounattend.xml` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

```xml
<ComputerName>lima-default</ComputerName>
<TimeZone>UTC</TimeZone>
...
<LocalAccount wcm:action="add">
  <Name>lima</Name>
  <DisplayName>lima</DisplayName>
...
<CommandLine>powershell.exe -NoProfile -Command "Set-Content -Path 'C:\ProgramData\ssh\administrators_authorized_keys' -Value 'ssh-rsa PLACEHOLDER'; icacls ..."</CommandLine>
```

Neither autounattend.xml contains a single `{{` directive, yet both pass through `textutil.ExecuteTemplate` in `ExecuteTemplateAutounattend` (`pkg/cidata/template.go:241`). The rendered output equals the file on disk byte for byte. `TemplateArgs.SSHPubKeys`, `TemplateArgs.User`, `TemplateArgs.Hostname`, and `TemplateArgs.TimeZone` are all populated by `templateArgs()` but never reach the guest. The PR description states "OpenSSH server is installed and started during OOBE so Lima's host-agent can connect" — with `ssh-rsa PLACEHOLDER` written to `administrators_authorized_keys` and the local account hardcoded to `lima`, Lima's host-agent cannot connect using the host's real key as the configured user.

Fix: substitute the dynamic values via the template engine. The SSH keys need careful quoting because they pass through both XML and PowerShell layers; the simplest form is a PowerShell array literal joined by newlines, with one entry per key:

```xml
<CommandLine>powershell.exe -NoProfile -Command "$keys = @(
{{- range .SSHPubKeys }}
'{{ . }}',
{{- end }}
''); Set-Content -Path 'C:\ProgramData\ssh\administrators_authorized_keys' -Value $keys; icacls ..."</CommandLine>
```

Apply the same pattern to the `<ComputerName>` (use `{{ .Hostname }}`, then truncate to 15 chars to satisfy NetBIOS limits), the `<LocalAccount>`/`<AutoLogon>` `<Name>` and `<Username>` blocks (use `{{ .User }}`), and the `<TimeZone>` element (`{{ .TimeZone }}`, with a IANA→Windows mapping table since Windows expects Microsoft IDs like `"Pacific Standard Time"`, not IANA names).

I2. **PowerShell backtick escapes inside single-quoted string land as literal characters; unanchored `-replace` regex breaks the admin Match block** — `pkg/cidata/cidata.TEMPLATE.d.Windows.amd64/autounattend.xml:205`, `pkg/cidata/cidata.TEMPLATE.d.Windows.arm64/autounattend.xml:211` [Claude Opus 4.7]

```xml
<CommandLine>powershell.exe -NoProfile -Command "$cfg = 'C:\ProgramData\ssh\sshd_config'; (Get-Content $cfg) -replace '#?AuthorizedKeysFile.*', 'AuthorizedKeysFile .ssh/authorized_keys' | Set-Content $cfg; Add-Content $cfg 'Match Group administrators`r`n  AuthorizedKeysFile C:/ProgramData/ssh/administrators_authorized_keys'; Restart-Service sshd"</CommandLine>
```

Two compounding problems in this single line:

1. PowerShell single-quoted strings are literal. The `` `r`n `` inside the `Add-Content` argument is therefore written to `sshd_config` as four literal characters (backtick, r, backtick, n) rather than CRLF. The result is one physical line containing `Match Group administrators`r`n  AuthorizedKeysFile C:/ProgramData/ssh/administrators_authorized_keys`. sshd parses `Match` criteria as whitespace-separated tokens and rejects or silently ignores the malformed block.

2. The `-replace '#?AuthorizedKeysFile.*'` regex is unanchored. It matches every line that contains an `AuthorizedKeysFile` token, with an optional leading `#`. Windows OpenSSH ships `sshd_config` with a default `Match Group administrators` block whose `AuthorizedKeysFile` line directs admin-group authentication at `__PROGRAMDATA__/ssh/administrators_authorized_keys`. The replacement rewrites that line to `AuthorizedKeysFile .ssh/authorized_keys`, so even if problem (1) were fixed, admin-group authentication would not look at the path the script claims to populate.

Combined with **I1**, both halves of the SSH-key plumbing are broken: no real key is written, and the location the script claims to write to is no longer consulted by sshd.

Fix: switch the `Add-Content` argument to a double-quoted PowerShell string so `` `r`n `` expands, and anchor the `-replace` regex so it only touches the top-level commented-out line. Inside an XML attribute the double quotes need escaping with `&quot;` or with a PowerShell `[char]` expression:

```diff
-          <CommandLine>powershell.exe -NoProfile -Command "$cfg = 'C:\ProgramData\ssh\sshd_config'; (Get-Content $cfg) -replace '#?AuthorizedKeysFile.*', 'AuthorizedKeysFile .ssh/authorized_keys' | Set-Content $cfg; Add-Content $cfg 'Match Group administrators`r`n  AuthorizedKeysFile C:/ProgramData/ssh/administrators_authorized_keys'; Restart-Service sshd"</CommandLine>
+          <CommandLine>powershell.exe -NoProfile -Command "$cfg = 'C:\ProgramData\ssh\sshd_config'; (Get-Content $cfg) -replace '^#?AuthorizedKeysFile .*', 'AuthorizedKeysFile .ssh/authorized_keys' | Set-Content $cfg; Add-Content $cfg ([string]::Join([Environment]::NewLine, @('Match Group administrators', '  AuthorizedKeysFile C:/ProgramData/ssh/administrators_authorized_keys'))); Restart-Service sshd"</CommandLine>
```

A simpler alternative is to overwrite `sshd_config` entirely from a heredoc-style PowerShell array, removing the dependency on the existing file's layout.

Make `ExecuteTemplateAutounattend`'s switch exhaustive afterward (return an error in the default branch rather than silently selecting amd64), so a future caller cannot reintroduce the silent path.

I6. **Windows path silently drops `mounts`, `provision`, `bootScripts` rather than rejecting them at validation** — `pkg/cidata/cidata.go:393-402` [Gemini 3.1 Pro]

```go
// Windows guests use autounattend.xml instead of cloud-init.

    ...

```

The Windows branch returns from `GenerateISO9660` immediately after writing the autounattend ISO. It does not run `drv.BootScripts()`, does not loop over `instConfig.Provision`, and does not include any `Mount` entries. ```diff

```diff
+    if *y.OS == limatype.WINDOWS {
+        if len(y.Provision) > 0 {
+            errs = errors.Join(errs, errors.New("field `provision` is not yet supported for Windows guests"))

+        if len(y.Mounts) > 0 {
+            errs = errors.Join(errs, errors.New("field `mounts` is not yet supported for Windows guests"))
+        }
+    }
```

When the integration in **I3** lands, lift the restrictions one at a time so the validation matches what the driver actually implements.

If `os.Stat` returns an error, `fi` is nil and `fi.Size()` panics. The failure is unlikely on the happy path, but easy to remove: use `len(...)` over the encoded payload, or check the error. Minor — this is a dev tool, not production code.


## Design Observations

### Concerns

- **The "template" infrastructure is unused** *(in-scope)* [Claude Opus 4.7] — Neither autounattend.xml contains a Go template directive, yet both are run through `textutil.ExecuteTemplate` and called "templates." Either delete the templating step (treat them as plain embedded XML rendered with `os.WriteFile`) or actually substitute the parts that vary per instance — SSH keys, hostname, username, timezone, image index, EFI partition size, account password. The current state is the worst of both worlds: the dispatcher pretends to render, but every per-instance value is lost. This is the root cause of **I1** and several suggestions.

- **The standalone hack tool tests a different system than `limactl start` runs** *(in-scope)* [Codex GPT 5.5] — `hack/gen-autounattend-iso.go` uses the tested SATA/NVMe/USB device choices in its `printAmd64Usage`/`printArm64Usage` output. Production startup goes through the generic QEMU path (see **I3**). Moving the Windows device selection into the QEMU driver so the dev tool and production path share the same device graph would make the dev tool's tested commands serve as a contract the driver must match.

- **`templateArgs` is doing two jobs at once** *(future)* [Claude Opus 4.7] — `templateArgs` is the Unix cloud-init prep stage and the single entry point for any future "build args for an OS-specific provisioning artifact." Splitting it into `templateArgsCommon` (Name, IID, Hostname, User, SSHPubKeys, TimeZone) and `templateArgsUnix` (everything else) would let the Windows path be self-contained and decouple it from the FreeBSD/9p/MountType branches. The TODO at `pkg/cidata/template.go:89` already flags this; **I7** turns it into a concrete dependency for fixing **I1**.

### Strengths

- The dispatch on `*instConfig.OS == limatype.WINDOWS` sits at the natural seam in `GenerateISO9660`, so the Linux/Darwin/FreeBSD paths are untouched [Claude Opus 4.7, Codex GPT 5.5].
- Joliet labeling is correct, and the inline comment at `cidata.go:520-522` explains *why* — ISO9660 Level 1's 8.3 truncation would hide `autounattend.xml` from Windows Setup [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro, Gemma4 31B].
- Using NVMe on arm64 instead of carrying `virtio-win.iso` is the right tradeoff: WinPE ARM64 has `stornvme.sys` built in, removing an external download and a license question [Claude Opus 4.7, Gemma4 31B].
- Keeping separate amd64 and arm64 templates is reasonable while the install device layout and image indexes differ. Once the substitution work in **I1** lands, a single template with `{{ if eq .Arch "arm64" }}` blocks would reduce divergence — but premature deduplication would tie the hands of arch-specific fixes [Codex GPT 5.5, Gemma4 31B].

# Code Review: PR #4985 — POC: Windows guest support via autounattend.xml

## Executive Summary

This POC introduces a `Windows` OS type, adds two arch-specific `autounattend.xml` templates (amd64 / arm64), and dispatches Windows guests through a new `generateWindowsISO()` path that produces a Joliet ISO containing the answer file. The templates are static XML strings (no Go template substitution actually happens), so per-instance values such as SSH public keys, hostname, user name, and timezone never reach the answer file — and the PR description's claim that "OpenSSH server is installed and started during OOBE so Lima's host-agent can connect" does not hold with the hardcoded `ssh-rsa PLACEHOLDER` key. ## Findings

### Critical Issues

### Important Issues

**I1. SSH public keys are never injected — host-agent cannot authenticate** — `pkg/cidata/cidata.TEMPLATE.d.Windows.amd64/autounattend.xml:200`, `pkg/cidata/cidata.TEMPLATE.d.Windows.arm64/autounattend.xml:206` (important, regression)

```xml
<CommandLine>powershell.exe -NoProfile -Command "Set-Content -Path 'C:\ProgramData\ssh\administrators_authorized_keys' -Value 'ssh-rsa PLACEHOLDER'; icacls ..."</CommandLine>
```

The literal string `ssh-rsa PLACEHOLDER` is written to `administrators_authorized_keys`. There are no Go template directives (`{{ }}`) anywhere in either `autounattend.xml`, so `ExecuteTemplateAutounattend` (`pkg/cidata/template.go:241`) cannot substitute `args.SSHPubKeys`. The PR description states "OpenSSH server is installed and started during OOBE so Lima's host-agent can connect" — that assertion is false: the host's real keys never reach the guest, so Lima's host-agent (which always uses key-based auth) cannot connect once Lima is taught to drive this path. The same applies to the `ComputerName`, `User`, `TimeZone`, and locale fields, which are all hardcoded despite `TemplateArgs` carrying the values.

Fix: substitute the keys via the template engine. The simplest form is to render the line as PowerShell with all keys joined by newlines, base64-encoded or otherwise quoted to survive the XML/PowerShell layers; e.g.

```xml
<CommandLine>powershell.exe -NoProfile -Command "$keys = @(
{{- range .SSHPubKeys }}
'{{ . }}',
{{- end }}
''); Set-Content -Path 'C:\ProgramData\ssh\administrators_authorized_keys' -Value $keys; icacls ..."</CommandLine>
```

and similarly use `{{ .Hostname }}` for `<ComputerName>` and `{{ .TimeZone }}` for `<TimeZone>`. If the POC's intent is to add the wiring later, add a TODO at the placeholder line and a matching note in the PR description so the SSH claim is not misread.

**I2. PowerShell backtick escapes inside single-quoted string are literal — sshd Match block is malformed** — `pkg/cidata/cidata.TEMPLATE.d.Windows.amd64/autounattend.xml:205`, `pkg/cidata/cidata.TEMPLATE.d.Windows.arm64/autounattend.xml:211` (important, regression)

```xml
<CommandLine>powershell.exe -NoProfile -Command "$cfg = 'C:\ProgramData\ssh\sshd_config'; (Get-Content $cfg) -replace '#?AuthorizedKeysFile.*', 'AuthorizedKeysFile .ssh/authorized_keys' | Set-Content $cfg; Add-Content $cfg 'Match Group administrators`r`n  AuthorizedKeysFile C:/ProgramData/ssh/administrators_authorized_keys'; Restart-Service sshd"</CommandLine>
```

Two compounding problems in this line:

1. The argument to `Add-Content` is single-quoted, so PowerShell does not interpret `` `r`n ``. The text appended to `sshd_config` is one physical line containing literal backticks: `Match Group administrators`r`n  AuthorizedKeysFile C:/ProgramData/ssh/administrators_authorized_keys`. sshd parses Match criteria as whitespace-separated tokens; the resulting block is rejected or silently ignored.
2. The preceding `-replace '#?AuthorizedKeysFile.*'` regex matches *every* line beginning with optional `#` followed by `AuthorizedKeysFile`, including the `AuthorizedKeysFile` line inside the default Match block shipped with Windows OpenSSH. That replacement rewrites the default admin Match to use `.ssh/authorized_keys`, so even without (1) admin-group authentication via `administrators_authorized_keys` is broken.

Combined with **I1**, even if a real key were substituted, the sshd configuration would not direct it where the script claims to put it.

Fix: switch the `Add-Content` argument to a double-quoted string so `` `r`n `` expands, and anchor the `-replace` regex so it does not touch the line inside the existing Match block. A minimal change:

```diff
-          <CommandLine>powershell.exe -NoProfile -Command "$cfg = 'C:\ProgramData\ssh\sshd_config'; (Get-Content $cfg) -replace '#?AuthorizedKeysFile.*', 'AuthorizedKeysFile .ssh/authorized_keys' | Set-Content $cfg; Add-Content $cfg 'Match Group administrators`r`n  AuthorizedKeysFile C:/ProgramData/ssh/administrators_authorized_keys'; Restart-Service sshd"</CommandLine>
+          <CommandLine>powershell.exe -NoProfile -Command "$cfg = 'C:\ProgramData\ssh\sshd_config'; (Get-Content $cfg) -replace '^#?AuthorizedKeysFile .*', 'AuthorizedKeysFile .ssh/authorized_keys' | Set-Content $cfg; Add-Content $cfg \"Match Group administrators`r`n  AuthorizedKeysFile C:/ProgramData/ssh/administrators_authorized_keys\"; Restart-Service sshd"</CommandLine>
```

The double-quoted Match block must escape the XML quotes (`\"...\"`) or be expressed as PowerShell `[char]13 + [char]10` or `"`r`n"`. A simpler alternative is to write the desired sshd_config from a heredoc-style array and overwrite the file entirely.

## Design Observations

### Concerns

- **The "template" infrastructure is unused** *(in-scope)*. Neither `cidata.TEMPLATE.d.Windows.amd64/autounattend.xml` nor `cidata.TEMPLATE.d.Windows.arm64/autounattend.xml` contains a single Go template directive, yet both are run through `textutil.ExecuteTemplate` and called "templates." Either delete the templating step (treat them as plain embedded XML and produce them with `os.WriteFile`-grade code), or actually substitute the parts that vary per instance (SSH keys, hostname, username, timezone, image index, EFI partition size, account password). Right now the worst of both worlds applies: the dispatcher pretends to render, but every per-instance value is lost. This is the root cause of **I1, S1, S2, S3, S12**.

- **The two arch templates are 95% duplicated** *(future)*. The only differences are `processorArchitecture` (4 occurrences each), EFI partition size (`100` vs `260`), and image index (`6` vs `3`). A single template with `{{ if eq .Arch "arm64" }} ... {{ end }}` blocks would halve the maintenance surface and stop divergent bug-fix drift (e.g., **I2** must currently be fixed in two places). Combine with the TODO at `template.go:89` so the same template can also render to an embedded representation per OS family.

- **`templateArgs` is doing two jobs at once** *(future)*. As noted in **I4**, the function is the Unix cloud-init prep stage but also the single entry point for any future "build args for an OS-specific provisioning artifact." Splitting it into `templateArgsCommon` (Name, IID, Hostname, User, SSHPubKeys, TimeZone) + `templateArgsUnix` (everything else) would make the Windows path self-contained and decouple it from the FreeBSD/9p/MountType branches.

### Strengths

- The dispatch on `*instConfig.OS == limatype.WINDOWS` is placed at the natural seam (`GenerateISO9660`), so the Linux/Darwin/FreeBSD paths are completely untouched.
- The Joliet-only ISO label choice is correct and the inline comment at `cidata.go:520-522` explains *why* (8.3 truncation would hide `autounattend.xml` from Windows Setup), which is exactly the kind of "why" the convention asks for.
- The decision to use NVMe on arm64 instead of carrying `virtio-win.iso` is the right call — WinPE ARM64 has `stornvme.sys` built-in, so it removes an external download dependency and a license question.
