Deep Code Review: limactl shell --sync

Date2026-03-13
TargetFunctionality review of limactl shell --sync (rsync-based host-guest directory synchronization)
ReviewersClaude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro

Consolidated Review

Executive Summary

The --sync feature mirrors a host working directory into the guest VM via rsync, runs a shell session, then offers an interactive accept/reject/diff workflow before syncing changes back. The core happy path works, but the implementation has serious bugs: two switch-case ordering issues leave destRsyncDir unset in valid configurations, and the guest cleanup deletes far more than it should. Several secondary issues affect robustness (silent failures, state mutation, SSH transport bypass).

Critical Issues

1. --sync combined with --workdir leaves destRsyncDir empty Claude

cmd/limactl/shell.go

	workDir, err := cmd.Flags().GetString("workdir")
	if err != nil {
		return err
	}
	switch {
	case workDir != "":
		changeDirCmd = fmt.Sprintf("cd %s || exit 1", shellescape.Quote(workDir))
		// FIXME: check whether y.Mounts contains the home, not just len > 0

		changeDirCmd = fmt.Sprintf("cd %s", shellescape.Quote(hostCurrentDir))
		hostHomeDir, err := os.UserHomeDir()
		if err == nil && runtime.GOOS == "windows" {
			hostHomeDir, err = mountDirFromWindowsDir(ctx, inst, hostHomeDir)
		}
		if err == nil {
			changeDirCmd = fmt.Sprintf("%s || cd %s", changeDirCmd, shellescape.Quote(hostHomeDir))
		} else {
			logrus.WithError(err).Warn("failed to get the home directory")
		}
	case syncHostWorkdir:
		destRsyncDir = *inst.Config.User.Home + hostCurrentDir

destRsyncDir is assigned only in the case syncHostWorkdir branch (line 253). When --workdir is also set, case workDir != "" matches first (line 239), so destRsyncDir stays as "". The subsequent rsync operations at lines 348-388 proceed with an empty destination path — mkdir -p '' via SSH, and rsync to instance: with no path. No validation prevents combining --sync and --workdir.

3. Guest cleanup deletes entire top-level subtree, not just the synced directory Claude Codex Gemini

cmd/limactl/shell.go
func askUserForRsyncBack(ctx context.Context, cmd *cobra.Command, inst *limatype.Instance, sshCmd *exec.Cmd, hostCurrentDir, destRsyncDir string, rsync copytool.CopyTool, tty bool) error {
	remoteSource := fmt.Sprintf("%s:%s", inst.Name, destRsyncDir)
	clean := filepath.Clean(hostCurrentDir)
	parts := strings.Split(clean, string(filepath.Separator))
	if len(parts) < 2 {
		return fmt.Errorf("invalid host current directory: %s", hostCurrentDir)
	}
	dirForCleanup := shellescape.Quote(fmt.Sprintf("%s/", *inst.Config.User.Home) + parts[1])

	rsyncBack := func() error {
		paths := []string{
			remoteSource,
			hostCurrentDir,
		}

		if err := rsyncDirectory(ctx, cmd, rsync, paths); err != nil {
			return fmt.Errorf("failed to sync back the changes from guest instance to host: %w", err)
		}
		logrus.Info("Successfully synced back the changes to host.")
		return nil
	}

	defer func() {
		// Clean up the guest synced workdir
		if err := executeSSHForRsync(ctx, *sshCmd, inst.SSHLocalPort, inst.SSHAddress, fmt.Sprintf("rm -rf %s", dirForCleanup)); err != nil {
			logrus.WithError(err).Warn("Failed to clean up guest synced workdir")
		}
	}()

dirForCleanup is derived from parts[1] — the first path component after root (e.g., Users from /Users/jan/repo). The deferred rm -rf at line 460 removes ~/Users/ in the guest, wiping every synced directory under that top-level component. Concurrent --sync sessions from different host directories under the same top-level path destroy each other’s data.

Fix: Clean up exactly destRsyncDir:

dirForCleanup := shellescape.Quote(destRsyncDir)

Note: if destRsyncDir was already quoted for older rsync versions (line 365), use the unquoted original to avoid double-quoting.

Important Issues

1. Permission-only and metadata-only changes silently dropped Codex

cmd/limactl/shell.go
	rawOutput, stats, err := getRsyncStats(ctx, remoteSource, filepath.Dir(hostCurrentDir))
	if err != nil {
		logrus.WithError(err).Warn("failed to get rsync stats")
	}
	if stats != nil && stats.String() == "" {
		logrus.Info("No changes detected")
		return nil
	}

parseRsyncStats only counts lines starting with <, >, or c as changes. Lines like .f...p..... file (permission-only changes) start with . and are ignored. When all changes are metadata-only, stats.String() returns "", and askUserForRsyncBack returns early with “No changes detected” — silently discarding the guest-side changes without prompting.

Fix: Count metadata-only updates (lines where positions 3-10 contain non-. characters) as a separate category, or at minimum do not short-circuit sync-back when rsync reports itemized output.

3. executeSSHForRsync appends to a value-copied slice, risking silent corruption Claude Gemini

cmd/limactl/shell.go
func executeSSHForRsync(ctx context.Context, sshCmd exec.Cmd, sshLocalPort int, sshAddress, command string) error {
	sshCmd.Args = append(sshCmd.Args,
		"-p", strconv.Itoa(sshLocalPort),
		sshAddress,
	)

	// Skip Args[0] (program name) to avoid duplication
	sshRmCmd := exec.CommandContext(ctx, sshCmd.Path, append(sshCmd.Args[1:], command)...)
	if err := sshRmCmd.Run(); err != nil {
		return err
	}
	return nil
}

The function takes sshCmd exec.Cmd by value and appends to its Args slice. This works only because exec.CommandContext currently allocates Args without spare capacity, so append creates a new backing array. If Go’s implementation changed or a caller constructed exec.Cmd differently, the append would overwrite the caller’s data. The function is called multiple times with the same original sshCmd.

Fix: Copy the args explicitly:

newArgs := make([]string, len(sshCmd.Args), len(sshCmd.Args)+4)
copy(newArgs, sshCmd.Args)
newArgs = append(newArgs, "-p", strconv.Itoa(sshLocalPort), sshAddress, command)
sshRmCmd := exec.CommandContext(ctx, sshCmd.Path, newArgs[1:]...)

4. Command() permanently mutates receiver options Claude Gemini

pkg/copytool/rsync.go
func (t *rsyncTool) Command(ctx context.Context, paths []string, opts *Options) (*exec.Cmd, error) {
	copyPaths, err := parseCopyPaths(ctx, paths)
	if err != nil {
		return nil, err
	}

	if opts != nil {
		t.Options = opts
	}

t.Options = opts replaces the tool’s options for all future calls. The CopyTool instance is reused across shellAction and askUserForRsyncBack, so any opts override persists beyond the call that set it. Currently all callers pass nil, so the bug is latent.

Fix: Use a local variable instead of mutating the receiver:

activeOpts := t.Options
if opts != nil {
	activeOpts = opts
}

5. Silent failure if diff executable is missing Gemini

cmd/limactl/shell.go
		if err := diffCmd.Run(); err != nil {
			// Command `diff` returns exit code 1 when files differ.
			var exitErr *exec.ExitError
			if errors.As(err, &exitErr) && exitErr.ExitCode() >= 2 {
				_ = pipeIn.Close()
				return fmt.Errorf("failed to run diff command: %w", err)
			}
		}

When the user selects “View the changed contents”, the error handling only checks for *exec.ExitError (exit code ≥ 2). If diff doesn’t exist on the host, diffCmd.Run() returns *exec.Error, which doesn’t match errors.As(err, &exitErr). The error is silently ignored and the user sees a blank pager.

Fix: Handle non-ExitError failures:

if err := diffCmd.Run(); err != nil {
	var exitErr *exec.ExitError
	if errors.As(err, &exitErr) {
		if exitErr.ExitCode() >= 2 {
			_ = pipeIn.Close()
			return fmt.Errorf("diff command failed: %w", err)
		}
	} else {
		_ = pipeIn.Close()
		return fmt.Errorf("failed to start diff: %w", err)
	}
}

6. Path spaces break with rsync 3.0.0 through 3.2.3 Gemini

cmd/limactl/shell.go
		// The macOS release of rsync (the latest being 2.6.9) does not support shell escaping of destination path but other versions do.
		rsyncVer, err := rsyncVersion(ctx)
		if err != nil {
			return fmt.Errorf("failed to get rsync version: %w", err)
		}
		if rsyncVer.LessThan(*semver.New("3.0.0")) {
			destRsyncDir = shellescape.Quote(destRsyncDir)
		}

The code quotes destRsyncDir for rsync < 3.0.0 (macOS’s ancient 2.6.9). But --protect-args did not become the default until rsync 3.2.4. Users running rsync 3.0.0–3.2.3 (common on older Linux, e.g. Ubuntu 20.04) will see failures when syncDirVal contains spaces, as the remote shell splits the unquoted path.

Fix: Change the version check to < 3.2.4, or always pass --protect-args for rsync ≥ 3.0.0.

7. Pager displays raw ANSI escape codes Claude

cmd/limactl/shell.go
		diffCmd := exec.CommandContext(ctx, "diff", "-ruN", "--color=always", hostCurrentDir, hostTmpDest)
		pager := os.Getenv("PAGER")

The diff command uses --color=always and the rsync stats headers use ANSI codes. The default pager less does not interpret ANSI escapes by default, so users see raw \033[0;90m text.

Fix:

if pager == "" {
	pager = "less -R"
}

Suggestions

1. Use path instead of filepath for guest directories Gemini

filepath.Clean applies OS-specific path logic. On Windows, it uses backslashes for paths destined for the Linux guest. Use path.Clean/path.Dir for guest-side paths.

2. Temp directory created before it’s needed Claude

cmd/limactl/shell.go

		return err
	}
	defer func() {

			logrus.WithError(err).Warnf("Failed to clean up temporary directory %s", baseDir)

The temp dir for diff viewing is created unconditionally, even when the user picks “Yes” or “No.” Move the creation into the case 2 block.

3. Redundant rsync guest availability check Claude

cmd/limactl/shell.go
func getRsyncStats(ctx context.Context, source, destination string) (string, *rsyncStats, error) {
	paths := []string{source, destination}
	rsync, err := copytool.New(ctx, string(copytool.BackendRsync), paths, &copytool.Options{
		Verbose: true,
		AdditionalArgs: []string{
			"--dry-run",
			"--itemize-changes",
			"-ah",
			"--delete",
		},
	})
	if err != nil {
		return "", nil, err
	}

getRsyncStats creates a new rsync tool via copytool.New, which SSHs into the guest to check rsync availability — already verified when the primary tool was created. Pass the existing tool or skip the check.

4. parseRsyncStats could miscount verbose output lines Claude

cmd/limactl/shell.go
func parseRsyncStats(output string) *rsyncStats {
	var s rsyncStats
	for line := range strings.SplitSeq(output, "\n") {
		if len(line) < 12 {
			continue
		}

		if strings.HasPrefix(line, "*deleting") {
			s.Deleted++
			continue
		}

		updateType := line[0]
		checksum := line[2]

		if updateType == '<' || updateType == '>' || updateType == 'c' {
			if checksum == '+' {
				s.Added++
			} else {
				s.Modified++
			}
		}
	}
	return &s
}

getRsyncStats creates its rsync tool with Verbose: true, adding -v output. If a filename is 12+ chars and starts with </>/c with + at index 2, it gets falsely counted. Either remove Verbose: true (the --itemize-changes flag already produces the needed output) or validate position 1 as a valid file-type indicator.

Testing Assessment

Covered well:

  • Happy-path sync with --yes: path preservation, special characters, file CRUD, permissions, missing ControlMaster. BATS tests are solid, testing real rsync against a real VM.
  • parseRsyncStats and rsyncStats.String() unit tests cover core parsing.

Gaps:

  • --sync + --workdir combination (would expose CRITICAL empty-destRsyncDir bug)
  • --sync on WSL2 with --mount-none (would expose the other CRITICAL bug)
  • Interactive prompt flow (Yes/No/View) — askUserForRsyncBack TTY path has zero test coverage
  • Cleanup scope — no test verifies dirForCleanup targets the correct directory
  • Permission-only / metadata-only changes syncing back
  • Concurrent --sync sessions
  • Missing diff or pager binary
  • Custom SSH transport ($SSH env var)
  • getRsyncStats, rsyncVersion, executeSSHForRsync have no unit tests
  • The unit test at shell_test.go:34 codifies incorrect behavior by asserting .f...p..... is “no changes” Codex

Documentation Assessment

  • The --sync flag help text is minimal. It doesn’t mention the --mount-none requirement, the minimum directory depth restriction, or the dependency on rsync and diff.
  • No documentation warns that host-side changes during an active --sync session will be overwritten if the user accepts the sync-back (rsync --delete runs in both directions). Gemini
  • The rsyncMinimumSrcDirDepth = 4 constant explains the value but not the safety rationale (preventing accidental sync of home or system directories). Claude

Skill Improvement Recommendations

  1. Claude nesting is broken. Running claude --print from within a Claude Code session produced empty output despite env -u CLAUDECODE. The skill must account for this — either document that the user needs to run the Claude agent from a separate terminal, or find a reliable workaround.
  2. The prompt was good but could push harder on edge-case analysis. Claude found the switch-case bugs by analyzing control flow; the other agents didn’t. Adding a prompt instruction like “Trace every code path through switch/if chains” could help all agents catch this class of bug.
  3. git blame instruction was effective for Codex but not Gemini. Codex cited specific commits; Gemini didn’t use git blame at all. Consider making the instruction more explicit.
  4. Consider giving each agent a slightly different angle. The prompt could assign focus areas to reduce redundancy while maintaining coverage. The unanimous finding of the cleanup bug shows convergence works, but specialization might surface more unique findings.
  5. Model name note. The Gemini model gemini-3.1-pro-preview worked but is a preview release. Monitor for the GA model name.
