## Executive Summary

This is round 2 of the plain-Windows enablement branch — round 1's three
Important findings (cygpath sibling threading, missing QEMU PATH, gzip
ctx cancellation) are all addressed in the current HEAD. The new
`cygpathForSSH` helper, `WindowsSubsystemPathWithCygpath` API split,

The only new Important is Codex's: `pkg/hostagent/mount.go:65` runs the
host mount path through `PathForSSH(sshExe)` and uses the result as the

default `SFTPDriver` is empty (`pkg/limayaml/defaults.go:758`), so

native Win32-OpenSSH `sftp-server.exe` resolved separately gets `/c/...`

during `limactl start`.

the cancellation goroutine double-closes `in` (benign, but seeds future

## Important Issues

I1. **Reverse-sshfs `LocalPath` is keyed to `ssh.exe`, not the local `sftp-server`** — `pkg/hostagent/mount.go:55-79`, `pkg/limayaml/defaults.go:757-758` [Codex GPT 5.5]

```go
if runtime.GOOS == "windows" {
    // PathForSSH tracks the toolchain Lima is using: Cygwin-style
    // (e.g. /c/Users/jan) when ssh comes from Git for Windows / MSYS2 (where
    // sftp-server is also Cygwin-based), and native forward-slash form
    // (e.g. C:/Users/jan) when ssh is native Windows OpenSSH.
    sshExe, err := sshutil.NewSSHExe()
    ...
    resolvedLocation, err = sshutil.PathForSSH(ctx, sshExe, m.Location)
    ...
}
...
rsf := &reversesshfs.ReverseSSHFS{
    Driver:    *m.SSHFS.SFTPDriver,
    LocalPath: resolvedLocation,
```

`resolvedLocation` is keyed off `ssh.exe`, but it is `sftp-server` that
ultimately consumes the path. The comment asserts the two stay paired —
"Cygwin ssh implies Cygwin sftp-server" — but the code does not enforce
it. `m.SSHFS.SFTPDriver` defaults to `""` (`pkg/limayaml/defaults.go:758`),
which lets sshocker PATH-resolve `sftp-server` independently of
`NewSSHExe()`. A host with Git/MSYS2 `ssh.exe` first on `PATH`, no
Cygwin `sftp-server` available, and native Win32-OpenSSH Server's
`C:\Windows\System32\OpenSSH\sftp-server.exe` resolving for the
unqualified lookup hands `/c/Users/...` to a native sftp-server that
needs `C:/Users/...`. Mount fails during `limactl start`.

The trigger is narrow — same-toolchain installs are the common case,
and `$SSH` plus a divergent PATH or an explicitly-set `SFTPDriver` is
what surfaces it — but the docs at
`website/content/en/docs/config/vmtype/wsl2.md:59-68` promise
"Lima adapts path forms to the SSH toolchain," and the comment at
`mount.go:57-60` extends that promise to the sftp-server side without
enforcement.

Fix: resolve the sftp-server binary up front and pick the path form
that matches its toolchain, not `ssh.exe`'s. For the common case
(empty `SFTPDriver`, native sftp-server on a Win32-OpenSSH host)
`filepath.ToSlash(m.Location)` is correct regardless of what
`PathForSSH` would have returned. Alternatively, set
`rsf.OpensshSftpServerBinary` to a binary in the same toolchain as
the detected `ssh.exe` so the pairing is explicit.

## Suggestions

S1. **gzip cancellation goroutine double-closes `in`** — `pkg/downloader/downloader.go:551,567-585` [Claude Opus 4.7]

```go
defer in.Close() // line 551
...
if decompressCmd == "gzip" {
    ...
    ctx, cancel := context.WithCancel(ctx)
    defer cancel()
    go func() {
        <-ctx.Done()
        in.Close()
    }()
    _, err = io.Copy(out, gz)
    return err
}
```

`defer cancel()` always fires on the way out, so the watcher goroutine
always closes `in`. The outer `defer in.Close()` at line 551 then closes
the same file again. `os.File.Close()` on a closed file returns
`*fs.PathError("file already closed")`; both deferred close errors are
discarded, so today this is benign. The trap is that a future contributor
adding error logging to either close will see spurious "file already
closed" errors on every successful gzip download.

Fix: gate the goroutine on a `done` channel that fires before the
function returns, so the goroutine only closes `in` when ctx is
actually canceled by the caller:

```diff
-        ctx, cancel := context.WithCancel(ctx)
-        defer cancel()
+        done := make(chan struct{})
+        defer close(done)
         go func() {
-            <-ctx.Done()
-            in.Close()
+            select {
+            case <-ctx.Done():
+                in.Close()
+            case <-done:
+            }
         }()
```

S2. **gzip cancellation surfaces as `os.ErrClosed` rather than `ctx.Err()`** — `pkg/downloader/downloader.go:573-585` [Gemini 3.1 Pro]

```go
// Close in on ctx cancellation so io.Copy returns instead of
// blocking on disk reads until decompression finishes. io.Copy
// will surface the close as an os.ErrClosed-wrapping error, not
// ctx.Err() — callers that need to distinguish cancellation from
// I/O faults should check ctx.Err() after this returns.
ctx, cancel := context.WithCancel(ctx)
defer cancel()
go func() {
    <-ctx.Done()
    in.Close()
}()
_, err = io.Copy(out, gz)
return err
```

The author documented this trade-off in the comment, so it is a
deliberate contract, not an oversight. The downside is that every
caller now has to remember the `ctx.Err()` check to distinguish a
user-initiated cancellation from a corrupt source file. `decompressLocal`
already holds `ctx`; performing the translation here keeps the public
contract aligned with the other decompressor branches (where
`exec.CommandContext` already surfaces cancellation through
`ctx.Err()`):

```diff
-_, err = io.Copy(out, gz)
-return err
+_, err = io.Copy(out, gz)
+if err != nil && ctx.Err() != nil {
+    return ctx.Err()
+}
+return err
```

S3. **`IsAvailableOnGuest` swallows a `parseCopyPaths` failure into "rsync unavailable"** — `pkg/copytool/copytool.go:66-89`, `pkg/copytool/rsync.go:38-42` [Claude Opus 4.7]

```go
func (t *rsyncTool) IsAvailableOnGuest(ctx context.Context, paths []string) bool {
    copyPaths, err := parseCopyPaths(ctx, paths)
    if err != nil {
        logrus.Debugf("failed to parse copy paths for rsync availability check: %v", err)
        return false
    }
```

When `parseCopyPaths` fails — for example, `NewSSHExe()` cannot find
`ssh.exe` on `PATH` while resolving an absolute Windows path — the
error is logged at Debug and the function returns `false`. `New()`
then falls through to scp, which surfaces `"scp not found on host"`,
hiding the actual root cause. The pre-PR code did not need ssh to
parse local paths, so this is a new failure mode introduced by
threading `PathForSSH` through `parseCopyPaths`.

The practical trigger is narrow — native Win32-OpenSSH is part of
the Windows 10/11 base since 2018, so `ssh.exe` rarely is missing
in the first place — but when it does fire, the user has no signal
pointing at SSH.

Fix: log the underlying error at a level higher than Debug, or
short-circuit in `New()` when `NewSSHExe()` fails so the caller sees
the SSH lookup error directly instead of a downstream "scp not
found" message.

S5. **CI composite actions skip `$LASTEXITCODE` checks after native commands** — `.github/actions/windows_msys2_prep/action.yml:14`, `.github/actions/windows_qemu_install/action.yml:9`, `.github/actions/windows_wsl2_setup/action.yml:13-29` [Claude Opus 4.7]

```pwsh
& 'C:\msys64\usr\bin\pacman.exe' -Sy --noconfirm openbsd-netcat diffutils socat w3m
```

`$ErrorActionPreference = 'Stop'` does not propagate to native commands
in PowerShell. If `pacman`, `winget`, `wsl --update`, `wsl --import`,
or `tar -cf` fails, the step still succeeds and the next step fails
with a confusing downstream symptom. The other plain-Windows actions
(`windows_plain_build`, `windows_plain_smoke_test`,
`windows_plain_templates`) all use `if ($LASTEXITCODE -ne 0) { throw ... }`;
these three should follow the same pattern.

Representative fix for `windows_msys2_prep`:

```diff
   - name: Prepend MSYS2 to PATH and install test dependencies
     shell: pwsh
     run: |
+      $ErrorActionPreference = 'Stop'
       Add-Content -Path $env:GITHUB_PATH -Value 'C:\msys64\usr\bin'
       & 'C:\msys64\usr\bin\pacman.exe' -Sy --noconfirm openbsd-netcat diffutils socat w3m
+      if ($LASTEXITCODE -ne 0) { throw "pacman failed: $LASTEXITCODE" }
```

Apply equivalent guards to `winget install` in `windows_qemu_install`
and to each `wsl` / `tar` invocation in `windows_wsl2_setup`.

S6. **`windows_wsl2_setup` writes `dummy/` and `dummy.tar` into the checked-out workspace** — `.github/actions/windows_wsl2_setup/action.yml:20-29` [Claude Opus 4.7]

```pwsh
mkdir dummy
mkdir dummy\bin
mkdir dummy\etc
echo "" >dummy\bin\sh
tar -cf dummy.tar --format ustar -C dummy .
```

Composite actions necessarily run after `actions/checkout`, so these
scratch paths land inside `$GITHUB_WORKSPACE` alongside the source
tree. On fresh GitHub-hosted runners it is harmless, but it makes
the action non-idempotent (a repeated local run fails on
`mkdir dummy` with "directory exists") and a future `git status`-based
lint would see the artifacts as untracked.

Fix: write to `$env:RUNNER_TEMP`, matching what
`windows_plain_smoke_test` already does for `LIMA_HOME`:

```diff
   - name: Install dummy WSL2 distro
     shell: pwsh
     run: |
-      mkdir dummy
-      mkdir dummy\bin
-      mkdir dummy\etc
-      echo "" >dummy\bin\sh
-      tar -cf dummy.tar --format ustar -C dummy .
-      wsl --import dummy $env:TEMP dummy.tar
+      $scratch = Join-Path $env:RUNNER_TEMP 'wsl2-dummy'
+      New-Item -ItemType Directory -Force -Path "$scratch\bin","$scratch\etc" | Out-Null
+      echo "" > "$scratch\bin\sh"
+      tar -cf "$scratch\dummy.tar" --format ustar -C $scratch .
+      wsl --import dummy $env:TEMP "$scratch\dummy.tar"
```

S7. **`cygwinDetectCache` "log once" promise breaks under contention** — `pkg/sshutil/sshutil.go:88-135` [Claude Opus 4.7]

The doc comment on `IsSSHCygwin`/`cygpathForSSH` says "Results are
cached per resolved absolute path and logged once per path at Debug
level," but the implementation is check-then-act between the
`RLock`/`RUnlock` and the `Lock`/`Unlock`:

```go
cygwinDetectCacheRW.RLock()
cached, ok := cygwinDetectCache[path]
cygwinDetectCacheRW.RUnlock()
if ok {
    return cached, cached != ""
}
cygpathExe := filepath.Join(filepath.Dir(path), "cygpath.exe")
if _, err := os.Stat(cygpathExe); err != nil {
    logrus.Debugf("ssh at %q detected as native Windows OpenSSH ...", path)
```

Two goroutines first-hitting the same path both miss the cache,
both `os.Stat`, both log, both write the same value. The cache is
correct, but the log line appears twice. Use `sync.Map` with a
`sync.Once` per key, or accept the contract gap by softening the
comment.

S9. **`ParseOpenSSHVersion` regex still misses the space-form Win32-OpenSSH banner** — `pkg/sshutil/sshutil.go:482` [Claude Opus 4.7]

```go
regex := regexp.MustCompile(`(?m)^OpenSSH(?:_for_Windows)?_(\d+\.\d+)(?:p(\d+))?\b`)
```

The `(?:_for_Windows)?_` group requires an underscore between
"Windows" and the version. Older Win32-OpenSSH releases reportedly
emit `OpenSSH_for_Windows 9.5p2` (space, not underscore); those
banners would silently return `0.0.0`, triggering the legacy-SSH
scp path in `pkg/copytool/scp.go:76`. The round-1 review raised a
related concern about 4-part version banners and the author
declined it as speculative ("no banner observed in 4-part shape").
Same precedent applies here — keep this open as a Suggestion that
becomes Important the moment a single field report shows a 0.0.0
downgrade in the wild.

Fix:

```diff
-regex := regexp.MustCompile(`(?m)^OpenSSH(?:_for_Windows)?_(\d+\.\d+)(?:p(\d+))?\b`)
+regex := regexp.MustCompile(`(?m)^OpenSSH(?:_for_Windows[_ ])?_?(\d+\.\d+)(?:p(\d+))?\b`)
```

and add `OpenSSH_for_Windows 9.5p2` to `TestParseOpenSSHVersion`.

## Design Observations

### Strengths

- **`PathForSSH` + `cygpathForSSH` clean split** *(in-scope)* [Claude Opus 4.7, Codex GPT 5.5] — Round 1's I1 fix introduces a clean separation: `cygpathForSSH` returns both the detection bool and the sibling `cygpath.exe` path, `PathForSSH` threads that sibling to `WindowsSubsystemPathWithCygpath`, and `WindowsSubsystemPath` (bare-PATH) is kept as a thin wrapper for callers without an ssh.exe in hand. The internal helper plus exported `WithCygpath` API split is the right shape.
- **Composite-action sequencing comments are load-bearing and explicit** *(in-scope)* [Claude Opus 4.7] — `windows_plain_host`'s "MUST run AFTER any step that needs git CLI" and `windows_plain_templates`'s "MUST run before `windows_plain_host`" lock in the implicit ordering between actions; a future job restructure that breaks the order fails loudly instead of producing a strange downstream symptom.
- **`windows_plain_host` four-layer verification holds up** *(in-scope)* [Claude Opus 4.7] — Filesystem, PATH across three scopes, smoking-gun binaries, and registry uninstall keys covered; the WSL `C:\Windows\System32\bash.exe` allowlist exception is in the right place.

### Concerns

- **Two distinct code paths for "convert a host path for a Linux guest"** *(future)* [Claude Opus 4.7] — `cmd/limactl/shell.go:714` and `pkg/limayaml/defaults.go:698` still call the bare-PATH `ioutilx.WindowsSubsystemPath`, while copytool / sshutil / hostagent funnel through `PathForSSH`. On plain Windows both paths produce the same drive-letter form; on a Cygwin install with a custom `fstab`, they would diverge. Worth a follow-up to either route the mount-point computation through `PathForSSH` or document why two paths are needed.
- **Toolchain coupling between ssh.exe and the local sftp-server is asserted but not enforced** *(in-scope)* [Codex GPT 5.5] — See I1. The right shape may be a small "Windows SSH toolchain" descriptor that carries the selected ssh executable and the matching sftp-server, so future call sites that need the pair (mount, future remote-execution helpers) cannot accidentally key them off only one half.

# Deep Code Review: `windows-plain-ci`

**Verdict:** Approve with minor changes. The PR adds plain-Windows CI coverage and the host-side runtime changes (native OpenSSH detection, gzip in-process decompression, drive-letter handling, ControlMaster stripping for scp/rsync) needed to make Lima work without MSYS2/Git-for-Windows. ## Executive Summary

The branch lets Lima run on a vanilla Windows 10/11 host (native OpenSSH, no Cygwin/MSYS2) by detecting the SSH toolchain via a sibling `cygpath.exe` probe, routing path conversions through `PathForSSH`, stripping mux options for scp/rsync on Windows (no Unix-domain sockets), and decompressing gzip in-process. **S1. Goroutine in gzip path causes benign double-close of `in` — `pkg/downloader/downloader.go:567-586`**

```go
if decompressCmd == "gzip" {
    gz, err := gzip.NewReader(bar.NewProxyReader(in))
    if err != nil {
        return err
    }
    defer gz.Close()
    ctx, cancel := context.WithCancel(ctx)
    defer cancel()
    go func() {
        <-ctx.Done()
        in.Close()
    }()
    _, err = io.Copy(out, gz)
    return err
}
```

When `io.Copy` finishes normally, `defer cancel()` fires, the watcher goroutine wakes up and calls `in.Close()`, then the outer `defer in.Close()` at line 551 closes the already-closed file. `os.File.Close()` on an already-closed file returns `*fs.PathError` with `Err = "file already closed"`. Both deferred closes drop the return value, so the second close error is silently swallowed — benign but adds noise to diagnostic stacks and is the sort of thing that traps a future contributor adding error logging to `defer in.Close()`.

The pattern also leaves a small window where the goroutine reads `in` (via the still-running progress-bar reader's reference) just as the outer `defer` is closing it. For an `*os.File` this is OS-safe; for a future change that swaps `in` to a non-`*os.File`, the race becomes real.

Fix: gate the goroutine on a `done` channel so it can be unblocked without closing `in`:

```diff
-        ctx, cancel := context.WithCancel(ctx)
-        defer cancel()
+        done := make(chan struct{})
+        defer close(done)
         go func() {
-            <-ctx.Done()
-            in.Close()
+            select {
+            case <-ctx.Done():
+                in.Close()
+            case <-done:
+            }
         }()
```

This removes the double-close and bounds the goroutine to the original `ctx`'s cancellation.

### Strengths

- **Toolchain detection is anchored on a verifiable sibling file** (`cygpath.exe` next to `ssh.exe`) rather than a fragile heuristic on the path string, with `EvalSymlinks` to handle Chocolatey/scoop shims. The cached, per-resolved-path result keeps repeated calls cheap.

### Concerns

- **`mountDirFromWindowsDir` and `pkg/limayaml/defaults.go:698` still call `ioutilx.WindowsSubsystemPath`** (the bare-PATH cygpath variant) and rely on the new native fallback in `WindowsSubsystemPathWithCygpath` (`pkg/ioutilx/ioutilx.go:80-91`) to keep working on plain Windows. On plain Windows they produce the same answer (forward-slash drive-letter form); on a Cygwin-with-fstab install they may diverge. Worth a follow-up to either route the mount-point side through `PathForSSH` or document why two paths are needed. `(future)`

### Executive Summary

This change implements robust support for vanilla Windows hosts by decoupling Lima from the MSYS2/Git for Windows Cygwin toolchain. It intelligently detects whether native Windows OpenSSH or a Cygwin-based OpenSSH is in use, routing paths and multiplexing options accordingly. The PR is extremely well-structured and thoughtfully tested, with only a minor error-handling leak in the new pure-Go downloader decompression path.

### Findings

#### Critical Issues

None.

#### Important Issues

I1. **Leaking cancellation error as an I/O error** — `pkg/downloader/downloader.go:584-585` (important, enhancement)

```go
		_, err = io.Copy(out, gz)
		return err
```

When `ctx` is cancelled, the deferred goroutine correctly closes `in` to interrupt `io.Copy`, which causes `io.Copy` to return an `os.ErrClosed`-wrapping error. The inline comment notes that callers needing to distinguish cancellation from I/O faults should manually inspect `ctx.Err()`. This forces every caller to understand the internal interrupt mechanism. Because `decompressLocal` already has access to `ctx`, it should perform this translation itself to provide a clean API boundary.

Fix:

```diff
-		_, err = io.Copy(out, gz)
-		return err
+		_, err = io.Copy(out, gz)
+		if err != nil && ctx.Err() != nil {
+			return ctx.Err()
+		}
+		return err
```

#### Strengths

- **Accurate Path Preservations:** The PR correctly retains the strict `filepath.IsAbs(path)` check in `parseCopyPaths` instead of falling into the trap of using `filepath.VolumeName(p) != ""`. This guarantees that drive-relative forms like `C:foo.txt` are preserved and successfully mapped to instance `C`, honoring pre-existing behaviors without creating local-cwd-relative side effects.
- **Clean Toolchain Isolation:** By detecting the OpenSSH provider and decoupling the `cygpath` conversions from a globally assumed `PATH`, the implementation smoothly handles both native OpenSSH and Git for Windows interchangeably without hacking global PATH manipulation.
