## Executive Summary

This PR replaces a TODO in the MCP toolset with mount validation logic: `TranslateHostPath` now verifies that a host path falls under a configured mount and translates it to the corresponding guest path. The core design (hard errors instead of warnings, `filepath.Rel` for prefix matching) is sound. The main issues are a first-match-wins bug with nested mounts, a `filepath.Dir` call on a POSIX guest path that breaks on Windows, and guest paths leaking from `Glob`/`SearchFileContent` results that cannot be fed back to other tools.

## Important Issues

1. **Nested mounts resolve to wrong guest path** — `pkg/mcp/toolset/toolset.go:124-136` (important, regression) [Codex GPT 5.4]

```go
for _, mount := range ts.inst.Config.Mounts {
	location := filepath.Clean(mount.Location)
	if cleanPath == location {
		return *mount.MountPoint, true
	}
	rel, err := filepath.Rel(location, cleanPath)
	if err == nil && !strings.HasPrefix(rel, "..") {
		guestPath := path.Join(*mount.MountPoint, rel)
		return guestPath, true
	}
}
```

The loop returns the first matching mount, not the most specific one. Lima supports nested mounts (e.g., `pkg/cidata/template_test.go:114-115` has both `/Users/dummy` and `/Users/dummy/lima`). With mounts ordered as `["/Users/dummy" → "/Users/dummy", "/Users/dummy/lima" → "/mnt/lima"]`, a host path `/Users/dummy/lima/file.txt` matches the parent mount at line 131-135 and translates to `/Users/dummy/lima/file.txt` instead of the correct `/mnt/lima/file.txt`. All MCP tool functions (`ListDirectory`, `ReadFile`, `WriteFile`, `Glob`, `SearchFileContent`, `RunShellCommand`) call `TranslateHostPath`, so every operation can target the wrong directory. Confirmed by `git blame` — lines 122-139 are all from commit `439c057f`. The unit tests at `toolset_test.go:95-108` only cover disjoint mounts.

Fix: Scan all mounts and select the longest matching `mount.Location`. Add a test with overlapping mounts in both parent-first and child-first order.

```diff
 func (ts *ToolSet) translateToGuestPath(hostPath string) (string, bool) {
 	cleanPath := filepath.Clean(hostPath)
+	bestGuestPath := ""
+	bestLen := -1
 	for _, mount := range ts.inst.Config.Mounts {
 		location := filepath.Clean(mount.Location)
-
-		if cleanPath == location {
-			return *mount.MountPoint, true
-		}
-
 		rel, err := filepath.Rel(location, cleanPath)
-		if err == nil && !strings.HasPrefix(rel, "..") {
-			rel = filepath.ToSlash(rel)
-			guestPath := path.Join(*mount.MountPoint, rel)
-			return guestPath, true
+		if err == nil && !strings.HasPrefix(rel, ".."+string(filepath.Separator)) && rel != ".." {
+			candidate := *mount.MountPoint
+			if rel != "." {
+				candidate = path.Join(candidate, filepath.ToSlash(rel))
+			}
+			if len(location) > bestLen {
+				bestGuestPath, bestLen = candidate, len(location)
+			}
 		}
 	}
-
-	return hostPath, false
+	return bestGuestPath, bestLen >= 0
 }
```

2. **`filepath.Dir` on guest path breaks on Windows** — `pkg/mcp/toolset/filesystem.go:95` (important, gap) [Claude Opus 4.6]

```go
dir := filepath.Dir(guestPath)
err = ts.sftp.MkdirAll(dir)
```

`guestPath` is now always a POSIX path (e.g., `/mnt/home-user/foo/bar.txt`) because `translateToGuestPath` at line 134 constructs it with `path.Join`. On Windows, `filepath.Dir` converts forward slashes to backslashes, producing `\mnt\home-user\foo`, which SFTP would send to the Linux guest as an invalid path. Confirmed by `git blame` — this line was authored by `a34f9b105` and is pre-existing, but the PR changes the semantics of `guestPath` from a host-native path to a POSIX path, making this `filepath.Dir` call produce a newly broken result on Windows. The `"path"` package is already imported at `filesystem.go:11`.

Fix:
```diff
-	dir := filepath.Dir(guestPath)
+	dir := path.Dir(guestPath)
```

3. **`HasPrefix(rel, "..")` falsely rejects paths named `..something`** — `pkg/mcp/toolset/toolset.go:132` (important, regression) [Gemini 3.1 Pro]

```go
rel, err := filepath.Rel(location, cleanPath)
if err == nil && !strings.HasPrefix(rel, "..") {
```

If a directory under a mount is named `..ssh` (technically valid on Unix), `filepath.Rel` returns `"..ssh"` and `strings.HasPrefix("..ssh", "..")` is `true`, incorrectly rejecting a valid mounted path. The practical likelihood is very low — such directory names are rare — but the fix is simple and eliminates the edge case. `filepath.IsLocal` (Go 1.20+) correctly handles this and also returns `true` for `"."` (exact mount match), removing the need for the separate `cleanPath == location` check at line 127-129.

Fix:
```diff
-		if cleanPath == location {
-			return *mount.MountPoint, true
-		}
-
 		rel, err := filepath.Rel(location, cleanPath)
-		if err == nil && !strings.HasPrefix(rel, "..") {
+		if err == nil && filepath.IsLocal(rel) {
```

Note: This fix composes well with Finding 1's longest-match approach.

4. **Glob and SearchFileContent return guest paths that fail re-entry** — `pkg/mcp/toolset/filesystem.go:137-141, 174-178` (important, gap) [Gemini 3.1 Pro]

```go
pattern := path.Join(guestPath, args.Pattern)
matches, err := ts.sftp.Glob(pattern)
```

`Glob` calls `ts.sftp.Glob(guestPath)` and returns absolute guest paths in the result. `SearchFileContent` runs `git grep ... <guestPath>`, which prefixes output lines with guest paths. When the AI reads these results and passes them to `ReadFile` or `WriteFile`, `TranslateHostPath` rejects them because they are guest paths, not host paths. This works today only because the default mount configuration maps `$HOME` → `$HOME` (same path), but breaks with any asymmetric mount (e.g., `/home/user` → `/mnt/home-user`).

This is pre-existing design — before this PR, there was no translation at all — but the PR makes it more consequential by introducing strict validation. The fix is outside the scope of this PR, but worth tracking.

## Design Observations

**Concerns**

- **Glob/SearchFileContent path round-tripping** (in-scope) — Finding 4 describes a design gap where tool outputs contain guest paths that cannot be fed back as inputs. A proper fix requires translating guest paths back to host paths before returning results, or making all tools accept both host and guest paths. This is worth addressing in a follow-up PR. [Claude Opus 4.6, Gemini 3.1 Pro]

- **Glob/SearchFileContent CWD dependency** (future) — Both `Glob` (filesystem.go:126) and `SearchFileContent` (filesystem.go:163) fall back to `os.Getwd()` when no explicit path is provided. With mount validation active, the MCP server's working directory must be under a configured mount or these tools fail. The BATS tests work because the MCP server starts from `$HOME` (which is mounted), but this is an implicit dependency. [Claude Opus 4.6]

- **Case-insensitive file systems** (future) — `translateToGuestPath` uses exact string matching and `filepath.Rel`. On macOS (case-insensitive by default), `/Users/Jan` and `/users/jan` are the same directory, but the mount check would reject the non-matching case. [Gemini 3.1 Pro]

**Strengths**

- The use of `filepath` for host-side operations and `path` for guest-side in `translateToGuestPath` (lines 123-134) correctly separates OS-specific vs. POSIX semantics, with `filepath.ToSlash` bridging the two. [Claude Opus 4.6]
- The `filepath.Rel` approach at line 131-132 correctly handles the `/home/user2` vs. `/home/user` edge case without string manipulation, validated by the unit test at line 80-92. [Claude Opus 4.6]
- Converting from soft warnings to hard errors is the right safety boundary for MCP tools. [Claude Opus 4.6, Codex GPT 5.4]

### Gemini 3.1 Pro Review

### Executive Summary

This PR implements strict host path validation for the MCP server, ensuring that the AI agent can only access paths explicitly mounted into the Lima VM. While the validation correctly prevents unauthorized guest filesystem access, it introduces a critical abstraction leak where tools like `Glob` and `SearchFileContent` still return guest paths, rendering their outputs unusable for chained operations. Additionally, the new path validation logic incorrectly rejects valid directories starting with `..` (such as `..ssh`).

### Critical Issues

1. **`!strings.HasPrefix(rel, "..")` falsely rejects valid local paths like `..ssh`** — `pkg/mcp/toolset/toolset.go:132-136` (critical, regression)

```go
		rel, err := filepath.Rel(location, cleanPath)
		if err == nil && !strings.HasPrefix(rel, "..") {
			rel = filepath.ToSlash(rel)
			guestPath := path.Join(*mount.MountPoint, rel)
```

`filepath.Rel` returns a clean relative path. If the target path resides within the base location but its name begins with `..` (for example, `/home/user/..ssh`), `rel` will be exactly `"..ssh"`. The condition `!strings.HasPrefix("..ssh", "..")` evaluates to `false`, incorrectly rejecting a valid mounted path.

Fix: Use `filepath.IsLocal` (introduced in Go 1.20) which correctly determines if a relative path is entirely within the directory. This simplifies the function since `filepath.IsLocal(".")` is also `true`, removing the need for the `cleanPath == location` exact match check:
```diff
-		if cleanPath == location {
-			return *mount.MountPoint, true
-		}
-
		rel, err := filepath.Rel(location, cleanPath)
-		if err == nil && !strings.HasPrefix(rel, "..") {
+		if err == nil && filepath.IsLocal(rel) {
			rel = filepath.ToSlash(rel)
```

2. **Output of `Glob` and `SearchFileContent` leak guest paths, breaking tool chaining** — `pkg/mcp/toolset/filesystem.go:139-143`, `174-177` (critical, regression)

```go
	pattern := path.Join(guestPath, args.Pattern)
	matches, err := ts.sftp.Glob(pattern)
```

This PR introduces strict path validation in `TranslateHostPath`, returning a hard error if a path is not mounted. However, `Glob` calls `ts.sftp.Glob(guestPath)`, returning absolute *guest paths*, and `SearchFileContent` invokes `git grep ... <guestPath>`, which prefixes output lines with the absolute *guest path*. When the AI reads these results and passes the paths to `ReadFile` or `WriteFile` (which expect valid host paths), `TranslateHostPath` will reject them as "not under any mounted directory" unless the host and guest mount points happen to be identical strings. The MCP server must translate these guest paths back to host paths.

Fix: For `Glob`, trim the `guestPath` prefix and join it with the host `pathStr`:
```go
	for i, match := range matches {
		rel := strings.TrimPrefix(match, guestPath)
		if strings.HasPrefix(rel, "/") {
			rel = strings.TrimPrefix(rel, "/")
		}
		matches[i] = filepath.Join(pathStr, filepath.FromSlash(rel))
	}
```

For `SearchFileContent`, instruct `git grep` to search a relative path so it outputs relative paths. The AI agent can then easily resolve them against the host directory since it already knows the starting path:
```diff
-	if args.Include != nil && *args.Include != "" {

-	}
+	searchPath := "."
+	if args.Include != nil && *args.Include != "" {
+		searchPath = *args.Include
+	}

-		Command:   []string{"git", "grep", "-n", "--no-index", args.Pattern, guestPath},
+		Command:   []string{"git", "grep", "-n", "--no-index", args.Pattern, searchPath},
		Directory: pathStr, // Directory must be always set

```
