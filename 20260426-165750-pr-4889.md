## Executive Summary

The PR retains the `*grpc.ClientConn` inside `GuestAgentClient`, exposes
`Close()`, closes the previous client in `getOrCreateClient` before
replacing it, and registers a shutdown cleanup. The leak fix is correct,
the new locking is consistent with the existing `clientMu`, and the diff
is small and well-commented.

Two important issues survive consolidation:

1. The new `a.client = nil` write under lock collides with two
   pre-existing unprotected reads of `a.client` in `watchGuestAgentEvents`.
   Pre-PR those reads were a benign race (the field was never nilled
   between writes); post-PR the field is observable as `nil` between
   `Close()` and the new assignment, so the second read in
   `if a.client == nil || !isGuestAgentSocketAccessible(ctx, a.client)`
   can dereference nil and crash the hostagent.
2. `pkg/portfwd/listener.go`'s listener map keys by `host/guest` and
   skips re-creation, so the dialer closure captured by
   `DialContextToGRPCTunnel(client)` outlives the client. Closing the
   stale client now permanently breaks dialing through any listener
   that was set up before the reconnect, even though pre-PR the gRPC
   reconnect machinery would have papered over a guest-agent restart
   on the same socket.

Structure: 0 critical, 2 important, 2 suggestions.

## Important Issues

I1. **Closing the stale client strands dialers held by existing port-forward listeners** — `pkg/hostagent/hostagent.go:843-848` [Codex GPT 5.5, Claude Opus 4.7]

```go
if a.client != nil {
    if err := a.client.Close(); err != nil {
        logrus.WithError(err).Debug("failed to close stale guest agent client")
    }
    a.client = nil
}
```

`processGuestAgentEvents` builds the dialer once per loop iteration as
`dialContext := portfwd.DialContextToGRPCTunnel(client)` and hands it to
`a.grpcPortForwarder.OnEvent` (`pkg/hostagent/hostagent.go:898-899`).
`DialContextToGRPCTunnel` captures `client` in a closure
(`pkg/portfwd/client.go:46-67`); each future TCP/UDP accept calls
`client.Tunnel(...)` against that captured pointer.

`ClosableListeners.forwardTCP` keys the listener map by
`tcp/host/guest` and early-returns when the same key already exists,
so the dialer is **not refreshed** on a follow-up event:

```go
_, ok := p.listeners[key]
if ok {
    p.listenersRW.Unlock()
    return
}
```
(`pkg/portfwd/listener.go:99-103`)

After a guest-agent restart with an unchanged guest port set, the
existing host listener keeps its old `dialContext`; new TCP accepts
call `Tunnel()` on the now-`Close()`d `ClientConn` and fail
immediately with `ErrClientConnClosing`. Pre-PR the same scenario
worked because gRPC's automatic reconnect (via the
`WithContextDialer` that re-runs `a.createConnection`) re-dialed the
forwarded `ga.sock` on transport failure — the cost was the leak this
PR fixes.

Fix: have the dialer resolve the current client on every dial, so
existing listeners pick up new clients automatically:

```go
dialContext := func(ctx context.Context, network, addr string) (net.Conn, error) {
    client, err := a.getOrCreateClient(ctx)
    if err != nil {
        return nil, err
    }
    return portfwd.DialContextToGRPCTunnel(client)(ctx, network, addr)
}
a.grpcPortForwarder.OnEvent(ctx, dialContext, ev)
```

The same problem applies more weakly to `startInotify`
(`pkg/hostagent/inotify.go:34-38`), which captures the client once
and never re-fetches. That stream is bound to a single `ctx` and dies
naturally on guest-agent restart, so it is less urgent, but the
re-fetch pattern would close that gap too.

I2. **Data race on unprotected reads of `a.client` can crash hostagent with nil-pointer panic** — `pkg/hostagent/hostagent.go:731,758` [Codex GPT 5.5, Gemini 3.1 Pro]

```go
for {
    if a.client == nil || !isGuestAgentSocketAccessible(ctx, a.client) {
```
(`pkg/hostagent/hostagent.go:758`)

Three goroutines call `getOrCreateClient` concurrently:
`watchGuestAgentEvents` (`hostagent.go:764`), `startInotify`
(`pkg/hostagent/inotify.go:34`), and the timesync loop
(`pkg/hostagent/timesync.go:49`). Inside `getOrCreateClient` the new
code at lines 843-847 explicitly nils `a.client` between
`Close()` and the new `NewGuestAgentClient` assignment — a window of
microseconds inside the locked region.

The watcher loop at lines 731 and 758 reads `a.client` twice in the
short-circuit `||` expression *without* taking `clientMu`. If
goroutine A enters that window while goroutine B's loop is mid-
expression, the first read can observe the prior pointer
(`a.client == nil` returns false) and the second read can observe
the in-window `nil`. `isGuestAgentSocketAccessible(ctx, nil)` calls
`client.Info(ctx)` → `c.cli.GetInfo(...)`, which dereferences nil
and panics, taking the hostagent down.

Pre-PR this race already existed for line 731 (the inotify
goroutine), but `a.client` was never assigned `nil` between successful
constructions, so the second read could only observe a stale-but-
non-nil pointer — a benign data race. The new `a.client = nil` in the
Close path turns it into a crashable race.

Fix: serialize every read of `a.client` through the same mutex. A
small accessor avoids touching the loop body shape:

```go
func (a *HostAgent) getClient() *guestagentclient.GuestAgentClient {
    a.clientMu.RLock()
    defer a.clientMu.RUnlock()
    return a.client
}
```

Then in `watchGuestAgentEvents`:

```diff
-        if a.client == nil || !isGuestAgentSocketAccessible(ctx, a.client) {
+        client := a.getClient()
+        if client == nil || !isGuestAgentSocketAccessible(ctx, client) {
```

Apply the same change at line 731 (inside the inotify-startup
goroutine). Note: keeping `a.client = nil` in `getOrCreateClient` is
fine once readers go through the accessor — the assignment at line
850 will repopulate it before the lock is released, but the explicit
nil makes the failure mode clear if `NewGuestAgentClient` itself
returns an error.

Severity calibration: Codex flagged this as Important; Gemini
escalated to Critical citing the panic. We keep it at Important
because the race window is narrow (Close + re-assign happen back-to-
back inside one critical section) and the second read at line 758 is
the same goroutine that calls `getOrCreateClient`, so only the
inotify/timesync interleaving exposes it. The crash is real, but
not deterministic.

## Suggestions

S1. **Cleanup hook fires before context cancellation, so the watch loop can recreate one final client that then leaks** — `pkg/hostagent/hostagent.go:746-755`, `505-518` [Claude Opus 4.7, Gemini 3.1 Pro]

```go
a.cleanUp(func() error {
    a.clientMu.Lock()
    defer a.clientMu.Unlock()
    if a.client == nil {
        return nil
    }
    err := a.client.Close()
    a.client = nil
    return err
})
```

`startRoutinesAndWait` (`pkg/hostagent/hostagent.go:514-517`) calls
`a.close()` first and *then* `cancelHA()`. `close()` walks the
cleanup stack in LIFO order, which means this hook closes the client
and nils it before the watcher's parent `ctx` is cancelled. If the
`watchGuestAgentEvents` for-loop is at the top of an iteration in
that microsecond gap, it observes `a.client == nil`, calls
`getOrCreateClient`, dials a fresh `ClientConn`, and then
`<-ctx.Done()` fires on the next select — the new client never gets
closed.

Practical impact: at most one leaked `ClientConn` at process exit,
which the OS reclaims anyway. Worth a guard for hygiene rather than
correctness. The least invasive fix is a `shuttingDown` flag set
inside the cleanup under `clientMu` and checked at the top of
`getOrCreateClient`. A simpler alternative is to drop the cleanup
entirely and rely on `cancelHA()` to shut everything down — the FDs
are released on process exit and the leak this PR fixes is the
*per-reconnect* growth, not the one-conn-at-shutdown case.

Severity calibration: Gemini flagged this as Important arguing the
PR's clean-shutdown intent is defeated; we downgrade to Suggestion
because the at-most-one-leaked-conn-at-exit blast radius does not
match Important.

S2. **`Close()` docstring promises idempotency that the implementation does not enforce** — `pkg/guestagent/api/client/client.go:53-60` [Claude Opus 4.7, Codex GPT 5.5]

```go
// Close releases the underlying gRPC connection. It is safe to call on a nil
// receiver or a client whose connection has already been closed.
func (c *GuestAgentClient) Close() error {
    if c == nil || c.conn == nil {
        return nil
    }
    return c.conn.Close()
}
```

`c.conn` is set once in the constructor and never cleared, so the
"already closed" branch is unreachable; the second `Close()` call
dispatches into `grpc.ClientConn.Close()` again. In current grpc-go
that returns `nil` after a full shutdown, but the implementation does
not actually rely on that. The docstring sets up an invariant the
code does not guarantee, and a future caller that takes the comment
at its word will be surprised when grpc-go semantics change.

Both current call sites (`getOrCreateClient` and the cleanup) set
`a.client = nil` after Close, so the duplicate-close path is
unreachable today. Fix is one of:

```diff
 func (c *GuestAgentClient) Close() error {
     if c == nil || c.conn == nil {
         return nil
     }
-    return c.conn.Close()
+    conn := c.conn
+    c.conn = nil
+    return conn.Close()
 }
```

or tighten the docstring to drop the "already-closed" promise.

### Suggestions

**S1. `Close()` docstring is broader than the implementation — `pkg/guestagent/api/client/client.go:53-60`** *(suggestion, regression)*

```go
// Close releases the underlying gRPC connection. It is safe to call on a nil
// receiver or a client whose connection has already been closed.
func (c *GuestAgentClient) Close() error {
	if c == nil || c.conn == nil {
		return nil
	}
	return c.conn.Close()
}
```

The comment promises that calling `Close()` "on a client whose connection has already been closed" is safe (i.e. returns nil and does no work). But `c.conn` is only ever set in `NewGuestAgentClient` and never cleared after `Close()`, so on a second call the `c.conn == nil` branch is never taken — it dispatches into `grpc.ClientConn.Close()` again. In current grpc-go that returns `nil` on a fully-shutdown conn, but the implementation does not actually rely on that — it simply doesn't enforce the promise.

Today both call sites (`getOrCreateClient` at line 843-848 and the cleanup at line 746-755) set `a.client = nil` after Close, so the duplicate-close path is unreachable. But the docstring sets up an invariant the implementation does not guarantee, which will mislead a future caller that takes the docstring at its word.

Fix: either tighten the docstring or make the implementation match. Matching is cheap:

```diff
 func (c *GuestAgentClient) Close() error {
 	if c == nil || c.conn == nil {
 		return nil
 	}
-	return c.conn.Close()
+	conn := c.conn
+	c.conn = nil
+	return conn.Close()
 }
```

Note this would not be safe for *concurrent* `Close()` callers without external synchronization, but every current caller is already protected by `a.clientMu`.

**S2. Tiny shutdown window can still leak one client — `pkg/hostagent/hostagent.go:746-755`, `505-518`** *(suggestion, regression)*

```go
a.cleanUp(func() error {
	a.clientMu.Lock()
	defer a.clientMu.Unlock()
	if a.client == nil {
		return nil
	}
	err := a.client.Close()
	a.client = nil
	return err
})
```

`startRoutinesAndWait` (line 514-517) calls `a.close()` first, then `cancelHA()`. After this cleanup runs, `a.client == nil` but the parent `ctx` is still live until `cancelHA()` runs on the next line. If the `watchGuestAgentEvents` for-loop happens to be re-entering its body in that microsecond gap (e.g. `processGuestAgentEvents` just returned an error and the `select { case <-ctx.Done(): ...; case <-time.After(10s): ... }` fired on time), it will call `getOrCreateClient` which constructs a fresh `GuestAgentClient` whose cleanup has already fired — that one leaks for the rest of the process lifetime.

This is a microsecond race, not a per-reconnect leak, so the practical impact is at most one leaked conn at shutdown. Worth a brief comment noting that the for-loop's check should be tightened (or the cleanup should set a "shutting down" flag) if the window ever widens — but I would not gate the merge on it.

If you want to close it now, the smallest fix is to make `getOrCreateClient` honor `ctx`:

```diff
 func (a *HostAgent) getOrCreateClient(ctx context.Context) (*guestagentclient.GuestAgentClient, error) {
 	a.clientMu.Lock()
 	defer a.clientMu.Unlock()
+	if err := ctx.Err(); err != nil {
+		return nil, err
+	}
 	if a.client != nil && isGuestAgentSocketAccessible(ctx, a.client) {
 		return a.client, nil
 	}
```

This still has the same gap (cleanup runs *before* `cancelHA`, so `ctx.Err()` is nil), so it does not actually close the window — it only helps if the cleanup itself cancels a per-loop context. The cleanest fix is a `shuttingDown` bool guarded by `clientMu`, set inside the cleanup. Given the practical impact is one leaked conn per process lifetime, I would defer this unless other tightening lands here.

## Design Observations

### Concerns

**Long-lived consumers still hold stale client references after close** *(future)* — `pkg/hostagent/inotify.go:34-38`, `pkg/portfwd/client.go:46-67`

`startInotify` calls `getOrCreateClient` once at line 34, opens a streaming `Inotify(ctx)` at line 38, then loops on `inotifyClient.Send(event)`. The captured `client` is never re-fetched. Likewise, `DialContextToGRPCTunnel(client)` (`pkg/portfwd/client.go:46`) captures a `client` reference into a closure that is stored inside long-lived TCP/UDP listeners (see `Forwarder.OnEvent` at `pkg/portfwd/forward.go:82`). Existing listeners are keyed by `protocol/host/guest` (`pkg/portfwd/listener.go:97-104`) and skip re-creation, so the closure with the stale client survives across reconnects.

Before this PR the stale references were merely pointing at a leaked-but-alive `ClientConn` (gRPC-go would even attempt auto-reconnect via `WithContextDialer`/`createConnection`, which sometimes papered over a guest-agent restart). After this PR the stale references point at a *closed* `ClientConn`, so any subsequent `Send`/`Tunnel` call fails immediately rather than transparently re-dialing.

This is not a regression in functional behavior in the documented case (when the guest-agent actually restarts, the underlying `ga.sock` connection is dead anyway), but it is a behavioral change in the *failure mode* and can bite when `isGuestAgentSocketAccessible` returns false for a transient reason while the underlying conn was still recoverable. The right long-term fix is a small reconnect loop in `startInotify` (re-call `getOrCreateClient` and reopen the `Inotify` stream on send error), and a way for `DialContextToGRPCTunnel` to pull `a.client` lazily on each tunnel open. Both are out of scope for this PR — flagging so the follow-up doesn't get lost.

### Strengths

- The `Close()` API is correctly placed on `GuestAgentClient` (the layer that owns the conn) rather than leaking `*grpc.ClientConn` out to the caller. Keeps the abstraction intact.
- Both Close→nil sequences are guarded by the existing `clientMu`, so the change introduces no new locking. Concurrent `getOrCreateClient` callers (`syncTimeOnce`, `startInotify`, `watchGuestAgentEvents`) cannot race the Close.
- The shutdown cleanup is registered alongside the symmetric `grpcPortForwarder.Close` at line 745, so the ordering is obvious to a future reader.
- The Debug-level log on close failure is the right call — Close errors here aren't actionable.

### Gemini 3.1 Pro

### Executive Summary

This PR successfully resolves a long-running resource leak by exposing a `Close()` method on `GuestAgentClient` and invoking it whenever the hostagent replaces an unreachable connection, thus preventing unbounded goroutine and file descriptor growth across guest restarts. However, the implementation introduces a deterministic nil-pointer crash window during concurrent connection replacements, and the newly added shutdown hook races against the still-running agent loop to recreate and immediately leak a fresh connection.

### Findings

#### Critical Issues

**C1. Data race causing nil-pointer panic via explicit `a.client = nil`** — `pkg/hostagent/hostagent.go:842-848`

```go
	if a.client != nil {
		if err := a.client.Close(); err != nil {
			logrus.WithError(err).Debug("failed to close stale guest agent client")
		}
		a.client = nil
	}
```

The PR explicitly nulls out `a.client` during client replacement. This introduces a deterministic crash window because `watchGuestAgentEvents` performs an unprotected, lock-free double-evaluation of `a.client` in two places (the main loop and the inotify goroutine):

```go
if a.client == nil || !isGuestAgentSocketAccessible(ctx, a.client) {
```

If a concurrent caller (such as `timesync` or `inotify`) triggers `getOrCreateClient`, it acquires `a.clientMu` and sets `a.client = nil`. If this happens between the two evaluations in the `if` statement—after `a.client == nil` evaluates to `false`, but before `!isGuestAgentSocketAccessible(ctx, a.client)` evaluates `a.client` again—the second evaluation reads `nil`. 

This `nil` is passed to `isGuestAgentSocketAccessible`, which invokes `client.Info(ctx)`. `GuestAgentClient.Info()` calls `c.cli.GetInfo()`, which instantly panics with a nil pointer dereference, crashing the entire hostagent. Prior to this PR, `a.client` was overwritten directly without an explicit `nil` state, avoiding this specific panic window.

**Fix**: Protect reads of `a.client` using the existing `a.clientMu` lock, and remove the unnecessary `a.client = nil` assignment.

```go
// Add a helper to safely retrieve the client
func (a *HostAgent) getClient() *guestagentclient.GuestAgentClient {
	a.clientMu.Lock()
	defer a.clientMu.Unlock()
	return a.client
}
```

Replace the unprotected reads in `watchGuestAgentEvents` (both the loop and the inotify goroutine):
```diff
-		if a.client == nil || !isGuestAgentSocketAccessible(ctx, a.client) {
+		client := a.getClient()
+		if client == nil || !isGuestAgentSocketAccessible(ctx, client) {
```

Finally, remove the `a.client = nil` step in `getOrCreateClient`:
```diff
 	if a.client != nil {
 		if err := a.client.Close(); err != nil {
 			logrus.WithError(err).Debug("failed to close stale guest agent client")
 		}
-		a.client = nil
 	}
```
*(critical, regression)*

#### Important Issues

**I1. Cleanup hook races with agent loop, recreating leaked client during shutdown** — `pkg/hostagent/hostagent.go:746-755`

```go
	a.cleanUp(func() error {
		a.clientMu.Lock()
		defer a.clientMu.Unlock()
		if a.client == nil {
			return nil
		}
		err := a.client.Close()
		a.client = nil
		return err
	})
```

The PR registers a cleanup hook to cleanly close the `GuestAgentClient` on shutdown. However, the `a.close()` sequence (which executes all `cleanUp` hooks) is called *before* `cancelHA()` cancels the main context. 

This means the `watchGuestAgentEvents` loop is still actively running. When the loop ticks after the cleanup hook nulls `a.client`, it evaluates `a.client == nil` and calls `getOrCreateClient()`, which instantly dials a *new* gRPC connection to replace the closed one. Moments later, the context fires, the loop exits, and the newly created `GuestAgentClient` is permanently leaked. While an OS process exit ultimately reclaims FDs, this completely defeats the clean shutdown logic intended by the PR.

**Fix**: Remove the `a.client.Close()` cleanup hook entirely. The final `ClientConn` will be safely reclaimed by the OS upon process exit, and active streams are inherently terminated by context cancellation. If a clean close is strictly required, it must be performed *after* the `cancelHA()` context cancellation has completely stopped the agent loops.
*(important, regression)*

#### Suggestions

**S1. Document `GuestAgentClient.conn` lifecycle** — `pkg/guestagent/api/client/client.go:22`

The `GuestAgentClient` struct now manages connection state, but its `conn` field isn't explicitly documented as thread-safe for the new `Close()` method. Adding a brief comment noting that `conn` is initialized exactly once in the constructor and thus safe to `Close()` concurrently would clarify its lifecycle.
*(suggestion, enhancement)*

### Design Observations

**Concerns**

- **Blocking RPC under mutex `a.clientMu` (pre-existing)**: In `getOrCreateClient`, the `isGuestAgentSocketAccessible` helper invokes a unary gRPC call (`client.Info(ctx)`) while holding `a.clientMu`. Performing network I/O under a mutex blocks all other goroutines (`timesync`, `inotify`, `watchGuestAgentEvents`) that need to access the client. If the guest agent becomes unresponsive without dropping the socket, the mutex will be held hostage until the caller's context times out. *(in-scope)*
