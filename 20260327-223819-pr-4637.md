## Executive Summary

This PR adds on-demand conversion of non-raw disk images to raw format in the download cache. Drivers declare which formats they support via a new `SupportedImageFormats` field, and the downloader converts only when necessary. The design is clean and preserves the original cached image for drivers that support it (e.g., QEMU with qcow2). The main issues center on the conversion logic overwriting `expectedDigest` and `shadDigest`, which breaks both user-provided digest validation and Last-Modified freshness checks for digest-less images. A crash between writing the raw file and its digest file leaves the cache in an unverifiable state.

## Important Issues

1. **User-provided expected digest silently overwritten** ŌĆö `pkg/downloader/downloader.go:305-313` (important, regression) [Gemini 3.1 Pro]

```go

	logrus.Infof("Converted raw image is missing or stale; (re)converting now.")
	converted, rawDigest, err := ensureRawInCache(ctx, shadData, imageFormat, o.expectedDigest)
	if err != nil {
		return nil, err
	}
	shadData = converted
	o.expectedDigest = rawDigest
	shadDigest = rawImgConvDigestPath
}
```

When an image requires conversion, lines 311-313 replace `o.expectedDigest` with the raw image's digest and redirect `shadDigest` to `rawImgConvDigestPath`. The original user-provided digest (from the template) is never validated against the cached source image. Later, `validateCachedDigest` at line 329 compares the raw digest against its own file ŌĆö a self-confirming check.

If an upstream image at the same URL is updated and the user changes the expected digest in their template, the stale cached image is served without detection.

Fix: Validate the original cached file against the caller's `expectedDigest` before running conversion logic. Move the image-conversion block after the existing digest validation (lines 329-340), or perform the original validation explicitly within the conversion block:

```diff

+    if o.expectedDigest != "" && shadDigest != "" {
+        if err := validateCachedDigest(shadDigest, o.expectedDigest); err != nil {
+            return nil, nil // force re-download
+        }
+    }
```

## Design Observations

### Concerns

- **(in-scope)** The conversion code overwrites `o.expectedDigest` and `shadDigest` in `getCached`, mixing two concerns: "what format should the cached file be?" and "is the cached file fresh and authentic?" A small refactoring ŌĆö validate the original cached file first, then convert ŌĆö would eliminate findings 1, 2, and 3 as a class. [Claude Opus 4.6, Codex GPT 5.4]

- **(in-scope)** The `Cached` function (`downloader.go:345-399`) returns `CachePath: shadData` without awareness of image conversions. Any future caller using `Cached` for images would receive the original qcow2 path instead of the converted raw. Currently safe because `Cached` is used only for nerdctl archives. A comment noting this limitation would help. [Claude Opus 4.6]

- **(future)** Cache conversion always targets `raw` regardless of driver preference. For VZ with `diskImageFormat: asif`, the flow is: download qcow2 -> convert to raw in cache -> copy to instance dir -> convert raw to asif in `EnsureDisk`. This double conversion wastes I/O. A more general cache keyed by target format (e.g., `imgconv/raw`, `imgconv/asif`) would avoid this, but is acceptable for a first iteration since asif is new. [Claude Opus 4.6]

### Strengths

- Drivers declare supported formats via `SupportedImageFormats`, the downloader decides when to convert, and `EnsureDisk` handles the final step. This separation respects the existing layering. [Claude Opus 4.6, Gemini 3.1 Pro]

- Preserving the original cached artifact alongside `imgconv/raw` solves the qemu-vs-vz determinism problem raised in the PR discussion. [Codex GPT 5.4]

- The write-tmp-then-rename pattern in `ensureRawInCache` prevents serving partial conversions to concurrent readers. [Claude Opus 4.6]

### Gemini 3.1 Pro

### Executive Summary
This change introduces a caching mechanism for converted raw disk images, aiming to avoid repeated `qcow2` to `raw` conversions for drivers like VZ. However, the implementation introduces critical regressions: it overwrites the user-provided expected digest (bypassing validation of the original image) and skips the `diskUtil.Convert` step during disk creation, which breaks VM disk resizing. Additionally, there are gaps in handling compressed images and crash resilience during the cache conversion process.

### Findings

#### Critical Issues

1. **Missing validation of original user-provided digest (critical, regression)** ŌĆö `pkg/downloader/downloader.go:283-324`

```go
					if needConvert {
						// ...
						o.expectedDigest = rawDigest
						shadDigest = rawImgConvDigestPath
					} else {
						// ...
							if d, err := digest.Parse(currentDigest); err == nil {
								o.expectedDigest = d
								shadDigest = rawImgConvDigestPath
							}
```

When an image requires conversion (or uses an existing converted cache), `o.expectedDigest` and `shadDigest` are reassigned to the digest and path of the *converted raw image*. Later in the function (line 334), `validateCachedDigest(shadDigest, o.expectedDigest)` is called to verify the cache. Because both variables were overwritten, the check blindly compares the raw image's digest against its own `.digest` file, completely skipping validation of the *original* downloaded file against the user's `expectedDigest` from `lima.yaml`. If a user updates their configuration to a new digest but the file URL remains unchanged, Lima will fail to detect the mismatch and erroneously boot with the stale cached image.

Fix: Perform the original file's validation against the caller's `expectedDigest` *before* executing the `isNonISOImage` conversion logic.

2. **Bypassing conversion skips disk resizing (critical, regression)** ŌĆö `pkg/driverutil/disk.go:103-105`

```go
		if format == string(diskImageFormat) {
			if err = os.Rename(imagePath, diskPath); err != nil {
				return err
			}
		} else {
```

In `EnsureDisk`, if the downloaded image's format matches `diskImageFormat`, the code uses `os.Rename` to bypass `diskUtil.Convert`. However, `diskUtil.Convert` was previously responsible for resizing the downloaded image to `diskSizeInBytes` (e.g., expanding a 2GB base image to a 100GB VM disk). By bypassing it, the VM will boot with a disk size equal to the base image's original size instead of the user's configured disk size, severely breaking VM functionality.

Fix: Revert the bypass and unconditionally call `diskUtil.Convert` (which already handles same-format conversions optimally while applying the resize), or explicitly call `diskUtil.ResizeDisk(ctx, diskPath, diskSizeInBytes)` after renaming:
```diff
 		if format == string(diskImageFormat) {
 			if err = os.Rename(imagePath, diskPath); err != nil {
 				return err
 			}
+			if err := diskUtil.ResizeDisk(ctx, diskPath, diskSizeInBytes); err != nil {
+				return err
+			}
 		} else {
```

#### Important Issues

1. **Crash window leaves cache in inconsistent state (important, gap)** ŌĆö `pkg/downloader/downloader.go:449-463`

```go
	if err := os.Rename(rawPathTmp, rawImgConvPath); err != nil {
		return "", "", fmt.Errorf("failed to replace original with raw image: %w", err)
	}

	algo := digest.Canonical
	// ...
	rawDigest, err := calculateFileDigest(rawImgConvPath, algo)
	if err != nil {
```

`ensureRawInCache` renames the temporary raw image to its final destination `rawImgConvPath` *before* computing its digest and writing the `raw.digest` file. Computing a digest on a large file takes significant time. If the process crashes or is interrupted during this window, the cache is left with a valid raw file but no digest file. On the next run, `needConvert` evaluates to `false` (since the raw file exists), but reading the digest file fails silently, leaving the cache state partially populated and unvalidated.

Fix: Calculate the digest on the temporary file (`rawPathTmp`) *before* renaming it. Then, ensure the rename and the digest file write occur sequentially such that the presence of the main image file guarantees the digest file's existence.

2. **Cache conversion optimization is defeated for compressed images (important, gap)** ŌĆö `pkg/downloader/downloader.go:284` and `390`

```go

		format, err := nativeimgutil.DetectFormat(shadData)
```

`isNonISOImage` attempts to detect the format of `shadData` (the cached file). However, if the user downloads a compressed image (e.g., `ubuntu.qcow2.gz` or `.xz`), `shadData` contains the raw compressed bytes. `DetectFormat` cannot read the magic header of a compressed file and defaults to returning `"raw"`. Because most drivers support `"raw"`, `slices.Contains` evaluates to true, completely skipping the cache conversion. Later, `copyLocal` decompresses the file to a `.qcow2` in the instance directory, forcing `EnsureDisk` to perform the conversion per-instance. This entirely defeats the PR's primary goal of caching the converted image across multiple instances.

Fix: The cache layer stores compressed bytes verbatim, so checking the format before decompression is flawed. Consider either decompressing into the cache layer first (if a shared decompressed cache is desired), or document this limitation clearly in the code.
