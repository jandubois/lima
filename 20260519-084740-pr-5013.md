## Executive Summary

This PoC introduces `pkg/windows/autounattend` (a typed answer-file generator) and `pkg/windows/cidata` (an ISO builder), together with a manual `poc-windows-guest/` test harness for end-to-end verification on a Linux QEMU host. The architectural direction is sound — a small, testable Go surface plus a shell-driven proof — and the test suite includes a `PasswordNeverPlaintext` security assertion. But the exported API hides two silent-failure bugs that the current tests do not exercise: a Go-template arity error in the `ExtraCommands` path that fails at execute time, and a byte-index-as-rune-index error in the UTF-16LE password encoder that produces invalid output for any non-ASCII password. The committed manual answer file at `poc-windows-guest/autounattend.xml` undoes the package's security guarantee by storing the password in plaintext. The new ISO builder reimplements behavior that already exists in `pkg/iso9660util` and invokes the detected tools with flags they do not accept, so the advertised `xorriso` and `mkisofs` fallbacks fail.

Structure: 2 Critical, 14 Important, 11 Suggestions, 3 Design Observations. Verdict is Rework: the API surface needs C1, C2, I1, I2 fixed before this lands.

## Critical Issues

C1. **Template arity error in `ExtraCommands` Order computation** — `pkg/windows/autounattend/template.xml:175` [Claude Opus 4.7, Codex GPT 5.5]

```xml
{{range $i, $cmd := .ExtraCommands}}
<SynchronousCommand wcm:action="add">
  <Order>{{inc $i 1 | inc}}</Order>
```

`inc` is registered as `func(n int) int` in `autounattend.go:20`. The expression `inc $i 1` invokes `inc` with two arguments. Go's `text/template` checks function arity at execute time, not parse time, so the template parses but the first caller with a non-empty `ExtraCommands` slice gets `wrong number of args for inc: want 1 got 2` from `Generate(cfg)`.

The current tests never populate `ExtraCommands`, so CI sees no failure. The bug is silent until someone uses the exported `ExtraCommands` field, at which point `Generate` fails and no answer file is produced.

Fix: drop the stray `1` and let the pipeline produce `i+2`. Add a test case that exercises `ExtraCommands` with at least one entry so the template body executes in CI.

```diff
-          <Order>{{inc $i 1 | inc}}</Order>
+          <Order>{{inc $i | inc}}</Order>
```

Codex graded this Important; Claude graded it Critical. I keep it Critical because the bug sits on the exported API surface and surfaces only in production — the test suite cannot catch it without an explicit ExtraCommands test.

C2. **UTF-16LE encoder uses UTF-8 byte index as destination index** — `pkg/windows/autounattend/autounattend.go:81-89` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro, Gemma4 31B, Qwen 3.6 35B]

```go
func encodeUTF16LE(plaintext string) string {
	s := plaintext + "Password"
	b := make([]byte, len(s)*2)
	for i, r := range s {
		b[i*2] = byte(r)
		b[i*2+1] = byte(r >> 8)
	}
	return base64.StdEncoding.EncodeToString(b)
}
```

`range s` over a Go string yields `i` as the byte offset of each rune, not the rune index. For an ASCII password the two agree, so the security test `TestGenerate_PasswordNeverPlaintext` passes with `"T3stP@ssw0rd!"`. The moment `s` contains a multi-byte rune the destination index `b[i*2]` skips ahead by the rune's UTF-8 length times two, leaving zero bytes between writes. Decoders see embedded U+0000 NULs in the password. Claude verified empirically that input `"abç"` produces `61006200e7000000...` instead of `61006200e700...`.

Runes outside the BMP need surrogate-pair encoding (4 bytes). The current code never emits surrogates, so it silently produces invalid UTF-16 for code points above U+FFFF.

Windows decodes the base64 blob as UTF-16LE for both `LocalAccount` and `AutoLogon`. Two concrete failures:

* The account is created with a password the user never set.
* The `AutoLogon` block contains a different value than the account, so first boot stalls at the login screen and the unattended install hangs.

All five agents flagged this. Claude and Gemini graded it Critical; Codex and Gemma graded it Important; Qwen described it without a severity tag.

Fix: use `unicode/utf16.Encode` over a rune slice.

```go
import (
    "encoding/binary"
    "unicode/utf16"
)

func encodeUTF16LE(plaintext string) string {
    u := utf16.Encode([]rune(plaintext + "Password"))
    b := make([]byte, 2*len(u))
    for i, v := range u {
        binary.LittleEndian.PutUint16(b[i*2:], v)
    }
    return base64.StdEncoding.EncodeToString(b)
}
```

Add a unit test that feeds a non-ASCII password and decodes the base64 back to UTF-16LE to verify the round trip.

I2. **`pkg/windows/cidata` reimplements ISO writing and uses flags the detected tools reject** — `pkg/windows/cidata/cidata.go:77-103` [Codex GPT 5.5, Claude Opus 4.7, Gemini 3.1 Pro, Gemma4 31B, Qwen 3.6 35B]

```go
for _, candidate := range []string{"genisoimage", "mkisofs", "xorriso"} {
    if p, err := exec.LookPath(candidate); err == nil {
        return p, nil
    }
}
...
cmd := exec.Command(toolPath,
    "-output", outputPath,
    "-joliet",
    "-rock",
    "-volid", "cidata",
    stagingDir,
)
```

Two distinct problems compound here:

The detection set names `xorriso`, but the args use the `mkisofs` emulation dialect that `xorriso` accepts only with `-as mkisofs` prepended. Without that, `xorriso` parses `-output` as its own command and the invocation fails.

Long-form `-output`, `-joliet`, `-rock`, `-volid` are not standard for `genisoimage` or `mkisofs` — those tools use `-o`, `-J`, `-r`, `-V`. The advertised "fallback" only works on `xorrisofs` (the mkisofs-emulation binary), which the code does not detect.

The existing helper at `pkg/iso9660util/joliet.go:57-67` already solves all of this:

```go
func writeJolietCommand(isoPath, label, workDir string) ([]string, error) {
    if runtime.GOOS == "darwin" {
        return []string{"hdiutil", "makehybrid", "-o", isoPath, "-iso", "-joliet", "-default-volume-name", label, workDir}, nil
    }
    candidates := []string{"xorrisofs", "genisoimage", "mkisofs"}
    for _, cmd := range candidates {
        if cmdAbs, err := exec.LookPath(cmd); err == nil {
            return []string{cmdAbs, "-o", isoPath, "--norock", "-J", "-V", label, workDir}, nil
        }
    }
    return nil, fmt.Errorf("none of %v is available", candidates)
}
```

This handles macOS (`hdiutil makehybrid`) and detects the correct binary set (`xorrisofs`, not `xorriso`) with short flags that all three tools accept.

Fix: call `pkg/iso9660util.Write(..., iso9660util.WithJoliet())` from `BuildISO`. Drop the in-package `detectISOTool` and `runISOTool`. If the existing helper is insufficient, extend it rather than duplicate it.

I3. **SSH public keys passed through a PowerShell double-quoted command-line argument** — `pkg/windows/autounattend/template.xml:171`, `poc-windows-guest/autounattend.xml:171`, `poc-windows-guest/run.sh:59-60` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

```xml
<CommandLine>powershell -NoProfile -NonInteractive -ExecutionPolicy Bypass -File "F:\first_logon.ps1" -SSHKey "{{joinLines .SSHPublicKeys}}"</CommandLine>
```

```bash
sed "s|REPLACE_WITH_YOUR_SSH_PUBLIC_KEY|${ssh_key}|g" \
    "$SCRIPT_DIR/autounattend.xml" > "$TMP_DIR/autounattend.xml"
```

Three failure modes:

* `joinLines` joins multiple keys with `\n`. XML preserves the literal newline inside the element text, and Windows Setup forwards that to `cmd.exe`, which truncates or rejects multi-line command strings. The single-key test in `autounattend_test.go:15` hides the problem.
* An SSH key comment containing `"`, backtick, or `$()` survives XML escaping and is interpreted by PowerShell. A malicious or merely unusual comment can break argument parsing or trigger expression evaluation as Administrator.
* In `run.sh`, `sed` substitutes the key through a `|` delimiter. A key containing `|`, `\`, or `&` corrupts the XML before boot.

Gemini graded this Critical; Codex and Claude graded it Important. I treat it as Important because the PR controls both ends (the autounattend template and `first_logon.ps1`), so the fix is local. The severity rises once external SSH keys are accepted from `limayaml`.

Fix: stop passing the key on the command line. Stage an `authorized_keys` file on the cidata ISO. Have `first_logon.ps1` read it from `$PSScriptRoot` and copy it to `C:\ProgramData\ssh\administrators_authorized_keys`. This removes the quoting hazard, the newline issue, and the `sed` substitution risk in one change.

I4. **`SanitiseHostname` can return a name ending in `-`** — `pkg/windows/autounattend/autounattend.go:102-110` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro, Gemma4 31B]

```go
s := strings.Trim(string(out), "-")
if len(s) > 15 {
    s = s[:15]
}
if s == "" {
    return "lima-windows"
}
return s
```

`Trim` runs once, before truncation. If the 15-byte truncation lands inside a hyphen run, the result has a trailing `-`. Concrete repro: `"abcdefghijklmn-xyz"` filters and trims to `"abcdefghijklmn-xyz"`, then `[:15]` is `"abcdefghijklmn-"`. Windows NetBIOS rejects names ending in `-`, and Sysprep fails on first boot with the answer file's `<ComputerName>` rejected.

The test table at `autounattend_test.go:91-97` has `"this-is-way-too-long-for-netbios"` — that input truncates cleanly because the 15th byte is `o`, not `-`. The boundary case is untested.

Fix: re-trim after truncation.

```diff
 s := strings.Trim(string(out), "-")
 if len(s) > 15 {
-    s = s[:15]
+    s = strings.TrimRight(s[:15], "-")
 }
```

Add `{"abcdefghijklmn-xyz", "abcdefghijklmn"}` to the test table.

I5. **Native command exit codes are not caught by `Invoke-Step`** — `poc-windows-guest/first_logon.ps1:64-66`, `poc-windows-guest/first_logon.ps1:91` [Codex GPT 5.5, Gemini 3.1 Pro]

```powershell
icacls $keyFile /inheritance:r                | Out-Null
icacls $keyFile /grant 'SYSTEM:(F)'           | Out-Null
icacls $keyFile /grant 'Administrators:(F)'   | Out-Null
...
& $chocoExe install winfsp -y --pre --no-progress
```

`$ErrorActionPreference = 'Stop'` does not promote native-executable non-zero exit codes to PowerShell exceptions on Windows PowerShell 5.1 (the default shell on Server 2025). If `icacls` fails to grant `SYSTEM:(F)` or `choco install winfsp` fails (transient network error, package signing change), the enclosing `Invoke-Step` still prints `OK`. The script then continues to the next step, register the VirtIO-FS service against a missing WinFSP install. The provisioning log claims success while the guest comes up broken.

Fix: wrap native commands in a helper that checks `$LASTEXITCODE` immediately.

```powershell
function Invoke-Native {
    param([string]$FilePath, [string[]]$Arguments)
    & $FilePath @Arguments
    if ($LASTEXITCODE -ne 0) {
        throw "$FilePath exited with $LASTEXITCODE"
    }
}
```

I6. **Chocolatey bootstrap downloads and evaluates an unpinned install script as Administrator** — `poc-windows-guest/first_logon.ps1:79-92` [Claude Opus 4.7, Codex GPT 5.5, Qwen 3.6 35B]

```powershell
$installScript = (New-Object Net.WebClient).DownloadString(
    'https://community.chocolatey.org/install.ps1'
)
Invoke-Expression $installScript
...
& $chocoExe install winfsp -y --pre --no-progress
```

The provisioning script downloads `install.ps1` from Chocolatey and runs it with `Invoke-Expression`, with no checksum, signature, or pinned version. It then installs a pre-release WinFSP package without pinning a specific version. A compromised upstream — or a TLS-interception path during first boot — yields Administrator code execution inside the guest. The PR is the PoC today, but the same script becomes the first-boot path for every Windows guest once `pkg/cidata` calls into it.

Fix:

* Bundle the WinFSP MSI onto the cidata ISO. Install with `msiexec /i ... /quiet`.
* If bundling is impossible, download a pinned WinFSP release URL and verify its SHA-256 against a hash compiled into the generator before invoking `msiexec`. Drop `--pre`.

I7. **`pkg/windows/autounattend` uses `html/template` for an XML document** — `pkg/windows/autounattend/autounattend.go:8,15-23` [Claude Opus 4.7]

```go
import "html/template"
...
var tmpl = template.Must(
    template.New("autounattend").
        Funcs(template.FuncMap{ ... }).
        Parse(rawTemplate),
)
```

`html/template` applies HTML-context-aware escaping driven by a state machine that does not match XML. For ASCII input the output is correct enough. As soon as a hostname, username, or extra command contains a quote, ampersand, or angle bracket, the escaping drifts — `&#34;` for `"`, `&#39;` for `'`, HTML comment handling rather than XML — in ways that downstream XSD validators on some Windows builds reject.

The fields that reach element text (`Username`, `Hostname`, `ExtraCommands`, `SSHPublicKeys`) are already constrained by validation upstream, so the practical risk today is small. The structural defect is that the package depends on `html/template`'s contextual rules to produce a non-HTML format.

Fix: switch to `text/template`. Escape XML element text and attribute values explicitly with `encoding/xml.EscapeString` for the free-form fields.

```diff
-"html/template"
+"text/template"
```

I8. **Hard-coded drive letters `E:` and `F:` couple the answer file to QEMU CD-ROM ordering** — `pkg/windows/autounattend/template.xml:14-27,171`, `poc-windows-guest/run.sh:100-109` [Claude Opus 4.7, Gemini 3.1 Pro]

```xml
<Path>E:\viostor\2k25\amd64</Path>
...
<CommandLine>powershell ... -File "F:\first_logon.ps1" -SSHKey ...</CommandLine>
```

```bash
-device ide-cd,bus=ide.1,drive=cdrom0 \   # Windows ISO
-device ide-cd,bus=ide.2,drive=cdrom1 \   # virtio ISO  → guest assigns E:
-device ide-cd,bus=ide.3,drive=cdrom2 \   # cidata ISO  → guest assigns F:
```

The answer file assumes the virtio driver ISO lands on `E:` and cidata on `F:`. Windows assigns drive letters in attach order, but the install ISO often takes the first letter, and any additional disks shift the assignments. The PoC works because `run.sh` controls attach order exactly. Wiring this into `pkg/qemu/qemu.go` (the PR's "Next Steps") and adding a user's `disk.additionalDisks` shifts the letters; installation then hangs at `winload.exe` because the driver paths no longer resolve.

Fix:

* In `windowsPE`, replace hard-coded `E:\...` paths with `%CDROM%\...` or scan all CD-ROMs via the `wcm:keyValue` mechanism.
* Set a known volume label (`cidata`) in `BuildISO` and resolve the script location at first logon via `wmic logicaldisk where VolumeName='cidata' get DeviceID`.

I9. **`Set-Content -Encoding UTF8` writes a BOM that OpenSSH refuses to parse** — `poc-windows-guest/first_logon.ps1:63` [Gemini 3.1 Pro]

```powershell
Set-Content -Path $keyFile -Value $SSHKey -Encoding UTF8 -Force
```

On Windows PowerShell 5.1, `-Encoding UTF8` writes a UTF-8 BOM. The Windows OpenSSH `authorized_keys` parser treats the BOM as part of the first key's algorithm string, so the key never matches and SSH authentication silently fails. The PoC reports "all steps completed successfully" because the `Set-Content` itself succeeds, but `ssh` from the host hangs at password prompt.

Fix: SSH keys are pure ASCII, so encode as ASCII (or `UTF8NoBOM` on PS 7+, or `[IO.File]::WriteAllText` with a no-BOM encoding).

```diff
-Set-Content -Path $keyFile -Value $SSHKey -Encoding UTF8 -Force
+Set-Content -Path $keyFile -Value $SSHKey -Encoding Ascii -Force
```

Note: if I3's recommended fix lands (drop the `-SSHKey` parameter and copy `authorized_keys` from the cidata ISO), apply the same no-BOM rule to whichever code emits the file.

I10. **`poc-windows-guest/run.sh` uses Ubuntu-specific QEMU machine type and a Linux-only accelerator** — `poc-windows-guest/run.sh:91-94` [Gemini 3.1 Pro]

I11. **`validate()` accepts UTF-8 multi-byte hostnames that NetBIOS rejects** — `pkg/windows/autounattend/autounattend.go:55-56` [Claude Opus 4.7]

```go
case len(c.Hostname) > 15:
```

`len()` on a Go string returns the byte count. A caller that bypasses `SanitiseHostname` and passes `"日本サーバ"` (15 UTF-8 bytes, 5 runes) passes validation. The template embeds those bytes into `<ComputerName>`, and Sysprep rejects the name at first boot — NetBIOS names are ASCII-only.

I12. **`run.sh` masks ISO tool failures with `|| true`** — `poc-windows-guest/run.sh:64-67` [Codex GPT 5.5]

```bash
"$ISO_TOOL" -output "$CIDATA_ISO" -joliet -rock -volid cidata "$TMP_DIR" \
    2>&1 | grep -v '^$' || true

[[ -s "$CIDATA_ISO" ]] || fatal "ISO tool produced an empty file"
```

The `|| true` after the pipe suppresses every non-zero exit, including a real failure from the ISO tool. The size check only catches an empty output; a tool that exits 1 after writing a partial non-empty file passes the check, and the script proceeds into a confusing Windows install failure 15 minutes later.

Fix: capture the output and check the exit code separately.

```bash
local iso_output
if ! iso_output=$("$ISO_TOOL" -output "$CIDATA_ISO" -joliet -rock -volid cidata "$TMP_DIR" 2>&1); then
    printf '%s\n' "$iso_output" | grep -v '^$' || true
    rm -f "$CIDATA_ISO"
    fatal "ISO tool failed"
fi
```

I14. **`validate()` rejects empty passwords but accepts passwords Windows will reject** — `pkg/windows/autounattend/autounattend.go:57-58` [Claude Opus 4.7]

```go
case c.Password == "":
    return fmt.Errorf("autounattend: Password is required")
```

## Design Observations

### Concerns

D1. **The hand-rolled XML template duplicates a complex Microsoft schema** (future) [Claude Opus 4.7]

`autounattend.xml` is a complex Microsoft schema with strict component-naming rules, pass ordering, and per-component XSDs. The current 185-line template already drops corner cases (`LocalAccount` vs `DomainAccount` password encoding, `UnsecureJoin` flags, FIDO/PIN policy elements). Hand-rolled templating works for the PoC but accumulates subtle bugs as the supported locale/edition matrix grows. Two alternatives before the production wiring lands:

* Define a typed model (`Unattend`, `Settings`, `Component`, ...) and marshal to XML via `encoding/xml`. This eliminates escape-context bugs (I7), provides field-level validation, and lets tests compare struct values rather than string contents.
* Survey existing Go libraries that produce answer files from structs and decide whether one is worth adopting.

The current `Generate` surface is small enough that a later refactor is not blocked by this decision; pinning the direction before `pkg/cidata` imports the package saves a rewrite.

---

D2. **`pkg/windows/cidata` overlaps with `pkg/cidata` in name and concept** (in-scope) [Claude Opus 4.7]

Lima already has a `pkg/cidata` (cloud-init seed builder for Linux guests). The new package is at `pkg/windows/cidata` with the same `cidata` leaf name. Importers must alias to disambiguate. Suggest renaming to `pkg/windows/answerfileiso` or merging the Windows-specific behavior into `pkg/cidata` behind a guest-OS branch.

---

D3. **Reimplementation of ISO writing instead of reuse** (in-scope) [Codex GPT 5.5]

This is the design observation behind I2: `pkg/windows/cidata` introduces a parallel implementation of staging and tool dispatch even though `pkg/iso9660util` already owns ISO writing and Joliet command selection. If `pkg/iso9660util` is missing a feature, extend it rather than fork.

* The Go package surface is small and testable: a single `Generate` and a single `BuildISO`.

## Review Process Notes

### Skill improvements

- **Add a dimension: validate template-engine choice against output format.** When a template renders XML, JSON, SQL, YAML, or any non-HTML format, the use of `html/template` is a defect, not an optimization. Flag any import of `html/template` whose `Parse` input has no `<html>` root or is fed by `//go:embed *.xml`/`*.json`/`*.yaml`. The principle: HTML-context-aware escaping rules misfit other markup, often silently. Reviewers should treat the template-engine/output-format mismatch as a first-class structural smell.

- **Add a dimension: function-arity bugs in Go templates surface at execute time, not parse time.** When a template uses `template.FuncMap` functions, check that every call site's argument count matches the function's declared arity, and that the test suite executes every conditional branch (`{{if}}`, `{{range}}`, `{{with}}`) at least once. A `template.Must(...).Parse(...)` that passes does not guarantee runtime correctness: an `inc $i 1` against a one-arg `inc` parses fine and only errors when the body executes. Flag any template path with conditional or range blocks whose body is not exercised by at least one test case.

- **Add a dimension: `range` over a Go string yields byte offset, not rune index.** A loop of the shape `for i, r := range s { dst[i*k] = ... }` is buggy whenever `s` may contain non-ASCII text. The variable `i` is the byte offset of the rune in the UTF-8 source. Using it to index a destination indexed by rune position skips slots for multi-byte runes. The test suite catches this only when the test data contains non-ASCII input, so flag any such loop where the test fixtures are ASCII-only.

### Repo context updates

- **Lima reuses ISO writing through `pkg/iso9660util`.** Any new package that needs to produce a Joliet ISO should call `pkg/iso9660util.Write` rather than implement its own tool detection and exec dispatch. The helper handles `darwin`/`hdiutil makehybrid`, detects `xorrisofs` (not `xorriso`), and uses short-form flags (`-o`, `-J`, `-V`, `--norock`) that are accepted by all three Linux tools (`xorrisofs`, `genisoimage`, `mkisofs`). New code that reimplements this should be flagged as duplicated infrastructure.

- **Lima already has a `pkg/cidata` (Linux cloud-init seed builder).** A new package whose path or leaf name collides with `pkg/cidata` forces aliasing at every import site. Flag any new `pkg/.../cidata` package and ask whether the new behavior belongs inside the existing `pkg/cidata` (perhaps behind a guest-OS branch) before creating a sibling.

## Executive Summary

This PoC introduces two Go packages (`pkg/windows/autounattend`, `pkg/windows/cidata`) plus a `poc-windows-guest/` directory of shell scripts and a static answer file used for the author's manual end-to-end verification. The Go code is clean and small, but contains two latent bugs that the test suite does not exercise: a Go-template arity error in the `ExtraCommands` path, and a byte-index-as-rune-index error in the UTF-16LE password encoder that corrupts any password containing a non-ASCII rune. The PoC shell tree duplicates the answer file in an insecure form (`PlainText>true</PlainText>` with `Value>limauser</Value>`), contradicting the encoded password the Go package goes out of its way to produce.

## Findings

### Critical Issues

**C1. Template arity error in `ExtraCommands` Order computation — `pkg/windows/autounattend/template.xml:173-178`** (critical, regression)

```xml
{{range $i, $cmd := .ExtraCommands}}
<SynchronousCommand wcm:action="add">
  <Order>{{inc $i 1 | inc}}</Order>
  <RequiresUserInput>false</RequiresUserInput>
  <CommandLine>{{$cmd}}</CommandLine>
</SynchronousCommand>
{{end}}
```

`inc` is registered as `func(n int) int` (autounattend.go:20). The expression `inc $i 1` invokes `inc` with two arguments, which Go's `text/template` rejects at execute time with `wrong number of args for inc: want 1 got 2`. I reproduced this against the function definitions in the diff:

```
ERROR: template: t:1:27: executing "t" at <inc>: wrong number of args for inc: want 1 got 2
```

Because `template.Funcs` arity is checked only when the body executes, parsing succeeds. The body only executes for non-empty `ExtraCommands` — every current test leaves `ExtraCommands` empty (autounattend_test.go:10-18), so the bug is invisible to CI. The first caller that passes `ExtraCommands` triggers a failure at `Generate(cfg)`, never producing an answer file.

Fix: drop the stray `1` and let the pipeline produce `i+2`.

```diff
-          <Order>{{inc $i 1 | inc}}</Order>
+          <Order>{{inc $i | inc}}</Order>
```

Also add a test that exercises `ExtraCommands` with at least one entry so the template body is executed during CI.

**C2. UTF-16LE encoder mis-indexes the destination buffer for non-ASCII passwords — `pkg/windows/autounattend/autounattend.go:81-89`** (critical, regression)

```go
func encodeUTF16LE(plaintext string) string {
	s := plaintext + "Password"
	b := make([]byte, len(s)*2)
	for i, r := range s {
		b[i*2] = byte(r)
		b[i*2+1] = byte(r >> 8)
	}
	return base64.StdEncoding.EncodeToString(b)
}
```

`range s` over a Go string yields `i` as the **byte offset** of each rune, not a rune index. For ASCII the two agree, but the moment `s` contains a multi-byte rune the destination index `b[i*2]` skips ahead by the rune's UTF-8 length × 2. Bytes between writes stay zero, so decoders see embedded U+0000 NUL runes. Additionally, runes outside the BMP need surrogate-pair encoding (4 bytes), which the function never produces.

Verified empirically (I ran the function from the diff against `"abç"` and compared against `unicode/utf16.Encode`):

```
Input "abç":
  buggy   hex = 61006200e7000000500061007300730077006f0072006400
  correct hex = 61006200e700500061007300730077006f0072006400
                              ^^^^ buggy inserts an extra U+0000
```

Windows reads the base64-decoded blob as UTF-16LE; the spurious NUL becomes part of the password (or the leading prefix), so:

* the `LocalAccount` is created with a password the user never set, and
* the `AutoLogon` block contains a different password than the account → first boot stalls at the login screen, defeating the unattended install.

The test `TestGenerate_PasswordNeverPlaintext` uses `"T3stP@ssw0rd!"` (ASCII), so this never surfaces. Any user password with `é`, `ñ`, `£`, etc., breaks silently.

Fix: use `unicode/utf16.Encode` over a rune slice, which handles BMP and supplementary planes correctly.

```diff
 import (
-	_ "embed"
 	"bytes"
+	_ "embed"
 	"encoding/base64"
+	"encoding/binary"
 	"fmt"
 	"html/template"
 	"strings"
+	"unicode/utf16"
 )
@@
 func encodeUTF16LE(plaintext string) string {
-	s := plaintext + "Password"
-	b := make([]byte, len(s)*2)
-	for i, r := range s {
-		b[i*2] = byte(r)
-		b[i*2+1] = byte(r >> 8)
-	}
-	return base64.StdEncoding.EncodeToString(b)
+	u := utf16.Encode([]rune(plaintext + "Password"))
+	b := make([]byte, 2*len(u))
+	for i, v := range u {
+		binary.LittleEndian.PutUint16(b[i*2:], v)
+	}
+	return base64.StdEncoding.EncodeToString(b)
 }
```

Add a unit test that feeds a non-ASCII password and decodes the base64 back to UTF-16LE to verify the round-trip.

**I1. `joinLines` inserts raw newlines into a `<CommandLine>` element — `pkg/windows/autounattend/autounattend.go:19` and `template.xml:171`** (important, regression)

```go
"joinLines":      func(ss []string) string { return strings.Join(ss, "\n") },
```

```xml

```

With more than one SSH key the rendered `<CommandLine>` contains a literal LF inside the double-quoted PowerShell argument. XML preserves newlines in element text, so the launcher reads a multi-line command string. Two failure modes:

1. Some Windows Setup builds reject newlines in `<CommandLine>` outright (the value is treated as a single command line).
2. Even if accepted, `first_logon.ps1` writes the SSH key with `Set-Content -Value $SSHKey` (first_logon.ps1:63). PowerShell will interpret the multi-line string as a single argument with embedded LF, which is what `administrators_authorized_keys` expects — but the upstream parsing is unreliable enough that this should not depend on it.

The single-key test (autounattend_test.go:15) hides the problem.

Fix: emit one `<SynchronousCommand>` per SSH key, or pass the keys to PowerShell as a comma-separated list and split inside the script:

```diff
-          <CommandLine>powershell -NoProfile -NonInteractive -ExecutionPolicy Bypass -File "F:\first_logon.ps1" -SSHKey "{{joinLines .SSHPublicKeys}}"</CommandLine>
+          <CommandLine>powershell -NoProfile -NonInteractive -ExecutionPolicy Bypass -File "F:\first_logon.ps1" -SSHKeyBase64 "{{ b64KeyList .SSHPublicKeys }}"</CommandLine>
```

…with a `b64KeyList` function that joins on `\n`, encodes the result in base64, and the script does `[Convert]::FromBase64String($SSHKeyBase64)`. That keeps the on-wire CommandLine free of newlines and quoting hazards.

Add a unit test with `SSHPublicKeys = []string{key1, key2}` that asserts the rendered XML has no LF inside `<CommandLine>...</CommandLine>`.

**I2. `SanitiseHostname` can return a string with a trailing `-` — `pkg/windows/autounattend/autounattend.go:93-111`** (important, regression)

```go
s := strings.Trim(string(out), "-")
if len(s) > 15 {
    s = s[:15]
}
if s == "" {
    return "lima-windows"
}
return s
```

`Trim` runs once, then the 15-byte truncation can chop in the middle of a hyphen run, leaving a trailing `-`. For example `"abcdefghijklmn-xyz"` after filtering and trim is `"abcdefghijklmn-xyz"`, and `[:15]` is `"abcdefghijklmn-"` — trailing hyphen, which Windows NetBIOS rejects (and which `Generate` will then send into a `<ComputerName>` that Sysprep treats as invalid).

The test table at autounattend_test.go:91-97 has no input that surfaces this — `"this-is-way-too-long-for-netbios"` truncates cleanly at `"this-is-way-too"` because the 16th character is a `-` (so the 15th boundary is at `o`).

```diff
 s := strings.Trim(string(out), "-")
 if len(s) > 15 {
-    s = s[:15]
+    s = strings.TrimRight(s[:15], "-")
 }
 if s == "" {
     return "lima-windows"
 }
```

**I5. `pkg/windows/autounattend` uses `html/template` for an XML document — `pkg/windows/autounattend/autounattend.go:8,15-23`** (important, regression)

```go
"html/template"
...
var tmpl = template.Must(
    template.New("autounattend").
        Funcs(template.FuncMap{ ... }).
        Parse(rawTemplate),
)
```

`html/template` performs context-aware escaping driven by an HTML state machine. The template is XML, so the state machine occasionally guesses wrong:

* It treats `<unattend xmlns="urn:schemas-microsoft-com:unattend">` as the document element and tracks attribute / element contexts based on HTML semantics.
* In attribute values it emits `&#34;` for `"`; in element text it emits `&#39;` for `'`. Both are legal XML, but the autounattend XSD validator on some Windows builds rejects numeric character references for the apostrophe.
* It strips `<!-- ... -->` HTML comments differently from XML processors.

For ASCII input this is fine. As soon as a hostname or username contains a quote, ampersand, or angle bracket, the output drifts in ways that are hard to predict from the source.

Fix: switch to `text/template` and do the (small) XML escaping yourself for the fields that need it. The inputs are already constrained (`SanitiseHostname`, base64 password), so XML escaping is only required for `Username`, `ExtraCommands`, and `SSHPublicKeys` element-text contexts.

```diff
-	"html/template"
+	"text/template"
```

…and have the generator call `xml.EscapeString` on free-form fields before passing them to `Execute`. (Or pre-validate the few fields that flow into element text.)


## Design Observations

### Concerns

**D1. The XML "template" approach reimplements a substantial spec by hand. (future)**

`autounattend.xml` is a complex Microsoft schema with strict component-naming rules, pass ordering, and per-component XSDs. The current template is roughly 185 lines and already drops corner cases (encoding of `Password` for `LocalAccount` vs `DomainAccount`, `UnsecureJoin` flags, FIDO/PIN policy elements, etc.). Hand-written templating is fine for a PoC but will accrue subtle bugs as the project supports more locales, editions, and provisioning needs.

Two alternatives worth evaluating before the production wiring lands:

1. **Marshal a Go struct to XML via `encoding/xml`.** Define a typed model (`Unattend`, `Settings`, `Component`, …) and let the standard library produce the output. This eliminates the entire class of escape-context bugs (I5), gives free validation at the field level (no `inc $i 1 | inc`), and the test suite can compare struct values rather than string contents.
2. **Reuse an existing Go library.** `github.com/cloudbase/garm-provider-azure/internal/spec/unattend` and a handful of community packages produce answer files from Go structs. Quality varies; worth a survey.

The whole layer is small enough (one rendering function) that a future refactor is not blocked by this decision, but pinning down the direction before wiring `pkg/cidata` to it would save a rewrite.

**D2. `pkg/windows/cidata` overlaps with `pkg/cidata` in name and concept. (in-scope)**

Lima already has a `pkg/cidata` (cloud-init seed builder). The new package is at `pkg/windows/cidata` with the same `cidata` leaf name. Importers must alias to disambiguate. Suggest renaming the new package — for example `pkg/windows/answerfileiso` or `pkg/windows/cidata` → `pkg/windows/seediso` — so neither importer has to alias.

### Strengths

* The Go package is deliberately small and testable, with a single entry point (`Generate`) and a single ISO builder (`BuildISO`). Hard to argue with that surface area.
* The `TestGenerate_PasswordNeverPlaintext` test is the right shape: it asserts a security property over the output, not the implementation. Even though `encodeUTF16LE` has bugs C2, this test correctly catches the regression that would happen if anyone re-introduced `PlainText>true`.
* `BuildISO` writes to `<path>.tmp` then renames — the right atomic-write pattern for a host-visible artifact.

I3. **Password encoder corrupts non-ASCII passwords** — `pkg/windows/autounattend/autounattend.go:81-87` (important, regression)

```go
b := make([]byte, len(s)*2)
for i, r := range s {
	b[i*2] = byte(r)
	b[i*2+1] = byte(r >> 8)
}
```

The loop at lines 83-87 treats Go string byte offsets as UTF-16 code unit indexes. For any non-ASCII password, `range` returns byte offsets, leaving gaps and producing bytes that are not UTF-16LE; for non-BMP characters it also misses surrogate-pair encoding. Windows will decode a different password than the caller supplied.

Fix:

```go
func encodeUTF16LE(plaintext string) string {
	units := utf16.Encode([]rune(plaintext + "Password"))
	b := make([]byte, len(units)*2)
	for i, u := range units {
		binary.LittleEndian.PutUint16(b[i*2:], u)
	}
	return base64.StdEncoding.EncodeToString(b)
}
```

### [Gemini] — Pass 1

### Executive Summary

This PR successfully demonstrates a fully unattended Windows Server 2025 installation pipeline using an injected `autounattend.xml` and an ISO delivery mechanism. However, the implementation contains critical flaws in UTF-16LE password encoding for non-ASCII characters, SSH key injection via `cmd.exe`, and silent error-handling gaps during native provisioning steps.

### Findings

#### Critical Issues

**C1. UTF-16LE Encoding Corruption for Multibyte Characters** — `pkg/windows/autounattend/autounattend.go:81-88`

```go
	b := make([]byte, len(s)*2)
	for i, r := range s {
		b[i*2] = byte(r)
		b[i*2+1] = byte(r >> 8)
	}
```

The `range` loop over a string yields the *byte index* in `i`, not the rune index. If `plaintext` contains multibyte characters, `len(s)` will allocate excess capacity, and `i` will jump by >1 on iteration. This leaves uninitialized `\x00` null bytes scattered in the output slice `b`. When Windows Setup decodes the Base64 string, it encounters null characters inside the password and either truncates it or rejects it, locking the user out. Furthermore, `byte(r >> 8)` incorrectly truncates surrogate pairs for runes outside the Basic Multilingual Plane. (critical, regression)

Fix: Use the standard library's `unicode/utf16` package.

```diff
-	b := make([]byte, len(s)*2)
-	for i, r := range s {
-		b[i*2] = byte(r)
-		b[i*2+1] = byte(r >> 8)
-	}
-	return base64.StdEncoding.EncodeToString(b)
+	encoded := utf16.Encode([]rune(s))
+	b := make([]byte, len(encoded)*2)
+	for i, u := range encoded {
+		b[i*2] = byte(u)
```
