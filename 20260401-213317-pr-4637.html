Executive Summary ¶

This PR adds driver-controlled image format conversion to Lima's download cache. When a driver declares its supported formats and the cached image is incompatible, the downloader converts it to RAW in an imgconv/raw subdirectory while preserving the original. The approach correctly addresses issue #4056 (avoid repeated qcow2-to-raw conversion for VZ), and the design — preserving originals, threading driver capabilities through SupportedImageFormats — is sound. However, three critical regressions block merge: the new rename optimization in EnsureDisk skips disk expansion to the configured size, format detection runs on still-compressed data, and the digest overwrite in getCached bypasses both digest-based and Last-Modified cache invalidation for converted images.

    // Try to convert to raw if it's an non-ISO image and not supported by the driver
    if isNonISOImage(o.description, remote, o.supportedImageFormats) {

        if !slices.Contains(o.supportedImageFormats, format) {

Both getCached and fetch call DetectFormat on the cached payload before copyLocal decompresses it. For common images like *.qcow2.gz or *.img.xz, DetectFormat reads compressed bytes and either fails or misidentifies the format. Even if conversion succeeds, copyLocal (downloader.go:335, downloader.go:406) still receives ext = path.Ext(remote) (e.g., .gz), so it attempts to decompress an already-raw file.

Fix: Defer format detection and conversion until after decompression. Alternatively, decompress in the cache before detecting the format, and clear the extension so copyLocal does not attempt decompression on the converted artifact.

C3. Cache invalidation bypassed for converted images ¶

Gemini — critical, regression

pkg/downloader/downloader.go:314-323
            } else {
                shadData = rawImgConvPath
                if currentDigestData, err := os.ReadFile(rawImgConvDigestPath); err == nil {
                    currentDigest := strings.TrimSpace(string(currentDigestData))
                    if d, err := digest.Parse(currentDigest); err == nil {
                        o.expectedDigest = d
                        shadDigest = rawImgConvDigestPath
                    }
                }
            }

When the raw cache exists and raw.digest is valid, o.expectedDigest is overwritten with the raw digest before validation at downloader.go:332. Two consequences:

  • Template digest updates ignored: If a user updates their template with a new digest, getCached replaces it with the cached raw digest and validates against itself — always passing. The stale converted image is served without re-download.
  • Last-Modified checks skipped: For digest-less images, shadDigest is initially empty, which would trigger a matchLastModified check at downloader.go:339. The overwrite at line 320 sets shadDigest to a path that exists, causing os.Stat(shadDigest) at downloader.go:329 to succeed and bypassing the freshness check entirely.

Fix: Validate the original cached image against the user's expected digest (or Last-Modified) before switching to the converted artifact. Only after validation passes should the code select imgconv/raw as the source.

Design Observations ¶

Concerns ¶

D1. Cache layer conflates transport encoding with image format conversion ¶

Codex — in-scope

The old model was simple: cache the downloaded bytes, then optionally decompress on copy-out. This PR adds image-format conversion inside the cache while keying decompression off the original URL extension. That creates state combinations like "raw bytes stored under a .qcow2.gz URL" — the root cause of Critical Issue 2. A cleaner split: first normalize transport encoding (decompress to the canonical image), then decide whether a driver-specific converted artifact is needed. This removes a whole class of extension/content mismatches.

D2. Conversion and disk creation split across two components with no coordination ¶

Claude Codex Gemini — in-scope

The cache layer converts format with nil size, while EnsureDisk handles both format conversion and sizing. The new rename optimization in EnsureDisk shows the risk: skipping format conversion inadvertently skipped sizing. A safer separation: the cache owns format (converting to a driver-compatible format at the image's natural size), and EnsureDisk always owns sizing (calling ResizeDisk regardless of whether it also converted).

Strengths ¶

Original image preservation. Keeping the original cached download in data and placing the conversion in imgconv/raw lets different drivers share the same cache entry without conflict, directly addressing the determinism concern raised by @AkihiroSuda. Claude Codex Gemini

Driver capability threading. Routing format decisions through DriverFeatures.SupportedImageFormats is a better ownership boundary than hard-coding VZ-specific behavior in the downloader. Codex

Rename optimization for matching formats. The idea of renaming instead of converting when formats match (driverutil/disk.go:104-109) is sound — it just needs the resize step added back. Gemini

Testing Assessment ¶

Untested scenarios, ranked by risk:

  1. Disk size expansion after cache conversion — the primary VZ workflow has no test verifying the disk reaches the configured size. checkDisk only asserts fi.Size() > 0.
  2. Compressed image download and conversion — no test covers .qcow2.gz, .img.xz, or .zst images through the new conversion path.
  3. Cache invalidation after template digest update — no test verifies that changing a template's expected digest for an already-cached-and-converted image triggers a re-download.
  4. Last-Modified freshness for digest-less converted images — no test verifies that digest-less images undergo Last-Modified checks after conversion.
  5. Crash recovery: raw.digest missing but raw image present — no test for the data integrity gap from Important Issue 1.
  6. Cross-driver cache sharing — no test for the QEMU-then-VZ same-URL sequence where QEMU downloads qcow2, then VZ needs it converted to raw.
  7. Same-format fast path in EnsureDisk — no test exercises a source raw image that is smaller than the configured disk size.

Documentation Assessment ¶

The internals.md update correctly documents the new imgconv/ directory structure. No other documentation gaps, though internals.md will need to stay aligned with whatever fix is chosen for raw.digest validation.

Commit Structure ¶

Single commit bundles all changes. This is appropriate for the scope — splitting would create non-functional intermediate states.

Repo context updates ¶

  • Lima's EnsureDisk function doubles as both format converter and disk resizer via diskUtil.Convert. Any optimization that short-circuits conversion must preserve the resize step. Future reviews of driverutil/disk.go should verify that disk size expansion occurs on every code path.
  • The proxyimgutil.ImageDiskManager proxy dispatches to qemu first and falls back to native only on exec.ErrNotFound. The qemu MakeSparse stub returns nil (not ErrNotFound), so the native fallback is never reached. Agents reviewing calls to proxy methods should trace the dispatch chain, not assume the native path runs.
