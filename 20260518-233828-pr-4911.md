## Executive Summary

The PR correctly diagnoses a long-standing race (issue #2227) where `watchGuestAgentEvents` and the inotify-startup goroutine both call `forwardSSH(verbForward, …)` on the same local socket path with no synchronization and no `-O cancel` of the prior forward, leaving `ga.sock` permanently missing after the first failed bind. The fix introduces `reForwardGuestAgentSock`, which serializes via a new `gaSockForwardMu` and issues a best-effort `verbCancel` before `verbForward`. The approach is sound and surgical (~40 lines, one file, no API changes).

The structure of this consolidated review: 0 Critical, 3 Important, 2 Suggestions, 1 design concern. All three Important findings were raised in Jan's most recent general comment on the PR and remain unaddressed in commit `8a7d899f`. None of the Important findings is a regression in user-facing behaviour — the fix is a net improvement — but they are correctness/hygiene gaps that should land before merge.

## Important Issues

I1. **`context.Background()` detaches the cancel call from caller lifecycle** — `pkg/hostagent/hostagent.go:984` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro, Qwen 3.6 35B, Gemma4 31B]

```go
// Best-effort teardown of any prior forward registered with the
// ControlMaster. Errors are expected (e.g. on the very first call when
// no forward exists yet) and intentionally ignored.
_ = forwardSSH(context.Background(), a.sshConfig, sshAddress, sshPort, localUnix, remoteUnix, verbCancel, false)
if err := forwardSSH(ctx, a.sshConfig, sshAddress, sshPort, localUnix, remoteUnix, verbForward, false); err != nil {
```

The in-source comment justifies ignoring the *error*, not detaching the *lifetime*. `forwardSSH` shells out via `exec.CommandContext(ctx, …)` and then waits in `cmd.Output()`, so passing `context.Background()` means an unresponsive SSH ControlMaster cannot be interrupted by hostagent shutdown: the goroutine sits in `cmd.Output()` while it holds `gaSockForwardMu`, blocking any concurrent caller of the helper for the full SSH command timeout. The immediately following `verbForward` call uses the real `ctx`, making the asymmetry harder to reason about.

Jan flagged this both inline at line 984 (https://github.com/lima-vm/lima/pull/4911#discussion_r3244931774) and as item #1 in his follow-up general comment; the latest commit has not addressed it.

Fix:

```diff
-	_ = forwardSSH(context.Background(), a.sshConfig, sshAddress, sshPort, localUnix, remoteUnix, verbCancel, false)
+	_ = forwardSSH(ctx, a.sshConfig, sshAddress, sshPort, localUnix, remoteUnix, verbCancel, false)
```

Qwen also suggested an alternative: wrap the cancel in a short `context.WithTimeout(ctx, 3*time.Second)` so the lock can never be held longer than that even if SSH hangs. This is reasonable defense-in-depth but not strictly needed — the plain `ctx` swap is enough to honor shutdown.

I2. **Shutdown cleanup bypasses `gaSockForwardMu`** — `pkg/hostagent/hostagent.go:727-731` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro, Qwen 3.6 35B, Gemma4 31B]

```go
if a.driver.ForwardGuestAgent(ctx) {
    if err := forwardSSH(ctx, a.sshConfig, sshAddress, sshPort, localUnix, remoteUnix, verbCancel, false); err != nil {
        errs = append(errs, err)
    }
}
```

This is the symmetric problem to the one the PR fixes. The cleanup callback registered at line 715 runs during `a.close()`, which executes **before** `cancelHA()` cancels the context for `watchGuestAgentEvents`. That means while cleanup is executing `forwardSSH(verbCancel, …)` on the guest-agent socket, the reconnect loop and the inotify goroutine are still alive and may concurrently call `reForwardGuestAgentSock`. The race the new mutex was introduced to prevent (concurrent `os.RemoveAll` / bind on the same path during a ControlMaster mutation) is reopened during the shutdown window.

The prior cleanup registered at line 762 sets `a.client = nil`, which causes the next iteration of the reconnect loop to take the `client == nil` branch and call `reForwardGuestAgentSock` — concurrently with the cleanup at line 728 doing `verbCancel` unsynchronized. The defer in `forwardSSH(verbCancel)` unlinks the socket file, so the worst case is shutdown leaving a fresh `ga.sock` on disk that the next start will recover from. Real but bounded.

This is item #3 in Jan's follow-up general comment.

Fix: extend the mutex to the cleanup path (or add a small locked helper):

```diff
 if a.driver.ForwardGuestAgent(ctx) {
+    a.gaSockForwardMu.Lock()
     if err := forwardSSH(ctx, a.sshConfig, sshAddress, sshPort, localUnix, remoteUnix, verbCancel, false); err != nil {
         errs = append(errs, err)
     }
+    a.gaSockForwardMu.Unlock()
 }
```

I3. **No automated regression test for a goroutine-timing fix** — `pkg/hostagent/` [Claude Opus 4.7, Codex GPT 5.5, Qwen 3.6 35B, Gemma4 31B]

The PR diffs only production code. The failure being fixed is sensitive to subtle goroutine timing on real SSH ControlMaster state, and it bit users for more than two years (#2227) before being correctly diagnosed. Without a test, the next refactor that touches this area — for example a "consolidate the two reconnect loops" change — can quietly reintroduce the bug, and the manual reproducer in the PR description is the only signal a regression occurred.

A cheap unit test would stub `forwardSSH` via a function variable and assert two invariants:

1. Under concurrent callers, `reForwardGuestAgentSock` is serialized — record observed concurrency depth, assert it stays at 1.
2. Within each call, the recorded `(verb, ctx)` pairs are `(verbCancel, _) → (verbForward, _)` in that order.

The package already has a `_test.go` file (`pkg/hostagent/inotify_test.go`), so adding `forward_test.go` does not require new test infrastructure.

This is item #2 in Jan's follow-up general comment.

## Suggestions

S1. **Helper name implies "re-" only, but it is also the first-call path** — `pkg/hostagent/hostagent.go:977` [Claude Opus 4.7, Gemini 3.1 Pro, Qwen 3.6 35B, Gemma4 31B]

```go
func (a *HostAgent) reForwardGuestAgentSock(ctx context.Context, localUnix, remoteUnix string) {
```

Both call sites (the inotify goroutine at line 742 and the reconnect loop at line 777) invoke this on the **first** tick as well as on every reconnect tick. The doc comment at line 956 reinforces this: "It must be used everywhere watchGuestAgentEvents (or any goroutine it spawns) needs to bring the forward back up …" — i.e., it is the canonical forward path, not a re-establishment specifically. `forwardGuestAgentSock` reads more accurately and matches the existing one-word helper name pattern in the file (`forwardSSH`, `executeSSH`).

This is item #4 in Jan's follow-up general comment.

Fix: rename `reForwardGuestAgentSock` → `forwardGuestAgentSock`, and drop the "re-" framing from the doc comment.

S2. **Startup-only race: both goroutines call the helper concurrently on first tick** — `pkg/hostagent/hostagent.go:742, 777` [Gemini 3.1 Pro]

When `mountInotify: true`, the inotify goroutine and the main reconnect loop both evaluate `client == nil || !isGuestAgentSocketAccessible(ctx, client)` near startup. At first tick `client` is `nil` in both, so both call `reForwardGuestAgentSock` and queue on `gaSockForwardMu`. The first call succeeds (cancel is a no-op, forward binds); the second call cancels the freshly-bound forward and binds again. Net effect: one redundant cancel+forward pair at startup. This is wasted work, not a correctness bug — the mutex still serializes the operations, and the second forward leaves the system in the same end state.

The race is confined to startup. The inotify goroutine calls `reForwardGuestAgentSock` exactly once (at lines 740-744, outside the `for` loop) and then enters a `startInotify` retry loop that does not re-establish the forward. So this is not a recurring "every reconnect" race as Gemini's report framed it.

Fix (optional, low priority): inside the helper, after acquiring the mutex, re-check `isGuestAgentSocketAccessible(ctx, a.getClient())` and short-circuit if the forward is already healthy. Skipping this is fine — the redundant pair only happens once per startup.

## Design Observations

### Concerns

- **Two parallel reconnect loops that both manage the same forward** *(future)* [Claude Opus 4.7]. The helper localizes the forward-setup race, but the deeper structural issue is that `watchGuestAgentEvents` and the inotify spawn goroutine both poll the same `client == nil || !isGuestAgentSocketAccessible` predicate on independent 10-second tickers, and the latter only does so at startup before entering a separate `startInotify` retry loop. The fix is correct but the mental model is fragile: a future contributor wiring a third consumer would have to know to call the helper too. A cleaner shape is a single "guest-agent supervisor" goroutine that owns the forward + client and exposes a channel to consumers (inotify, event reader). Out of scope for this PR.

### Strengths

- **Doc comment captures the full failure mode** [Claude Opus 4.7, Qwen 3.6 35B]. The comment at lines 956-976 explains both halves of the bug (ControlMaster duplicate registration, defer-unlink-on-failure) and the rationale for each fix step, all in one place where a future reader needs it. A meaningfully better artifact than the typical "fix race" commit message.
- **Cancel before forward over making `forwardSSH` idempotent** [Codex GPT 5.5, Claude Opus 4.7]. Issuing `verbCancel` before `verbForward` keeps the change local and avoids touching `forwardSSH`'s existing error-handling semantics, which other callers depend on.
- **Surgical scope** [Qwen 3.6 35B, Gemma4 31B]. ~40 lines, one file, one mutex field, no API changes, no architectural overhaul. Easy to review and easy to revert.

### Repo context updates

**ADD: SSH ControlMaster forwards require cancel-before-forward.** Whenever code calls `forwardSSH(verbForward, …)` on a path that already has an active forward registered with the SSH ControlMaster, the bind fails with `forwarding for listen path X already exists` and `forwardSSH`'s failure branch unlinks the local socket — leaving the path permanently broken until a `verbCancel` releases the registration. Any new caller of `forwardSSH(verbForward, …)` that may race with a prior forward (reconnect loops, retry goroutines, lifecycle hooks) must call `verbCancel` first and serialize via a mutex if multiple goroutines can trigger it. Reviewers should flag any direct call to `forwardSSH(verbForward, …)` that is not preceded by a corresponding `verbCancel` on the same `(local, remote)` pair, and any path that mutates a forward without holding the relevant serialization mutex.

**ADD: HostAgent cleanup callbacks run before context cancellation.** Cleanup callbacks registered via `a.cleanUp(...)` execute inside `a.close()`, which runs *before* `cancelHA()` cancels the agent's context. Any background goroutine that uses the same `ctx` is therefore still alive while cleanups run. If a cleanup function and a still-live goroutine touch the same resource, the synchronization that protects the goroutine's access must also apply to the cleanup callsite. Reviewers should check that any mutex introduced to serialize hostagent operations is also held at every cleanup-time callsite that touches the same resource.

## Design Observations

### Concerns

- **Two parallel reconnect loops that both manage the same forward** *(future)*. The helper localizes the forward-setup race, but the deeper structural issue is that `watchGuestAgentEvents` (lines 773-797) and the inotify spawn goroutine (lines 735-758) both poll the same `client == nil || !isGuestAgentSocketAccessible` predicate on independent 10-second tickers and both call `reForwardGuestAgentSock`. The inotify goroutine never re-runs that check after the initial iteration — it only calls `startInotify` in a retry loop and relies on the main reconnect loop to repair the forward. The fix here is correct but the mental model is fragile: a future contributor wiring a third consumer would have to know to call the helper too. A cleaner shape is a single "guest agent supervisor" goroutine that owns the forward + client and exposes a channel to consumers (inotify, event reader). Out of scope for this PR, but worth noting.

### Strengths

- The doc comment at lines 956-976 captures both halves of the bug and the rationale for each fix step in one place — exactly where a future reader needs it. This is a meaningfully better artifact than the typical "fix race" commit message.
- Issuing `verbCancel` before `verbForward` (rather than relying on `forwardSSH` to be made idempotent against `ControlMaster` state) keeps the change local and avoids touching `forwardSSH`'s existing error-handling semantics, which other callers depend on.
