Consolidated Review

Executive Summary

This PR adds a conversion cache layer to Lima's downloader. When a VM driver (VZ, krunkit) cannot boot the downloaded image format (typically QCOW2), the downloader converts the image to RAW once in a new imgconv/raw subdirectory alongside the cached original, avoiding repeated per-instance conversions. The approach is sound and addresses the disk-space and startup-time problems described in issue #4056.

However, the implementation overwrites the caller's expected digest with the derived raw digest before validation, so a cached raw image is never re-validated against the original template digest. If an upstream image changes and the user updates their digest, Lima silently reuses the stale cached conversion. Two additional important issues affect compressed-image handling and a potential nil-pointer panic.

Critical Issues

1. Cached raw conversion bypasses the caller's digest validation Codex Gemini

When a cached raw image exists at line 315, getCached overwrites o.expectedDigest (the original digest from the template) with the raw image's digest at line 319 and points shadDigest at raw.digest at line 320. The subsequent validateCachedDigest call at line 332 then compares raw.digest against itself — a tautology that always passes.

The original digest file (e.g., sha256:<original>) is never consulted. If the upstream image changes and the user updates their template digest, getCached still returns the old raw conversion. This violates the stated requirement in issue #4056 that "the cache must retain the checksum of the original image for lookup."

The same overwrite occurs in the needConvert branch at line 312 after ensureRawInCache, and again in fetch() at line 400. In every path, the original digest is lost.

Claude Opus 4.6 identified a narrower variant (missing raw.digest file) at the same location but rated it important rather than critical. The broader issue — that digest validation is bypassed even when raw.digest exists — warrants critical severity because it silently serves wrong data.

Fix: Keep the original and derived digests separate. First validate the cached original image against the caller's expected digest (or force a re-download on mismatch). Only then validate imgconv/raw against raw.digest before copying it.

2. Compressed images bypass cache conversion Codex Gemini

DetectFormat at lines 283 and 390 runs on the cached file before decompression. Compressed downloads (.qcow2.gz, .img.xz) store compressed bytes in shadData. The QCOW2 magic number is obscured by the compression wrapper, so DetectFormat falls through to "raw." Because "raw" appears in every driver's supported list, the conversion logic is skipped entirely.

pkg/downloader/downloader.go
	if isNonISOImage(o.description, remote, o.supportedImageFormats) {
		imageFormat, err := nativeimgutil.DetectFormat(shadData)
		if err != nil {
			return nil, err
		}
pkg/downloader/downloader.go
	// Try to convert to raw if it's an non-ISO image and not supported by the driver
	if isNonISOImage(o.description, remote, o.supportedImageFormats) {
		format, err := nativeimgutil.DetectFormat(shadData)
		if err != nil {
			return nil, err
		}

This defeats the PR's purpose for compressed images — the most common distribution format for cloud images (e.g., Ubuntu .img files distributed as .xz).

Fix: Run format detection on uncompressed bytes. Either decompress to a temporary file before calling DetectFormat when o.decompress is true, or check the inner extension (e.g., strip .gz/.xz before checking path.Ext). Add tests for compressed inputs.

3. isNonISOImage relies on display-text string matching Claude

Whether cache conversion fires depends on the description string containing "image". This display string is constructed at pkg/fileutils/download.go:32. A caller that passes a different description (e.g., "the disk") silently disables conversion.

pkg/downloader/downloader.go
func isNonISOImage(description, remote string, supportedImageFormats []string) bool {
	return strings.Contains(description, "image") && supportedImageFormats != nil && path.Ext(remote) != ".iso"
}

The check is duplicated at pkg/fileutils/download.go:35, creating a second point of failure:

pkg/fileutils/download.go
// DownloadFile downloads a file to the cache, optionally copying it to the destination. Returns path in cache.
func DownloadFile(ctx context.Context, dest string, f limatype.File, decompress bool, description string, expectedArch limatype.Arch, supportedImageFormats []string) (string, error) {
	if f.Arch != expectedArch {
		return "", fmt.Errorf("%w: %q: unsupported arch: %q", ErrSkipped, f.Location, f.Arch)
	}
	fields := logrus.Fields{"location": f.Location, "arch": f.Arch, "digest": f.Digest}
	logrus.WithFields(fields).Infof("Attempting to download %s", description)
	opts := []downloader.Opt{
		downloader.WithCache(),
		downloader.WithDecompress(decompress),
		downloader.WithDescription(fmt.Sprintf("%s (%s)", description, path.Base(f.Location))),
		downloader.WithExpectedDigest(f.Digest),
	}
	if strings.Contains(description, "image") && supportedImageFormats != nil {
		opts = append(opts, downloader.WithImageFormats(supportedImageFormats))
	}

Fix: Replace the string-matching heuristic with a typed flag. The WithImageFormats option already signals intent — if supportedImageFormats is non-nil, the caller wants conversion. The description check is redundant.

When qemu-img is installed (the common case), qemu-img convert -O raw already produces sparse output, so this is harmless. When qemu-img is absent, the native converter copies bytes without sparsification, and the subsequent MakeSparse call — the one place that should fix this — is silently skipped.

Fix: Either call p.native.MakeSparse directly in ensureRawInCache, or fix the proxy stub to return a sentinel error (e.g., ErrNotImplemented) that triggers the fallback. The proxy bug is pre-existing, but this new call site makes it load-bearing for cache disk usage.

Design Observations

1. o.expectedDigest serves two meanings Codex

o.expectedDigest starts as "the caller's expected digest for the original download" and later becomes "the derived digest of imgconv/raw." Keeping these as separate variables would eliminate the digest-bypass bug (critical finding #1) and simplify the mental model. The original digest governs cache lookup and revalidation; the raw digest protects only the derived artifact's integrity.

2. Hardcoded RAW-only cache layout Claude

ensureRawInCache always converts to raw, and the cache layout is hardcoded to imgconv/raw. All current drivers that set SupportedImageFormats include "raw", so this works today. If a future driver needs a different format, the layout would need generalization (e.g., imgconv/<format>/data).

3. DownloadFile parameter sprawl Claude

Adding supportedImageFormats []string as a positional parameter to DownloadFile at pkg/fileutils/download.go:23 forces every call site to pass it, even when nil. Six call sites were touched. An options pattern (like downloader.Download already uses) would prevent churn as the function grows.

Testing Assessment

The PR adds three test functions covering the core conversion flow. Untested scenarios, ranked by risk:

  1. Digest rotation with existing cached raw conversion — no test asserts that Download re-downloads when the caller provides a new digest for a URL whose imgconv/raw already exists. This gap masks the critical digest-bypass bug.
  2. Compressed image inputs (.qcow2.gz, .img.xz) through both fresh and cached paths — DetectFormat runs on compressed bytes and silently misclassifies the format.
  3. Corrupt or missing raw.digest with existing raw image — the fallback path at lines 316-322 has no coverage.
  4. Re-conversion on stale raw image — the origStat.ModTime().After(rawStat.ModTime()) path at line 300 has no test.
  5. Native conversion (no qemu-img) — all tests skip when qemu-img is absent. The native convert + make-sparse path has no coverage.
  6. QEMU driver retains original format — no test verifies that a driver listing qcow2 as supported skips conversion and returns the original.

Documentation Assessment

The internals.md update describing imgconv/ is appropriate. It should also state that the original digest remains authoritative for cache lookup, while raw.digest protects only the derived artifact — once the implementation matches that contract.

Skill Improvement Recommendations

  • Compressed-image handling: The prompt should explicitly ask agents to trace compressed vs. uncompressed paths when the codebase has a decompress option. Two of three agents found this; adding a hint would ensure all three do.
  • Cross-agent severity calibration: Claude rated the digest bypass as important while Codex and Gemini rated it critical. The prompt's severity guidance ("reserve CRITICAL for issues that will or easily can happen in production") is clear enough; the gap is in agent judgment, not prompt wording.
  • False-positive rate: Gemini's false critical (disk resizing) shows that asking agents to "trace the full lifecycle" is not sufficient. Consider adding: "Before flagging a function for missing behavior, check whether the caller handles it after the call returns."
  • All agents missed: No agent flagged that ensureRawInCache writes raw.digest non-atomically (line 467) after the atomic rename of the raw image (line 453). A crash between these two operations leaves a raw image without a digest file. Claude noted this as a precondition for a milder bug but did not flag the non-atomicity itself.
