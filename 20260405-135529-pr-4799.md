## Executive Summary

This PR replaces the overly broad `len(inst.Config.Mounts) > 0` check in `limactl shell` with a precise `mountsContainPath` function that verifies the host working directory falls under a configured mount before generating a `cd` command in the guest. Round 2 addresses three of five issues from the prior review: the home-directory fallback when CWD is outside mounts, symlink resolution via `filepath.EvalSymlinks`, and the root-mount double-separator edge case. Two prior issues remain open: `mountsContainPath` still has no unit tests, and the Windows non-WSL2 path format mismatch is unresolved (though masked by the WSL2 short-circuit for the common case).

I1. **Windows non-WSL2 path format mismatch remains unresolved** — `cmd/limactl/shell.go:257,264,637-657` [Codex GPT 5.4, Gemini 3.1 Pro] (important, regression)

```go
case mountsContainPath(inst.Config.Mounts, hostCurrentDir) || inst.VMType == limatype.WSL2:
```

On Windows, `hostCurrentDir` (line 257) and `hostHomeDir` (line 264) are converted to guest-format paths (e.g., `/c/Users/alice/project`) via `mountDirFromWindowsDir` at lines 247 and 212/676. Mount locations in `inst.Config.Mounts` remain in native Windows format (e.g., `C:\Users\alice`). The string comparison in `mountsContainPath` (line 652) always fails because a Unix-style guest path never matches a Windows-style host path.

The `|| inst.VMType == limatype.WSL2` clause at line 257 masks this for WSL2 (the primary Windows backend), but the new home-directory fallback at line 264 lacks a WSL2 short-circuit:

```go
case hostHomeDir != "" && mountsContainPath(inst.Config.Mounts, hostHomeDir):
    changeDirCmd = fmt.Sprintf("cd %s", shellescape.Quote(hostHomeDir))
```

On WSL2, this case is unreachable because case 2 (line 257) always matches via `|| inst.VMType == limatype.WSL2`. On a hypothetical non-WSL2 Windows VM (QEMU), both the CWD and home fallback branches would fail silently. Lima currently only supports WSL2 on Windows, so the practical impact is nil today.

This was issue #5 in the prior review and remains unaddressed.

Fix: Pass the unconverted native host path to `mountsContainPath`. Capture `os.Getwd()` and `os.UserHomeDir()` before Windows conversion and use those for mount matching, while continuing to use the converted paths for the `cd` commands themselves.

I1. **`mountsContainPath()` still misclassifies mounted paths on Windows non-WSL2** — `cmd/limactl/shell.go:257-265,637-654` (important, regression)

```go
case mountsContainPath(inst.Config.Mounts, hostCurrentDir) || inst.VMType == limatype.WSL2:
	changeDirCmd = fmt.Sprintf("cd %s", shellescape.Quote(hostCurrentDir))
...
loc, err := localpathutil.Expand(m.Location)
...

	resolvedLoc == string(os.PathSeparator) || strings.HasPrefix(resolvedPath, resolvedLoc+string(os.PathSeparator)) {
	return true

```

`git blame` shows both the branch at `cmd/limactl/shell.go:257-265` and the helper at `cmd/limactl/shell.go:637-654` were introduced by `c3ae08ec`. The problem is that `hostCurrentDir` is converted into subsystem syntax in `hostCurrentDirectory()` at `cmd/limactl/shell.go:673-678` via `mountDirFromWindowsDir()` at `cmd/limactl/shell.go:709-714`, but `mountsContainPath()` compares that converted string against `m.Location` after only `localpathutil.Expand()` at `cmd/limactl/shell.go:643-649`. On Windows, `FillDefault()` keeps `Mount.Location` in native form and only converts the guest mount target at `pkg/limayaml/defaults.go:694-700`; `pkg/hostagent/mount.go:55-60` does the same conversion later at mount time. In practice this means a converted current directory like `/c/Users/alice/project` will never match a configured mount location like `C:\Users\alice`, so `limactl shell` will incorrectly skip `cd hostCurrentDir` even when the directory is mounted. This is exactly the previously raised Windows format-mismatch issue.

Fix: normalize mount locations with the same Windows conversion that is applied to `hostCurrentDir`, not just `Expand()`. ### [Gemini] — Pass 1

### Executive Summary

This PR attempts to fix an issue where `limactl shell` unconditionally changes into the host's current directory whenever *any* mount exists. It correctly introduces a check to verify whether the host directory is actually under one of the mounted locations. However, the implementation introduces a critical regression on Windows by passing guest-converted paths into host-side symlink evaluation functions, breaking the `cd` behavior entirely. It also leaves `mountsContainPath` untested, leaving previous review feedback unaddressed.

### Findings

#### Critical Issues

C1. **Host paths are converted to guest paths before matching on Windows** — `cmd/limactl/shell.go:257-264` (critical, regression)

```go
	case mountsContainPath(inst.Config.Mounts, hostCurrentDir) || inst.VMType == limatype.WSL2:
		changeDirCmd = fmt.Sprintf("cd %s", shellescape.Quote(hostCurrentDir))
		if hostHomeDir != "" {
			changeDirCmd = fmt.Sprintf("%s || cd %s", changeDirCmd, shellescape.Quote(hostHomeDir))
		}
	case syncHostWorkdir:
		changeDirCmd = fmt.Sprintf("cd %s", shellescape.Quote(destRsyncDir))
	case hostHomeDir != "" && mountsContainPath(inst.Config.Mounts, hostHomeDir):
```

`hostCurrentDir` and `hostHomeDir` are converted from Windows host paths to Unix guest paths (via `mountDirFromWindowsDir` at lines 212/676 and 247) *before* they are passed to `mountsContainPath`. When `mountsContainPath` calls `filepath.EvalSymlinks(path)` on Windows (line 638), it fails because the Unix-style path (e.g., `/mnt/c/Users/...` or `/c/Users/...`) is not a valid Windows path. The function falls back to the unconverted string but then compares it against `resolvedLoc`, which is derived from `m.Location` (a valid Windows host path, e.g., `C:\Users\...`). 

Since a Unix-style guest path will never match a Windows-style host path, `mountsContainPath` will always return `false` on Windows for both `hostCurrentDir` (except on WSL2, where it's masked by `|| inst.VMType == limatype.WSL2`) and `hostHomeDir` (where the WSL2 fallback is missing). This completely breaks the `cd` behavior on Windows that this PR aimed to fix.

Fix: Keep the unconverted host paths for the containment check, and only use the guest-converted paths for the actual `changeDirCmd`. For example:

```go
	// Retrieve unconverted native host paths
	nativeHostCurrentDir, _ := os.Getwd()
	nativeHostHomeDir, _ := os.UserHomeDir()
	
	switch {
	case workDir != "":
		changeDirCmd = fmt.Sprintf("cd %s || exit 1", shellescape.Quote(workDir))
	case mountsContainPath(inst.Config.Mounts, nativeHostCurrentDir) || inst.VMType == limatype.WSL2:
		changeDirCmd = fmt.Sprintf("cd %s", shellescape.Quote(hostCurrentDir)) // hostCurrentDir is the converted guest path
		if hostHomeDir != "" {
			changeDirCmd = fmt.Sprintf("%s || cd %s", changeDirCmd, shellescape.Quote(hostHomeDir))
		}
	case syncHostWorkdir:
		changeDirCmd = fmt.Sprintf("cd %s", shellescape.Quote(destRsyncDir))
	case nativeHostHomeDir != "" && mountsContainPath(inst.Config.Mounts, nativeHostHomeDir):
		changeDirCmd = fmt.Sprintf("cd %s", shellescape.Quote(hostHomeDir))
```

#### Important Issues

I1. **Path comparisons are case-sensitive** — `cmd/limactl/shell.go:652-653` (important, gap)

```go

```

Path comparison (`==` and `strings.HasPrefix`) is case-sensitive. On case-insensitive file systems (the default for Windows and macOS), if the casing of the current directory differs from the mounted path casing (e.g., `/Users/foo` vs `/Users/Foo`), `mountsContainPath` incorrectly returns `false` and fails to `cd` into the directory. 

Fix: Use a case-insensitive comparison on macOS and Windows. Since paths have already been cleaned and evaluated for symlinks, `strings.EqualFold` and case-insensitive prefix matching is safer for cross-platform interoperability.
