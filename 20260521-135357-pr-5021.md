## Executive Summary

This POC adds swtpm-backed TPM 2.0 support to Lima's QEMU driver behind a top-level `tpm.enabled` knob. The arch dispatch picks `tpm-crb` on x86_64 and `tpm-tis-device` on ARM, and the swtpm subprocess mirrors the existing virtiofsd lifecycle. The headline gaps are validate-time gating (no driver or arch check on `tpm.enabled`), a `templates/experimental/windows.yaml` that fails `limactl create` validation as documented, and swtpm lifecycle paths that leak the process on Start failure and orphan it on `limactl stop -f`.

The PR also folds in a `WindowsSubsystemPath` change in `defaults.go` that the description calls a fix. It swaps one broken behaviour (empty mount point) for another (a native Windows path in `mount.MountPoint`); the fix belongs in a separate PR and needs more work.

## Important Issues

I1. **swtpm process leaks if QEMU Start fails after swtpm is running** — `pkg/driver/qemu/qemu_driver.go:315-374, 458-462` [Claude Opus 4.7]

```go
if err := qCmd.Start(); err != nil {
    return nil, err            // swtpm still running; l.swtpmCmd never set
}
l.qCmd = qCmd
l.swtpmCmd = swtpmCmd          // assigned only after qCmd.Start() succeeds
```

`swtpmCmd` is stored on the driver only after `qCmd.Start()`. Any failure between starting swtpm (line ~340) and assigning `l.swtpmCmd` (line 462) — vhost Start, vhost-socket timeout, `qCmd.Start()`, the `applier` template loop — returns from `Start()` without `killSwtpm`, leaves swtpm running with its socket and state directory held open, and a later `Stop()` finds `l.swtpmCmd == nil` and exits early.

Fix: assign `l.swtpmCmd = swtpmCmd` immediately after the wait-for-socket loop succeeds, so the existing `killSwtpm` paths cover every later failure. The pre-existing `l.vhostCmds` assignment has the same shape and deserves the same fix.

I2. **swtpm is orphaned by `limactl stop -f` and may corrupt NVRAM when killed** — `pkg/driver/qemu/qemu.go:1135-1141`, `pkg/instance/stop.go:111-153` [Codex GPT 5.5, Gemini 3.1 Pro]

```go
swtpmCmd.SysProcAttr = executil.BackgroundSysProcAttr  // Setpgid: true
```

`executil.BackgroundSysProcAttr` is `&syscall.SysProcAttr{Setpgid: true}` on Unix, so swtpm runs in its own process group. `StopForcibly` (`pkg/instance/stop.go:112,147`) sends SIGKILL to `DriverPID` and `HostAgentPID` only; the signal does not reach swtpm. The swtpm args at `qemu.go:1135-1141` carry no `--terminate`, so swtpm waits for a new client after QEMU dies and stays alive indefinitely. Even on the graceful `killSwtpm` path the `Process.Kill()` (SIGKILL) at `qemu_driver.go:602` denies swtpm the chance to flush NVRAM, which can corrupt the TPM state of a Windows guest that holds BitLocker keys.

Fix: add `--terminate` to `SwtpmCmdline` so swtpm exits on QEMU disconnect, and replace `Process.Kill()` in `killSwtpm` with SIGTERM-then-SIGKILL (matching the QEMU shutdown sequence).

I3. **`tpm.enabled` is silently ignored on non-QEMU drivers** — `pkg/limayaml/validate.go`, `pkg/limatype/lima_yaml.go:233-237` [Claude Opus 4.7, Gemini 3.1 Pro]

`templates/default.yaml:638` documents "Only supported with vmType: qemu", but `validate.go` carries no `tpm`-related check and the vz/wsl2 drivers never inspect `Config.TPM`. A user who sets `tpm.enabled: true` with `vmType: vz` passes validation, boots a TPM-less VM, and meets the "This PC can't run Windows 11" message from inside the guest with no host-side warning.

Fix: in `Validate()`, when `*y.TPM.Enabled` is true, reject `*y.VMType != limatype.QEMU`:
```go
if y.TPM.Enabled != nil && *y.TPM.Enabled && *y.VMType != limatype.QEMU {
    errs = errors.Join(errs, errors.New("`tpm.enabled` is only supported with `vmType: qemu`"))
}
```

I4. **TPM architecture rejection fires after additional disks are locked** — `pkg/driver/qemu/qemu.go:713,753-768` [Claude Opus 4.7, Codex GPT 5.5]

```go
default:
    return "", nil, fmt.Errorf("TPM is not supported for architecture %q", *y.Arch)
```

`Cmdline()` locks `AdditionalDisks` via `disk.LockForInstance(cfg.InstanceDir)` at `qemu.go:713`, then reaches the new TPM switch at line 753. When the architecture is unsupported (`riscv64`, `ppc64le`, `s390x`), the function returns an error after the disks already carry an `in_use_by` lock for this instance; the hostagent unlock path in `pkg/hostagent/hostagent.go:592-607` is never registered because `Start()` (`qemu_driver.go:293-295`) returns first. Users then need `limactl disk unlock` to recover. The same error should fire during `limactl create`, before any persistent state changes.

Fix: in `Validate()`, when `*y.TPM.Enabled` is true, reject `*y.Arch` outside `{X8664, AARCH64, ARMV7L}`. The runtime switch in `Cmdline()` can stay as defence in depth.

I5. **`templates/experimental/windows.yaml` fails `limactl create` validation** — `templates/experimental/windows.yaml:5-20` [Codex GPT 5.5, Gemini 3.1 Pro]

I6. **`WindowsSubsystemPath` "fix" trades one broken mount point for another** — `pkg/limayaml/defaults.go:694-703` [Gemini 3.1 Pro, Claude Opus 4.7]

```go
if runtime.GOOS == "windows" {
    if wPath, err := ioutilx.WindowsSubsystemPath(ctx, mountLocation); err != nil {
        logrus.WithError(err).Warnf("Couldn't convert location %q into mount target", mount.Location)
    } else {
        mountLocation = wPath
    }
}
```

`WindowsSubsystemPath` (`pkg/ioutilx/ioutilx.go:53-60`) shells out to `cygpath` and returns `("", err)` on failure. The old code overwrote `mountLocation` with that empty string; the new code preserves the original Windows path like `C:\Users\foo`. Both reach `mount.MountPoint = ptr.Of(mountLocation)` and propagate to the Linux guest, where neither a literal `C:\Users\foo` nor an empty string is a valid mount point. The PR description calls this a fix, but the behaviour is still broken whenever `cygpath` is missing.

Fix: either fail explicitly when cygpath fails, or fall back to `WindowsSubsystemPathForLinux` (which uses `wsl wslpath`) to produce a valid POSIX path. Also see S2 — this change should not ride with the TPM commit.

## Suggestions

S1. **`tpm/` state directory is copied on instance clone** — `pkg/limatype/filenames/filenames.go:107-109`, `pkg/instance/clone.go:60-72` [Claude Opus 4.7]

```go
var SkipOnClone = []string{
    Protected,
}
```

`SkipOnClone` carries only `Protected`. Cloning an instance with TPM enabled copies the swtpm state directory verbatim, so both instances share the same EK seed, NVRAM, and sealed-key state. Two VMs with identical TPM state defeats TPM-bound disk encryption (the clone can unseal the original's disk) and duplicates EK certificates on the network.

Fix: add `SwtpmDir` to `SkipOnClone`. Note that `WalkDir` callbacks need `fs.SkipDir` (not `return nil`) to skip a directory's contents, so the existing `slices.Contains` branch in `clone.go:62-64` also needs to return `fs.SkipDir` when the entry is a directory; verify against an integration test.

S5. **`TestTPMQEMUArgs` writes firmware blobs outside `t.TempDir()` and leaves directories behind** — `pkg/driver/qemu/tpm_test.go:99-139` [Claude Opus 4.7]

```go

shareDir2 := filepath.Join(localDir, "share", "qemu")
```

S7. **`SwtpmCmdline` does not clear stale swtpm state-directory locks** — `pkg/driver/qemu/qemu.go:1126-1133` [Claude Opus 4.7]

```go
swtpmSock := filepath.Join(cfg.InstanceDir, filenames.SwtpmSock)
// Remove stale socket

```

The function removes the stale Unix socket, but swtpm also writes a `.lock` file inside the state directory on every run. After a host crash or SIGKILL, the lock survives, and the next start fails with `TPM already in use`. The virtiofsd analogue (`VirtiofsdCmdline`) does not need this because virtiofsd holds no on-disk lock; swtpm does. S8. **A clean swtpm exit during VM runtime is logged silently** — `pkg/driver/qemu/qemu_driver.go:369-373` [Claude Opus 4.7]

```go
go func() {
    if err := <-swtpmWaitCh; err != nil {
        logrus.Errorf("swtpm exited with error: %v", err)
    }
}()
```

swtpm normally runs until killed by `killSwtpm`. A nil-error exit during runtime — for example, swtpm dying from a signal — leaves QEMU running with a dead TPM and is invisible to the operator. Log at `Info` on nil-error exit too, so guest-side TPM failures correlate with a host-side line.

S9. **`time.NewTimer` inside the swtpm-startup loop never stops** — `pkg/driver/qemu/qemu_driver.go:345-350` [Gemini 3.1 Pro]


## Design Observations

### Concerns

**`TPM` as a top-level field vs. inside `vmOpts.qemu`** (future) [Claude Opus 4.7] — Every other driver-specific feature (vz's Rosetta, wsl2-specific paths, qemu-only accel knobs) lives under `vmOpts.<driver>`. A top-level `tpm.enabled` reads as "portable across drivers," which is not the case today. The PR description calls this a deliberate architectural choice. If vz/krunkit ever grow TPM backends, the top-level location is right; until then, the field reads as misplaced. Either way, gate it against the active driver (I3).

**Helper subprocess lifecycle is spread across QEMU, virtiofsd, and now swtpm** (future) [Codex GPT 5.5] — Each helper carries its own pidfile, signal handling, and cleanup expectations. A small helper registry with `Start`, `Wait`, `Pidfile`, and `Cleanup` hooks would make force-stop and restart behaviour less dependent on each helper remembering its own cleanup path. The swtpm leak in I1, the orphan in I2, and the pre-existing virtiofsd parallel all share this shape.

**swtpm dying mid-run leaves QEMU with a dead TPM** (future) [Claude Opus 4.7] — The lifecycle mirrors virtiofsd, which has no death-of-helper → shutdown-of-QEMU plumbing. A follow-up that ties helper exit into a supervisor that escalates to `qCmd.Process.Signal(SIGTERM)` would catch swtpm crashes, vhost crashes, and future helpers.

### Strengths

**Explicit arch dispatch with a `default` error** (in-scope) [Codex GPT 5.5, Claude Opus 4.7] — The TPM device switch at `qemu.go:761-769` rejects unsupported architectures with a clear error instead of silently emitting an invalid `-device` to QEMU.

**Windows-host guard in `SwtpmCmdline`** (in-scope) [Claude Opus 4.7] — `runtime.GOOS == "windows"` is checked first and returns a precise message; the rest of the function never runs with an inappropriate transport.

**Defaults follow the established triple-merge pattern** (in-scope) [Claude Opus 4.7, Gemini 3.1 Pro] — TPM defaults use the same `y → d → o → fallback` pattern as `NestedVirtualization` and `Plain`, with matching assertions at all four checkpoints in `defaults_test.go`.

## Review Process Notes

**Repo context updates:**

- `[repo]` `pkg/limayaml/validate.go` is the single source of truth for cross-field validation in this repo. New top-level configuration fields with cross-cutting requirements (driver gating, arch gating, mutually-exclusive options) must add their own checks here, even when the documentation in `templates/default.yaml` describes the constraint. Documentation-only constraints get bypassed silently.
- `[repo]` `pkg/instance/stop.go::StopForcibly` only sends SIGKILL to `DriverPID` and `HostAgentPID`. Any helper subprocess launched with `executil.BackgroundSysProcAttr` (`Setpgid: true`) runs in its own process group and will be orphaned by `limactl stop -f` unless it has its own cleanup path or exits when the parent disconnects.
- `[repo]` `pkg/instance/clone.go` uses `filenames.SkipOnClone`, matched against the filesystem entry's basename. Returning `nil` from the walk function continues into a skipped directory's contents; only `fs.SkipDir` actually skips the children. New per-instance secret directories (TPM state, future SEV launch blobs, etc.) need both an entry in `SkipOnClone` and a directory-aware fix in the walk callback.
- `[repo]` `pkg/driver/qemu/qemu.go::Cmdline` acquires per-disk locks via `disk.LockForInstance` before later validation steps. Any new error path added to `Cmdline` should return before the disk-locking block at line 713, otherwise the hostagent's unlock path never runs and disks need manual `limactl disk unlock` recovery.

**Skill improvements:**

- Add a review dimension on "validation gating for new top-level YAML fields": every new field that documents a constraint in the template (driver, arch, host OS, mutually exclusive flag) should be checked in `Validate()` of the relevant package; a documentation-only constraint is a bug magnet because the field will be silently ignored when the constraint is violated. The principle generalises: any time a PR adds a config knob whose docs say "Only supported with X," verify the validate layer enforces it.
- Add a review dimension on "ordering of failure paths against persistent state": when a new error path is added to a function that already mutates persistent state (file locks, directories, network state), verify the error fires before the mutation. The principle generalises beyond Lima: any function that combines validation and resource acquisition should fail before acquiring resources.

# Code Review: PR #5021 — POC: Add QEMU TPM (swtpm) support

## Executive Summary

This POC adds optional swtpm-backed TPM 2.0 support to the QEMU driver via a new `tpm.enabled` field, with arch-specific device selection (`tpm-crb` / `tpm-tis-device`). The implementation mirrors the existing virtiofsd lifecycle pattern. The main concerns are a swtpm-process leak when downstream startup (virtiofsd or QEMU itself) fails after swtpm is running, the absence of any validation tying `tpm.enabled` to `vmType: qemu` (silently ignored on vz/wsl2), and several test-hygiene issues including filesystem pollution outside `t.TempDir()`.

### Important Issues

**I1. swtpm process leaks if vhost or QEMU startup fails after swtpm is running** — `pkg/driver/qemu/qemu_driver.go:315-374, 458-462`

```go
if err := qCmd.Start(); err != nil {
    return nil, err            // swtpm still running, l.swtpmCmd never set
}
l.qCmd = qCmd
l.swtpmCmd = swtpmCmd          // set only after qCmd.Start() succeeds
```

`swtpmCmd` is only assigned to the driver field on line 462, *after* `qCmd.Start()`. If any of the following fails between lines 412 and 458 — vhost `Start()` (414), `vhostSockExists==false` (446), `qCmd.Start()` (458) — `Start()` returns early without invoking `killSwtpm()`, and `l.swtpmCmd` is still nil, so any later `Stop()` returns immediately. The swtpm process is orphaned with its socket and state directory held open. The same anti-pattern exists for vhostCmds (pre-existing); the new code replicates the gap. Tag: (important, gap).

Fix:
```diff
 if err := qCmd.Start(); err != nil {
+    if swtpmCmd != nil && swtpmCmd.Process != nil {
+        _ = swtpmCmd.Process.Kill()
+    }
+    for _, v := range vhostCmds { if v.Process != nil { _ = v.Process.Kill() } }
     return nil, err
 }
```
Or assign `l.swtpmCmd = swtpmCmd` and `l.vhostCmds = vhostCmds` immediately after each successful `Start()` so the existing `killSwtpm`/`killVhosts` paths can clean up.

**I3. Test pollutes filesystem outside `t.TempDir()` and leaks directories** — `pkg/driver/qemu/tpm_test.go:115-139`


## Design Observations

### Concerns

**`TPM` as a top-level field vs. inside `vmOpts.qemu`** (future) — Every other driver-specific feature (qemu-mode-specific accel, vz-specific Rosetta, wsl2-specific tmpfs) lives under `vmOpts.<driver>`. TPM as a top-level toggle implies "this is portable across drivers," which it is not today (only QEMU). The author noted this choice in the PR description as a deliberate architectural decision; it is worth revisiting before stabilization. If vz ever grows a TPM device, the top-level location is correct; if it never does, the field will read as misplaced. Either way, validate it against the active driver (see I2).

**Lifecycle parity with virtiofsd is preserved**, but `virtiofsd`'s lifecycle is itself thin (no death-of-vhost → shutdown-of-QEMU plumbing). Copying that thinness means swtpm dying mid-VM-run leaves QEMU running with a dead TPM, surfacing only as guest-side TPM failures. Consider a follow-up that ties any "process supporting QEMU" (vhost, swtpm, future devices) into a shared supervisor that escalates to `qCmd.Process.Signal(SIGTERM)` on unexpected exit.

### Strengths

- The arch dispatch uses an explicit `switch` with a `default` error case rather than silently emitting an invalid device.
- The Windows-host guard in `SwtpmCmdline` fails fast with an actionable message rather than producing a broken command line.
- Defaults follow the established `d → o → fallback` triple-merge pattern exactly, with matching tests at all three checkpoints.

### Executive Summary

This PR introduces QEMU TPM (`swtpm`) emulation support as a prerequisite for Windows 11 guests, covering configuration, command-line generation, and process lifecycle management. ### Findings

#### Critical Issues

C1. **cygpath failure preserves native Windows path for Linux mount point** — `pkg/limayaml/defaults.go:697-702` (critical, regression)

```diff
 			if runtime.GOOS == "windows" {
-				var err error
-				mountLocation, err = ioutilx.WindowsSubsystemPath(ctx, mountLocation)
-				if err != nil {
+				if wPath, err := ioutilx.WindowsSubsystemPath(ctx, mountLocation); err != nil {
 					logrus.WithError(err).Warnf("Couldn't convert location %q into mount target", mount.Location)
+				} else {
+					mountLocation = wPath
 				}
 			}
```

When `ioutilx.WindowsSubsystemPath` (which requires `cygpath`) fails (e.g., when the user runs native Win32-OpenSSH), the fix preserves `mount.Location` (e.g., `C:\Users\foo`) and assigns it to `mount.MountPoint`. A native Windows path containing `:` is an invalid POSIX path for the Linux guest and breaks provisioning when the guest agent attempts to `mkdir -p` the mount point. The fix trades a previous empty-string failure for a broken guest directory structure.

Fix: Do not pass native Windows paths to the Linux guest. If `cygpath` fails, either fail explicitly or fallback to `WindowsSubsystemPathForLinux` to produce a valid POSIX path (e.g., `/mnt/c/Users/foo`).

#### Important Issues

I1. **TPM enabled silently ignored on non-QEMU drivers** — `pkg/limayaml/validate.go` (important, gap)

The documentation explicitly states TPM is "Only supported with vmType: qemu." However, the validation logic lacks any check to enforce this. If a user sets `vmType: vz` and `tpm.enabled: true`, the validation passes, the driver ignores the setting, and the VM silently boots without a TPM. This will cause Windows 11 installations to fail later without clear feedback from Lima.

Fix: Add a check in `Validate()` to enforce the driver constraint:
```go

		errs = errors.Join(errs, errors.New("field `tpm.enabled` is only supported with `vmType: qemu`"))
	}
```

I2. **Forceful termination of swtpm may corrupt TPM NVRAM** — `pkg/driver/qemu/qemu.go:1135-1142` (important, regression)

```diff
 		"socket",
 		"--tpmstate", "dir=" + stateDir,
 		"--ctrl", "type=unixio,path=" + swtpmSock,
 		"--tpm2",
 		"--log", "level=1",
+		"--terminate",
 	}
```

The `killSwtpm` function (`qemu_driver.go:602`) forcefully terminates `swtpm` using `Process.Kill()` (SIGKILL). Force-killing `swtpm` prevents it from cleanly flushing its NVRAM state to the disk, which can corrupt the TPM state for the guest.

Fix: Since QEMU acts as the client and gracefully shuts down before `killSwtpm` is called, pass the `--terminate` flag to `swtpm`. This instructs `swtpm` to exit gracefully when the client disconnects, making `killSwtpm` safely act as a backstop.
