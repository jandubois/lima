## Executive Summary

Round 3 reviews the same commit (`45823748`) as Round 2 — no code has changed
since 2026-03-29, so the previously identified gaps remain. The core
implementation in `TranslateHostPath` is sound (longest-match on cleaned
mount locations, `filepath.Rel` + `filepath.IsLocal` containment, hard
errors), but mount locations and host paths are not symlink-resolved. On
macOS this rejects legitimate paths whenever `/tmp` → `/private/tmp`,
`$HOME` lives on an external volume, or any caller canonicalizes a path
through `readlink` first. The `bats` instance now unconditionally mounts
`$HOME`, which affects every other test that reuses that instance and
creates host-side cleanup hazards on assertion failure. Finally, `Glob`
and `SearchFileContent` still return raw guest paths that the agent
cannot feed back into `ReadFile`/`WriteFile` because the new strict
validation rejects them — pre-existing design, newly consequential.

## Important Issues

I1. **Mount locations and host paths are not symlink-resolved** — `pkg/mcp/toolset/toolset.go:121-145` [Claude Opus 4.7, Codex GPT 5.4, Gemini 3.1 Pro] (round 2 I1 unaddressed)

```go
func (ts *ToolSet) translateToGuestPath(hostPath string) (string, bool) {
	cleanPath := filepath.Clean(hostPath)
	bestGuestPath := ""
	bestLen := -1
	for _, mount := range ts.inst.Config.Mounts {
		if mount.MountPoint == nil {
			continue
		}
		location := filepath.Clean(mount.Location)
		rel, err := filepath.Rel(location, cleanPath)
		if err == nil && filepath.IsLocal(rel) {
```

`filepath.Clean` canonicalizes separators and collapses `..`/`.` components
but does not follow symlinks. Lima stores mount locations cleaned but not
resolved (`pkg/limayaml/defaults.go:682`), and the existing hostagent code
at `pkg/hostagent/inotify.go:75-81` explicitly calls `filepath.EvalSymlinks`
on every mount location for exactly this reason. The new MCP validation
diverges from that precedent.

Common failure cases on macOS:

- `/tmp` is a symlink to `/private/tmp`. If the mount location is `/tmp`
  and the agent passes `/private/tmp/foo`, `filepath.Rel("/tmp", "/private/tmp/foo")`
  returns `"../private/tmp/foo"`, `IsLocal` returns false, and the call
  fails with `path "..." is not under any mounted directory`.
- `$HOME` on an external volume (e.g. `/Users/jan` → `/Volumes/Data/Users/jan`).
  The default template mounts `${HOME}` as `/Users/jan`, so any caller that
  canonicalizes through the symlink target fails validation.

Every MCP filesystem and shell tool routes through `TranslateHostPath`, so
this regression blocks `read_file`, `write_file`, `list_directory`, `glob`,
`search_file_content`, and `run_shell_command` on affected paths. The new
unit tests in `toolset_test.go` exercise only lexical parent/child cases;
no test covers a symlinked prefix.

Fix: resolve symlinks on both the host path and each mount location before
comparison. Tolerate `EvalSymlinks` errors (fall back to the cleaned path)
so that write_file targeting a not-yet-existing file under a symlinked
parent still works — resolve the deepest existing ancestor and re-append
the unresolved suffix.

```diff
 func (ts *ToolSet) translateToGuestPath(hostPath string) (string, bool) {
-	cleanPath := filepath.Clean(hostPath)
+	cleanPath := resolveExistingAncestor(filepath.Clean(hostPath))
 	bestGuestPath := ""
 	bestLen := -1
 	for _, mount := range ts.inst.Config.Mounts {
 		if mount.MountPoint == nil {
 			continue
 		}
-		location := filepath.Clean(mount.Location)
+		location, err := filepath.EvalSymlinks(filepath.Clean(mount.Location))
+		if err != nil {
+			continue
+		}
 		rel, err := filepath.Rel(location, cleanPath)
```

Add a regression test with a symlinked host prefix that mirrors the cases in
`pkg/hostagent/inotify_test.go`. Consider extracting a shared mount-translation
helper so both call sites use the same symlink semantics (see Design
Observations below).

I2. **`Glob` and `SearchFileContent` return guest paths that the agent cannot feed back to other tools** — `pkg/mcp/toolset/filesystem.go:130-139, 165-170` [Gemini 3.1 Pro] (round 1 I4 deferred)

```go

`Glob` returns absolute guest paths from `ts.sftp.Glob`. `SearchFileContent`
runs `git grep` with the guest path, so stdout lines begin with guest paths.
When the AI agent feeds either result back into `ReadFile`, `WriteFile`, or
any other tool, `TranslateHostPath` rejects the guest path because it does
not sit under any host mount. The workflow happens to succeed today only
when the mount is symmetric (e.g. `$HOME` → `$HOME`, as the BATS harness
arranges); any asymmetric mount breaks it.

This issue is pre-existing (no translation was performed before this PR)
but the new strict validation makes it user-visible. Round 1 flagged it
(I4) and deferred to a follow-up; it remains open.

Fix: in `Glob`, translate each match from guest path back to host path
before returning. In `SearchFileContent`, pass a path relative to the
search directory so `git grep` emits host-relative output.

-	matches, err := ts.sftp.Glob(pattern)
+	pattern := path.Join(guestPath, args.Pattern)
+	matches, err := ts.sftp.Glob(pattern)

 		matches = []string{}
 	}
 	if err != nil {
 		return nil, nil, err
 	}
+	guestPrefix := strings.TrimRight(guestPath, "/") + "/"
+	for i, m := range matches {
+		if rel, ok := strings.CutPrefix(m, guestPrefix); ok {
+			matches[i] = filepath.Join(pathStr, filepath.FromSlash(rel))
+		}
+	}
```

Gemini raised this at critical severity; we consolidate to important
because Round 1 and Round 2 both reviewed this as a deferred design gap
rather than a merge blocker, and every caller has a usable error message
rather than silent corruption.

I3. **Changing the shared `bats` instance to mount `$HOME:w` affects every other test that reuses the instance** — `hack/bats/helpers/limactl.bash:31` [Claude Opus 4.7] (round 2 I2 unaddressed)

```bash
bats)          limactl start --yes --name "$instance" --mount "$HOME:w" template:default 3>&- 4>&- ;;
```

`copy.bats`, `path.bats`, `preserve-env.bats`, and `mcp.bats` all set
`INSTANCE=bats` and share the same VM. Two concrete risks:

1. **Stale instances with `LIMA_BATS_REUSE_INSTANCE`.** `ensure_instance`
   at `hack/bats/helpers/limactl.bash:24-27` reuses the instance when it is
   already Running. A developer with a previously created `bats` instance
   (no `$HOME` mount) silently runs `mcp.bats` against a VM missing the
   required mount, and every test fails with
   `path "..." is not under any mounted directory`. The failure points at
   the tool output, not the helper change.
2. **Test output depends on the developer's `$HOME`.** `mcp.bats` writes
   to `$HOME/mcp.test`, `$HOME/tmp_mcp_test`, `$HOME/tmp_mcp_ro`, etc.
   A path collision with existing files produces confusing failures, and
   assertion failures mid-test leave artifacts behind (see I4).

Fix: verify the `$HOME` mount exists in `local_setup` and fail fast with a
clear diagnostic, or rename the instance (e.g. `bats-mount-home`) so that
tests requiring `$HOME` do not force the mount onto unrelated tests. Add a
trailing comment on the `--mount "$HOME:w"` line explaining the coupling.

## Suggestions

S1. **`TranslateHostPath` / `translateToGuestPath` split adds indirection without clarity** — `pkg/mcp/toolset/toolset.go:106-145` [Claude Opus 4.7]

```go
func (ts *ToolSet) TranslateHostPath(hostPath string) (guestPath string, err error) {
	if hostPath == "" {
		return "", errors.New("path is empty")
	}
	if !filepath.IsAbs(hostPath) {
		return "", fmt.Errorf("expected an absolute path, got a relative path: %q", hostPath)
	}

	guestPath, isMounted := ts.translateToGuestPath(hostPath)
	if !isMounted {
		return "", fmt.Errorf("path %q is not under any mounted directory", hostPath)
	}
	return guestPath, nil
}
```

Nothing else calls `translateToGuestPath`. The two names differ only by a
preposition, so a reader must compare both bodies to see which does what.
Either inline the loop into `TranslateHostPath`, or rename the private
helper to `lookupMount` so the boolean return reads naturally (`guest, ok := ts.lookupMount(host)`).

S2. **`filepath.Rel` compares paths case-sensitively on case-insensitive filesystems** — `pkg/mcp/toolset/toolset.go:130` [Claude Opus 4.7]

```go
rel, err := filepath.Rel(location, cleanPath)
if err == nil && filepath.IsLocal(rel) {
```

macOS APFS (default case-insensitive) and Windows NTFS treat `/Users/Jan`
and `/users/jan` as the same directory, but `filepath.Rel` compares
lexically. An agent that lowercases paths, or one that happens to emit a
different casing than the mount configuration, fails validation even
though the file is reachable.

Fix: on darwin and windows, compare case-insensitively, or resolve both
sides through `EvalSymlinks` against the real filesystem — which
canonicalizes case and composes with the symlink fix in I1.

S3. **Test coverage for the longest-match tiebreaker is order-sensitive only** — `pkg/mcp/toolset/toolset_test.go:38-84` [Claude Opus 4.7]

```go
{
    name:     "child mount listed first",
    hostPath: absPath("/parent/child/file.txt"),
    ...
    Mounts: []limatype.Mount{
        {Location: absPath("/parent/child"), MountPoint: &mountPointChild},
        {Location: absPath("/parent"),       MountPoint: &mountPointParent},
    },
```

All three overlapping-mount cases use the same `/parent` + `/parent/child`
pair. Add a case with two mounts whose locations have equal string length
but different directories (e.g. `/foo/aa` and `/bb/cc`) to confirm
selection follows real prefix containment rather than lexical ordering,
and a three-deep case (`/a`, `/a/b`, `/a/b/c`) to confirm `bestLen`
scales past two entries.

S4. **`read_file returns an error when path does not exist` depends on the developer's `$HOME` contents** — `hack/bats/tests/mcp.bats:214-218` [Claude Opus 4.7]

```bash
@test 'read_file returns an error when path does not exist' {
    tools_call read_file '{"path":"'"$HOME/os-release-info"'"}'
```

The test assumes `$HOME/os-release-info` does not exist. A developer with
such a file (however unlikely) sees `isError: null` instead of `true`.

Fix: use a randomized filename, e.g. `$HOME/os-release-info.$$.$RANDOM`,
or `$HOME/mcp-nonexistent-$BATS_TEST_NUMBER`. `$BATS_TEST_TMPDIR` lives
under `/private/var/folders/...` on macOS (outside the `$HOME` mount), so
it is not a drop-in replacement.

S5. **`WriteFile` ignores the error returned by `Close`** — `pkg/mcp/toolset/filesystem.go:95-103` [Gemini 3.1 Pro]

```go
f, err := ts.sftp.Create(guestPath)
if err != nil {
    return nil, nil, err
}
defer f.Close()
_, err = f.Write([]byte(args.Content))
if err != nil {
    return nil, nil, err
}
```

SFTP servers may defer flush to `Close`; a failed close can mean the data
never reached disk but the tool reports success. Capture the close error
and return it as the operation result when `Write` itself succeeded.

S6. **MCP tool schema descriptions still advertise "absolute path" without the mount constraint** — `pkg/mcp/msi/filesystem.go:24,50,65,76,91` (round 2 S1 unaddressed)

```go
Path string `json:"path" jsonschema:"The absolute path to the file to read."`
```

After the new hard error at `toolset.go:116`, tools reject absolute paths
that fall outside every configured mount. The schema text advertises the
old contract, so the AI agent receives tool-call errors it cannot predict
from the tool description, causing avoidable retries and confusing
failure modes. The same applies to `run_shell_command`'s `directory`
parameter (`pkg/mcp/msi/shell.go`).

Fix: update the jsonschema strings to something like
`The absolute host path to the file to read. The path must lie under a
mount location configured for this Lima instance.`

S7. **Public `TranslateHostPath` has no nil-instance guard** — `pkg/mcp/toolset/toolset.go:106` (round 2 S2 unaddressed)

```go
guestPath, isMounted := ts.translateToGuestPath(hostPath)
```

`translateToGuestPath` dereferences `ts.inst.Config.Mounts` at line 125.
Every current caller (`ListDirectory`, `ReadFile`, `WriteFile`, `Glob`,
`SearchFileContent`, `RunShellCommand`) checks `ts.inst == nil` first, so
the current code is safe. The precondition is undocumented, and a future
caller that reuses `TranslateHostPath` directly crashes with a nil-pointer
panic rather than a clear error.

Fix: either add `if ts.inst == nil { return "", errors.New("instance not registered") }` at the top of `TranslateHostPath`, or add a godoc line
stating that the method requires `RegisterInstance` to have succeeded.

## Design Observations

### Concerns

- **Mount translation is duplicated across packages** *(in-scope)* — `pkg/mcp/toolset/toolset.go:121-145` reimplements host-to-guest translation already present in `pkg/hostagent/inotify.go:95-109`. The two copies have diverged: the hostagent version resolves symlinks, the toolset version does not. I1 is the first concrete bug caused by that divergence. Extracting a shared helper in a small mount-translation utility would unify symlink handling and give both call sites the same regression tests. [Claude Opus 4.7, Codex GPT 5.4, Gemini 3.1 Pro]

### Strengths

- **Containment via `filepath.Rel` + `filepath.IsLocal`** *(in-scope)* — handles trailing slashes, `..` components, same-prefix-different-directory cases (`/home/user` vs `/home/user2`), and Windows cross-drive paths with one primitive. Naive `strings.HasPrefix` would get all of these wrong. [Claude Opus 4.7, Codex GPT 5.4, Gemini 3.1 Pro]
- **Centralized enforcement** *(in-scope)* — every MCP tool routes through `TranslateHostPath` rather than each tool repeating the policy. [Claude Opus 4.7, Codex GPT 5.4]
- **Longest-match heuristic for overlapping mounts** *(in-scope)* — after `filepath.Clean`, a longer canonical location string is always the more specific mount. Simple and correct. [Claude Opus 4.7]
- **POSIX/native separator split** *(in-scope)* — `filepath.ToSlash(rel)` + `path.Join(*mount.MountPoint, ...)` correctly produces a Linux path for the guest regardless of host OS. [Claude Opus 4.7]

# Deep Code Review — PR #4419: Hostpath mount validation

**Verdict:** Request changes — the new validation is sound, but a pre-existing repo convention for symlink resolution is not followed, which will cause legitimate paths to be rejected on macOS (e.g., `/tmp` → `/private/tmp`) and on any host where `$HOME` is a symlink.

## Executive Summary

The PR replaces a `TODO` stub in `TranslateHostPath` with a real mount-matching implementation that walks the configured mounts, picks the longest matching location, and returns an error when the host path is outside every mount. The translation logic itself is correct and well tested, but it does not resolve symlinks on the host path or on mount locations — Lima already has a matching helper (`pkg/hostagent/inotify.go:70-93`) that does resolve them, so the two mount-matching call sites will now diverge.

- macOS default: `/tmp` is a symlink to `/private/tmp`.
