Executive Summary ¶

This PR adds an experimental Windows guest on QEMU plus its Secure Boot and TPM 2.0 prerequisites. Five fixable issues stand out: a 1.5-second sleep races the Windows installer boot prompt on slow hosts; a single unreadable firmware descriptor aborts the entire descriptor lookup; the UEFI variable store persists across Secure Boot mode changes; the 200×3-second per-requirement retry budget under-caps Windows install time relative to the 60-minute outer timeout; and swtpm shutdown produces a spurious error on every Stop.

The change isolates Windows-specific code paths cleanly: descriptor selection lives in pkg/driver/qemu/firmware.go, Windows cidata sits beside the existing Linux template in pkg/cidata, and the schema additions (Firmware.SecureBoot, Firmware.TPM2, GuestDrivers) carry validation. Three independent reviewers (Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro) ran the same prompt; the boot-key race was raised by all three, lending strong consensus to the Critical severity. Suggestions cluster around polishing the descriptor-matching heuristics, routing virtio-win ISOs through the existing download pipeline, and aligning shell-script and template defaults across host OSes.

func (a *HostAgent) waitForRequirements(label string, requirements []requirement) error {
	const (
		retries       = 200
		sleepDuration = 3 * time.Second
	)
	var errs []error

	for i, req := range requirements {

				break retryLoop
			}
			if req.fatal {
				logrus.Infof("No further %s requirements will be checked", label)
				errs = append(errs, fmt.Errorf("failed to satisfy the %s requirement %d of %d %q: %s; skipping further checks: %w", label, i+1, len(requirements), req.description, req.debugHint, err))
				return errors.Join(errs...)
			}
			if j == retries-1 {
				errs = append(errs, fmt.Errorf("failed to satisfy the %s requirement %d of %d %q: %s: %w", label, i+1, len(requirements), req.description, req.debugHint, err))
				break retryLoop
			}
			time.Sleep(sleepDuration)
		}
	}
	return errors.Join(errs...)
}

	}
	return codePath, varsPath, codeFormat, varsFormat, nil
}

S2. firmwareUses4M substring check matches "4m" anywhere in the filename Claude¶

A descriptor whose code path contains "4m" anywhere (e.g., a vendor name) is selected over a non-4M alternative. Anchor the match to the conventional suffix: _4M., -4m., or filename suffix.

S3. Descriptor-not-found error inlines every candidate path Claude¶

With dozens of system descriptors, fmt.Errorf("... from %v ...", descriptorPaths, ...) prints a multi-line list. Report a count plus an actionable hint ("install distribution OVMF packages or set firmware.descriptors") instead.

S4. qemuFirmwareDescriptorDirs adds Unix-only paths on Windows hosts Claude¶

/etc/qemu/firmware and /usr/share/qemu/firmware are appended unconditionally; on Windows the localDir != "/usr" check is a no-op and the path itself is meaningless. Gate the Unix entries on runtime.GOOS != "windows".

S5. Boot-key marker uses 0o644 while adjacent secret files use 0o600 Claude¶

The marker carries no secret, but windowsUserPassword writes its file with 0o600. Use 0o600 for both to keep the instance directory's mode story uniform.

S6. generateAlphaNumericPassword may sample passwords lacking required character classes Codex¶

24 chars from [A-Za-z0-9] make a missing-class outcome statistically rare but possible. If the host or guest enforces Windows local-account complexity policy (Server SKUs, custom GPO), unattend may fail. Guarantee at least one upper, one lower, and one digit, then shuffle.

S7. templates/windows.yaml declares minimumLimaVersion: 2.0.0 Codex¶

The template depends on the new Firmware.SecureBoot, Firmware.TPM2, and GuestDrivers fields that don't exist in 2.0.0. Bump to the first release that contains these schema and runtime changes.

S8. QEMU firmware descriptors can carry relative paths Gemini¶

Per interop/firmware.json, filename may be relative to the descriptor's directory. The current code passes the raw path to osutil.FileExists, rejecting valid system descriptors with relative paths. Resolve relative paths against filepath.Dir(descriptorPath) before the existence check.

Design Observations ¶

Concerns ¶

  • Windows setup state is inferred from host-side timers and marker files (in-scope) Codex. I1 (retry budget) and C1 (boot-key marker) are symptoms of the same model. A small state machine tied to guest-observable milestones — setup.done on the cidata oem/ share, sshd reaching Running, OpenSSH cap install completing — would let the host wait on real progress instead of fixed budgets. The PR already writes setup.done from windows-setup.ps1; build the rest of the state on top of it.
  • Hand-rolled QEMU descriptor parser covers a partial slice of interop/firmware.json (future) Claude. Multiple flash variants per target, stateless mode, and additional feature flags are absent. Lima only needs to find a usable file pair, so the partial implementation is correct for now; document the limited subset in a docstring on qemuFirmwareDescriptor with a link to the schema.
  • GuestDrivers is generically named but currently serves only Windows virtio drivers (future) Claude. If future guests reuse the field, the name fits; if not, WindowsDrivers reads truer. Pick one and document the contract.

Strengths ¶

  • Descriptor-based firmware selection generalises beyond Windows; Linux guests with Secure Boot also benefit, and the legacy Firmware.Images fallback stays intact Claude.
  • qemuFirmwareDescriptorPaths correctly implements QEMU's zero-byte masking rule, letting users override system descriptors via a same-named empty file in a more-specific directory Gemini.
  • ensureFirmwareFileIfMissing preserves per-instance NVRAM across restarts so enrolled keys survive — verified by the unit test Claude. (Pair with I2's caveat for cross-mode persistence.)
  • Mounting cidata.iso over IDE plus $WinPEDriver$ driver injection avoids slipstreaming a custom Windows ISO Gemini.
  • Validation in pkg/limayaml/validate.go ships with the feature: TPM/Secure Boot/Legacy BIOS combinations and architecture / vmType gating are enforced Claude.

Testing Assessment ¶

Unit tests cover getFirmwareFromDescriptorFiles and ensureFirmwareFileIfMissing. Gaps ranked by risk:

  1. C1 (sendkey race): no test asserts the marker isn't written before the guest reaches the prompt.
  2. C2 (descriptor abort): no test feeds a malformed JSON alongside a valid descriptor.
  3. I1 (retry budget): no test exercises a Windows install whose SSH appears after 10 minutes.
  4. I2 (vars persistence): no test toggles secureBoot or preEnrollSecureBootKeys between runs and checks NVRAM identity.
  5. I3 (kill error): no test verifies Stop() returns nil on a TPM-enabled instance.
  6. templateArgsForWindows: no coverage for generated-password reuse vs. regeneration, or XML-special characters in User.Name.
  7. S1/S2/S8: firmwareTargetsQ35 and firmwareUses4M predicates and the descriptor relative-path resolver have no unit tests.

Documentation Assessment ¶

  • templates/default.yaml documents the new firmware.secureBoot, firmware.tpm2, firmware.preEnrollSecureBootKeys, firmware.descriptors, and guestDrivers keys adequately.
  • New exported APIs in pkg/cidata/template.go and pkg/limatype/lima_yaml.go lack godoc (see S10).
  • pkg/driver/qemu/firmware.go has no file- or package-level docstring explaining the descriptor selection algorithm.
  • The docs/ and website/content/ trees gain no user-facing material; an experimental feature is hard to discover without it. Add a short page describing prerequisites (swtpm, VirtIO driver ISO, OVMF packages), the host OSes that work today, and the documented limitations from the PR description.

Acknowledged Limitations ¶

The PR description states the change is experimental and lists scope cuts that the consolidated findings respect:

  • Non-QEMU drivers, non-Linux hosts (other than macOS implications surfaced in I4), and non-x86_64 architectures are not in scope.
  • Legacy BIOS is untested.
  • Guest agent for Windows is not implemented; SSH/PowerShell is the readiness path (pkg/instance/start.go:62-64). I1 lands harder because of this.
  • pkg/driver/qemu/qemu.go:805-807 documents the boot-order workaround on first boot; C1 flags the workaround's timing, not the documented choice.
  • pkg/limatype/lima_yaml.go:115 notes "Password is currently used only for Windows guests"; the corresponding fields are scoped accordingly.

Review Process Notes ¶

Repo context updates ¶

  • ADD: When reviewing a new pkg/driver/qemu change that calls cmd.Process.Kill() followed by cmd.Wait() (directly or via a channel), check that the calling code distinguishes "the wait error followed a deliberate kill" from "the process exited on its own with an error." Otherwise Wait returns signal: killed on every clean shutdown and propagates through errors.Join to user-facing return values. The pattern appears in killProcess, but any new subprocess (virtiofsd, swtpm, sidecar VMMs) using the same Kill+Wait shape carries the same risk. Flag findings only when the caller-facing return value is observable (a Stop() method, an HTTP/CLI error path).
  • ADD: When a Lima driver allocates per-instance state files keyed by content type (e.g., pkg/driver/qemu/firmware.go's QemuEfiVarsFD / QemuEfiVarsQCOW2), check whether the path key encodes every dimension that affects compatibility. Switching between secureBoot=false and secureBoot=true, or between preEnrollSecureBootKeys=false and =true, currently produces the same per-instance vars path. Any new firmware/NVRAM/state file added to pkg/driver/qemu (or any other driver) must either include the relevant feature flags in its filename or carry a sidecar identity check that refuses to start on mismatch.
  • ADD: When a Lima feature uses pkg/hostagent/requirements.go's waitForRequirement for first-boot probes whose budget might exceed 10 minutes (Windows install, large image conversion, slow distribution package installs), check whether the call site also adjusts the per-requirement retry constants (retries = 200, sleepDuration = 3 * time.Second) or makes them context-aware. The outer watchHostAgentTimeout is irrelevant when the inner budget caps the wait. Today only Windows installs hit this; future heavy first-boot flows will too.

Skill improvements ¶

  • ADD to dimensions.md (review dimensions list): When a finding describes a defer-based cleanup tied to a struct field's nil/non-nil state, confirm the field's actual assignment lifecycle by reading the function from "field declared" to "function returns" before classifying severity. Speculative cleanup races ("if X were already set...") need a concrete code path that produces the state; without one, the finding is a false positive and should be dropped rather than carried as Important. Three independent agents can still all reach the same wrong conclusion if the prompt does not require this verification step.
