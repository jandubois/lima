## Executive Summary

This branch adds first-class support for "plain Windows" hosts â€” Windows
10/11 without Git for Windows, MSYS2, or Cygwin â€” across SSH path
translation, copy tooling, decompression, and CI. The core mechanism is
`sshutil.IsSSHCygwin` (probes for `cygpath.exe` alongside `ssh.exe`) and
`sshutil.PathForSSH` (a single chokepoint that picks Cygwin-style paths
for Cygwin ssh and forward-slash native paths for Win32-OpenSSH). The
host-agent reverse-sshfs mount, identity files, control paths, and
`limactl copy` all funnel through this helper. Two new CI jobs
(`windows-plain-wsl2`, `windows-plain-qemu`) uninstall MSYS2 / Git for
Windows from `windows-2025` runners and verify across filesystem, PATH,
binaries, and registry that the host is genuinely vanilla before exercising
a create/start/shell/copy/stop/delete cycle. The branch also drops the
external `gzip` dependency (in-process `compress/gzip`) and removes the
deprecated `_LIMA_WINDOWS_EXTRA_PATH` env var.

Three issues warrant attention before merge: `PathForSSH` invokes a bare
`cygpath` from `PATH` rather than the sibling next to the detected
`ssh.exe`, breaking the toolchain-tracking contract when `$SSH` points
outside `PATH`; the plain QEMU CI job never adds `C:\Program Files\QEMU`
to `PATH` after `windows_plain_host` recomposes it, so `qemu-system-x86_64`
likely is not resolvable for the smoke test; and the new in-process gzip
path drops context cancellation responsiveness that the external
subprocess provided.

## Important Issues

I1. **`PathForSSH` invokes a `cygpath` outside the detected ssh toolchain** â€” `pkg/sshutil/sshutil.go:136-137`, `pkg/ioutilx/ioutilx.go:60-61` [Codex GPT 5.5]

```go
if IsSSHCygwin(sshExe) {
    return ioutilx.WindowsSubsystemPath(ctx, orig)
}
```

```go
out, err := exec.CommandContext(ctx, "cygpath", filepath.ToSlash(orig)).CombinedOutput()
```

`IsSSHCygwin` confirms a Cygwin install by spotting `cygpath.exe` next to
the resolved `ssh.exe` (`sshutil.go:111-113`), but `PathForSSH` then
discards that sibling and delegates to `WindowsSubsystemPath`, which runs
a bare `"cygpath"` resolved through `PATH`. When `$SSH` points at a
Git-for-Windows / MSYS2 / Cygwin `ssh.exe` whose directory is not first on
`PATH`, the bare lookup either fails (falling through to the native
drive-letter conversion in `ioutilx.go:65-75`, which ignores any custom
MSYS2 `fstab`) or resolves to a different installation's `cygpath` with
different fstab rules. The result is a path that does not match what the
selected ssh expects â€” silently wrong identity files, control paths, copy
arguments, and reverse-sshfs `LocalPath`.

The common path (no `$SSH` set; `ssh.exe` discovered through `PATH`) does
not trip the bug, because the sibling `cygpath.exe` and `ssh.exe` share a
`PATH` entry. The bug is narrow but real and silent â€” the wrong path is
returned without an error.

Fix: thread the sibling `cygpath.exe` discovered in `IsSSHCygwin` through
to `WindowsSubsystemPath` (e.g., a new
`WindowsSubsystemPathWithCygpath(ctx, cygpathExe, orig)`), and keep the
bare-`cygpath` fallback only for callers that have no specific ssh
binary to anchor against.

I2. **Plain QEMU CI never adds `C:\Program Files\QEMU` to PATH** â€” `.github/workflows/test.yml:272-279`, `.github/actions/windows_plain_host/action.yml:68-71` [Codex GPT 5.5]

```yaml
- uses: ./.github/actions/windows_qemu_install
- uses: ./.github/actions/windows_plain_templates
- uses: ./.github/actions/windows_plain_host
- uses: ./.github/actions/windows_plain_build
- uses: ./.github/actions/windows_plain_smoke_test
```

The QEMU driver resolves `qemu-system-*` through `$QEMU_SYSTEM_*` first
and `exec.LookPath` otherwise (`pkg/driver/qemu/qemu.go:1104-1121`). The
existing `windows-qemu` job prepends `C:\Program Files\QEMU` to `$env:PATH`
explicitly (`test.yml:215`); the new `windows-plain-qemu` job does not.
`windows_plain_host` then recomposes `PATH` from machine + user scopes
and writes the result to `$GITHUB_ENV`, discarding any process-only
augmentation an installer step might have done. The `windows_qemu_install`
action runs only `winget install â€¦QEMU` (`windows_qemu_install/action.yml:9`);
the SoftwareFreedomConservancy QEMU manifest does not register a PATH
entry.

Net effect: when the smoke test launches `limactl start`, the QEMU driver
calls `exec.LookPath("qemu-system-x86_64")` and gets `executable file
not found in %PATH%`. The test fails at first VM launch.

Fix: add a PATH prepend step between `windows_plain_host` and
`windows_plain_build` (or set `QEMU_SYSTEM_X86_64` for the smoke step):

```diff
     - uses: ./.github/actions/windows_qemu_install
     - uses: ./.github/actions/windows_plain_templates
     - uses: ./.github/actions/windows_plain_host
+    - name: Add QEMU to PATH
+      shell: pwsh
+      run: Add-Content -Path $env:GITHUB_PATH -Value 'C:\Program Files\QEMU'
     - uses: ./.github/actions/windows_plain_build
```

I3. **In-process gzip decompression ignores context cancellation** â€” `pkg/downloader/downloader.go:566-574` [Gemini 3.1 Pro]

```go
if decompressCmd == "gzip" {
    gz, err := gzip.NewReader(bar.NewProxyReader(in))
    if err != nil {
        return err
    }
    defer gz.Close()
    _, err = io.Copy(out, gz)
    return err
}
```

The previous code path forked `gzip -d` with `exec.CommandContext`, so
`ctx` cancellation killed the subprocess and unblocked the parent's
stdin / stdout pipes within a few ms. The new in-process path uses
`io.Copy(out, gz)` where both `out` and the underlying reader are
`*os.File` handles â€” neither honours `ctx`, so a `^C` while decompressing
a multi-GB VM image (the practical case this code path handles) is
ignored until the copy finishes. The other branches (`bzip2`, `xz`,
`zstd`) still get prompt cancellation through `exec.CommandContext`,
so this is an asymmetric regression introduced by `abae6d76`.

Fix: spawn a watcher goroutine that closes `in` (and `out`) when `ctx`
is done, so a blocked `io.Copy` returns with an error:

```diff
     if decompressCmd == "gzip" {
         gz, err := gzip.NewReader(bar.NewProxyReader(in))
         if err != nil {
             return err
         }
         defer gz.Close()
+        ctx, cancel := context.WithCancel(ctx)
+        defer cancel()
+        go func() {
+            <-ctx.Done()
+            in.Close()
+        }()
         _, err = io.Copy(out, gz)
         return err
     }
```

## Suggestions

S1. **`parseCopyPaths` re-resolves `ssh.exe` and forks `cygpath` per absolute Windows path, twice per `limactl copy`** â€” `pkg/copytool/copytool.go:111-125`, `pkg/copytool/rsync.go:37-58` [Claude Opus 4.7]

```go
for _, path := range paths {
    cp := &Path{}
    isLocalAbs := runtime.GOOS == "windows" && filepath.IsAbs(path)
    if isLocalAbs {
        sshExe, err := sshutil.NewSSHExe()
        if err != nil {
            return nil, err
        }
        path, err = sshutil.PathForSSH(ctx, sshExe, path)
```

`rsyncTool.IsAvailableOnGuest` (`rsync.go:38`) and `rsyncTool.Command`
(`rsync.go:93`) each call `parseCopyPaths`, so the loop above runs twice
per copy. For every absolute Windows path in `paths`, `NewSSHExe()` does
an `exec.LookPath("ssh")` and `PathForSSH` may fork `cygpath` (a ~50â€“200 ms
subprocess). A two-arg `limactl copy C:\src C:\dst-of-instance-mount` on
Cygwin-backed ssh ends up forking `cygpath` four times â€” purely from
path classification. Pre-PR, only `cygpath` ran (no ssh resolution per
path).

Fix: resolve `sshExe` once per `parseCopyPaths` call (or hoist into the
caller and pass down), and reuse it for every absolute path.

S5. **`ParseOpenSSHVersion` regex captures only `MAJOR.MINOR`; multi-segment Win32-OpenSSH banners lose precision** â€” `pkg/sshutil/sshutil.go:464-481` [Claude Opus 4.7]

```go
regex := regexp.MustCompile(`(?m)^OpenSSH(?:_for_Windows)?_(\d+\.\d+)(?:p(\d+))?\b`)
```

If Win32-OpenSSH ever ships a banner like `OpenSSH_for_Windows_9.5.0.0p1`
(its release tags sometimes use 4-part versions), the regex captures
`9.5` and treats `.0.0p1` as a non-match for `(?:p(\d+))?` â€” the parsed
version is `9.5.0`, two integers adrift from the source. Current
version-gated checks (`< 8.0.0`, `< 8.1.0`) are unaffected, but the test
pins `OpenSSH_for_Windows_9.5p2 â†’ 9.5.2` without covering the longer
form. Either widen the regex to consume extra numeric segments before
`p`, or note the limitation in the comment.

Suggested regex (with a test case for 4-part input):

```diff
-regex := regexp.MustCompile(`(?m)^OpenSSH(?:_for_Windows)?_(\d+\.\d+)(?:p(\d+))?\b`)
+regex := regexp.MustCompile(`(?m)^OpenSSH(?:_for_Windows)?_(\d+\.\d+)(?:\.\d+(?:\.\d+)?)?(?:p(\d+))?\b`)
```

S6. **`windows_plain_host` rejects any `git.exe`, even a standalone install outside the Git-for-Windows tree** â€” `.github/actions/windows_plain_host/action.yml:106-110` [Claude Opus 4.7]

```pwsh
$forbiddenCmds = @('cygpath','pacman','mintty','git','git-bash','git-cmd')
```

The PATH layer above already rejects only entries matching
`\\Git\\(cmd|bin|usr)`, which catches Git-for-Windows specifically. The
binary layer is stricter: it fails for any `git.exe` on `PATH`, including
a standalone MinGit drop or a winget / Chocolatey install at
`C:\ProgramData\â€¦`. If a future runner image ships such a git, the
verify step fails despite the host being genuinely Cygwin-free. Gate the
`git` check on the same path regex used in the PATH layer, so the test
detects Git for Windows rather than git in general.

S7. **`git ls-tree` exit code is checked after the foreach has already iterated possibly-partial output** â€” `.github/actions/windows_plain_templates/action.yml:30-40` [Claude Opus 4.7]

```pwsh
foreach ($line in (git ls-tree -r HEAD $srcRoot)) {
  ...
  $target = (git cat-file blob $sha).Trim()
  if ($LASTEXITCODE -ne 0) { throw "git cat-file $sha failed: $LASTEXITCODE" }
  ...
}
if ($LASTEXITCODE -ne 0) { throw "git ls-tree failed: $LASTEXITCODE" }
```

If `git ls-tree` failed *with no output*, the foreach is a no-op and the
post-loop check catches it. But if it failed with *partial* output, the
loop iterates the partial output, each `git cat-file` overwrites
`$LASTEXITCODE`, and the post-loop check inspects the last `cat-file`'s
exit code instead of the original failure. Capture the output into a
variable and check `$LASTEXITCODE` before iterating:

```diff
-foreach ($line in (git ls-tree -r HEAD $srcRoot)) {
+$lsTreeOutput = git ls-tree -r HEAD $srcRoot
+if ($LASTEXITCODE -ne 0) { throw "git ls-tree failed: $LASTEXITCODE" }
+foreach ($line in $lsTreeOutput) {
   if ($line -notmatch '^120000\s') { continue }
   ...
 }
-if ($LASTEXITCODE -ne 0) { throw "git ls-tree failed: $LASTEXITCODE" }
```

S8. **`WindowsSubsystemPath` drops cygpath's stderr before falling back** â€” `pkg/ioutilx/ioutilx.go:60-65` [Claude Opus 4.7]

```go
out, err := exec.CommandContext(ctx, "cygpath", filepath.ToSlash(orig)).CombinedOutput()
if err == nil {
    return strings.TrimSpace(string(out)), nil
}
logrus.WithError(err).Debugf("cygpath unavailable for %q, attempting native conversion", orig)
```

`CombinedOutput` captures cygpath's stderr in `out`, but the debug log
shows only the Go error. When the fallback also errors (UNC path, no
cygpath), the user sees `cannot convert ... to a Cygwin-style path`
with no trace of *why* cygpath actually failed (missing? refused the
input? crashed?). Include `out` in the debug log:

```diff
-logrus.WithError(err).Debugf("cygpath unavailable for %q, attempting native conversion", orig)
+logrus.WithError(err).Debugf("cygpath unavailable for %q (output: %q), attempting native conversion", orig, strings.TrimSpace(string(out)))
```

This is a gap rather than a regression â€” the pre-PR code also dropped
the stderr buffer â€” but it is the right time to fix it given the new
fallback path swallows the failure silently.

S9. **Native cygpath fallback silently turns drive-relative paths into absolute ones** â€” `pkg/ioutilx/ioutilx.go:66-74` [Gemini 3.1 Pro]

```go
if vol := filepath.VolumeName(orig); len(vol) == 2 && vol[1] == ':' {
    tail := strings.TrimPrefix(filepath.ToSlash(orig[2:]), "/")
    converted := "/" + strings.ToLower(vol[:1]) + "/" + tail
```

The defensive `TrimPrefix` keeps `C:foo` from becoming `/cfoo`, but the
result `/c/foo` is now treated as if it were absolute. cygpath proper
resolves drive-relative paths against the current drive directory; the
fallback cannot reach that information. The current comment acknowledges
the tradeoff, and Lima always passes absolute paths into
`WindowsSubsystemPath`, so the code path is unreachable in practice.
Guarding the conversion with `filepath.IsAbs(orig)` (and returning the
existing "cannot convert" error otherwise) prevents a future caller
from quietly producing a wrong path:

```diff
-if vol := filepath.VolumeName(orig); len(vol) == 2 && vol[1] == ':' {
+if filepath.IsAbs(orig) {
+    if vol := filepath.VolumeName(orig); len(vol) == 2 && vol[1] == ':' {
        tail := strings.TrimPrefix(filepath.ToSlash(orig[2:]), "/")
        converted := "/" + strings.ToLower(vol[:1]) + "/" + tail
        logrus.Debugf("native cygpath fallback: %q -> %q", orig, converted)
        return converted, nil
+    }
 }
```

## Design Observations

### Strengths

- **`PathForSSH` as a single chokepoint** *(in-scope)* [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro] â€” Funnelling every Windows path translation through one helper that consults `IsSSHCygwin` is much easier to reason about than scattering the cygpath-vs-native decision across copy, identity, control path, and reverse-sshfs sites. Cache is keyed by resolved absolute path (not first-call-wins), which is correct.
- **Four-layer verification in `windows_plain_host`** *(in-scope)* [Claude Opus 4.7] â€” Checking filesystem, PATH, smoking-gun binaries, and registry uninstall keys is genuinely robust against runner-image drift. A future runner-image change that reinstalls Git for Windows under a new path fails the job loudly instead of silently regressing coverage.
- **Reading symlink targets from git's object store in `windows_plain_templates`** *(in-scope)* [Claude Opus 4.7] â€” Short-circuits the NTFS-symlink-vs-stub representation question. The `StartsWith("$srcRoot/")` guard against `..` underflow is good defence-in-depth.
- **Windows mux stripping applied uniformly** *(in-scope)* [Codex GPT 5.5] â€” `SSHOptsRemovingControlPath` is now invoked from limactl shell, hostagent, scp, and rsync paths so native Windows OpenSSH (no mux support) and Cygwin ssh (unreliable mux over sftp) get the same treatment.
- **`Invoke-Limactl` wrapper for native exit codes** *(in-scope)* [Claude Opus 4.7] â€” `$ErrorActionPreference = 'Stop'` doesn't propagate to native commands, and the wrapper makes the smoke test fail closed.

### Concerns

- **`PathForSSH`'s API is still partly Boolean** *(in-scope)* [Codex GPT 5.5] â€” `IsSSHCygwin` answers "Cygwin or native," but the correct conversion depends on the exact installation. Carrying the resolved sibling `cygpath.exe` through `PathForSSH` (see I1) makes multiple Windows SSH toolchains predictable.

## Testing Assessment

The added coverage matches the scope of the change well: two new CI jobs
exercise the create/start/shell/copy/stop/delete cycle on a verified-vanilla
host for both QEMU and WSL2 drivers, and `TestParseCopyPathsWindowsDriveLetter`
pins the `C:\foo` vs `C:foo` semantic split. Open gaps:

1. **In-process gzip is not covered on Windows.** The very platform that
   motivated the change still skips the test (S2).
2. **`PathForSSH`'s native (non-Cygwin) branch has no unit test.** A
   `t.TempDir()`-based test stubbing `sshExe.Exe` to a path without a
   sibling `cygpath.exe` would catch a future regression cheaply (S4).
3. **UNC path handling is documented but unverified** (S3).
4. **`Win32-OpenSSH` 4-part version banners are not in the test table** (S5).
5. **No CI assertion verifies `qemu-system-x86_64` resolves after
   `windows_plain_host`.** A `pwsh -c "Get-Command qemu-system-x86_64"`
   between the plain-host step and the smoke test would catch I2.
6. **Plain-Windows smoke test never sets up a mount.** The default
   template's `mounts: []` means the QEMU plain job never exercises
   native sftp-server â€” the very feature the docs explicitly mention
   needing OpenSSH.Server for. Adding `--mount` (or a template with a
   `mounts:` entry) to the QEMU smoke test would close this gap.
7. **No test covers `$SSH` pointing at a Cygwin `ssh.exe` whose directory
   is not first on `PATH`.** That scenario is what triggers I1.

### [Gemini 3.1 Pro]

### Executive Summary

This PR introduces robust support for "plain Windows" (running Lima natively on Windows without requiring MSYS2, Cygwin, or Git for Windows). It achieves this by adding a native `PathForSSH` translator, correctly stripping SSH multiplexing options that native OpenSSH does not support, replacing the external `gzip` invocation with an in-process native Go implementation, and reorganizing GitHub Actions to test these scenarios. The implementation is thoughtfully structured, particularly in how it respects existing MSYS2 toolchains when they are detected.

One important regression was introduced: the in-process `gzip` decompression loop ignores context cancellation, which can cause the process to block indefinitely if canceled during a large file download/extraction. 

### Findings

#### Critical Issues
None.

#### Important Issues

I1. **Blocking call inside cancellation-aware wrapper** â€” `pkg/downloader/downloader.go:564-572`

```go
	if decompressCmd == "gzip" {
		gz, err := gzip.NewReader(bar.NewProxyReader(in))
		if err != nil {
			return err
		}
		defer gz.Close()
		_, err = io.Copy(out, gz)
		return err
	}
```

The PR optimizes gzip decompression by replacing the external `exec.CommandContext` with in-process `gzip.NewReader` and `io.Copy`. However, unlike the external command which is killed immediately by Go's `os/exec` when `ctx` is canceled, `io.Copy` on local files does not respect `context.Context`. If the user cancels the operation (e.g., via Ctrl+C) during a long decompression of a large VM image, the process will ignore the cancellation and block until `io.Copy` completes. This introduces a regression in cancellation responsiveness for the `gzip` path while `bzip2`/`xz`/`zstd` remain context-aware.

Fix: Start a goroutine that waits on `ctx.Done()` and explicitly closes the underlying file descriptors to break the blocking read/write operations.

```diff
 	if decompressCmd == "gzip" {
 		gz, err := gzip.NewReader(bar.NewProxyReader(in))
 		if err != nil {
 			return err
 		}
 		defer gz.Close()
+
+		ctx, cancel := context.WithCancel(ctx)
+		defer cancel()
+		go func() {
+			<-ctx.Done()
+			in.Close()
+			out.Close()
+		}()
+
 		_, err = io.Copy(out, gz)
 		return err
 	}
```
*(important, regression)*

MSYS2 `cygpath` dynamically resolves drive-relative paths to their absolute equivalent (e.g., `/c/Users/jan/foo`); the fallback cannot reliably do this without querying the working directory. While this code path is unreachable in practice because Lima only passes absolute paths to `WindowsSubsystemPath`, it fails to meet its documented intent.
