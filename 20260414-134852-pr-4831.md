## Executive Summary

PR #4831 fixes a real race condition on Alpine ramdisk VMs where `growpart`
triggers a kernel partition-table re-read that races with `e2fsck`, causing
udevd's libblkid probe to fail and delete the `/dev/disk/by-label/data-volume`
symlink. Under unlucky timing, the symlink never exists at all, and the
`else` branch misidentifies an already-populated data disk as unused and
reformats it — silent data loss.

The fix has three parts: (1) resolve the data volume via `blkid` output
directly, eliminating the udev-symlink dependency; (2) add `udevadm settle`
after `growpart` to prevent the concurrent-probe race; and (3) replace the
`/proc/mounts`-only "disk in use" check with an `lsblk` partition check so
a partitioned-but-unmounted data disk is no longer reformatted. Any one of
these fixes alone would close the reported bug; together they make the
script robust across the failure modes the issue describes.

All three reviewers approve the change. No critical or important issues were
identified. The suggestions below are defensive improvements against edge
cases (PARTLABEL false match, multi-device corner cases, missing udev tooling)
and one pre-existing gap around raw-formatted disks. Test coverage for the
boot path remains absent, which is noted but is a general Lima constraint
rather than a regression. Structure: 0 critical, 0 important, 7 suggestions.

S1. **`udevadm settle` is unguarded under `set -e`** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:71,115` [Gemini 3.1 Pro]

```bash
# growpart triggers a partition table re-read; settle udev before
# touching the device to avoid racing with the re-probe.
udevadm settle
```

The script runs with `set -eux -o pipefail`. `udevadm settle` can exit
non-zero in two ways: the `udevadm` binary may be absent (hypothetical on
lima-alpine, which ships eudev, but possible on a user-supplied minimal
image using `mdev`), or the settle may time out (default 120 s) if udevd
has a stuck event. Either failure crashes the boot script with the same
user-visible symptom this PR set out to fix — `/mnt/data` does not mount,
tmpfs fills up. The `udevadm settle` call is defense-in-depth against the
race; the `blkid` resolution already eliminates the core dependency on
udev state, so making the settle tolerant of failure does not weaken the
fix.

Fix: guard or silence the call on both lines.

```diff
-        udevadm settle
+        udevadm settle || true
```

S2. **`blkid` regex can match `PARTLABEL`** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:61` [Gemini 3.1 Pro]

```bash
DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')
```

The regex `.*LABEL="data-volume".*` also matches lines carrying
`PARTLABEL="data-volume"`, because the leading `.*` greedily consumes the
`PART` prefix. BusyBox `blkid` (the implementation on lima-alpine) does
not emit `PARTLABEL`, so this is currently latent; util-linux `blkid` does
emit it for GPT partitions, so a user who installs `util-linux-blkid` or
attaches a disk with a GPT partition labeled `data-volume` but no
filesystem label would see the script misidentify the device and fall
over at `e2fsck`. Fix is a one-character anchor.

```diff
-DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')
+DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*[[:space:]]LABEL="data-volume".*/\1/p')
```

S3. **Multi-line `blkid` output breaks downstream commands** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:61-64` [Claude Opus 4.6, Gemini 3.1 Pro]

```bash
DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')

if [ -n "${DATA_VOLUME}" ]; then
    DATA_DISK="${DATA_VOLUME%[0-9]}"
```

If `blkid` reports more than one device with `LABEL="data-volume"` (user
error, disk cloning, or combined with S2 if `PARTLABEL` collisions creep
in), `DATA_VOLUME` becomes a newline-separated list. The `[ -n ]` test
still succeeds, `${DATA_VOLUME%[0-9]}` only strips the final digit, and
`growpart`/`e2fsck`/`mount` receive a multi-line argument. The script
then crashes under `set -e`. This is not a regression — the old
`blkid | grep | awk` pipeline had the same behavior — but the defensive
fix is trivial: quit sed after the first match.

Fix:

```diff
-DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')
+DATA_VOLUME=$(blkid | sed -n '/[[:space:]]LABEL="data-volume"/{s/^\([^:]*\):.*/\1/p;q;}')
```

This also incorporates the anchor from S2 so both issues collapse into a
single edit.

S4. **`DATA_DISK="${DATA_VOLUME%[0-9]}"` strips only one digit** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:64` [Claude Opus 4.6]

```bash
DATA_DISK="${DATA_VOLUME%[0-9]}"
```

This strips a single trailing digit from the device path to derive the
parent disk. It is correct for `/dev/vda1` → `/dev/vda`, which covers every
Lima VM today (virtio-blk on VZ/QEMU always names disks `vd[a-z]`). It is
wrong for NVMe naming (`/dev/nvme0n1p1` → `/dev/nvme0n1p`, not
`/dev/nvme0n1`) and for double-digit partitions (`/dev/sda10` →
`/dev/sda1`). Lima does not use these devices for the Alpine data volume
path today, so this is a robustness issue rather than a live bug.

Worth noting from `git blame`: the old code at this location used
`sub(/\d$/, "", s[1])` inside awk, and `\d` is not a POSIX character
class, so BusyBox awk treated it as a literal `d`. The digit strip in the
old code never actually ran; the new code is the first time `DATA_DISK`
is correctly derived at all, which is why `growpart` now functions on the
happy path.

Fix (optional, only if robustness for other device types is wanted):

```diff
-DATA_DISK="${DATA_VOLUME%[0-9]}"
+DATA_DISK="/dev/$(lsblk --noheadings --output pkname "${DATA_VOLUME}")"
```

S5. **Unpartitioned disks with raw filesystems are still reformatted** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:103-108` [Gemini 3.1 Pro]

```bash
if lsblk --list --noheadings --output type /dev/"${DISK}" | grep --quiet "part"; then
    continue
fi
if awk '/^\/dev\// {gsub("/dev/", ""); print $1}' /proc/mounts | grep --quiet "^${DISK}$"; then
    continue
fi
```

The new `else`-branch check correctly protects partitioned-but-unmounted
data disks from being reformatted — that is the data-loss path the PR
set out to close. A related gap remains: if a user attaches a disk that
was formatted directly as a raw filesystem (no partition table, e.g.
`mkfs.ext4 /dev/vdb`), the `lsblk` partition check finds nothing, and if
the disk is not mounted the `/proc/mounts` check also passes, so the
script continues and reformats it. This is pre-existing — the old code
had the same gap — and it is not the user scenario the issue describes,
but it is the next nearest data-loss path in the same branch.

Fix: also reject disks that already carry a filesystem signature.

```diff
         # A disk is in use if it has any partitions or is mounted directly.
         # Check lsblk for partitions, not just /proc/mounts; an unmounted
         # but partitioned disk (e.g. the data volume after a failed boot)
         # must not be reformatted.
+        if lsblk --list --noheadings --output fstype /dev/"${DISK}" | grep -q "[^[:space:]]"; then
+            continue
+        fi
         if lsblk --list --noheadings --output type /dev/"${DISK}" | grep --quiet "part"; then
             continue
         fi
```

S6. **`gsub("/dev/", "")` is imprecise in the mount-path awk** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:106` [Gemini 3.1 Pro]

```bash
if awk '/^\/dev\// {gsub("/dev/", ""); print $1}' /proc/mounts | grep --quiet "^${DISK}$"; then
```

`gsub` operates on the whole record, so a mount point containing `/dev/`
(e.g. `/mnt/dev/foo`) gets its substring rewritten before `print $1`. The
output of `print $1` is unaffected because only the first field is
printed, so this is purely cosmetic today. Pre-existing: the old code had
the same awk expression. Worth tidying while touching the area.

Fix: scope the substitution to `$1`.

```diff
-        if awk '/^\/dev\// {gsub("/dev/", ""); print $1}' /proc/mounts | grep --quiet "^${DISK}$"; then
+        if awk '/^\/dev\// {sub("^/dev/", "", $1); print $1}' /proc/mounts | grep --quiet "^${DISK}$"; then
```

## Review Process Notes

### Skill improvements

- **Shell safety: flag unguarded calls under `set -e` whose failure reintroduces the bug a PR is fixing.** When a shell PR fixes a bug and adds defense-in-depth instrumentation (e.g., `udevadm settle`, `sync`, `flush`), check whether the added command can fail and whether `set -e` is active. If the command's failure produces the same user-visible symptom the PR was meant to eliminate, flag the unguarded call. Rationale: a defense-in-depth command whose failure reintroduces the class of bug it was added to defend against is strictly worse than a well-chosen primary fix alone.

- **Shell safety: check `sed`/`grep`/`awk` patterns with leading `.*` for substring bleed into structurally distinct fields.** Patterns of the form `.*KEY="value"` can inadvertently match lines where `KEY` is a suffix of another key name (e.g., `LABEL` matches lines carrying `PARTLABEL`). When reviewing shell code that parses field-oriented tool output via unanchored regex, verify that the field name cannot appear as a suffix of another known field in the same output, and suggest a leading whitespace or start-of-field anchor when it can.

### Repo context updates

- **[repo] `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/` scripts run under BusyBox ash with `ASH_BASH_COMPAT`, not GNU bash.** The shebang is `#!/bin/bash` but on lima-alpine `/bin/bash` is BusyBox ash with ASH_BASH_COMPAT enabled. When reviewing scripts in this directory, do not assume GNU-bash-only features (process substitution, `[[ ]]` regex, `mapfile`, `readarray`, associative arrays) will work. Additionally, the `blkid`, `awk`, and other utilities on lima-alpine are BusyBox implementations and have reduced flag coverage compared to util-linux/GNU equivalents (e.g., BusyBox `blkid` lacks `--label`, BusyBox `awk` does not support `\d`). Flag code that depends on GNU-only syntax, and when in doubt, verify against BusyBox documentation rather than GNU man pages.

S1. **Fragile disk-from-partition derivation** — `04-persistent-data-volume.sh:64` (suggestion, gap)

```bash
DATA_DISK="${DATA_VOLUME%[0-9]}"
```

This strips a single trailing digit to derive the disk from the partition device (e.g., `/dev/vda1` → `/dev/vda`). It would produce wrong results for NVMe naming (`/dev/nvme0n1p1` → `/dev/nvme0n1p`) or double-digit partition numbers (`/dev/sda10` → `/dev/sda1`). Per `git blame`, the old code at this location used `sub(/\d$/, "", s[1])` in awk, which was *worse* — `\d` is not a valid POSIX regex class, so BusyBox awk treated it as literal `d`, meaning the digit strip never actually worked. The new code is strictly better, and Lima currently creates only partition 1 on virtio-blk, so this doesn't affect real deployments.

Fix: if robustness is desired for future device types, use `lsblk` to query the parent device directly:

```diff
-DATA_DISK="${DATA_VOLUME%[0-9]}"
+DATA_DISK="/dev/$(lsblk --noheadings --output pkname "${DATA_VOLUME}")"
```

This handles any device naming scheme. However, it adds a `lsblk` dependency to the if-branch, which may not be worth the cost for a theoretical scenario.

S2. **Consider limiting blkid output to first match** — `04-persistent-data-volume.sh:61` (suggestion, enhancement)

```bash
DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')
```

If multiple partitions somehow carry the `LABEL="data-volume"` label (filesystem corruption, duplicate disks), `DATA_VOLUME` would contain multiple lines. Downstream commands (`growpart`, `e2fsck`, `mount`) would receive multi-line arguments and fail. This is not a regression — the old `blkid | grep` had the same behavior — and duplicate labels are pathological. But adding a defensive limit is trivial:

```diff
-DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')
+DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p' | head -1)
```

Note: with `set -o pipefail`, this is safe — `head -1` closing the pipe early causes `sed` to receive SIGPIPE, but `head` itself exits 0, and `pipefail` takes the rightmost non-zero status or 0.

Actually, to avoid SIGPIPE complications entirely, the sed itself can be made to quit after the first match:

```diff
-DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')
+DATA_VOLUME=$(blkid | sed -n '/LABEL="data-volume"/{s/^\([^:]*\):.*/\1/p;q;}')
```

### Executive Summary

This PR effectively eliminates the race condition between `growpart`, `e2fsck`, and `udevd` by directly resolving the data volume device via `blkid` instead of relying on the transient `udev` symlink. However, the direct parsing introduces new edge cases: the `blkid` regex can accidentally match `PARTLABEL`, multiple matching devices will crash the script, and the newly added `udevadm settle` commands will abort the boot process entirely on minimal Alpine images missing the `udevadm` binary. Additionally, a pre-existing data-loss gap remains where unpartitioned but formatted disks are blindly overwritten.

### Findings

#### Critical Issues
None.

#### Important Issues

**I1. `udevadm settle` crashes script if udev is missing or stuck** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:71`, `115`

```bash
		udevadm settle
```

The script runs with `set -eux -o pipefail` and targets Alpine Linux. While Lima's official `alpine-lima` images include `eudev`, users can provide minimal Alpine ISOs or older images that use `mdev` instead of `eudev`. On such systems, `udevadm` is not installed, causing the command to exit with `127` and fatally crashing the boot script. Even on systems with `eudev`, if the daemon has a stuck event, `settle` will block for 120 seconds and exit non-zero, triggering the same crash.

Fix: Make the settle command conditional and ignore failures to ensure the script safely proceeds.

```diff
 		# growpart triggers a partition table re-read; settle udev before
 		# touching the device to avoid racing with the re-probe.
-		udevadm settle
+		command -v udevadm >/dev/null 2>&1 && udevadm settle || true
 		# Only resize when filesystem is in a healthy state
```
*(Apply to both line 71 and line 115)*

**I2. `blkid` regex incorrectly matches `PARTLABEL`** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:61` (important, gap)

```bash
DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')
```

The regex `.*LABEL="data-volume".*` inadvertently matches `PARTLABEL="data-volume"` because the leading `.*` aggressively consumes the `PART` prefix. If a partition happens to have this partition label but lacks an ext4 filesystem label, the script will misidentify it, causing `e2fsck` and `mount` to fail. Standard `blkid` (present in eudev environments) outputs `PARTLABEL` for GPT partitions.

Fix: Require whitespace before `LABEL=` to explicitly exclude `PARTLABEL=`.

```diff
 # BusyBox blkid doesn't support --label, so we parse the output instead.
-DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')
+DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*[[:space:]]LABEL="data-volume".*/\1/p')
 
 if [ -n "${DATA_VOLUME}" ]; then
```

**I3. Multiple `blkid` matches cause script crash** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:61-64` (important, gap)

```bash
DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')

if [ -n "${DATA_VOLUME}" ]; then
	DATA_DISK="${DATA_VOLUME%[0-9]}"
```

If `blkid` returns multiple matches (e.g., if a user attached a backup disk, or if the regex flaw in I2 matched multiple properties), `DATA_VOLUME` will contain newlines. The `[ -n ... ]` check evaluates to true, but `${DATA_VOLUME%[0-9]}` only strips a digit from the final line. The script will then pass the multi-line string to `growpart`, `e2fsck`, and `mount`, all of which will fail and trigger a fatal crash under `set -e`.

Fix: Use `sed`'s `q` command to cleanly quit after the first match (incorporating the fix from I2).

```diff
 # BusyBox blkid doesn't support --label, so we parse the output instead.
-DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')
+DATA_VOLUME=$(blkid | sed -n '/[[:space:]]LABEL="data-volume"/ { s/^\([^:]*\):.*/\1/p; q; }')
 
 if [ -n "${DATA_VOLUME}" ]; then
```

**I4. Unpartitioned disks with existing filesystems are formatted** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:103-105` (important, gap)

```bash
		if lsblk --list --noheadings --output type /dev/"${DISK}" | grep --quiet "part"; then
			continue
		fi
```

The author correctly fixed the `else` branch to identify "partitioned-but-unmounted" disks. However, if a user provides an unpartitioned raw disk that already contains a filesystem (e.g., initialized directly via `mkfs.ext4 /dev/vdb`) and it is not mounted, `lsblk` will output `disk` with no `part` entries. The `grep "part"` check fails, and the script silently reformats the disk via `sfdisk` and `mkfs.ext4`, resulting in data loss. 

Fix: Use `lsblk` to explicitly check for any existing filesystem signatures (`fstype`) before formatting.

```diff
 		# A disk is in use if it has any partitions or is mounted directly.
 		# Check lsblk for partitions, not just /proc/mounts; an unmounted
 		# but partitioned disk (e.g. the data volume after a failed boot)
 		# must not be reformatted.
+		if lsblk --list --noheadings --output fstype /dev/"${DISK}" | grep -q "[^[:space:]]"; then
+			continue
+		fi
 		if lsblk --list --noheadings --output type /dev/"${DISK}" | grep --quiet "part"; then
 			continue
 		fi
```

**S1. Imprecise awk substitution scope** — `pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh:106` (suggestion, gap)

```bash
		if awk '/^\/dev\// {gsub("/dev/", ""); print $1}' /proc/mounts | grep --quiet "^${DISK}$"; then
```

`gsub("/dev/", "")` operates on the entire `/proc/mounts` line, indiscriminately replacing `/dev/` inside the mount path as well (e.g., `/mnt/dev/foo` becomes `/mntfoo`). While practically harmless because `print $1` extracts only the first field, it is conceptually imprecise. 

Fix: Apply `sub` specifically to `$1`.

```diff
-		if awk '/^\/dev\// {gsub("/dev/", ""); print $1}' /proc/mounts | grep --quiet "^${DISK}$"; then
+		if awk '/^\/dev\// {sub("^/dev/", "", $1); print $1}' /proc/mounts | grep --quiet "^${DISK}$"; then
```
