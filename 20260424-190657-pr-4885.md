## Executive Summary

This PR makes `limactl` usable on plain Windows 10/11 hosts by teaching `sshutil` to dispatch between native OpenSSH and Cygwin-based OpenSSH, stripping SSH mux options on Windows, decompressing gzip in-process, and adding two "plain Windows" CI jobs (one per driver). The refactor is scoped well and the research-PR framing is honest about what it does and does not cover.

Two real issues surfaced. The new rsync/scp mux-stripping is applied in `rsync.Command` and `scp.Command` but was omitted from the `checkRsyncOnGuest` probe, so an installed-but-gated rsync can be rejected on native Windows OpenSSH (I1). The drive-letter check in `parseCopyPaths` uses `filepath.VolumeName` instead of `filepath.IsAbs`, which silently breaks `limactl copy` for users whose instance is named with a single letter like `C` (I2) — a regression from the pre-PR code that used `filepath.IsAbs`.

The remaining items are documentation, logging, and defensive-coding suggestions. The author's `DO NOT MERGE` framing is appropriate for a research PR; the issues above are small enough to address in a follow-up before merging for real.

## Important Issues

I1. **`checkRsyncOnGuest` does not strip SSH mux options on Windows** — `pkg/copytool/rsync.go:61-84` [Codex GPT 5.5, Claude Opus 4.7]

```go
func checkRsyncOnGuest(ctx context.Context, inst *limatype.Instance) bool {
    sshExe, err := sshutil.NewSSHExe()
    ...
    sshOpts, err := sshutil.SSHOpts(ctx, sshExe, inst.Dir, *inst.Config.User.Name, false, false, false, false)
    ...
    sshArgs := append([]string{}, sshExe.Args...)
    sshArgs = append(sshArgs, sshutil.SSHArgsFromOpts(sshOpts)...)
    checkCmd := exec.CommandContext(ctx, sshExe.Exe, sshArgs...)
```

`rsyncTool.Command` at `pkg/copytool/rsync.go:131-138` and `scpTool.Command` at `pkg/copytool/scp.go:125-134` both call `sshutil.SSHOptsRemovingControlPath` on Windows. The probe at `pkg/copytool/rsync.go:67` builds the same `sshOpts` but skips that step, so the probe inherits `ControlMaster=auto`, `ControlPath=…`, and `ControlPersist=yes` from `SSHOpts`.

Native Windows OpenSSH does not support Unix-domain mux sockets (PR description). Cygwin-based ssh has documented reliability issues over a mux socket (`pkg/copytool/scp.go:126-131`). The probe therefore hits the same failure mode the `Command` method works around. When a user installs rsync on a plain-Windows host (e.g., via `winget`), the probe fails before running `command -v rsync` on the guest, and `BackendAuto` falls back to scp while `BackendRsync` reports "rsync not available on guest(s)". `go test ./pkg/copytool/...` does not cover this path because CI does not install rsync on the Windows runners.

Fix: apply the same mux-stripping in `checkRsyncOnGuest`:

```diff
 sshOpts, err := sshutil.SSHOpts(ctx, sshExe, inst.Dir, *inst.Config.User.Name, false, false, false, false)
 if err != nil {
     logrus.Debugf("failed to get SSH options for rsync check: %v", err)
     return false
 }
+if runtime.GOOS == "windows" {
+    sshOpts = sshutil.SSHOptsRemovingControlPath(sshOpts)
+}

 sshArgs := append([]string{}, sshExe.Args...)
 sshArgs = append(sshArgs, sshutil.SSHArgsFromOpts(sshOpts)...)
```

I2. **`parseCopyPaths` drive-letter check shadows single-letter instance names** — `pkg/copytool/copytool.go:104` [Gemini 3.1 Pro]

```go
isLocalAbs := runtime.GOOS == "windows" && filepath.VolumeName(path) != ""
if isLocalAbs {
    sshExe, err := sshutil.NewSSHExe()
    ...
    path, err = sshutil.PathForSSH(ctx, sshExe, path)
    ...
    continue
}
```

On Windows `filepath.VolumeName("C:foo.txt")` returns `"C:"` (drive-relative paths still carry a volume). The pre-PR code at the merge-base used `filepath.IsAbs(path)`, which correctly returns `false` for `"C:foo.txt"` and routes the string into the colon-split where it is recognised as a remote path on instance `C`. After this change, `limactl copy C:foo.txt .` on an instance named `C` is mis-parsed as a local Windows path and runs through `PathForSSH` instead of being routed to the guest.

Single-letter instance names are rare but valid, and the failure is silent (cygpath will happily translate the bogus path). The fix restores pre-PR behaviour for the instance-name case while keeping the intended bypass for `C:\foo` and `C:/foo`, which `filepath.IsAbs` accepts on Windows. UNC paths also pass `filepath.IsAbs` and therefore remain routed through `PathForSSH`, matching the pre-PR behaviour.

Fix:

```diff
-isLocalAbs := runtime.GOOS == "windows" && filepath.VolumeName(path) != ""
+isLocalAbs := runtime.GOOS == "windows" && filepath.IsAbs(path)
```

A unit test on `parseCopyPaths` with `C:\foo`, `C:/foo`, `C:foo`, and `instance:path` would lock the intended branching in place.

## Suggestions

S1. **`IsSSHCygwin` uses a process-global `sync.Once` tied to the first caller's `sshExe`** — `pkg/sshutil/sshutil.go:72-110` [Claude Opus 4.7]

```go
var cygwinDetect struct {
    sync.Once
    isCygwin bool
}

func IsSSHCygwin(sshExe SSHExe) bool {
    if runtime.GOOS != "windows" || sshExe.Exe == "" {
        return false
    }
    cygwinDetect.Do(func() {
        ...
    })
    return cygwinDetect.isCygwin
}
```

The docstring documents this: "Subsequent calls return the same answer regardless of the sshExe passed". The concern is blast radius, not correctness in the current flow. A unit test that wants to exercise both the Cygwin and native branches of `PathForSSH` in one binary cannot do so, and any future caller that resets `SSH` mid-process would silently keep the old detection result.

The cache in `openSSHInfos` at `pkg/sshutil/sshutil.go:479-481` already uses `map[sshExecutable]*openSSHInfo` guarded by an `sync.RWMutex`. Mirror that pattern for cygwin detection: key by the resolved absolute ssh path and drop `sync.Once`. The change is ~10 lines and removes the "first call wins" subtlety.

S2. **Preserve the `mount-home` skip rationale after removal** — `hack/test-templates.sh` [Claude Opus 4.7]

```diff
 case "$NAME" in
 "default")
-    # CI failure:
-    # "[hostagent] failed to confirm whether /c/Users/runneradmin [remote] is successfully mounted"
-    [ "${OS_HOST}" = "Msys" ] && CHECKS["mount-home"]=
     [ "${OS_HOST}" = "Darwin" ] && CHECKS["ssh-over-vsock"]="1"
```

S3. **Resolve symlinks before looking for `cygpath.exe`** — `pkg/sshutil/sshutil.go:91-108` [Claude Opus 4.7]

```go
cygwinDetect.Do(func() {
    path := sshExe.Exe
    if !filepath.IsAbs(path) {
        resolved, err := exec.LookPath(path)
        ...
        path = resolved
    }
    cygpathPath := filepath.Join(filepath.Dir(path), "cygpath.exe")
```

`exec.LookPath` returns the path as it appears in `PATH`. If that entry is a symlink or a shim (chocolatey, scoop, user shortcut), `filepath.Dir` points at the shim's directory, not the real binary's neighbourhood, and the Cygwin detection returns the wrong answer. A `filepath.EvalSymlinks(path)` before `filepath.Dir` aligns the check with the actual binary location.

S6. **Assert the resolved symlink target stays inside `$srcRoot` before `Copy-Item`** — `.github/workflows/test.yml:536-553` [Claude Opus 4.7]

```powershell
$resolved = if ($slash -ge 0) { "$($resolved.Substring(0, $slash))/$target" } else { $target }
$segs = New-Object System.Collections.ArrayList
foreach ($s in ($resolved -split '/')) {
  if ($s -eq '' -or $s -eq '.') { continue }
  if ($s -eq '..') {
    if ($segs.Count -gt 0) { $segs.RemoveAt($segs.Count - 1) | Out-Null }

  }

```

The loop silently swallows `..` underflow. A symlink chain with enough `..` segments produces a path relative to the repo root; the `Copy-Item` on line 552 then copies whichever file is there into `_output\share\lima\templates\…`. Lima's templates do not use such symlinks today, so this is defensive only, but a single post-loop check (`if ($resolved -notlike "$srcRoot/*") { throw ... }`) closes the loop on future template edits.

S7. **Toolchain swap between `limactl create` and `limactl start` changes reverse-sshfs `LocalPath`** — `pkg/hostagent/mount.go:55-70` [Claude Opus 4.7]

```go
if runtime.GOOS == "windows" {
    sshExe, err := sshutil.NewSSHExe()
    ...
    resolvedLocation, err = sshutil.PathForSSH(ctx, sshExe, m.Location)
    ...
}
```

The PR description already flags this: `mount.go` resolves `LocalPath` with `PathForSSH` (toolchain-aware) on every start, while `pkg/limayaml/defaults.go` computes the default `MountPoint` once at create time via `WindowsSubsystemPath` (Cygwin-style, no fallback variation). A user who creates an instance with Git for Windows on PATH and then removes it would see `LocalPath` change between restarts. Either document "do not swap Windows SSH toolchain between create and start" in the WSL2/QEMU docs, or persist the resolved `LocalPath` at create time so the restart path does not recompute it.

S8. **Harden the native cygpath fallback against relative drive-letter inputs** — `pkg/ioutilx/ioutilx.go:66-70` [Gemini 3.1 Pro]

```go
if vol := filepath.VolumeName(orig); len(vol) == 2 && vol[1] == ':' {
    converted := "/" + strings.ToLower(vol[:1]) + filepath.ToSlash(orig[2:])
    logrus.Debugf("native cygpath fallback: %q -> %q", orig, converted)
    return converted, nil
}
```

`orig[2:]` is expected to begin with a separator. For `orig = "C:foo"`, the result is `"/cfoo"` — not a valid Cygwin-style path. All current callers pass absolute paths, but the API contract is fragile for future callers. Insert the slash explicitly:

```diff
-converted := "/" + strings.ToLower(vol[:1]) + filepath.ToSlash(orig[2:])
+converted := "/" + strings.ToLower(vol[:1]) + "/" + strings.TrimPrefix(filepath.ToSlash(orig[2:]), "/")
```

S9. **Docs overstate bundled OpenSSH component availability** — `website/content/en/docs/config/vmtype/wsl2.md:48-50` [Codex GPT 5.5]

```md
Lima uses native Windows OpenSSH (`C:\Windows\System32\OpenSSH\`, included in
Windows 10 build 1803 and later) for `ssh`, `scp`, `ssh-keygen`, and `sftp-server`.
No additional Cygwin-style toolchain is required.
```

OpenSSH Client (ssh, scp, ssh-keygen) ships by default on Windows 10 build 1803+, but OpenSSH Server — which provides `sftp-server.exe` — is an optional Feature on Demand (https://learn.microsoft.com/en-us/windows-server/administration/OpenSSH/openssh-overview). The new CI at `.github/workflows/test.yml:357` and `:584` treats `sftp-server` as required, so the reverse-sshfs path assumes the server component is installed. Users without it will see a mismatch between the docs and runtime behaviour. Soften to: Lima works with native Windows OpenSSH when the required components are installed; QEMU with reverse-sshfs additionally requires `sftp-server.exe` from the OpenSSH Server optional feature.

S10. **`parseCopyPaths` silently accepts UNC paths through `PathForSSH`** — `pkg/copytool/copytool.go:104-118` [Claude Opus 4.7]

```go
isLocalAbs := runtime.GOOS == "windows" && filepath.VolumeName(path) != ""
if isLocalAbs {
    ...
    path, err = sshutil.PathForSSH(ctx, sshExe, path)
    ...
}
```

`filepath.VolumeName(`\\server\share\foo`)` returns `\\server\share` on Windows, so UNC paths hit this branch. On native OpenSSH they pass through `filepath.ToSlash` as `//server/share/foo`, which scp may or may not accept. On the Cygwin branch `ioutilx.WindowsSubsystemPath` falls back to the drive-letter handler (`len(vol) == 2`) and returns the new explicit error. UNC support is neither exercised in tests nor mentioned in the comment. Either document the intent (`// UNC paths flow through PathForSSH; native ssh accepts //server/share/foo`) or reject UNC early with a clean error. The fix for I2 (switching to `filepath.IsAbs`) preserves this branching because UNC paths are absolute, so UNC behaviour is unchanged either way — but the contract remains worth documenting.

## Design Observations

**Concerns**

- *Toolchain detection is implicit* (in-scope) [Claude Opus 4.7]. `IsSSHCygwin` equates "is `cygpath.exe` next to `ssh.exe`?" with "is this ssh a Cygwin build?". The heuristic works for the two real toolchains (Git for Windows, MSYS2) and is documented, but a user who drops `cygpath.exe` into `C:\Windows\System32\OpenSSH\` for unrelated reasons would flip native OpenSSH to be mis-detected. `detectOpenSSHInfo` already runs `ssh -V`; the banner (`OpenSSH_for_Windows_…` vs. `OpenSSH_…p… Cygwin|MSYS`) is a more direct signal.
- *Mux-stripping duplicated across call sites* (future) [Claude Opus 4.7]. `scp.Command`, `rsync.Command`, `hostagent/mount.go`, and the probe in `checkRsyncOnGuest` each decide whether to strip `ControlMaster`/`ControlPath`/`ControlPersist` on Windows using two slightly different helpers (`SSHOptsRemovingControlPath`, `DisableControlMasterOptsFromSSHArgs`). A single choke point — `SSHOpts` taking a `withControlMaster bool`, or a dedicated "give me Windows-safe SSH options" constructor — would prevent the inconsistency that surfaced as I1 and make future additions uniform.

**Strengths**

- The `PathForSSH` / `IsSSHCygwin` pair cleanly separates "which path form should ssh consume?" from "is the toolchain Cygwin-based?", and either function is reusable independently [Codex GPT 5.5, Claude Opus 4.7].
- Moving gzip decompression in-process via `bar.NewProxyReader(in)` removes a high-frequency external dependency without changing the rest of the decompressor dispatch or the progress-bar UX [Codex GPT 5.5, Gemini 3.1 Pro].
- The new `TestParseOpenSSHVersion` case for `OpenSSH_for_Windows_9.5p2` guards against the exact silent-0.0.0 regression the regex extension was meant to fix [Claude Opus 4.7].
- The two plain-Windows CI jobs print `PATH` before and after scrubbing, inventory required/forbidden/optional binaries, assert `ssh` resolves to `C:\Windows\System32\OpenSSH\ssh.exe`, and dump `ha.stderr.log`/`ha.stdout.log`/`serial.log` on failure — so CI failures are self-diagnosable without a local Windows repro [Claude Opus 4.7].

## Review Process Notes

**Skill improvements**

- When reviewing a change that adds a helper function applied at multiple call sites (e.g., mux-stripping, path translation, error wrapping), enumerate every call site in the touched packages that uses the same precondition and check whether the helper is applied consistently. Helpers introduced to fix a narrow symptom are easy to leave off one path when the symptom did not manifest there during local testing. Grep for the helper's name and for the `runtime.GOOS` branch it guards, and cross-check the list against all functions that build equivalent arguments — because an inconsistency between the "do X" call site and the "check whether X works" call site is a silent regression that local runs rarely reveal.

- When a diff replaces a `filepath.IsAbs` check with a different absoluteness test (e.g., `filepath.VolumeName != ""`), verify that the replacement agrees with the original on drive-relative paths, UNC paths, and bare drive letters. Go's `filepath.VolumeName` accepts drive-relative inputs like `C:foo` that `filepath.IsAbs` rejects, so the switch can silently change how paths split downstream.

**Repo context updates**

- [repo] When reviewing Windows-related code paths in Lima, note that the Windows SSH landscape has two valid toolchains (native OpenSSH, Cygwin-based OpenSSH via Git for Windows or MSYS2) with different path conventions and mux-socket semantics. Code paths that shell out to `ssh`, `scp`, `ssh-keygen`, or `sftp-server` must handle both and any probe that builds SSH options must strip `ControlMaster`/`ControlPath`/`ControlPersist` on Windows, because native OpenSSH does not support Unix-domain mux sockets and Cygwin-based ssh has known reliability issues over one. **I2. In-process gzip decompression swallows `gz.Close()` errors** — `pkg/downloader/downloader.go:568-576`

```go
if decompressCmd == "gzip" {
    gz, err := gzip.NewReader(bar.NewProxyReader(in))
    if err != nil {
        return err
    }
    defer gz.Close()
    _, err = io.Copy(out, gz)
    return err
}
```

`gzip.Reader.Close` returns the cumulative error state of the reader — in particular, it is the documented place where CRC/length mismatches detected at the footer surface if `io.Copy` happened to read past the last data byte without noticing. By deferring `gz.Close()` and returning the `io.Copy` error, the code can silently accept a gzip stream whose trailer reports a bad CRC or length. The downloader verifies the digest of the *compressed* file earlier, so an intact-but-tampered compressed blob is rejected upstream — but for files without a digest (e.g., `LIMA_HOME`-local tars, or any decompression reached from a path that bypasses digest verification) the CRC check is the last line of defense and this code discards it.

Out of scope for a research PR, but worth fixing when this lands for real.

Fix:

```diff
     if decompressCmd == "gzip" {
         gz, err := gzip.NewReader(bar.NewProxyReader(in))
         if err != nil {
             return err
         }
-        defer gz.Close()
-        _, err = io.Copy(out, gz)
-        return err
+        if _, err := io.Copy(out, gz); err != nil {
+            gz.Close()
+            return err
+        }
+        return gz.Close()
     }
```

## Design Observations

**Concerns**

- *Toolchain detection is implicit* (in-scope). `IsSSHCygwin`'s heuristic ("does `cygpath.exe` live next to `ssh.exe`?") is a proxy for "is this ssh a Cygwin build?" — it works for the two real toolchains (Git for Windows, MSYS2) but it's brittle. A user who installs `cygpath.exe` into `C:\Windows\System32\OpenSSH\` for some unrelated reason (e.g., script packaging) would flip native OpenSSH to be misdetected as Cygwin. A more robust signal would be running `ssh -V` once (already done by `detectOpenSSHInfo`) and matching on the banner (`OpenSSH_for_Windows_…` → native; `OpenSSH_…p… Cygwin|MSYS` — the banner format varies). Not urgent, but worth considering before this lands for real.

- *Mux-stripping duplicated across call sites* (future). `scp.Command`, `rsync.Command`, `hostagent/mount.go`, and `cmd/limactl/shell.go` each strip mux options on Windows via slightly different mechanisms (`SSHOptsRemovingControlPath`, `DisableControlMasterOptsFromSSHArgs`). A single choke point — e.g., `SSHOpts` taking a `withControlMaster bool` parameter, or `sshutil` owning "give me Windows-safe SSH options" as a named constructor — would prevent the kind of inconsistency flagged in I1. Consider centralising during cleanup.

**Strengths**

- The `PathForSSH` / `IsSSHCygwin` split cleanly separates the "which path form should the ssh family consume?" question from the "is the toolchain Cygwin-based?" question, leaving either reusable independently.
- The ParseOpenSSHVersion test adds the Win32-OpenSSH banner as a new positive case, guarding against the exact silent 0.0.0 regression that motivated the regex extension.
- The in-process gzip path is a small, well-isolated win — preserves the progress bar via `bar.NewProxyReader(in)`, keeps the external-tool fallback for other compressors, and is transparent on non-Windows platforms.
- The two plain-Windows CI jobs print the PATH before/after scrub, inventory every expected binary, and dump `ha.*.log` / `serial.log` on failure. If something breaks on these jobs, the log should be self-diagnosing without a local Windows repro.

### Executive Summary

This PR successfully decouples Lima from the MSYS2/Git for Windows ecosystem by natively supporting the Windows OpenSSH toolchain, substituting the external `gzip` dependency with a pure-Go implementation, and providing a cygpath fallback for drive letters. The implementation robustly detects the installed SSH variant to selectively strip multiplexing flags unsupported by native Windows OpenSSH, and the CI additions thoroughly exercise the new "plain Windows" path. The overall design is solid, with only a minor regression identified where the new path-parsing logic unintentionally shadows single-letter instance names during `limactl copy` operations.

### Findings

#### Critical Issues
None.

#### Important Issues

**I1. Drive-letter check shadows single-letter instance names** — `pkg/copytool/copytool.go:104`

```go
		isLocalAbs := runtime.GOOS == "windows" && filepath.VolumeName(path) != ""
		if isLocalAbs {
			sshExe, err := sshutil.NewSSHExe()
```
The new `isLocalAbs` check evaluates `filepath.VolumeName(path) != ""` rather than `filepath.IsAbs(path)`. While this successfully intercepts absolute Windows paths (e.g., `C:/Users`) so they aren't incorrectly split on the colon, it also intercepts *relative* paths that include a drive letter (e.g., `C:foo.txt`). 

Because of this, if a user has an instance named `C`, the command `limactl copy C:foo.txt .` will now incorrectly parse `C:foo.txt` as a local Windows path rather than the remote path `foo.txt` on instance `C`. Before this PR, `filepath.IsAbs("C:foo.txt")` correctly evaluated to false, routing the string to the split logic where it was successfully recognized as a remote instance reference.

Fix: Revert the condition back to `filepath.IsAbs(path)`. This maintains the necessary bypass for `C:\` and `C:/` without breaking access to single-letter instance names.
```diff
-		isLocalAbs := runtime.GOOS == "windows" && filepath.VolumeName(path) != ""
+		isLocalAbs := runtime.GOOS == "windows" && filepath.IsAbs(path)
```

#### Suggestions

**S1. Native cygpath fallback produces invalid paths for relative drive-letter inputs** — `pkg/ioutilx/ioutilx.go:61`

```go
	if vol := filepath.VolumeName(orig); len(vol) == 2 && vol[1] == ':' {
		converted := "/" + strings.ToLower(vol[:1]) + filepath.ToSlash(orig[2:])
		logrus.Debugf("native cygpath fallback: %q -> %q", orig, converted)
		return converted, nil
	}
```
The fallback string concatenation implicitly assumes that `orig[2:]` begins with a directory separator. If `orig` is a relative path like `C:foo`, `filepath.ToSlash(orig[2:])` yields `foo`, resulting in the invalid path `/cfoo` instead of `/c/foo`. 

While current callers (such as `limayaml.Validate` and `os.Getwd()` via `shell.go`) strictly enforce or return absolute paths, relying on implicit slashes makes the API contract of `WindowsSubsystemPath` fragile for future use.

Fix: Ensure a slash explicitly separates the drive letter from the rest of the path:
```diff
 	if vol := filepath.VolumeName(orig); len(vol) == 2 && vol[1] == ':' {
-		converted := "/" + strings.ToLower(vol[:1]) + filepath.ToSlash(orig[2:])
+		converted := "/" + strings.ToLower(vol[:1]) + "/" + strings.TrimPrefix(filepath.ToSlash(orig[2:]), "/")
 		logrus.Debugf("native cygpath fallback: %q -> %q", orig, converted)
```

Validating that `gzip.NewReader` operates seamlessly with `io.Copy` through the progress bar pipeline would act as a safeguard against edge cases (like trailing checksum errors).
