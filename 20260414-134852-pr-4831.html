Executive Summary ¶

PR #4831 fixes a real race condition on Alpine ramdisk VMs where growpart triggers a kernel partition-table re-read that races with e2fsck, causing udevd's libblkid probe to fail and delete the /dev/disk/by-label/data-volume symlink. Under unlucky timing, the symlink never exists at all, and the else branch misidentifies an already-populated data disk as unused and reformats it — silent data loss.

The fix has three parts: (1) resolve the data volume via blkid output directly, eliminating the udev-symlink dependency; (2) add udevadm settle after growpart to prevent the concurrent-probe race; and (3) replace the /proc/mounts-only "disk in use" check with an lsblk partition check so a partitioned-but-unmounted data disk is no longer reformatted. Any one of these fixes alone would close the reported bug; together they make the script robust across the failure modes the issue describes.

All three reviewers approve the change. No critical or important issues were identified. The suggestions below are defensive improvements against edge cases (PARTLABEL false match, multi-device corner cases, missing udev tooling) and one pre-existing gap around raw-formatted disks. Test coverage for the boot path remains absent, which is noted but is a general Lima constraint rather than a regression. Structure: 0 critical, 0 important, 7 suggestions.

S1. udevadm settle is unguarded under set -e Gemini¶
pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh
	if command -v growpart >/dev/null 2>&1 && command -v resize2fs >/dev/null 2>&1; then
		# Automatically expand the data volume filesystem
		growpart "$DATA_DISK" 1 || true
		# growpart triggers a partition table re-read; settle udev before
		# touching the device to avoid racing with the re-probe.
		udevadm settle
		# Only resize when filesystem is in a healthy state
		if e2fsck -f -p "${DATA_VOLUME}"; then
			resize2fs "${DATA_VOLUME}" || true
		fi
	fi

The script runs with set -eux -o pipefail. udevadm settle can exit non-zero in two ways: the udevadm binary may be absent (hypothetical on lima-alpine, which ships eudev, but possible on a user-supplied minimal image using mdev), or the settle may time out (default 120 s) if udevd has a stuck event. Either failure crashes the boot script with the same user-visible symptom this PR set out to fix — /mnt/data does not mount, tmpfs fills up. The udevadm settle call is defense-in-depth against the race; the blkid resolution already eliminates the core dependency on udev state, so making the settle tolerant of failure does not weaken the fix.

Fix: guard or silence the call on both lines.

-        udevadm settle
+        udevadm settle || true
S2. blkid regex can match PARTLABEL Gemini¶

The regex .*LABEL="data-volume".* also matches lines carrying PARTLABEL="data-volume", because the leading .* greedily consumes the PART prefix. BusyBox blkid (the implementation on lima-alpine) does not emit PARTLABEL, so this is currently latent; util-linux blkid does emit it for GPT partitions, so a user who installs util-linux-blkid or attaches a disk with a GPT partition labeled data-volume but no filesystem label would see the script misidentify the device and fall over at e2fsck. Fix is a one-character anchor.

-DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*LABEL="data-volume".*/\1/p')
+DATA_VOLUME=$(blkid | sed -n 's/^\([^:]*\):.*[[:space:]]LABEL="data-volume".*/\1/p')

S4. DATA_DISK="${DATA_VOLUME%[0-9]}" strips only one digit Claude¶
pkg/cidata/cidata.TEMPLATE.d/boot.Linux/04-persistent-data-volume.sh

This strips a single trailing digit from the device path to derive the parent disk. It is correct for /dev/vda1 → /dev/vda, which covers every Lima VM today (virtio-blk on VZ/QEMU always names disks vd[a-z]). It is wrong for NVMe naming (/dev/nvme0n1p1 → /dev/nvme0n1p, not /dev/nvme0n1) and for double-digit partitions (/dev/sda10 → /dev/sda1). Lima does not use these devices for the Alpine data volume path today, so this is a robustness issue rather than a live bug.

Worth noting from git blame: the old code at this location used sub(/\d$/, "", s[1]) inside awk, and \d is not a POSIX character class, so BusyBox awk treated it as a literal d. The digit strip in the old code never actually ran; the new code is the first time DATA_DISK is correctly derived at all, which is why growpart now functions on the happy path.

Fix (optional, only if robustness for other device types is wanted):

-DATA_DISK="${DATA_VOLUME%[0-9]}"
+DATA_DISK="/dev/$(lsblk --noheadings --output pkname "${DATA_VOLUME}")"

S7. No automated coverage for the boot path this PR fixes Codex¶

There is no BATS or integration test in hack/bats/tests/ that exercises 04-persistent-data-volume.sh on an Alpine ramdisk VM. The script is timing-sensitive early-boot infrastructure and the race itself is not reproducible on demand, so this is understandable — but it leaves the three highest-risk behaviors this patch protects unguarded: the blkid-based device resolution, the settle-then-e2fsck sequencing, and the partition-presence check in the else branch. A later refactor could silently reintroduce the symlink dependency or the "partitioned but unmounted" reformat bug without any test breaking.

Fix: if practical, add a boot-path scenario that exercises both an already-formatted data-volume partition (must still mount after the resize/fsck path) and a partitioned-but-unmounted disk (must be skipped rather than reformatted). If the current BATS harness cannot drive an Alpine ramdisk VM directly, a minimal repo-local harness that asserts the resulting mount state would be sufficient.

Strengths ¶

  • Defense-in-depth layering (in-scope) Claude Codex Gemini — The three fixes target independent failure points: blkid eliminates the udev-state dependency, udevadm settle prevents the growpart/e2fsck race, and the lsblk partition check closes the reformat path in the else branch. Any one fix alone would have been sufficient for the reported bug; together they make the script robust against related regressions.
  • Else-branch restructuring reduces nesting (in-scope) Claude — Replacing the IN_USE flag with continue-based early exits (lines 103–108) cuts one level of nesting and lifts the format/mount/break work out of a conditional. The control flow reads top-to-bottom instead of inside-out.
  • Comment quality on the race explanation (in-scope) Claude — The block comment at lines 55–60 explains both the why (udev re-probe during growpart races with e2fsck's superblock modification) and the what (BusyBox blkid lacks --label) concisely. A future reader has enough context to understand the design without having to find the issue or PR.
  • Topological disk discovery (in-scope) Gemini — The else branch uses lsblk --output name,type to enumerate disks rather than hardcoding vda/sda prefixes. This keeps the script adaptable across virtualization backends that might present different block-device names.

Testing Assessment ¶

No automated tests exercise 04-persistent-data-volume.sh. The script runs during early boot inside an Alpine ramdisk VM, and the race itself is timing-dependent and not reproducible on demand — both genuine obstacles to integration testing. Three scenarios are worth validating by hand or in a minimal harness:

  1. Core race fix. Boot an Alpine ramdisk VM and confirm /mnt/data mounts successfully. Observe that /dev/disk/by-label/data-volume may still be absent post-boot (the known udev behavior), and that the mount used the blkid-resolved path regardless.
  2. Partition-safety path. Boot a VM whose data disk is partitioned but unmounted (e.g. after a failed previous boot) and confirm the script enters the else branch, skips the disk via the new lsblk partition check, and does not reformat it.
  3. Fresh-format path. Boot a new VM with no data volume and confirm sfdisk → lsblk → mkfs.ext4 → udevadm settle → mount completes and the subsequent mount --bind loop succeeds. Per the \d-in-awk observation in S4, this may be the first boot on which growpart and the derived DATA_DISK actually function correctly; a re-resize after a template disk size bump is worth specifically exercising.

Review Process Notes ¶

Skill improvements ¶

  • Shell safety: flag unguarded calls under set -e whose failure reintroduces the bug a PR is fixing. When a shell PR fixes a bug and adds defense-in-depth instrumentation (e.g., udevadm settle, sync, flush), check whether the added command can fail and whether set -e is active. If the command's failure produces the same user-visible symptom the PR was meant to eliminate, flag the unguarded call. Rationale: a defense-in-depth command whose failure reintroduces the class of bug it was added to defend against is strictly worse than a well-chosen primary fix alone.
  • Shell safety: check sed/grep/awk patterns with leading .* for substring bleed into structurally distinct fields. Patterns of the form .*KEY="value" can inadvertently match lines where KEY is a suffix of another key name (e.g., LABEL matches lines carrying PARTLABEL). When reviewing shell code that parses field-oriented tool output via unanchored regex, verify that the field name cannot appear as a suffix of another known field in the same output, and suggest a leading whitespace or start-of-field anchor when it can.

Repo context updates ¶

  • [repo] pkg/cidata/cidata.TEMPLATE.d/boot.Linux/ scripts run under BusyBox ash with ASH_BASH_COMPAT, not GNU bash. The shebang is #!/bin/bash but on lima-alpine /bin/bash is BusyBox ash with ASHBASHCOMPAT enabled. When reviewing scripts in this directory, do not assume GNU-bash-only features (process substitution, [[ ]] regex, mapfile, readarray, associative arrays) will work. Additionally, the blkid, awk, and other utilities on lima-alpine are BusyBox implementations and have reduced flag coverage compared to util-linux/GNU equivalents (e.g., BusyBox blkid lacks --label, BusyBox awk does not support \d). Flag code that depends on GNU-only syntax, and when in doubt, verify against BusyBox documentation rather than GNU man pages.
