Executive Summary ¶

This is round 2 of the plain-Windows enablement branch — round 1's three Important findings (cygpath sibling threading, missing QEMU PATH, gzip ctx cancellation) are all addressed in the current HEAD. The new cygpathForSSH helper, WindowsSubsystemPathWithCygpath API split, explicit Add QEMU to PATH step in windows-plain-qemu, and gzip in-process cancellation goroutine all check out against the code.

The only new Important is Codex's: pkg/hostagent/mount.go:65 runs the host mount path through PathForSSH(sshExe) and uses the result as the LocalPath that the locally-spawned sftp-server consumes — but the default SFTPDriver is empty (pkg/limayaml/defaults.go:758), so sshocker PATH-resolves sftp-server.exe independently of which ssh.exe Lima selected. A host with Git/MSYS2 ssh.exe in front and native Win32-OpenSSH sftp-server.exe resolved separately gets /c/... fed to a native sftp-server, which fails the reverse-sshfs mount silently during limactl start.

The residue around the round-1 gzip fix accounts for most suggestions: the cancellation goroutine double-closes in (benign, but seeds future log-noise traps), and the function still surfaces cancellation as an os.ErrClosed-wrapping error rather than ctx.Err() despite already holding ctx. The remaining suggestions cluster in two areas: CI hygiene (unchecked $LASTEXITCODE after native commands in three composite actions; smoke test only exercises host→guest copy) and small contract/style cleanups in parseCopyPaths and cygwinDetectCache.

The outer defer in.Close() at line 551 then closes the same file again. os.File.Close() on a closed file returns *fs.PathError("file already closed"); both deferred close errors are discarded, so today this is benign. +        defer close(done)
         go func() {
-            <-ctx.Done()
-            in.Close()
+            select {
+            case <-ctx.Done():

+            }

S7. cygwinDetectCache "log once" promise breaks under contention Claude¶

The doc comment on IsSSHCygwin/cygpathForSSH says "Results are cached per resolved absolute path and logged once per path at Debug level," but the implementation is check-then-act between the RLock/RUnlock and the Lock/Unlock:

pkg/sshutil/sshutil.go
// Windows OpenSSH. Detection looks for cygpath.exe in the same directory
// as ssh.exe (the layout Git for Windows and MSYS2 use), after resolving
// symlinks so shims from chocolatey, scoop, etc. do not throw the check
// off. Always returns false on non-Windows.
//
// Results are cached per resolved absolute path and logged once per path
// at Debug level.
func IsSSHCygwin(sshExe SSHExe) bool {
	_, ok := cygpathForSSH(sshExe)
	return ok
}

// cygpathForSSH locates the cygpath.exe that belongs to the same install
// as sshExe and reports whether sshExe is Cygwin-based. The returned
// path is the resolved sibling, so callers that pass it to exec.Command
// invoke the toolchain's own cygpath even when $SSH points at an ssh
// outside PATH. On non-Windows or empty input, returns ("", false).
//
// Results are cached per resolved absolute path and logged once per path
// at Debug level.
func cygpathForSSH(sshExe SSHExe) (string, bool) {
	if runtime.GOOS != "windows" || sshExe.Exe == "" {
		return "", false
	}
	path := sshExe.Exe
	if !filepath.IsAbs(path) {
		resolved, err := exec.LookPath(path)
		if err != nil {
			logrus.WithError(err).Debugf("cygpathForSSH: cannot resolve %q via PATH; assuming native", sshExe.Exe)
			return "", false
		}
		path = resolved
	}
	if resolved, err := filepath.EvalSymlinks(path); err == nil {
		path = resolved
	}
	cygwinDetectCacheRW.RLock()
	cached, ok := cygwinDetectCache[path]
	cygwinDetectCacheRW.RUnlock()
	if ok {
		return cached, cached != ""
	}
	cygpathExe := filepath.Join(filepath.Dir(path), "cygpath.exe")
	if _, err := os.Stat(cygpathExe); err != nil {
		logrus.Debugf("ssh at %q detected as native Windows OpenSSH (no cygpath.exe alongside)", path)
		cygpathExe = ""
	} else {
		logrus.Debugf("ssh at %q detected as Cygwin-based (found %q alongside)", path, cygpathExe)
	}
	cygwinDetectCacheRW.Lock()
	cygwinDetectCache[path] = cygpathExe
	cygwinDetectCacheRW.Unlock()
	return cygpathExe, cygpathExe != ""
}

Design Observations ¶

Strengths ¶

  • PathForSSH + cygpathForSSH clean split (in-scope) Claude Codex — Round 1's I1 fix introduces a clean separation: cygpathForSSH returns both the detection bool and the sibling cygpath.exe path, PathForSSH threads that sibling to WindowsSubsystemPathWithCygpath, and WindowsSubsystemPath (bare-PATH) is kept as a thin wrapper for callers without an ssh.exe in hand. The internal helper plus exported WithCygpath API split is the right shape.
  • Composite-action sequencing comments are load-bearing and explicit (in-scope) Claude — windows_plain_host's "MUST run AFTER any step that needs git CLI" and windows_plain_templates's "MUST run before windows_plain_host" lock in the implicit ordering between actions; a future job restructure that breaks the order fails loudly instead of producing a strange downstream symptom.
  • windows_plain_host four-layer verification holds up (in-scope) Claude — Filesystem, PATH across three scopes, smoking-gun binaries, and registry uninstall keys covered; the WSL C:\Windows\System32\bash.exe allowlist exception is in the right place.

Concerns ¶

  • Two distinct code paths for "convert a host path for a Linux guest" (future) Claude — cmd/limactl/shell.go:714 and pkg/limayaml/defaults.go:698 still call the bare-PATH ioutilx.WindowsSubsystemPath, while copytool / sshutil / hostagent funnel through PathForSSH. On plain Windows both paths produce the same drive-letter form; on a Cygwin install with a custom fstab, they would diverge. Worth a follow-up to either route the mount-point computation through PathForSSH or document why two paths are needed.
  • Toolchain coupling between ssh.exe and the local sftp-server is asserted but not enforced (in-scope) Codex — See I1. The right shape may be a small "Windows SSH toolchain" descriptor that carries the selected ssh executable and the matching sftp-server, so future call sites that need the pair (mount, future remote-execution helpers) cannot accidentally key them off only one half.

Testing Assessment ¶
