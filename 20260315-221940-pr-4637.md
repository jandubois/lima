## Consolidated Review

### Executive Summary

This PR adds a conversion cache layer to Lima's downloader. When a VM driver
(VZ, krunkit) cannot boot the downloaded image format (typically QCOW2), the
downloader now converts the image to RAW once in a new `imgconv/raw`
subdirectory alongside the cached original, avoiding repeated per-instance
conversions. The approach is sound and addresses the disk-space and startup-time
problems described in issue #4056.

However, the implementation overwrites the caller's expected digest with the
derived raw digest before validation, so a cached raw image is never
re-validated against the original template digest. If an upstream image changes
and the user updates their digest, Lima silently reuses the stale cached
conversion. Two additional important issues affect compressed-image handling
and a potential nil-pointer panic.

### Critical Issues

1. **Cached raw conversion bypasses the caller's digest validation** — `pkg/downloader/downloader.go:311-322` [Codex GPT 5.4, Gemini 3.1 Pro]

```go
} else {
    shadData = rawImgConvPath
    if currentDigestData, err := os.ReadFile(rawImgConvDigestPath); err == nil {
        currentDigest := strings.TrimSpace(string(currentDigestData))
        if d, err := digest.Parse(currentDigest); err == nil {
            o.expectedDigest = d
            shadDigest = rawImgConvDigestPath
        }
    }
}
```

When a cached raw image exists at line 315, `getCached` overwrites
`o.expectedDigest` (the original digest from the template) with the raw
image's digest at line 319 and points `shadDigest` at `raw.digest` at
line 320. The subsequent `validateCachedDigest` call at line 332 then
compares `raw.digest` against itself — a tautology that always passes.

The original digest file (e.g., `sha256:<original>`) is never consulted.
If the upstream image changes and the user updates their template digest,
`getCached` still returns the old raw conversion. This violates the stated
requirement in issue #4056 that "the cache must retain the checksum of the
original image for lookup."

The same overwrite occurs in the `needConvert` branch at line 312 after
`ensureRawInCache`, and again in `fetch()` at line 400. In every path, the
original digest is lost.

Claude Opus 4.6 identified a narrower variant (missing `raw.digest` file)
at the same location but rated it important rather than critical. The
broader issue — that digest validation is bypassed even when `raw.digest`
exists — warrants critical severity because it silently serves wrong data.

Fix: Keep the original and derived digests separate. First validate the
cached original image against the caller's expected digest (or force a
re-download on mismatch). Only then validate `imgconv/raw` against
`raw.digest` before copying it.

### Important Issues

1. **Nil pointer dereference when `os.Stat(shadData)` fails** — `pkg/downloader/downloader.go:299-300` [Claude Opus 4.6, Gemini 3.1 Pro]

```go
origStat, _ := os.Stat(shadData)
if origStat.ModTime().After(rawStat.ModTime()) {
```

The error from `os.Stat` at line 299 is discarded. Although `shadData` was
verified to exist at line 278, a concurrent cache deletion or filesystem
error between the two stat calls would leave `origStat` nil, causing a
panic at line 300.

Fix: Check the error. If stat fails, set `needConvert = true`.

```go
origStat, err := os.Stat(shadData)
if err != nil {
    needConvert = true
} else if origStat.ModTime().After(rawStat.ModTime()) {
    needConvert = true
}
```

2. **Compressed images bypass cache conversion** — `pkg/downloader/downloader.go:283,390` [Codex GPT 5.4, Gemini 3.1 Pro]

```go
imageFormat, err := nativeimgutil.DetectFormat(shadData)
```

`DetectFormat` at lines 283 and 390 runs on the cached file before
decompression. Compressed downloads (`.qcow2.gz`, `.img.xz`) store
compressed bytes in `shadData`. The QCOW2 magic number is obscured by the
compression wrapper, so `DetectFormat` falls through to "raw." Because
"raw" appears in every driver's supported list, the conversion logic is
skipped entirely.

This defeats the PR's purpose for compressed images — the most common
distribution format for cloud images (e.g., Ubuntu `.img` files distributed
as `.xz`).

Fix: Run format detection on uncompressed bytes. Either decompress to a
temporary file before calling `DetectFormat` when `o.decompress` is true,
or check the inner extension (e.g., strip `.gz`/`.xz` before checking
`path.Ext`). Add tests for compressed inputs.

3. **`isNonISOImage` relies on display-text string matching** — `pkg/downloader/downloader.go:360-361` [Claude Opus 4.6]

```go
func isNonISOImage(description, remote string, supportedImageFormats []string) bool {
    return strings.Contains(description, "image") && supportedImageFormats != nil && path.Ext(remote) != ".iso"
}
```

Whether cache conversion fires depends on the `description` string
containing `"image"`. This display string is constructed at
`pkg/fileutils/download.go:32`. A caller that passes a different
description (e.g., `"the disk"`) silently disables conversion.

The check is duplicated at `pkg/fileutils/download.go:35`, creating a
second point of failure.

Fix: Replace the string-matching heuristic with a typed flag. The
`WithImageFormats` option already signals intent — if
`supportedImageFormats` is non-nil, the caller wants conversion. The
`description` check is redundant.

4. **`MakeSparse` call is a no-op through the proxy** — `pkg/downloader/downloader.go:446-448` [Claude Opus 4.6, Gemini 3.1 Pro]

```go
if err := diskUtil.MakeSparse(ctx, rawTmpF, fi.Size()); err != nil {
    logrus.WithError(err).Warnf("Failed to make %q sparse (non-fatal)", rawPathTmp)
}
```

`proxyimgutil.MakeSparse` at `pkg/imgutil/proxyimgutil/proxyimgutil.go:74`
delegates to `QemuImageUtil.MakeSparse` at
`pkg/qemuimgutil/qemuimgutil.go:190`, which is a stub returning nil. The
proxy treats nil as success and never falls through to the native
implementation that does the actual `seek + truncate`.

When `qemu-img` is installed (the common case), `qemu-img convert -O raw`
already produces sparse output, so this is harmless. When `qemu-img` is
absent, the native converter copies bytes without sparsification, and the
subsequent `MakeSparse` call — the one place that should fix this — is
silently skipped.

Fix: Either call `p.native.MakeSparse` directly in `ensureRawInCache`, or
fix the proxy stub to return a sentinel error (e.g.,
`ErrNotImplemented`) that triggers the fallback. The proxy bug is
pre-existing, but this new call site makes it load-bearing for cache disk
usage.

### Design Observations

1. **`o.expectedDigest` serves two meanings** (in-scope) [Codex GPT 5.4]

`o.expectedDigest` starts as "the caller's expected digest for the original
download" and later becomes "the derived digest of `imgconv/raw`." Keeping
these as separate variables would eliminate the digest-bypass bug (critical
finding #1) and simplify the mental model. The original digest governs
cache lookup and revalidation; the raw digest protects only the derived
artifact's integrity.

2. **Hardcoded RAW-only cache layout** (future) [Claude Opus 4.6]

`ensureRawInCache` always converts to raw, and the cache layout is
hardcoded to `imgconv/raw`. All current drivers that set
`SupportedImageFormats` include `"raw"`, so this works today. If a future
driver needs a different format, the layout would need generalization
(e.g., `imgconv/<format>/data`).

3. **`DownloadFile` parameter sprawl** (in-scope) [Claude Opus 4.6]

Adding `supportedImageFormats []string` as a positional parameter to
`DownloadFile` at `pkg/fileutils/download.go:23` forces every call site to
pass it, even when nil. Six call sites were touched. An options pattern
(like `downloader.Download` already uses) would prevent churn as the
function grows.

### Testing Assessment

The PR adds three test functions covering the core conversion flow.
Untested scenarios, ranked by risk:

1. **Digest rotation with existing cached raw conversion** — no test asserts
   that `Download` re-downloads when the caller provides a new digest for a
   URL whose `imgconv/raw` already exists. This gap masks the critical
   digest-bypass bug.
2. **Compressed image inputs** (`.qcow2.gz`, `.img.xz`) through both fresh
   and cached paths — `DetectFormat` runs on compressed bytes and silently
   misclassifies the format.
3. **Corrupt or missing `raw.digest`** with existing raw image — the
   fallback path at lines 316-322 has no coverage.
4. **Re-conversion on stale raw image** — the
   `origStat.ModTime().After(rawStat.ModTime())` path at line 300 has no
   test.
5. **Native conversion** (no `qemu-img`) — all tests skip when `qemu-img`
   is absent. The native convert + make-sparse path has no coverage.
6. **QEMU driver retains original format** — no test verifies that a driver
   listing `qcow2` as supported skips conversion and returns the original.

### Documentation Assessment

The `internals.md` update describing `imgconv/` is appropriate. It should
also state that the original digest remains authoritative for cache lookup,
while `raw.digest` protects only the derived artifact — once the
implementation matches that contract.

### Codex GPT 5.4

#### Executive Summary

This PR teaches the downloader to keep the original image in the cache while creating `imgconv/raw` for drivers that cannot boot the source format directly, and it threads driver-supported formats from `instance.Prepare()` into the download path. The main regression is that once `imgconv/raw` exists, cache reuse no longer honors the caller's original digest, so stable URLs like `*-latest.qcow2` can silently return an older cached image; there is also an untested compressed-image path where format detection runs on compressed bytes, so the new raw-cache behavior is skipped for `.qcow2.gz`/`.img.xz` style inputs.

#### Findings

##### Critical Issues

1. **Existing `imgconv/raw` bypasses the caller's original digest** — `pkg/downloader/downloader.go:305` (critical, regression)

```go
if needConvert {
    converted, rawDigest, err := ensureRawInCache(ctx, shadData, imageFormat, o.expectedDigest)
    ...
    o.expectedDigest = rawDigest
    shadDigest = rawImgConvDigestPath
} else {
    shadData = rawImgConvPath
    ...
    o.expectedDigest = d
    shadDigest = rawImgConvDigestPath
}
```

`instance.Prepare()` still passes the template's original digest into the downloader at line 89 in `pkg/instance/start.go`, and `fetch()` still stores that original digest in `<algo>.digest` at lines 382-384 in `pkg/downloader/downloader.go`. But `getCached()` overwrites `o.expectedDigest` with `raw.digest` at lines 312 and 319, then validates `raw.digest` at line 332 instead of the caller's requested digest. That means a cached raw image built from an older `latest.qcow2` will be reused even after the template digest changes, because the original `<algo>.digest` is never consulted once `imgconv/raw` exists. This directly violates the stated requirement that "the cache must retain the checksum of the original image for lookup."

Fix: keep the caller's original digest and the derived raw digest separate. First validate the cached original image against the caller's `<algo>.digest` (or force re-download on mismatch), then optionally validate `imgconv/raw` against `raw.digest` before copying it.

##### Important Issues

1. **Compressed images are inspected before decompression, so raw-cache conversion is skipped** — `pkg/downloader/downloader.go:360` (important, regression)

```go
func isNonISOImage(description, remote string, supportedImageFormats []string) bool {
    return strings.Contains(description, "image") && supportedImageFormats != nil && path.Ext(remote) != ".iso"
}
...
if isNonISOImage(o.description, remote, o.supportedImageFormats) {
    format, err := nativeimgutil.DetectFormat(shadData)
```

The new logic classifies "non-ISO images" from the URL suffix at lines 360-361, and `fetch()`/`getCached()` call `nativeimgutil.DetectFormat()` on the cached file at lines 390 and 283 before `copyLocal()` does any decompression at line 406. `DetectFormat()` traces into `go-qcow2reader@v0.7.1/qcow2reader.go:32-41`, where unknown headers fall back to `raw`, and `image/raw/raw.go:55-57`, where `raw.Open()` accepts any file. So a compressed cloud image like `.qcow2.gz` or `.img.xz` is treated as already-`raw`, the new `imgconv/raw` cache is never populated, and the PR's optimization disappears for exactly the "compressed wrapper" case discussed in issue #4056.

Fix: only run format detection on uncompressed bytes. Either decompress to a temporary path before `DetectFormat()` when `o.decompress` is true, or explicitly skip cache conversion for compressed sources until the code can inspect the unwrapped payload correctly. Add tests for `.qcow2.gz`/`.img.xz` with and without an existing cache entry.

None.

3. Existing `imgconv/raw` with a missing or unreadable `raw.digest`, to confirm the downloader does not silently trust an unverifiable derived file.

#### Documentation Assessment

The new cache-layout docs in `website/content/en/docs/dev/internals.md` describe `imgconv/raw` and `raw.digest`, but they do not state the crucial contract from issue #4056: the original digest remains authoritative for cache lookup, while `raw.digest` is only for the derived artifact's integrity. That needs to be spelled out once the implementation matches it.

### Gemini 3.1 Pro

#### Executive Summary

This PR introduces an optimization to preemptively convert non-ISO disk images to their target formats within the Lima cache, saving time and disk space across multiple instances. However, the implementation introduces critical regressions: it bypasses digest validation for cached images (ignoring user updates to digests), and it breaks root disk resizing for pre-converted images, causing VMs to boot with incorrectly sized disks.

#### Findings

##### Critical Issues

1. **`getCached` bypasses digest validation for cached images, ignoring user's `expectedDigest`** — `pkg/downloader/downloader.go:311-318` (critical, regression)

```go
			} else {
				shadData = rawImgConvPath
				if currentDigestData, err := os.ReadFile(rawImgConvDigestPath); err == nil {
					currentDigest := strings.TrimSpace(string(currentDigestData))
					if d, err := digest.Parse(currentDigest); err == nil {
						o.expectedDigest = d
						shadDigest = rawImgConvDigestPath
					}
				}
			}
```

When a converted image is found in the cache, the code overwrites `o.expectedDigest` (the original digest requested by the user) and `shadDigest` with the digest of the converted `raw` image *before* validation occurs. Later, `validateCachedDigest` is called with these overwritten values, causing it to validate the raw digest against itself (which always passes). Consequently, if a user updates the image digest in `lima.yaml` (e.g., to fetch a newer image from the same URL), Lima will silently ignore the mismatch and reuse the old cached image.

Fix: Validate the *original* cached file's digest (`shadDigest`) against the *original* user-provided `o.expectedDigest` *before* switching `shadData` to the converted image. Only if the original file passes validation should the cache be used.

2. **`EnsureDisk` bypasses disk resizing for pre-converted images** — `pkg/driverutil/disk.go:102-106` (critical, regression)

```go
		if format == string(diskImageFormat) {
			if err = os.Rename(imagePath, diskPath); err != nil {
				return err
			}
		} else {
```

When the downloaded or cached image natively matches the required driver format (e.g., a cached `raw` image for the VZ driver), this change bypasses `diskUtil.Convert` and simply renames the file. While this correctly avoids an unnecessary conversion, it completely skips resizing the disk to the requested `diskSizeInBytes`. As a result, the VM will boot with a root disk size equal to the original downloaded image (often 2-10GB) instead of the user's requested size (e.g., 100GiB) in `lima.yaml`.

Fix: Call `diskUtil.ResizeDisk(ctx, diskPath, diskSizeInBytes)` immediately after successfully renaming the file.

**[Consolidation note: This finding is a false positive. `prepareDisk()` at `pkg/instance/start.go:126` handles disk resizing after `CreateDisk()` returns, for all image formats including raw.]**

##### Important Issues

1. **Potential nil pointer dereference on `origStat`** — `pkg/downloader/downloader.go:294-297` (important, regression)

```go
			} else {
				origStat, _ := os.Stat(shadData)
				if origStat.ModTime().After(rawStat.ModTime()) {
					needConvert = true
				}
			}
```

The error from `os.Stat(shadData)` is explicitly ignored. While `shadData` was checked earlier in the function, it could be deleted or altered by a concurrent process. If `os.Stat` fails, `origStat` will be `nil`, and calling `origStat.ModTime()` will cause a runtime panic.

Fix: Explicitly check the error returned by `os.Stat` and handle it gracefully (e.g., `if err != nil { return nil, err }`).

##### Suggestions

1. **Redundant `MakeSparse` call** — `pkg/downloader/downloader.go:446-448`

```go
	if err := diskUtil.MakeSparse(ctx, rawTmpF, fi.Size()); err != nil {
		logrus.WithError(err).Warnf("Failed to make %q sparse (non-fatal)", rawPathTmp)
	}
```

Calling `diskUtil.MakeSparse` on the recently converted image essentially executes `f.Truncate(fi.Size())` (via `nativeimgutil.makeSparse`), which is a no-op on an already fully written file and does not actively punch holes for existing zeros. `qemu-img convert -O raw` natively handles sparsity during conversion on APFS and modern Linux filesystems.

Fix: Remove the manual `MakeSparse` block to simplify the code.

#### Design Observations

**Cache conversion bypasses compressed images** (in-scope)

The stated goal of this PR is to convert non-ISO images to RAW in the cache. However, when downloading compressed images (e.g., `.qcow2.gz`), the cached file `shadData` remains compressed. Because `nativeimgutil.DetectFormat` does not parse compressed files, it silently defaults to returning `"raw"`. For drivers like VZ that natively support `"raw"`, conversion in the cache is entirely skipped. The file is decompressed into the instance directory, and `EnsureDisk` is forced to perform the conversion on every startup — completely defeating the caching optimization for compressed sources.

#### Testing Assessment

1. **Missing test for digest mismatch**: `downloader_test.go` adds a test case (`TestDownloadImageConversion/with-digest`) which successfully exercises a cache hit. However, it lacks a test asserting that `Download` **fails** when provided a new, *incorrect* `expectedDigest` for an already converted cache entry. This missing scenario masks the critical digest validation regression described above.
2. **Missing test for pre-converted disk resizing**: `driverutil/disk_test.go` should be updated to assert that a downloaded image whose format naturally matches `diskImageFormat` is properly resized to the requested size after processing by `EnsureDisk`.

#### Documentation Assessment

The documentation updates in `internals.md` accurately reflect the addition of the `imgconv/` directory to the cache structure. No further changes needed.
