## Consolidated Review

### Executive Summary

The `--sync` feature mirrors a host working directory into the guest VM via rsync, runs a shell session, then offers an interactive accept/reject/diff workflow before syncing changes back. The core happy path works, but the implementation has serious bugs: two switch-case ordering issues leave `destRsyncDir` unset in valid configurations, and the guest cleanup deletes far more than it should. Several secondary issues affect robustness (silent failures, state mutation, SSH transport bypass).

### Critical Findings

**[CRITICAL] cmd/limactl/shell.go:238-258 — `--sync` combined with `--workdir` leaves `destRsyncDir` empty** [Claude Opus 4.6]

Problem: `destRsyncDir` is assigned only in the `case syncHostWorkdir` branch (line 253). When `--workdir` is also set, `case workDir != ""` matches first (line 239), so `destRsyncDir` stays as `""`. The subsequent rsync operations at lines 348-388 proceed with an empty destination path — `mkdir -p ''` via SSH, and rsync to `instance:` with no path. No validation prevents combining `--sync` and `--workdir`.

Fix: Reject the combination explicitly:
```go
if syncHostWorkdir && workDir != "" {
    return errors.New("cannot use --sync and --workdir together")
}
```

**[CRITICAL] cmd/limactl/shell.go:242,253 — `--sync` on WSL2 instances without mounts silently breaks** [Claude Opus 4.6]

Problem: The switch case at line 242 checks `len(inst.Config.Mounts) > 0 || inst.VMType == limatype.WSL2`. On a WSL2 instance started with `--mount-none`, mounts are empty but `inst.VMType == limatype.WSL2` is true. This case matches before `case syncHostWorkdir` at line 253, so `destRsyncDir` is never assigned. The validation at line 198 only checks for mounts, so it passes. Same empty-path failure as above.

Fix: Compute `destRsyncDir` before the switch (it depends only on `syncHostWorkdir` and `hostCurrentDir`, not on the mount/workdir logic):
```go
if syncHostWorkdir {
    destRsyncDir = *inst.Config.User.Home + hostCurrentDir
}
```

**[CRITICAL] cmd/limactl/shell.go:438-443,460 — Guest cleanup deletes entire top-level subtree, not just the synced directory** [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro]

Problem: `dirForCleanup` is derived from `parts[1]` — the first path component after root (e.g., `Users` from `/Users/jan/repo`). The deferred `rm -rf` at line 460 therefore removes `~/Users/` in the guest, wiping every synced directory under that top-level component. Concurrent `--sync` sessions from different host directories under the same top-level path destroy each other's data.

Fix: Clean up exactly `destRsyncDir`:
```go
dirForCleanup := shellescape.Quote(destRsyncDir)
```
Note: if `destRsyncDir` was already quoted for older rsync versions (line 365), use the unquoted original to avoid double-quoting.

### Important Findings

**[IMPORTANT] cmd/limactl/shell.go:469-475 — Permission-only and metadata-only changes silently dropped** [Codex GPT 5.4]

Problem: `parseRsyncStats` only counts lines starting with `<`, `>`, or `c` as changes. Lines like `.f...p..... file` (permission-only changes) start with `.` and are ignored. When all changes are metadata-only, `stats.String()` returns `""`, and `askUserForRsyncBack` returns early with "No changes detected" — silently discarding the guest-side changes without prompting.

Fix: Count metadata-only updates (lines where positions 3-10 contain non-`.` characters) as a separate category, or at minimum do not short-circuit sync-back when rsync reports itemized output.

**[IMPORTANT] pkg/copytool/rsync.go:72-80, 120-136 — Rsync backend bypasses Lima's SSH transport configuration** [Codex GPT 5.4]

Problem: `checkRsyncOnGuest()` hardcodes `exec.CommandContext(ctx, "ssh")` and `Command()` hardcodes `ssh` in the `-e` string. This bypasses the `$SSH` environment variable override and any wrapper args from `sshutil.NewSSHExe()`. Users with custom SSH commands or config wrappers will find that `limactl shell` works but `limactl shell --sync` fails.

Fix: Use `sshutil.NewSSHExe()` in both `checkRsyncOnGuest` and `Command`. Build the `-e` argument from `sshExe.Exe` + `sshExe.Args` + SSH opts, shell-escaping each token.

**[IMPORTANT] cmd/limactl/shell.go:589-596 — `executeSSHForRsync` appends to a value-copied slice, risking silent corruption** [Claude Opus 4.6, Gemini 3.1 Pro]

Problem: The function takes `sshCmd exec.Cmd` by value and appends to its `Args` slice. This works only because `exec.CommandContext` currently allocates Args without spare capacity, so `append` creates a new backing array. If Go's implementation changed or a caller constructed `exec.Cmd` differently, the append would overwrite the caller's data. The function is called multiple times with the same original `sshCmd`.

Fix: Copy the args explicitly:
```go
newArgs := make([]string, len(sshCmd.Args), len(sshCmd.Args)+4)
copy(newArgs, sshCmd.Args)
newArgs = append(newArgs, "-p", strconv.Itoa(sshLocalPort), sshAddress, command)
sshRmCmd := exec.CommandContext(ctx, sshCmd.Path, newArgs[1:]...)
```

**[IMPORTANT] pkg/copytool/rsync.go:90-92 — `Command()` permanently mutates receiver options** [Claude Opus 4.6, Gemini 3.1 Pro]

Problem: `t.Options = opts` replaces the tool's options for all future calls. The `CopyTool` instance is reused across `shellAction` and `askUserForRsyncBack`, so any `opts` override persists beyond the call that set it. Currently all callers pass `nil`, so the bug is latent.

Fix: Use a local variable instead of mutating the receiver:
```go
activeOpts := t.Options
if opts != nil {
    activeOpts = opts
}
```

**[IMPORTANT] cmd/limactl/shell.go:571-578 — Silent failure if `diff` executable is missing** [Gemini 3.1 Pro]

Problem: When the user selects "View the changed contents", the error handling only checks for `*exec.ExitError` (exit code >= 2). If `diff` doesn't exist on the host, `diffCmd.Run()` returns `*exec.Error`, which doesn't match `errors.As(err, &exitErr)`. The error is silently ignored and the user sees a blank pager.

Fix: Handle non-ExitError failures:
```go
if err := diffCmd.Run(); err != nil {
    var exitErr *exec.ExitError
    if errors.As(err, &exitErr) {
        if exitErr.ExitCode() >= 2 {
            _ = pipeIn.Close()
            return fmt.Errorf("diff command failed: %w", err)
        }
    } else {
        _ = pipeIn.Close()
        return fmt.Errorf("failed to start diff: %w", err)
    }
}
```

**[IMPORTANT] cmd/limactl/shell.go:363-366 — Path spaces break with rsync 3.0.0 through 3.2.3** [Gemini 3.1 Pro]

Problem: The code quotes `destRsyncDir` for rsync < 3.0.0 (macOS's ancient 2.6.9). But `--protect-args` did not become the default until rsync 3.2.4. Users running rsync 3.0.0-3.2.3 (common on older Linux, e.g. Ubuntu 20.04) will see failures when `syncDirVal` contains spaces, as the remote shell splits the unquoted path.

Fix: Change the version check to `< 3.2.4`, or always pass `--protect-args` for rsync >= 3.0.0.

**[IMPORTANT] cmd/limactl/shell.go:532-539 — Pager displays raw ANSI escape codes** [Claude Opus 4.6]

Problem: The diff command uses `--color=always` and the rsync stats headers use ANSI codes. The default pager `less` does not interpret ANSI escapes by default, so users see raw `\033[0;90m` text.

Fix:
```go
if pager == "" {
    pager = "less -R"
}
```

### Suggestions

**[SUGGESTION] cmd/limactl/shell.go:438 — Use `path` instead of `filepath` for guest directories** [Gemini 3.1 Pro]

`filepath.Clean` applies OS-specific path logic. On Windows, it uses backslashes for paths destined for the Linux guest. Use `path.Clean`/`path.Dir` for guest-side paths.

**[SUGGESTION] cmd/limactl/shell.go:491-504 — Temp directory created before it's needed** [Claude Opus 4.6]

The temp dir for diff viewing is created unconditionally, even when the user picks "Yes" or "No." Move the creation into the `case 2` block.

**[SUGGESTION] cmd/limactl/shell.go:674-700 — Redundant rsync guest availability check** [Claude Opus 4.6]

`getRsyncStats` creates a new rsync tool via `copytool.New`, which SSHs into the guest to check rsync availability — already verified when the primary tool was created. Pass the existing tool or skip the check.

**[SUGGESTION] cmd/limactl/shell.go:720-744 — `parseRsyncStats` could miscount verbose output lines** [Claude Opus 4.6]

`getRsyncStats` creates its rsync tool with `Verbose: true`, adding `-v` output. If a filename is 12+ chars and starts with `<`/`>`/`c` with `+` at index 2, it gets falsely counted. Either remove `Verbose: true` (the `--itemize-changes` flag already produces the needed output) or validate position 1 as a valid file-type indicator.

## Executive Summary

The `limactl shell --sync` feature syncs a host directory to a guest VM via rsync, runs a shell command, then optionally syncs changes back with a user prompt. The core happy path works, but the implementation has two correctness bugs where `destRsyncDir` is never set (combining `--sync` with `--workdir`, or running on WSL2 without mounts), an overly broad guest cleanup that deletes more than what was synced, and a pager that renders color escapes as raw text.

## Findings

### CRITICAL

No validation prevents combining `--sync` and `--workdir`.

Fix: Add a mutual-exclusion check after both flags are parsed:
```go
if syncHostWorkdir && workDir != "" {
    return errors.New("cannot use --sync and --workdir together")
}
```

**[CRITICAL] cmd/limactl/shell.go:242,253 — `--sync` on WSL2 instances without mounts silently breaks**

```go
if syncHostWorkdir {
    destRsyncDir = *inst.Config.User.Home + hostCurrentDir
}
```

### IMPORTANT

**[IMPORTANT] cmd/limactl/shell.go:438-443,460 — Guest cleanup deletes more than the synced directory**

Problem: `dirForCleanup` is computed as `<guestHome>/<parts[1]>`, where `parts[1]` is the first path component after root (e.g., `Users` on macOS, `home` on Linux). For a host path like `/Users/jan/projects/myapp`, the cleanup removes the entire `/home/guestuser/Users/` tree via `rm -rf`, not just the specific `/home/guestuser/Users/jan/projects/myapp/` directory that was created.

If two concurrent `--sync` sessions use directories under the same top-level component (e.g., `/Users/jan/project-a` and `/Users/jan/project-b`), the first cleanup destroys the second session's data.

Fix: Clean up `destRsyncDir` directly instead of the top-level component:
```go
dirForCleanup := shellescape.Quote(destRsyncDir)
```
If the intent is to avoid leaving intermediate directories, use a separate mechanism (e.g., `find` empty dirs).

**[IMPORTANT] cmd/limactl/shell.go:532-539 — Pager displays raw ANSI escape codes**

Problem: The diff command uses `--color=always` (line 532) and the rsync stats headers use ANSI escape codes (lines 81-82). The default pager is `less` (line 536), which by default does not interpret ANSI color escapes. Users see `\033[0;90m--- rsync dry-run statistics ---\033[0m` as literal text.

Fix: Pass `-R` (raw control chars) when using the default pager:
```go
if pager == "" {
    pager = "less -R"
}
```

**[IMPORTANT] cmd/limactl/shell.go:589-601 — `executeSSHForRsync` takes `exec.Cmd` by value, relying on fragile slice semantics**

Problem: The function takes `sshCmd exec.Cmd` by value (not pointer), then appends to `sshCmd.Args` at line 590. This works only because `exec.CommandContext` allocates the Args slice with no spare capacity, so `append` always creates a new backing array and the caller's original is unaffected. If Go's `exec.CommandContext` implementation ever changes its allocation strategy, or if a future caller constructs the `exec.Cmd` differently, this function silently corrupts the caller's Args.

The function also has a misleading name (`sshRmCmd` at line 596) — it runs arbitrary commands, not just `rm`.

Fix: Don't append to the copied slice. Build the args list from scratch:
```go
func executeSSHForRsync(ctx context.Context, sshCmd exec.Cmd, sshLocalPort int, sshAddress, command string) error {
    args := append(sshCmd.Args[1:], "-p", strconv.Itoa(sshLocalPort), sshAddress, command)
    cmd := exec.CommandContext(ctx, sshCmd.Path, args...)
    return cmd.Run()
}
```

**[IMPORTANT] pkg/copytool/rsync.go:90-91 — `Command()` mutates receiver state when `opts != nil`**

Problem: `t.Options = opts` permanently replaces the tool's options. Any subsequent call to `Command(ctx, paths, nil)` uses the last non-nil opts, not the options from construction. The `CopyTool` interface's `Command` docstring says opts "completely overrides" but doesn't mention this side effect persists. Today's callers always pass `nil`, so the bug is latent.

Fix: Use the passed opts without mutating the receiver:
```go
activeOpts := t.Options
if opts != nil {
    activeOpts = opts
}
```

## Appendix B: Codex GPT 5.4 Review

### Executive Summary

`limactl shell --sync` is a useful safety feature: it mirrors a host working tree into the VM, runs the shell there, then offers an accept/reject/diff workflow before syncing back. The overall design is sound, but it is not solid as-is: there is one real deletion-scope bug in guest cleanup, and a metadata-sync regression in the "no changes" fast path. The rsync backend refactor also regressed SSH transport handling for this feature.

### Findings

**[CRITICAL] cmd/limactl/shell.go:438 — Guest cleanup deletes a whole top-level subtree, not just this sync session**
Problem: `dirForCleanup` is derived from only `parts[1]`, so a host path like `/Users/jan/proj` turns into guest cleanup of `~/Users`, not `~/Users/jan/proj`. The deferred `rm -rf` at lines 458-462 therefore removes every synced directory under that first path segment, including unrelated prior syncs and concurrent `--sync` sessions. This scope bug was introduced in the original `--sync` feature commit (`02c7dd55`).
Fix: Clean up exactly `destRsyncDir` instead of reconstructing a broader path from `hostCurrentDir`. Add an integration test that runs two sync sessions under the same top-level host segment and verifies one exit does not delete the other guest shadow tree.

**[IMPORTANT] cmd/limactl/shell.go:469 — Permission-only changes are dropped as "No changes detected"**
Problem: `getRsyncStats()` feeds `parseRsyncStats()`, and `askUserForRsyncBack()` returns early when `stats.String() == ""` (lines 473-475). But the parser intentionally ignores itemized lines like `.f...p... file`, which are real rsync changes for mode/permission updates. I verified locally that a chmod-only difference produces `.f...p... f`. In that case Lima skips the prompt and skips `rsyncBack()`, so guest-side permission changes are silently lost. This regression was introduced by the stats/diff work (`495a472f`) and preserved in the March follow-up (`3674e55f`).
Fix: Treat any itemized rsync line that is not just the summary/header noise as a real change, or at minimum count metadata-only updates separately and do not short-circuit sync-back on them. Add tests for chmod-only and timestamp-only changes, and remove the current unit-test expectation that `.f...p.....` means "no changes".

**[IMPORTANT] pkg/copytool/rsync.go:59 — The rsync backend no longer honors Lima's configured SSH transport**
Problem: `checkRsyncOnGuest()` hardcodes `exec.CommandContext(ctx, "ssh")` (lines 72-80), and `Command()` hardcodes `ssh` again in the rsync `-e` string (lines 120-136). That bypasses the `SSH` env override advertised by `limactl shell`, drops any wrapper args returned by `sshutil.NewSSHExe()`, and also concatenates SSH options without shell-escaping. The pre-refactor implementation used the actual resolved SSH command and escaped each token; the `copytool` refactor (`0cd036a6`) regressed this. Result: `limactl shell --sync` can fail in environments where plain `limactl shell` works, especially with custom SSH commands/config wrappers.
Fix: Use `sshutil.NewSSHExe()` end-to-end in both guest availability checks and rsync command construction. Build the `-e` command from `sshExe.Exe` plus `sshExe.Args` plus SSH opts, shell-escaping each token. Add tests that set `SSH='ssh -F <temp-config>'` or a wrapper script and verify `--sync` still works.

### Testing Assessment

Current coverage is not sufficient for this feature.

I ran `go test ./cmd/limactl -run 'Test(ParseRsyncStats|RsyncStatsString)'`, and those unit tests pass. I did not run the BATS suite because it boots real VMs.

Specific gaps:
- No test covers cleanup scope in the guest shadow tree. There is nothing that would catch the current `rm -rf ~/Users` behavior.
- No test covers permission-only or other metadata-only guest changes syncing back to the host.
- No test exercises the interactive accept/reject/view flow in a real TTY. The existing BATS tests all use `--yes`, so the path that already required follow-up fixes in PRs #4569 and #4696 is still effectively untested.
- No test covers custom SSH transport via the `SSH` environment variable, or Windows-specific rsync transport behavior.
- The unit test in cmd/limactl/shell_test.go:28 currently codifies the wrong behavior by asserting `.f...p..... existing.txt` is "no changes".

### Documentation Assessment

The AI docs in website/content/en/docs/examples/ai.md describe the happy path, but they do not mention host-side dependencies for the review path (`diff`, pager) or any limitations around custom SSH transports. After the fixes above, the docs should also be updated to match actual behavior around metadata changes and sync-session cleanup semantics.

### Executive Summary

The `--sync` functionality provides a convenient and interactive way to temporarily share host directories with the guest and sync them back upon exit. While the user prompt, diff integration, and rsync wrapper are well put together, the feature contains a critical flaw that risks deleting unrelated synced projects in the guest. Additionally, there are important stability issues regarding shared state mutation, silent executable failures, and version-specific `rsync` behavior with spaces.

### Findings

**[CRITICAL] cmd/limactl/shell.go:438-444, 460 — Data loss via overly aggressive directory cleanup**
* Problem: To construct the cleanup path `dirForCleanup`, the code splits `hostCurrentDir` by the path separator and extracts the top-level directory (`parts[1]`, e.g., `"Users"` from `/Users/jan/repo`). It then executes `rm -rf /home/lima/Users` on the guest. This completely wipes out *all* projects the user may have synced from their `Users` directory, rather than just the specific active project.
* Fix: Target the specific synced directory for cleanup. Change `dirForCleanup := shellescape.Quote(fmt.Sprintf("%s/", *inst.Config.User.Home) + parts[1])` to simply use the already calculated destination directory:
  ```go
  dirForCleanup := shellescape.Quote(destRsyncDir)
  ```
  *(Note: Be sure not to use a mutated `destRsyncDir` if it was already quoted earlier for older rsync versions to avoid double-quoting).*

**[IMPORTANT] pkg/copytool/rsync.go:91-93 — `CopyTool.Command()` modifies options persistently**
* Problem: When `opts != nil` is passed into `Command`, it assigns `t.Options = opts`. Because the `copytool.CopyTool` instance can be reused across different calls (like in `askUserForRsyncBack` which reuses the instance from `shellAction`), this mutates shared state permanently.
* Fix: Merge the options into a local variable without mutating the struct:
  ```go
  currentOpts := t.Options
  if opts != nil {
      currentOpts = opts
  }
  ```

**[IMPORTANT] cmd/limactl/shell.go:572-578 — Silent failure if `diff` executable is missing**
* Problem: When the user selects "View the changed contents", the code executes the `diff` command. The error handling exclusively checks for `*exec.ExitError`. If the `diff` binary does not exist, `diffCmd.Run()` returns an `*exec.Error`, which doesn't match, and the error is silently ignored.
* Fix: Add a fallback to catch executable startup errors.

**[IMPORTANT] cmd/limactl/shell.go:590-593 — Unsafe slice mutation in `executeSSHForRsync`**
* Problem: `sshCmd` is passed by value, but its internal `Args` slice is modified via `append(sshCmd.Args, ... , command)`. Because slices share backing arrays, appending to it can silently overwrite memory in the caller's backing array if capacity allows.
* Fix: Explicitly copy the arguments to a new slice.

**[IMPORTANT] cmd/limactl/shell.go:363-366 — Path spaces will break with `rsync` versions 3.0.0 to 3.2.3**
* Problem: The code explicitly quotes `destRsyncDir` to support paths with spaces on older `rsync` versions (`< 3.0.0`). However, `rsync` did not make `--protect-args` the default until version 3.2.4. Users running rsync 3.0.0 through 3.2.3 will encounter failures if `syncDirVal` contains spaces.
* Fix: Change the version check to `< 3.2.4`, or uniformly append `--protect-args` for rsync >= 3.0.0.

**[SUGGESTION] cmd/limactl/shell.go:438, 720 — Use `path` instead of `filepath` for guest directories**
* Problem: `filepath.Clean` and `filepath.Dir` are used on `hostCurrentDir`. Because `hostCurrentDir` is converted to a WSL path on Windows, `filepath` incorrectly applies Windows path logic to a Linux-style path.
* Fix: Use the `path` package for paths destined for the guest.

### Testing Assessment

The BATS integration tests provide solid coverage of the happy paths. However, test coverage is missing for:
1. Interactive edge cases (missing diff/pager binaries).
2. Empty diff state (no files modified).
3. Data isolation regression test (concurrent sync sessions).

### Documentation Assessment

The `--sync` flag help text lacks a critical behavioral warning: changes made on the host during the active shell session will be overwritten if the user accepts the sync-back. A note about this two-way sync risk and its reliance on `rsync --delete` should be added. The inline documentation for `parseRsyncStats` is excellent.
