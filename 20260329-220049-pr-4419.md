## Executive Summary

This PR implements mount validation for MCP host paths, filling a long-standing TODO in `TranslateHostPath`. The approach — `filepath.Rel` + `filepath.IsLocal` with longest-match selection — is sound and well-tested. The main gap is that mount locations are matched lexically without symlink resolution, so paths like `/private/tmp/foo` fail to match a `/tmp` mount on macOS. The BATS test infrastructure needs minor cleanup resilience improvements.

## Important Issues

1. **Symlink aliases of mounted directories are rejected** — `pkg/mcp/toolset/toolset.go:129-131` [Codex GPT 5.4]

```go
location := filepath.Clean(mount.Location)
rel, err := filepath.Rel(location, cleanPath)
if err == nil && filepath.IsLocal(rel) {
```

The new validation matches `hostPath` against `mount.Location` after `filepath.Clean`, but `Clean` does not resolve symlinks. On macOS, `/tmp` is a symlink to `/private/tmp` and `/var` to `/private/var`. If a mount is configured with location `/tmp` (as Lima's defaults store it), a host path `/private/tmp/foo` will not match — `filepath.Rel("/tmp", "/private/tmp/foo")` yields `"../../private/tmp/foo"`, which `filepath.IsLocal` rejects. The codebase already handles this exact problem for inotify watchers: `pkg/hostagent/inotify.go:75` calls `filepath.EvalSymlinks(m.Location)` and maintains a symlink-to-original mapping at lines 79–81. Lima's defaults processing (`pkg/limayaml/defaults.go:682`) only calls `filepath.Clean`, not `EvalSymlinks`, so mount locations retain their symlinked form.

Fix: resolve symlinks for both the mount location and the host path's deepest existing ancestor. Either precompute resolved mount locations in `RegisterInstance` (preferred — avoids per-call `EvalSymlinks` overhead), or reuse the alias-map approach from `inotify.go:95-108`.

2. **Shared BATS instance configuration change is under-documented** — `hack/bats/helpers/limactl.bash:31` [Claude Opus 4.6]

```bash
bats)  limactl start --yes --name "$instance" --mount "$HOME:w" template:default 3>&- 4>&- ;;
```

The `bats` instance is shared across `copy.bats`, `path.bats`, `preserve-env.bats`, and `mcp.bats`. Adding `--mount "$HOME:w"` changes the instance configuration for all of them. The `Glob` and `SearchFileContent` tools default to `os.Getwd()` (`pkg/mcp/toolset/filesystem.go:119`), so MCP tests implicitly depend on the repo being checked out under `$HOME`. If the repo lives elsewhere, those tests will fail with "path not under any mounted directory."

Fix: add a comment explaining the mount is required for MCP host-path validation, and note the `$HOME`-containment assumption.

3. **BATS read-only directory test lacks resilient cleanup** — `hack/bats/tests/mcp.bats:261-274` [Claude Opus 4.6]

```bash
mkdir -p "$HOME/tmp_mcp_ro"
chmod 444 "$HOME/tmp_mcp_ro"
tools_call write_file '{"path":"'"$HOME/tmp_mcp_ro/tmp"'","content":"tmp"}'
# ...
chmod 755 "$HOME/tmp_mcp_ro"   # skipped on test failure
rm -rf "$HOME/tmp_mcp_ro"      # skipped on test failure
```

If this test fails at any assertion, the `chmod` and `rm` cleanup at lines 272–273 never executes, leaving a read-only `$HOME/tmp_mcp_ro` directory in the user's home. The old tests had defensive cleanup (`chmod -R 777 /tmp/tmp || true`) at the start of the next test.

Fix: add defensive cleanup at the start of the test:
```diff
 @test 'write_file returns an error when the directory is not writable' {
+    chmod 755 "$HOME/tmp_mcp_ro" 2>/dev/null || true
+    rm -rf "$HOME/tmp_mcp_ro"
     mkdir -p "$HOME/tmp_mcp_ro"
```

    2.  On Unix, this check is insufficient. A path like `/../etc/passwd` is a valid absolute path but escapes the intended directory.

    Fix: Use `filepath.IsAbs` for a cross-platform absolute path check.


### Design Observations

#### Concerns

*   **(in-scope) Error vs. Warning Philosophy**: The initial design choice to use warnings for unmounted paths (as seen in commit `45823748`) reflects a misunderstanding of the security boundary. Allowing an AI agent to silently operate on the guest filesystem when it intended to access the host is a significant design flaw. The pivot to hard errors, as discussed in the PR comments, is the correct direction and should be fully implemented by removing all warning logic related to mount validation.

#### Strengths

*   **Centralized Validation**: The introduction of `isMounted` provides a single, central place to validate host paths. This is a good design choice, as it ensures that all filesystem operations (`ReadFile`, `ListDirectory`, etc.) will share the same validation logic, reducing code duplication and the risk of inconsistent behavior.

### Testing Assessment

The test coverage is insufficient and misses several key scenarios:
1.  **Panic on Nil Map**: There are no tests for the warning path, which is why the `nil map` panic at `filesystem.go:50` was not caught.
2.  **Home Directory**: The special case for the user's home directory (`toolset.go:130`) is not tested.
3.  **Path Traversal**: There are no explicit tests for directory traversal attacks (e.g., using `..` to escape a mount point). While the current tests might indirectly cover some cases, they are not explicit.
4.  **Windows Paths**: The tests are written with Unix-style paths and will fail on Windows, as noted by `AkihiroSuda`. Tests should be written to be platform-agnostic, using `filepath` constants and functions. The test failures on Windows confirm that the path logic is not robust.
