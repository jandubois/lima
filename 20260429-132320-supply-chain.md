## Open issues

### Critical — `git clone` / templated self-fetch in CI install steps

- `.github/workflows/test.yml:460` — `git clone https://github.com/abiosoft/colima` followed by `git checkout "$(git tag --sort=-v:refname | head -n1)"`. The checkout target is whatever colima tagged most recently, then `make && sudo make install` runs the resulting tree. Anyone with push access to colima can mint a tag and own this CI host. Pin to a 40-hex SHA, or vendor the build artefact. (wiki §3.6 — git clone in scripts.)
- `.github/workflows/test.yml:515` — `git clone https://github.com/lima-vm/socket_vmnet` then `git checkout v1.2.2` and `sudo make install`. Tag-pinned, not SHA-pinned: a force-push or re-tag at upstream silently changes the bytes installed. Pin to a 40-hex SHA. (wiki §3.6.)
- `.github/workflows/test.yml:640` — `git clone https://github.com/AkihiroSuda/gomodjail` with no `--branch` and no `git checkout`, then `make binaries install`. The default branch is fetched and built. Pin to a 40-hex SHA. (wiki §3.6.)
- `.github/workflows/test.yml:562` — `template: https://raw.githubusercontent.com/lima-vm/lima/${{ matrix.oldver }}/examples/ubuntu-lts.yaml`. The matrix-driven ref is a tag (`v0.x.x`) rather than `master`, but tags are mutable upstream and this workflow consumes the YAML to drive subsequent VM image downloads. Pin the matrix entries to 40-hex commit SHAs, or copy the historical templates into `hack/test-templates/`. (wiki §3.5 — self-fetch from a moving ref.)

### Inline downloads — verifier missing

- `hack/test-templates.sh:479` — `curl -Lso nerdctl-full.tgz "${nerdctl}"` followed by `tar xzf "${PWD}/nerdctl-full.tgz" -C /usr/local`. The URL comes from `limactl info | jq -r .defaultTemplate.containerd.archives[].location`, and the same template entry carries a `.digest` field, but this script never fetches or checks it. Read the matching `.digest` and pipe through `sha256sum -c` before extracting. (wiki §3.5.)

### Advisory — bytes verified downstream

- `hack/cache-common-inc.sh:27` — `download_template_if_needed`: `curl -sSLf "${template}" >"${tmp_yaml}"` fetches a remote YAML template before parsing it with `yq`. The bytes are parsed as data (not executed), but the parsed values then drive `download_to_cache` for VM disk images. A poisoned template can substitute image URLs whose `.digest` is `null`. The CI callers always pass URLs that resolve to lima's own repo at a tag — see the §`test.yml:562` finding for the upstream pin.
- `hack/cache-common-inc.sh:400` — `curl -SL ... -o "${cache_path}/data" "${url}"` downloads VM disk images into a cache. The script writes neither the digest nor a `sha256sum -c` step inline (the comment at line 411 acknowledges that `sha256.digest` is "expected to exist" if the template provided one). Verification happens later in `limactl` when the VM is started, but only when the template has a `digest:` value — the riscv64 example template has `null` for kernel/initrd. Fail closed in this script when the template digest is missing rather than relying on `limactl` semantics.
- `hack/brew-install-version.sh:33` — `curl -s "${RAW}/${SHA}/Formula/..."` fetches a Homebrew formula `.rb` by a 40-hex commit SHA looked up locally from `homebrew-core`. The SHA pin authenticates the URL path, but raw.githubusercontent.com does not cryptographically prove that bytes match the git object — only HTTPS-protected delivery. The downstream `brew install` then verifies bottle digests, so the `.rb` file is the only un-checked surface. Mostly fine; consider a one-line `sha256sum -c` against a stored digest for completeness.

## Retro: dependency types not explicitly covered

The script seeded no retro candidates (no unrecognised manifest filenames). The judgments below surface false-positive shapes worth tightening rather than missing ecosystems.

1. **What was seen** — `cmd/limactl/start.go:71` was flagged as `self_fetch_unpinned`: the URL `https://raw.githubusercontent.com/lima-vm/lima/master/templates/alpine.yaml` appears inside a Cobra `Example` string (CLI help text), not as a download.
   **Why current rules missed it** — Phase 13's self-fetch detector scans every text file for raw URLs at our remotes; it does not exclude application source files where the URL is documentation, not a fetch.
   **Proposed change** — Restrict `self_fetch_unpinned[]`'s `url_in_source` source to build-path file globs (`scripts/**`, `hack/**`, `Dockerfile*`, `Makefile*`, `.github/**`, `.ci/**`, `ci/**`, `packaging/**`, `installer/**`, `setup/**`) — the same allowlist `language_source_fetches` already uses. Skip URLs in `.go`/`.py`/`.ts`/`.rs` source under `cmd/`, `pkg/`, `internal/`, etc. unless that source actually invokes a fetch primitive.

2. **What was seen** — `hack/cache-common-inc.sh:16` and `hack/calculate-cache.sh:79` were flagged as `self_fetch_unpinned` for URLs sitting inside doc-comment example blocks (`# $ download_template_if_needed https://raw.githubusercontent.com/lima-vm/lima/v0.15.1/...`).
   **Why current rules missed it** — The detector matches the URL substring without checking whether the line is inside a comment.
   **Proposed change** — In shell-file URL extraction, skip lines whose first non-whitespace byte is `#`. (Same idea applies to Python/Ruby `#` and `//` for Go/Node — but the shell-only fix already covers these cases.)

3. **What was seen** — Inline-download detector flagged HEAD-only `curl -sIL ... -o /dev/null` invocations as candidate downloads (`hack/cache-common-inc.sh:153`, `hack/cache-common-inc.sh:377`, `hack/inject-cmdline-to-template.sh:47`). These are content-type / size probes; nothing is written.
   **Why current rules missed it** — The "download-like uses" filter accepts any `-o` flag, including `-o /dev/null`, and accepts `-I` (HEAD).
   **Proposed change** — In Phase 2's curl/wget filter, drop candidates that combine `-I`/`--head` with `-o /dev/null` (or with no body-capturing redirect). Add `-o /dev/null` to the already-excluded outputs even without `-I`, since nothing on disk means nothing to verify.

4. **What was seen** — Inline-download detector flagged `curl -sSf http://127.0.0.1:29080` and `:29070` in `hack/test-plain-static-port-forward.sh` as candidate downloads.
   **Why current rules missed it** — The filter matches `curl -s` invocations against any URL; localhost-targeted probes are never supply-chain risks.
   **Proposed change** — Skip URLs whose host is `127.0.0.1`, `localhost`, `[::1]`, or matches `localhost:` followed by a port. These are functional tests, not downloads.

5. **What was seen** — `Makefile:632`/`Makefile:633` and `.github/workflows/test.yml:80` were tagged "(floating)" / "verifier ✗" by the run-from-network detector even though they go through the Go module proxy with concrete versions (`@v1.6.0`) or a `-modfile=./hack/tools/go.mod` lockfile.
   **Why current rules missed it** — The script writes "(floating)" whenever no `@<version>` literal is on the line, regardless of `-modfile=`. And the verifier scan does not look up the sibling `go.sum` to call out that hashing is satisfied.
   **Proposed change** — When `go install` carries `-modfile=<path>`, treat the install as pinned if `<path>` has a sibling `go.sum`. Drop the "(floating)" tag in that case and surface "pinned via `<path>`" in the digest. (Bonus: the same logic could surface "pinned via Go module proxy" when the `@version` is concrete, instead of "verifier ✗".)
