## Executive Summary

This PR adds an experimental Windows guest on QEMU plus its Secure Boot and TPM 2.0 prerequisites. Five fixable issues stand out: a 1.5-second sleep races the Windows installer boot prompt on slow hosts; a single unreadable firmware descriptor aborts the entire descriptor lookup; the UEFI variable store persists across Secure Boot mode changes; the 200√ó3-second per-requirement retry budget under-caps Windows install time relative to the 60-minute outer timeout; and `swtpm` shutdown produces a spurious error on every `Stop`.

The change isolates Windows-specific code paths cleanly: descriptor selection lives in `pkg/driver/qemu/firmware.go`, Windows cidata sits beside the existing Linux template in `pkg/cidata`, and the schema additions (`Firmware.SecureBoot`, `Firmware.TPM2`, `GuestDrivers`) carry validation. Three independent reviewers (Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro) ran the same prompt; the boot-key race was raised by all three, lending strong consensus to the Critical severity. Suggestions cluster around polishing the descriptor-matching heuristics, routing virtio-win ISOs through the existing download pipeline, and aligning shell-script and template defaults across host OSes.

## Critical Issues

C1. **Installer boot-key sleeps a fixed 1.5 s before the UEFI prompt, then marks done before the keypress lands** ‚Äî `pkg/driver/qemu/qemu_driver.go:450-468` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

```go
if err := waitFileExists(qmpSockPath, 5*time.Second); err != nil { ... return }
// Wait for the prompt.
time.Sleep(1500 * time.Millisecond)
if _, err := sendHmpCommand(cfg, "sendkey", "ret"); err != nil { ... return }
if err := os.WriteFile(markerPath, []byte("sent\n"), 0o644); err != nil { ... }
```

A QEMU+OVMF guest typically reaches the "Press any key to boot from CD or DVD" prompt 5‚Äì15 s after start, depending on host load and firmware path. A 1.5 s sleep after the QMP socket appears fires the keypress before the firmware reaches that screen, so the installer hangs at the prompt forever. The marker is written immediately after `sendkey` succeeds ‚Äî not after a guest-observable milestone ‚Äî so the next `limactl start` skips the workaround and reinstall is also stuck.

Fix: poll until `setup.done` or hold off on writing the marker until first reboot, and send `sendkey ret` several times across ~30 s (e.g., every 3 s for 10 iterations) so the keypress survives a slow firmware path.

C2. **An unreadable or malformed firmware descriptor aborts the entire descriptor search** ‚Äî `pkg/driver/qemu/firmware.go:146-174` [Claude Opus 4.7]

```go
for _, descriptorPath := range descriptorPaths {
    b, err := os.ReadFile(descriptorPath)
    if err != nil {
        return qemuFirmware{}, err
    }
    var descriptor qemuFirmwareDescriptor
    if err = json.Unmarshal(b, &descriptor); err != nil {
        return qemuFirmware{}, err
    }
}
```

`getFirmwareFromDescriptorFiles` walks every descriptor returned by `qemuFirmwareDescriptorPaths` plus user-supplied `firmware.descriptors`. A single permission-denied file or malformed JSON anywhere on the search path bubbles up as a hard error, masking every valid descriptor in the same and lower-priority directories. Distribution package updates (Fedora `edk2-ovmf`, Arch `edk2-ovmf`) or a user-supplied stale path are sufficient to break Lima's firmware lookup on the host. With `secureBoot=true`, the caller returns the error directly ‚Äî no fallback path runs.

Fix: log and `continue` per-file on read/parse failure; only return an error when every candidate fails.

```diff
-    b, err := os.ReadFile(descriptorPath)
-    if err != nil {
-        return qemuFirmware{}, err
-    }
+    b, err := os.ReadFile(descriptorPath)
+    if err != nil {
+        logrus.WithError(err).Warnf("skipping unreadable firmware descriptor %q", descriptorPath)
+        continue
+    }
```

## Important Issues

I1. **Per-requirement retry budget caps Windows install at 10 minutes** ‚Äî `pkg/hostagent/requirements.go:20-46` [Codex GPT 5.5]

```go
const (
    retries       = 200
    sleepDuration = 3 * time.Second
)
```

`pkg/instance/start.go:43-44` extends the outer watch timeout to 60 minutes for Windows guests, but `waitForRequirement` still gives up after 200 √ó 3 s = 10 minutes per requirement. The first essential Windows requirement is SSH connectivity; a Windows install that needs more than 10 minutes (slow disk, busy host, slipstreaming) fails that probe permanently and `errors.Join` carries the failure to the end of `waitForRequirements`, even when the later requirements succeed. `watchHostAgentEvents` then reports `degraded` and `limactl start` returns non-zero. The 60-minute outer timeout is dead code as long as the inner budget is shorter.

Fix: make per-requirement retries context-aware (loop until the surrounding `ctx` deadline) or expose a Windows-specific retry budget that matches the 60-minute outer envelope.

I2. **`ensureFirmwareFileIfMissing` preserves stale UEFI vars across Secure Boot mode changes** ‚Äî `pkg/driver/qemu/firmware.go:295-321,349-357` [Codex GPT 5.5]

```go
codePath = firmware.instanceCodePath(instanceDir)
varsPath = firmware.instanceVarsPath(instanceDir)
...
if err = ensureFirmwareFile(codePath, firmware.Code); err != nil { ... }
if err = ensureFirmwareFileIfMissing(varsPath, *firmware.Vars); err != nil { ... }
```

`instanceVarsPath` keys on file format only ‚Äî `QemuEfiVarsQCOW2` vs `QemuEfiVarsFD`. Toggling `firmware.secureBoot`, `firmware.preEnrollSecureBootKeys`, or swapping a 2 MB descriptor for a 4 MB one keeps the same vars path, so the new firmware code pairs with the old NVRAM. The user thinks they enabled enrolled keys; the guest boots without them, silently. Preserving NVRAM across runs is correct for the same descriptor ‚Äî losing enrolled keys would break the install ‚Äî but switching descriptors must invalidate stale NVRAM.

Fix: hash or record the descriptor identity beside the vars file. When the recorded identity disagrees with the selected descriptor, refuse to start (with an actionable reset hint) rather than silently mismatch firmware code and NVRAM.

I3. **`killSwtpm` returns `signal: killed` as an error on every normal shutdown** ‚Äî `pkg/driver/qemu/qemu_driver.go:731-751` [Gemini 3.1 Pro]

```go
if cmd.ProcessState == nil {
    if err := cmd.Process.Kill(); err != nil && !errors.Is(err, os.ErrProcessDone) {
        return fmt.Errorf("failed to kill %s: %w", name, err)
    }
}
if waitCh == nil { return nil }
if err, ok := <-waitCh; ok && err != nil {
    return fmt.Errorf("%s exited with error: %w", name, err)
}
```

`shutdownQEMU` calls `errors.Join(qWaitErr, l.killVhosts(), l.killSwtpm())` on multiple branches. When `cmd.Wait` returns after a `Kill`, Go reports `signal: killed` as a non-nil error. `killProcess` propagates that into `Stop`'s return value on every shutdown of a TPM-enabled instance. The user sees a spurious error and `limactl stop` exits non-zero.

Fix: track whether the caller forced the kill; suppress the resulting `signal: killed` from the wait channel in that case.

```diff
+    killed := false
     if cmd.ProcessState == nil {
         if err := cmd.Process.Kill(); ...
+        killed = true
     }
-    if err, ok := <-waitCh; ok && err != nil {
+    if err, ok := <-waitCh; ok && err != nil && !killed {
         return fmt.Errorf("%s exited with error: %w", name, err)
     }
```

I4. **`GuestDrivers.Images` accepts the standard `FileWithVMType` schema but skips the download pipeline; `windows.yaml` pins a Linux-only path** ‚Äî `pkg/cidata/cidata.go:571-583,609-623`, `pkg/limayaml/defaults.go:~325-340`, `templates/windows.yaml:30-39` [Codex GPT 5.5, Gemini 3.1 Pro]

```go
for _, image := range instConfig.GuestDrivers.Images {
    ...
    return localpathutil.Expand(image.Location)
}
...
isoFile, err := os.Open(virtioDriverISO)
```

The schema is `FileWithVMType` (shared with `Firmware.Images` and instance `Images`), which validates URL + digest forms via `validateFileObject`. `guestDriverISO` calls `localpathutil.Expand` and then `os.Open`, so any non-`file://` location fails at runtime even though validation accepted it. Compounding this, `defaults.go` injects `/usr/share/virtio-win/virtio-win.iso` only on Linux hosts, and `templates/windows.yaml` repeats the same hardcoded path, so the template fails out-of-the-box on macOS.

Fix: route `GuestDrivers.Images` through `fileutils.DownloadFile` (mirroring the `Firmware.Images` path in `pkg/driver/qemu/qemu.go`), or fail validation when `location` is not a local path. Drop the explicit `guestDrivers.images` block from `templates/windows.yaml`; let the Linux default carry it.

I5. **`html.EscapeString` writes `&#39;` into XML where `&apos;` is the canonical form** ‚Äî `pkg/cidata/template.go:285-291` [Claude Opus 4.7]

```go
args.WindowsUser = html.EscapeString(args.WindowsUser)
args.WindowsPassword = html.EscapeString(args.WindowsPassword)
args.WindowsImageIndex = html.EscapeString(args.WindowsImageIndex)
for i := range args.WindowsDriverPaths {
    args.WindowsDriverPaths[i] = html.EscapeString(args.WindowsDriverPaths[i])
}
```

`autounattend.xml` is XML, not HTML; the repo context flags this pattern explicitly. `html.EscapeString` happens to escape the five chars XML cares about, so the output parses, but the import is misleading and `&#39;` for `'` (instead of `&apos;`) reads as an oddity to anyone auditing the template. The intent shows up wrong in code search.

Fix: switch to `encoding/xml`'s `xml.EscapeText` (writes to an `io.Writer`) or a 6-line helper that replaces only `<`, `>`, `&`, `'`, `"`. The runtime behavior is equivalent for current inputs; the new code documents intent.

I6. **`GenerateCloudConfig` for Windows silently drops `os.RemoveAll` errors** ‚Äî `pkg/cidata/cidata.go:366-370` [Claude Opus 4.7]

```go
if *instConfig.OS == limatype.WINDOWS {
    os.RemoveAll(filepath.Join(instDir, filenames.CloudConfig)) // delete existing
    return nil
}
```

The function returns nil regardless of whether the stale cloud-config was actually removed. A locked file, restrictive directory permissions, or an antivirus quarantine on a re-used instance directory leaves the old config in place while the caller proceeds, paired with the wrong cidata for the new guest.

Fix: propagate the error.

```diff
-    os.RemoveAll(filepath.Join(instDir, filenames.CloudConfig)) // delete existing
-    return nil
+    if err := os.RemoveAll(filepath.Join(instDir, filenames.CloudConfig)); err != nil {
+        return err
+    }
+    return nil
```

## Suggestions

S1. **`firmwareTargetsQ35` matches any machine containing `"q35"` as a substring** ‚Äî `pkg/driver/qemu/firmware.go:270-285` [Claude Opus 4.7]

```go
if strings.HasPrefix(machine, "pc-q35-") || strings.Contains(machine, "q35") {
    return true
}
```

The `HasPrefix("pc-q35-")` arm already covers every QEMU-shipped Q35 machine name (`pc-q35-7.2`, `pc-q35-8.0`, ‚Ä¶). The `Contains` arm matches any future or vendor name with `q35` anywhere in the string. Replace with `machine == "q35"` for the bare alias.

S2. **`firmwareUses4M` substring check matches `"4m"` anywhere in the filename** ‚Äî `pkg/driver/qemu/firmware.go:287-293` [Claude Opus 4.7]

A descriptor whose code path contains "4m" anywhere (e.g., a vendor name) is selected over a non-4M alternative. Anchor the match to the conventional suffix: `_4M.`, `-4m.`, or filename suffix.

S3. **Descriptor-not-found error inlines every candidate path** ‚Äî `pkg/driver/qemu/firmware.go:173` [Claude Opus 4.7]

With dozens of system descriptors, `fmt.Errorf("... from %v ...", descriptorPaths, ...)` prints a multi-line list. Report a count plus an actionable hint ("install distribution OVMF packages or set `firmware.descriptors`") instead.

S4. **`qemuFirmwareDescriptorDirs` adds Unix-only paths on Windows hosts** ‚Äî `pkg/driver/qemu/firmware.go:116-144` [Claude Opus 4.7]

`/etc/qemu/firmware` and `/usr/share/qemu/firmware` are appended unconditionally; on Windows the `localDir != "/usr"` check is a no-op and the path itself is meaningless. Gate the Unix entries on `runtime.GOOS != "windows"`.

S5. **Boot-key marker uses `0o644` while adjacent secret files use `0o600`** ‚Äî `pkg/driver/qemu/qemu_driver.go:463` [Claude Opus 4.7]

The marker carries no secret, but `windowsUserPassword` writes its file with `0o600`. Use `0o600` for both to keep the instance directory's mode story uniform.

S6. **`generateAlphaNumericPassword` may sample passwords lacking required character classes** ‚Äî `pkg/cidata/template.go:320-330` [Codex GPT 5.5]

24 chars from `[A-Za-z0-9]` make a missing-class outcome statistically rare but possible. If the host or guest enforces Windows local-account complexity policy (Server SKUs, custom GPO), unattend may fail. Guarantee at least one upper, one lower, and one digit, then shuffle.

S7. **`templates/windows.yaml` declares `minimumLimaVersion: 2.0.0`** ‚Äî `templates/windows.yaml:1-6` [Codex GPT 5.5]

The template depends on the new `Firmware.SecureBoot`, `Firmware.TPM2`, and `GuestDrivers` fields that don't exist in 2.0.0. Bump to the first release that contains these schema and runtime changes.

S8. **QEMU firmware descriptors can carry relative paths** ‚Äî `pkg/driver/qemu/firmware.go:243-260` [Gemini 3.1 Pro]

Per `interop/firmware.json`, `filename` may be relative to the descriptor's directory. The current code passes the raw path to `osutil.FileExists`, rejecting valid system descriptors with relative paths. Resolve relative paths against `filepath.Dir(descriptorPath)` before the existence check.

S9. **`windowsVirtioDriverNames` rebuilds the slice on every call** ‚Äî `pkg/cidata/cidata.go:639-641` [Claude Opus 4.7]

```go
func windowsVirtioDriverNames() []string {
    return []string{"viostor", "vioscsi", "NetKVM"}
}
```

Promote to a package-level `var`. Callers iterate, never mutate.

S10. **New exported APIs lack godoc** ‚Äî `pkg/cidata/template.go:247,266`; `pkg/limatype/lima_yaml.go:124-131` [Claude Opus 4.7]

`TemplateArgsForWindows`, `ExecuteTemplateWindowsCIDataISO`, and the `WindowsUser` / `WindowsPassword` / `WindowsImageIndex` / `WindowsDriverPaths` fields are exported without docstrings. Add short godoc lines that start with the symbol name (Go convention).

S11. **`firmwareTargetsQ35` rejects every non-x86_64 architecture** ‚Äî `pkg/driver/qemu/firmware.go:270-285` [Gemini 3.1 Pro]

The PR description acknowledges that "non-x86_64 architectures may be incomplete," so this is not a regression, but the function name promises Q35-specific behavior and silently filters out `aarch64`/`virt` callers. Rename to `firmwareTargetsArchAndMachine` and parameterize, or add a comment that the function is x86_64-only.

S12. **Test fixture binds two variables to the same path** ‚Äî `pkg/driver/qemu/qemu_test.go:106-110` [Claude Opus 4.7]

```go
secureVarsWithoutKeys := filepath.Join(dir, "OVMF_VARS_4M.qcow2")
plainVars := filepath.Join(dir, "OVMF_VARS_4M.qcow2")
```

The two variables refer to the same file; the test relies on the second `os.WriteFile` overwriting the first. Pick distinct filenames or consolidate.

## Design Observations

### Concerns

- **Windows setup state is inferred from host-side timers and marker files** (in-scope) [Codex GPT 5.5]. I1 (retry budget) and C1 (boot-key marker) are symptoms of the same model. A small state machine tied to guest-observable milestones ‚Äî `setup.done` on the cidata `oem/` share, `sshd` reaching `Running`, OpenSSH cap install completing ‚Äî would let the host wait on real progress instead of fixed budgets. The PR already writes `setup.done` from `windows-setup.ps1`; build the rest of the state on top of it.
- **Hand-rolled QEMU descriptor parser covers a partial slice of `interop/firmware.json`** (future) [Claude Opus 4.7]. Multiple flash variants per target, `stateless` mode, and additional feature flags are absent. Lima only needs to *find* a usable file pair, so the partial implementation is correct for now; document the limited subset in a docstring on `qemuFirmwareDescriptor` with a link to the schema.
- **`GuestDrivers` is generically named but currently serves only Windows virtio drivers** (future) [Claude Opus 4.7]. If future guests reuse the field, the name fits; if not, `WindowsDrivers` reads truer. Pick one and document the contract.

### Strengths

- Descriptor-based firmware selection generalises beyond Windows; Linux guests with Secure Boot also benefit, and the legacy `Firmware.Images` fallback stays intact [Claude Opus 4.7].
- `qemuFirmwareDescriptorPaths` correctly implements QEMU's zero-byte masking rule, letting users override system descriptors via a same-named empty file in a more-specific directory [Gemini 3.1 Pro].
- `ensureFirmwareFileIfMissing` preserves per-instance NVRAM across restarts so enrolled keys survive ‚Äî verified by the unit test [Claude Opus 4.7]. (Pair with I2's caveat for cross-mode persistence.)
- Mounting `cidata.iso` over IDE plus `$WinPEDriver$` driver injection avoids slipstreaming a custom Windows ISO [Gemini 3.1 Pro].
- Validation in `pkg/limayaml/validate.go` ships with the feature: TPM/Secure Boot/Legacy BIOS combinations and architecture / vmType gating are enforced [Claude Opus 4.7].

## Review Process Notes

### Repo context updates

- **ADD:** When reviewing a new `pkg/driver/qemu` change that calls `cmd.Process.Kill()` followed by `cmd.Wait()` (directly or via a channel), check that the calling code distinguishes "the wait error followed a deliberate kill" from "the process exited on its own with an error." Otherwise `Wait` returns `signal: killed` on every clean shutdown and propagates through `errors.Join` to user-facing return values. The pattern appears in `killProcess`, but any new subprocess (virtiofsd, swtpm, sidecar VMMs) using the same Kill+Wait shape carries the same risk. Flag findings only when the caller-facing return value is observable (a `Stop()` method, an HTTP/CLI error path).

- **ADD:** When a Lima driver allocates per-instance state files keyed by content type (e.g., `pkg/driver/qemu/firmware.go`'s `QemuEfiVarsFD` / `QemuEfiVarsQCOW2`), check whether the path key encodes every dimension that affects compatibility. Switching between `secureBoot=false` and `secureBoot=true`, or between `preEnrollSecureBootKeys=false` and `=true`, currently produces the same per-instance vars path. Any new firmware/NVRAM/state file added to `pkg/driver/qemu` (or any other driver) must either include the relevant feature flags in its filename or carry a sidecar identity check that refuses to start on mismatch.

```go
		// ...
		return localpathutil.Expand(image.Location)
	}
```

While the PR description notes that non-Linux host support "may be incomplete," the `guestDrivers.images` block shares the `FileWithVMType` schema with standard `images`, creating the expectation that HTTP URLs will be automatically fetched via `fileutils.DownloadFile`. Bypassing the downloader and directly calling `os.Open` means URLs fail entirely. Furthermore, because `defaults.go` hardcodes the Linux-specific `/usr/share/virtio-win/virtio-win.iso` path, the `windows.yaml` template fails out-of-the-box on macOS unless the user manually downloads the ISO and points to it.

I3. **`swtpm` shutdown pollutes logs with false-positive errors** ‚Äî `pkg/driver/qemu/qemu_driver.go`

```go
	if cmd.ProcessState == nil {
		if err := cmd.Process.Kill(); err != nil && !errors.Is(err, os.ErrProcessDone) {
			return fmt.Errorf("failed to kill %s: %w", name, err)
		}
	}
	if waitCh == nil {
		return nil
	}
	if err, ok := <-waitCh; ok && err != nil {
		return fmt.Errorf("%s exited with error: %w", name, err)
	}
```

When `shutdownQEMU` calls `l.killSwtpm()`, it terminates the `swtpm` subprocess. `killProcess` then waits on `waitCh`. Because `cmd.Wait()` yields an exit error (`signal: killed`) when `Kill()` is invoked, `killProcess` bubbles this up. This forces `Stop()` to return an error on every normal shutdown of a TPM-enabled instance where `swtpm` didn't exit instantly.

**Fix:** Suppress the error from `waitCh` if `Kill()` was explicitly triggered.
```diff
 	if cmd == nil {
 		return nil
 	}
+	killed := false
 	if cmd.ProcessState == nil {
 		if err := cmd.Process.Kill(); err != nil && !errors.Is(err, os.ErrProcessDone) {
 			return fmt.Errorf("failed to kill %s: %w", name, err)
 		}
+		killed = true
 	}
 	if waitCh == nil {
 		return nil
 	}
 	if err, ok := <-waitCh; ok && err != nil {
+		if killed {
+			return nil
+		}
 		return fmt.Errorf("%s exited with error: %w", name, err)
 	}
```

S1. **QEMU firmware descriptors support relative paths** ‚Äî `pkg/driver/qemu/firmware.go`

```go
	code := descriptor.Mapping.Executable.Filename
	vars := descriptor.Mapping.NVRAMTemplate.Filename
// ...
	if !osutil.FileExists(code) || !osutil.FileExists(vars) {
		return qemuFirmware{}, false
	}
```

Per the QEMU `firmware.json` specification, `filename` values can be relative paths, in which case they resolve relative to the directory containing the descriptor file. Checking the raw path against `limactl`'s working directory will incorrectly reject valid system descriptors that use relative references.

**Fix:** If the extracted path is not absolute (`!filepath.IsAbs(code)`), prefix it with `filepath.Dir(descriptorPath)` before checking `osutil.FileExists`.

### Design Observations

#### Strengths
- **Firmware Descriptor Masking:** The implementation in `qemuFirmwareDescriptorPaths` correctly honors the QEMU specification's zero-byte masking rule, robustly shadowing descriptors from lower-priority system directories.
- **Virtio Driver Injection:** Mounting the `cidata.iso` via the IDE bus is an elegant workaround for Windows PE's lack of in-box VirtIO SCSI drivers. Placing the drivers inside `$WinPEDriver$` or explicitly targeting them via `<DriverPaths>` ensures the installer can dynamically load the storage driver and see the underlying VirtIO disk without requiring a slipstreamed, custom Windows ISO.

### Acknowledged Limitations

- **Non-QEMU and non-Linux host support:** The PR description acknowledges that `windows` template behavior may be incomplete on non-Linux hosts and that non-QEMU drivers aren't supported. I promoted the lack of automatic download for `guestDrivers.images` to finding **I2** because it directly contradicts the expected schema contract for `limatype.FileWithVMType`.
