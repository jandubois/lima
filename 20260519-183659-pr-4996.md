## Executive Summary

This PoC adds `poc-windows-guest/` to boot Windows Server 2025 under QEMU with autounattend-driven install, OpenSSH, and VirtIO-FS host-share. End-to-end functional on Ubuntu 24.04, but `run.sh` deletes `windows.qcow2` on every launch, the README's `tag=hoge` disagrees with `run.sh`'s `tag=lima`, and backgrounded `virtiofsd` orphans across runs. Eight Important issues span lifecycle, error handling, drive-letter portability, and supply chain.

Reviewers: 5 (Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro, Gemma4 31B, Qwen 3.6 27B). Qwen ran twice: MXFP8 produced hallucinated XML; bf16 produced clean findings. Both included below; bf16 is the kept run.

## Important Issues

I1. **`run.sh` deletes the installed `windows.qcow2` on every launch** — `poc-windows-guest/run.sh:7-9` [Codex GPT 5.5, Qwen 3.6 27B]

```bash
# Cleanup
rm -f images/windows.qcow2
rm -f images/win-cidata.iso
```

The README presents `run.sh` as the way to launch the VM. The first two lines unconditionally delete the installed qcow2 disk image, so the second invocation destroys the Windows install (and any data created in it) and forces a full reinstall. The cleanup block belongs in a separate `rebuild` mode, not at the top of the launch path.

Fix: gate the deletion behind a flag (`--rebuild` or similar), or only create the disk when absent.

```diff
-# Cleanup
-rm -f images/windows.qcow2
-rm -f images/win-cidata.iso
+# Build the cidata ISO fresh each run (small, regenerable)
+rm -f images/win-cidata.iso
+
+# Create disk only on first run; preserve installed state otherwise
+if [ ! -f images/windows.qcow2 ]; then
+    qemu-img create -f qcow2 ./images/windows.qcow2 40G
+fi
```

(and remove the unconditional `qemu-img create` further down).

I2. **`virtiofsd` is backgrounded with no readiness check, no PID tracking, and no cleanup trap** — `poc-windows-guest/run.sh:22` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro, Gemma4 31B, Qwen 3.6 27B]

```bash
# Run virtiofsd in background
virtiofsd --shared-dir=./ --socket-path=./virtiofsd.sock --sandbox none &

# Run QEMU
qemu-system-x86_64 \
    ...
```

Three failure modes compound here. (1) QEMU starts immediately after the `&`, so if `virtiofsd` is still binding the socket QEMU's `-chardev socket,...,path=./virtiofsd.sock` can race and fail to connect. (2) When `virtiofsd` itself fails immediately (missing binary, bad arg, stale socket already bound), `set -e` does not catch background-process exits; the script continues into QEMU and prints a confusing "connection refused." (3) When QEMU exits — clean shutdown, Ctrl-C, or crash — `virtiofsd` is never killed and the socket is never removed; subsequent runs race against the orphan or fail with "address already in use."

Fix: store the PID, install an `EXIT` trap, remove stale sockets, and verify readiness before launching QEMU.

```diff
+rm -f ./virtiofsd.sock
+trap 'kill "${VFS_PID:-0}" 2>/dev/null || true; rm -f ./virtiofsd.sock' EXIT
 virtiofsd --shared-dir=./ --socket-path=./virtiofsd.sock --sandbox none &
+VFS_PID=$!
+while [ ! -S ./virtiofsd.sock ]; do kill -0 "$VFS_PID" || exit 1; sleep 0.1; done
```

The existing Lima QEMU driver under `pkg/hostagent` follows this pattern for its vhost sockets — reuse that lifecycle when this graduates from PoC.

I4. **`first_logon.ps1` has no error handling — silent partial-provisioning failures** — `poc-windows-guest/first_logon.ps1:1-40` [Claude Opus 4.7, Codex GPT 5.5, Qwen 3.6 27B]

```powershell
Add-WindowsCapability -Online -Name OpenSSH.Server~~~~0.0.1.0
Start-Service sshd
Set-Service -Name sshd -StartupType Automatic
...
C:\ProgramData\chocolatey\choco.exe install winfsp -y --pre
...
New-Service -Name VirtioFsSvc -BinaryPathName 'E:\viofs\2k25\amd64\virtiofs.exe' -DisplayName VirtioFsSvc -StartupType Automatic
```

PowerShell does not stop on cmdlet failure by default, and native commands (`choco.exe`, `icacls`) are not checked through `$LASTEXITCODE`. If `Add-WindowsCapability` fails with a transient network error pulling FoD from Microsoft, `Start-Service sshd` runs anyway and produces a separate, unrelated error; the VM boots into a half-provisioned state where SSH never works. The same applies to chocolatey install failure (no WinFSP → `New-Service` succeeds but the service binary is missing → `Start-Service VirtioFsSvc` silently fails). The transcript log captures everything but a user has to scroll through hundreds of lines to notice.

Fix: set strict error handling at the top of the script, wrap the body in `try/finally`, and check `$LASTEXITCODE` after each native command.

```diff
+$ErrorActionPreference = 'Stop'
+try {
   # ... existing body ...
+  if ($LASTEXITCODE -ne 0) { throw "choco install winfsp failed: $LASTEXITCODE" }
+} catch { Write-Error $_; exit 1 }
+finally { Stop-Transcript }
```

I5. **`virtiofsd --shared-dir=./` exposes the host ISOs and the live qcow2 to the guest** — `poc-windows-guest/run.sh:22`, `poc-windows-guest/README.md:55` [Claude Opus 4.7, Codex GPT 5.5]

```bash
virtiofsd --shared-dir=./ --socket-path=./virtiofsd.sock --sandbox none &
```

The shared directory is the PoC root, which contains `images/` (the Windows ISO, the virtio-win ISO, and the live `windows.qcow2` that QEMU is using as the guest disk), `autounattend.xml` (with the plaintext password), `first_logon.ps1`, and the `virtiofsd.sock` itself. The README's `dir Z:\` screenshot at line 109-119 confirms — the guest can list and modify its own backing disk. Combined with `--sandbox none` (no chroot/user-namespace isolation), this gives the guest a direct path to corrupt its own VM image and tamper with the installation media. For a PoC this is tolerable, but the default share root should be a dedicated subdirectory.

Fix:

```diff
+mkdir -p ./shared
-virtiofsd --shared-dir=./ --socket-path=./virtiofsd.sock --sandbox none &
+virtiofsd --shared-dir=./shared --socket-path=./virtiofsd.sock --sandbox none &
```

and a note in the README that `--sandbox none` plus a parent-directory share means the guest can tamper with its own VM image.

I6. **Chocolatey bootstrap is `iex`-piped from the internet with no integrity check** — `poc-windows-guest/first_logon.ps1:29` [Claude Opus 4.7]

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force; [System.Net.ServicePointManager]::SecurityProtocol = [System.Net.ServicePointManager]::SecurityProtocol -bor 3072; iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))
```

This is the canonical `iex (DownloadString …)` supply-chain anti-pattern: every VM provisioning run executes whatever code currently lives at `community.chocolatey.org/install.ps1`, as Administrator, with no checksum and no pinned commit. A compromise of the chocolatey CDN (or a man-in-the-middle between the guest and the CDN — the guest does its first network reach through QEMU's user-mode NAT) is full code execution on every Lima Windows guest provisioned from this PoC. WinFSP is the only package needed; downloading the MSI directly from `github.com/winfsp/winfsp` releases and verifying a pinned SHA-256 removes both the chocolatey dependency and the unpinned-fetch risk.

Fix:

```powershell
$winfspUrl    = 'https://github.com/winfsp/winfsp/releases/download/v2.0/winfsp-2.0.23075.msi'
$winfspSha256 = '<sha256-of-the-msi>'
$msi = "$env:TEMP\winfsp.msi"
Invoke-WebRequest -Uri $winfspUrl -OutFile $msi -UseBasicParsing
$got = (Get-FileHash -Algorithm SHA256 $msi).Hash
if ($got -ne $winfspSha256) { throw "winfsp checksum mismatch: $got" }
Start-Process msiexec.exe -Wait -ArgumentList "/i `"$msi`" /qn"
```

I7. **Drive letters E:\ and F:\ are hardcoded throughout — install-time and at runtime** — `poc-windows-guest/autounattend.xml:9-22,134`, `poc-windows-guest/first_logon.ps1:19,36` [Claude Opus 4.7, Gemini 3.1 Pro, Gemma4 31B]

```powershell
$pubkey = Get-Content -Path F:\user.pub
New-Service -Name VirtioFsSvc -BinaryPathName 'E:\viofs\2k25\amd64\virtiofs.exe' -DisplayName VirtioFsSvc -StartupType Automatic
```

Two failure surfaces share one cause. At install time, the answer file driver paths (autounattend.xml:9-22), the cidata `first_logon.ps1` invocation (autounattend.xml:134), and the cidata `user.pub` read (first_logon.ps1:19) all assume `E:` = virtio-win ISO and `F:` = cidata ISO. That mapping comes from the `cdrom0/ide.1 → D:`, `cdrom1/ide.2 → E:`, `cdrom2/ide.3 → F:` Windows convention — not a contract. Reorder the `-device ide-cd,bus=ide.N` lines in `run.sh`, or add another removable device, and every reference silently breaks.

At runtime, `first_logon.ps1:36` registers `VirtioFsSvc` with its `BinaryPathName` on the virtio-win CD-ROM. The service persists across reboots, but its binary lives on removable media — detaching the ISO or any drive-letter reshuffle leaves the service registered with a missing binary.

Fix: copy the service binary to `C:\` before registering, and locate the cidata ISO by volume label (`Get-Volume -FileSystemLabel autounattend`) rather than by drive letter.

```diff
-New-Service -Name VirtioFsSvc -BinaryPathName 'E:\viofs\2k25\amd64\virtiofs.exe' -DisplayName VirtioFsSvc -StartupType Automatic
+Copy-Item -Path 'E:\viofs\2k25\amd64\virtiofs.exe' -Destination 'C:\Windows\System32\virtiofs.exe'
+New-Service -Name VirtioFsSvc -BinaryPathName 'C:\Windows\System32\virtiofs.exe' -DisplayName VirtioFsSvc -StartupType Automatic
```

I8. **`pc-q35-noble` machine type is Ubuntu-noble-specific** — `poc-windows-guest/run.sh:27`, `poc-windows-guest/README.md:67` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

```bash
-machine pc-q35-noble,usb=off,vmport=off,dump-guest-core=off,memory-backend=pc.ram,hpet=off,acpi=on \
```

The `pc-q35-noble` alias ships with Canonical's Ubuntu 24.04 ("noble") QEMU package. On Fedora, Arch, Debian-stable, RHEL, or any non-Ubuntu host, QEMU rejects this with `unsupported machine type` before Windows setup even starts. The README's "Tested on Ubuntu 24.04" note understates the constraint — this is a hard prerequisite, not a "tested on" annotation. The portable name `q35` selects the newest variant available on every distro.

Fix:

```diff
-    -machine pc-q35-noble,usb=off,vmport=off,dump-guest-core=off,memory-backend=pc.ram,hpet=off,acpi=on \
+    -machine q35,usb=off,vmport=off,dump-guest-core=off,memory-backend=pc.ram,hpet=off,acpi=on \
```

Apply the same change in `README.md:67`.

Severity note: Gemini ranked this as Important on portability grounds; Claude and Codex ranked it as a Suggestion since the README explicitly says Ubuntu 24.04 was used. Settling on Important here because the script fails immediately on non-Ubuntu hosts, which is the first thing many readers will try.

S7. **`tmp/win-cidata/` is not cleaned before each `mkisofs` run** — `poc-windows-guest/run.sh:7-12` [Claude Opus 4.7]

```bash
# Cleanup
rm -f images/windows.qcow2
rm -f images/win-cidata.iso

# Build win-cidata.iso
mkdir -p tmp/win-cidata
```

`mkdir -p` does not clear the directory. A previous run that left a renamed `first_logon.ps1.old` (or any other artifact from manual editing) gets packaged into the next ISO, because `mkisofs ... ./tmp/win-cidata/` writes the directory's current contents verbatim. Either `rm -rf tmp/win-cidata` before the `mkdir -p`, or use `mktemp -d` for a fresh staging area each run.

Fix:

```diff
+# Build win-cidata.iso (fresh staging dir each time)
+rm -rf tmp/win-cidata
 mkdir -p tmp/win-cidata
```

## Design Observations

**Concerns**

- **Plaintext credentials in `autounattend.xml`** `(future)` [Claude Opus 4.7] — `poc-windows-guest/autounattend.xml:111,142`. The local-account password and the autologon password both sit in plaintext, intended as a one-shot bootstrap before the public key is installed. Once the SSH key is in place the password becomes the weak link, and the same file ships into the cidata ISO that gets attached to the running VM. For the eventual non-PoC integration, generate a strong random password per provision, or use `<UserAccounts>\AdministratorPassword` with the base64 obfuscation Microsoft documents (still trivially decoded, but less visible than `<Value>limauser</Value>` next to the username `limauser`).

- **`autounattend.xml` and `first_logon.ps1` are pinned to Windows Server 2025** `(future)` [Claude Opus 4.7] — `poc-windows-guest/autounattend.xml:9-22`, `poc-windows-guest/first_logon.ps1:36`. Driver paths reference `\2k25\` literally. The eventual Lima integration needs a templating layer that substitutes both the Windows version (`2k22`, `2k25`, `11`, etc.) and the virtio-win release. The current PoC layout — one fixed `2k25` subdir — should not get codified as the long-term answer-file format.

- **The PoC reimplements QEMU/virtiofsd lifecycle handling instead of reusing Lima's** `(future)` [Codex GPT 5.5]. The in-tree QEMU driver under `pkg/hostagent` already handles socket readiness, vhost-process shutdown, and structured logging. If this graduates into Lima proper, the Windows guest path should reuse that machinery; the current `run.sh` failures around `virtiofsd` (I2) are exactly the failure modes the driver already solves.

**Strengths**

- The PoC keeps all new code under one directory and touches no existing Lima Go or template code, making the experiment trivial to delete or evolve.
- VirtIO-FS (rather than 9p or SMB) on the host side matches the QEMU recommendation in #4852 and the existing Lima VirtIO-FS path on macOS, so the host-side plumbing is reusable.
- `<SynchronousCommand>` for first-logon (`autounattend.xml:131`) is the correct primitive — Windows blocks boot until the script completes, so installation failures are observable rather than racing against early SSH access.
- SSH key placement at `C:\ProgramData\ssh\administrators_authorized_keys` with the `icacls /inheritance:r` + `SYSTEM:F` + `Administrators:F` ACL set [Gemma4 31B] correctly handles the specific OpenSSH-for-Windows rule that admin-group accounts authorize via a system-wide file, not `%USERPROFILE%\.ssh`.

## Testing Assessment

No tests added. Acceptable for a PoC, but the eventual feature needs:

1. **First-logon failure-mode coverage**: a smoke check inside `first_logon.ps1` that asserts the WinFSP MSI install succeeded and `virtiofs.exe` exists before `New-Service` runs. Addresses I4, I6, I7.
2. **Drive-letter-independence test**: either a runtime probe (find ISOs by volume label) or an integration test that boots the guest and asserts `Z:\` is mounted after first reboot. Addresses I7.
3. **Idempotent re-launch test**: running `run.sh` a second time against an already-installed VM should preserve the qcow2. Addresses I1.
4. **virtiofsd lifecycle test**: verify the socket is removed and no orphan `virtiofsd` remains after a clean QEMU exit or Ctrl-C. Addresses I2.
5. **Cross-distro QEMU smoke**: at minimum, document which distros are known to work; better, replace `pc-q35-noble` with `q35` so the script runs on Fedora/Debian/Arch without modification. Addresses I8.
6. **Offline / proxy-blocked install path**: the chocolatey bootstrap (I6) and `Add-WindowsCapability` (which pulls from Microsoft FoD) both require unrestricted internet — CI behind a corporate proxy will fail. Document the network requirements or pre-bundle the dependencies.

## Documentation Assessment

`README.md` is the only documentation, written as a runbook rather than a design note. That fits a PoC, but two improvements would significantly raise the floor for the next reader:

- **Distinguish "create/rebuild" from "launch existing"**: once I1 lands, the README should make the two flows explicit. Today's text reads as "launch the VM," but the script actually means "wipe and reinstall."
- **State hard prerequisites alongside package names**: the existing "Tested on Ubuntu 24.04" note understates the `pc-q35-noble` and `--accel kvm` constraints. List them as hard requirements (Ubuntu 24.04 QEMU, KVM-enabled host, user in the `kvm` group) so a reader knows up-front whether their setup will work.
- **Note the host-data-exposure caveat**: the README should flag that the guest can read and write any file under the shared directory, including the qcow2 backing its own disk (I5).
- **Fix the find-and-replace artifact noise**: see I3 (tag mismatch). Once that and S4 are fixed, the README and `run.sh` agree, and the volume label is no longer misspelled.

The Lima top-level documentation does not yet reference this directory (confirmed via grep) — which is correct for a PoC. Do not link it from the main README until the design issues above are settled.

### Skill improvements

- **Per-agent wall-clock is undercounted for local (opencode-backed) agents and may be undercounted for cloud agents too.** `collect-metrics/session_opencode.go` computes `duration_s` as the span between the first and last `timestamp` field in `raw-{kind}-pass-{N}.ndjson`. opencode's first `step_start` event fires only when the model begins streaming its first response tokens — that excludes model load AND all server-side prompt processing on every preceding HTTP request. On this review, the qwen pass showed an ndjson event window of 5 min 25 s while Ollama's own server log showed actual model-side wall-clock of 13 min 48 s (~60 % undercount; ~8 min of model load and multi-turn prompt processing were invisible to the ndjson stream). The same event-bounds approach applies to Claude/Codex/Gemini; cloud-agent CLIs emit their first session event much earlier, so the undercount is smaller, but it is still nonzero. Capture the actual process start and end times in `launch.go` (immediately before and after `cmd.Run()`) and write them to e.g. `times-{kind}-pass-{N}.json`; `collect-metrics` should prefer those timestamps over event bounds for all agent kinds.

- **Local-model format adherence is nondeterministic across conversation length and context size, not a fixed property of the model.** This review saw a dense 27 B local model (Qwen 3.5 dense decoder per Ollama's `arch=…` log) produce hallucinated XML-shaped markup on a multi-turn 16 KB-diff review while the same model produced clean structured findings on a 7 KB single-turn review three hours earlier. Do not treat a single bad run as evidence of a persistent model failure: the skill's existing failure handling already drops empty/malformed local-model passes from consolidation, and one such miss is the expected cost of including a local-model slot. Before swapping a model line in `launch.go` on the strength of one run, check the sidecar JSON's `consolidated_findings` for the same agent across 3+ prior reviews of varying diff sizes — if the agent regularly joins consensus findings on small/single-turn reviews but fails on larger/multi-turn ones, the right response is to scope the agent's use, not to swap the model.

I2. **`virtiofsd` is never cleaned up; orphans across runs** — `poc-windows-guest/run.sh:22`

```bash
# Run virtiofsd in background
virtiofsd --shared-dir=./ --socket-path=./virtiofsd.sock --sandbox none &

# Run QEMU
qemu-system-x86_64 \
    ...
```

`virtiofsd` is launched with `&` and never killed. When QEMU exits (clean shutdown, Ctrl-C, or crash), the helper stays alive holding `./virtiofsd.sock`. The next run then races with the orphan: the new `virtiofsd` may `bind` over the leftover socket, or QEMU may connect to the wrong one. The `set -e` at line 5 also does not catch a backgrounded process failure — if `virtiofsd` exits immediately (missing binary, bad arg, sock already busy), the script continues and QEMU prints a confusing "connection refused".

Fix:

```diff
+trap 'kill "${VIRTIOFSD_PID:-0}" 2>/dev/null || true; rm -f ./virtiofsd.sock' EXIT
+
 virtiofsd --shared-dir=./ --socket-path=./virtiofsd.sock --sandbox none &
+VIRTIOFSD_PID=$!
+# Give it a moment to bind, then confirm it's still alive.
+sleep 1
+kill -0 "$VIRTIOFSD_PID" 2>/dev/null || { echo "virtiofsd failed to start" >&2; exit 1; }
```

I4. **Chocolatey installer pulled from the internet without pinning** — `poc-windows-guest/first_logon.ps1:29`

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force; [System.Net.ServicePointManager]::SecurityProtocol = [System.Net.ServicePointManager]::SecurityProtocol -bor 3072; iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))
```

The chocolatey bootstrap is `iex`-ed straight from the public CDN every time a VM is provisioned, with no checksum, no pinned commit, and no fallback. Any compromise of `community.chocolatey.org` (or a man-in-the-middle on the install URL) is full code-execution as Administrator on the guest. WinFSP is the only package needed; consider downloading the WinFSP MSI directly from the project's GitHub release (pinned to a SHA-256 digest of the MSI bytes) and skipping chocolatey entirely. This also drops a heavy dependency.

Fix: replace the chocolatey-based path with a direct, pinned WinFSP download:

```powershell
$winfspUrl    = 'https://github.com/winfsp/winfsp/releases/download/v2.0/winfsp-2.0.23075.msi'
$winfspSha256 = '<sha256-of-the-msi>'
$msi = "$env:TEMP\winfsp.msi"
Invoke-WebRequest -Uri $winfspUrl -OutFile $msi -UseBasicParsing
$got = (Get-FileHash -Algorithm SHA256 $msi).Hash
if ($got -ne $winfspSha256) { throw "winfsp checksum mismatch: $got" }
Start-Process msiexec.exe -Wait -ArgumentList "/i `"$msi`" /qn"
```

## Design Observations

**Concerns**

- **Plaintext credentials in the answer file** (autounattend.xml:111, 142). The local-account password and the autologon password are stored in plaintext, intended only for the one-shot SSH-key bootstrap. Once the public key is installed, the password is the weakest link in the guest's auth surface. For the eventual non-PoC integration, the account should be created with a strong random password, or with a `<Password><Value>` field that was base64-encoded with the `UserAccounts\AdministratorPassword` convention Microsoft documents (still trivially decoded, but reduces accidental over-the-shoulder leakage). Calling it out now so the production design accounts for it. `(future)`

- **`autounattend.xml` and `first_logon.ps1` will diverge per Windows version**. Driver paths reference `\2k25\` literally (autounattend.xml:9-22; first_logon.ps1:36). The eventual Lima integration will need a templating layer that substitutes the Windows version and the virtio-win release. Worth keeping in mind so the PoC layout (one fixed `2k25` subdir) doesn't get codified as the answer-file format. `(future)`

**Strengths**

- The PoC keeps everything in one directory and does not touch any existing Lima Go or template code, making the experiment trivial to delete or evolve. The `.gitignore` correctly excludes all generated artifacts.
- Using VirtIO-FS (rather than 9p or SMB) on the host side aligns with the QEMU recommendation referenced in #4852 and with the existing Lima VirtIO-FS path on macOS, so the host-side plumbing is reusable.
- Use of `SynchronousCommand` for first-logon (autounattend.xml:131) is the correct primitive — Windows will block boot until the script completes, so installation failures are observable rather than races against early SSH access.
