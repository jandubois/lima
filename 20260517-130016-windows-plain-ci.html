Executive Summary ¶

This branch adds first-class support for "plain Windows" hosts — Windows 10/11 without Git for Windows, MSYS2, or Cygwin — across SSH path translation, copy tooling, decompression, and CI. The core mechanism is sshutil.IsSSHCygwin (probes for cygpath.exe alongside ssh.exe) and sshutil.PathForSSH (a single chokepoint that picks Cygwin-style paths for Cygwin ssh and forward-slash native paths for Win32-OpenSSH). The host-agent reverse-sshfs mount, identity files, control paths, and limactl copy all funnel through this helper. Two new CI jobs (windows-plain-wsl2, windows-plain-qemu) uninstall MSYS2 / Git for Windows from windows-2025 runners and verify across filesystem, PATH, binaries, and registry that the host is genuinely vanilla before exercising a create/start/shell/copy/stop/delete cycle. The branch also drops the external gzip dependency (in-process compress/gzip) and removes the deprecated _LIMA_WINDOWS_EXTRA_PATH env var.

Three issues warrant attention before merge: PathForSSH invokes a bare cygpath from PATH rather than the sibling next to the detected ssh.exe, breaking the toolchain-tracking contract when $SSH points outside PATH; the plain QEMU CI job never adds C:\Program Files\QEMU to PATH after windows_plain_host recomposes it, so qemu-system-x86_64 likely is not resolvable for the smoke test; and the new in-process gzip path drops context cancellation responsiveness that the external subprocess provided.

func WindowsSubsystemPath(ctx context.Context, orig string) (string, error) {
	out, err := exec.CommandContext(ctx, "cygpath", filepath.ToSlash(orig)).CombinedOutput()
	if err == nil {
		return strings.TrimSpace(string(out)), nil
	}
	logrus.WithError(err).Debugf("cygpath unavailable for %q, attempting native conversion", orig)
	if vol := filepath.VolumeName(orig); len(vol) == 2 && vol[1] == ':' {
		// We expect orig[2:] to begin with a separator for absolute drive
		// paths (C:\foo, C:/foo). Insert the slash explicitly so a
		// drive-relative input like "C:foo" does not collapse to "/cfoo".
		tail := strings.TrimPrefix(filepath.ToSlash(orig[2:]), "/")
		converted := "/" + strings.ToLower(vol[:1]) + "/" + tail
		logrus.Debugf("native cygpath fallback: %q -> %q", orig, converted)
		return converted, nil
	}
	return "", fmt.Errorf("cannot convert %q to a Cygwin-style path: cygpath unavailable and input is not a drive-letter path", orig)
}

func ParseOpenSSHVersion(version []byte) *semver.Version {
	// Matches "OpenSSH_8.4p1 ..." (upstream) and "OpenSSH_for_Windows_9.5p2 ..."
	// (Win32-OpenSSH). The optional "_for_Windows" prefix is recognized so native
	// Windows OpenSSH is not misdetected as version 0.0.0.
	regex := regexp.MustCompile(`(?m)^OpenSSH(?:_for_Windows)?_(\d+\.\d+)(?:p(\d+))?\b`)
	matches := regex.FindSubmatch(version)
	if len(matches) == 3 {
		if len(matches[2]) == 0 {
			matches[2] = []byte("0")
		}
		return semver.New(fmt.Sprintf("%s.%s", matches[1], matches[2]))
	}
	// Version-gated behaviour (cipher selection, scp URL form) silently
	// downgrades when we return 0.0.0, so log the unparsed banner to make
	// the cause traceable in --debug output.
	logrus.Debugf("ParseOpenSSHVersion: no match in %q; returning 0.0.0", string(version))
	return &semver.Version{}
}

Current version-gated checks (< 8.0.0, < 8.1.0) are unaffected, but the test pins OpenSSH_for_Windows_9.5p2 → 9.5.2 without covering the longer form. When the fallback also errors (UNC path, no cygpath), the user sees cannot convert ... to a Cygwin-style path with no trace of why cygpath actually failed (missing? refused the input? crashed?). Include out in the debug log:

-logrus.WithError(err).Debugf("cygpath unavailable for %q, attempting native conversion", orig)
+logrus.WithError(err).Debugf("cygpath unavailable for %q (output: %q), attempting native conversion", orig, strings.TrimSpace(string(out)))

This is a gap rather than a regression — the pre-PR code also dropped the stderr buffer — but it is the right time to fix it given the new fallback path swallows the failure silently.

Design Observations ¶

Strengths ¶

  • PathForSSH as a single chokepoint (in-scope) Claude Codex Gemini — Funnelling every Windows path translation through one helper that consults IsSSHCygwin is much easier to reason about than scattering the cygpath-vs-native decision across copy, identity, control path, and reverse-sshfs sites. Cache is keyed by resolved absolute path (not first-call-wins), which is correct.
  • Four-layer verification in windows_plain_host (in-scope) Claude — Checking filesystem, PATH, smoking-gun binaries, and registry uninstall keys is genuinely robust against runner-image drift. A future runner-image change that reinstalls Git for Windows under a new path fails the job loudly instead of silently regressing coverage.
  • Reading symlink targets from git's object store in windows_plain_templates (in-scope) Claude — Short-circuits the NTFS-symlink-vs-stub representation question. The StartsWith("$srcRoot/") guard against .. underflow is good defence-in-depth.
  • Windows mux stripping applied uniformly (in-scope) Codex — SSHOptsRemovingControlPath is now invoked from limactl shell, hostagent, scp, and rsync paths so native Windows OpenSSH (no mux support) and Cygwin ssh (unreliable mux over sftp) get the same treatment.
  • Invoke-Limactl wrapper for native exit codes (in-scope) Claude — $ErrorActionPreference = 'Stop' doesn't propagate to native commands, and the wrapper makes the smoke test fail closed.

Concerns ¶

  • PathForSSH's API is still partly Boolean (in-scope) Codex — IsSSHCygwin answers "Cygwin or native," but the correct conversion depends on the exact installation. Carrying the resolved sibling cygpath.exe through PathForSSH (see I1) makes multiple Windows SSH toolchains predictable.

Testing Assessment ¶

The added coverage matches the scope of the change well: two new CI jobs exercise the create/start/shell/copy/stop/delete cycle on a verified-vanilla host for both QEMU and WSL2 drivers, and TestParseCopyPathsWindowsDriveLetter pins the C:\foo vs C:foo semantic split. Open gaps:

  1. In-process gzip is not covered on Windows. The very platform that motivated the change still skips the test (S2).
  2. PathForSSH's native (non-Cygwin) branch has no unit test. A t.TempDir()-based test stubbing sshExe.Exe to a path without a sibling cygpath.exe would catch a future regression cheaply (S4).
  3. UNC path handling is documented but unverified (S3).
  4. Win32-OpenSSH 4-part version banners are not in the test table (S5).
  5. No CI assertion verifies qemu-system-x86_64 resolves after windows_plain_host. A pwsh -c "Get-Command qemu-system-x86_64" between the plain-host step and the smoke test would catch I2.
  6. Plain-Windows smoke test never sets up a mount. The default template's mounts: [] means the QEMU plain job never exercises native sftp-server — the very feature the docs explicitly mention needing OpenSSH.Server for. Adding --mount (or a template with a mounts: entry) to the QEMU smoke test would close this gap.
  7. No test covers $SSH pointing at a Cygwin ssh.exe whose directory is not first on PATH. That scenario is what triggers I1.

Documentation Assessment ¶

The "don't swap toolchains mid-instance-life" warning is in the right place.
