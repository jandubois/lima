## Executive Summary

This PR adds driver-controlled image format conversion to Lima's download cache. When a driver declares its supported formats and the cached image is incompatible, the downloader converts it to RAW in an `imgconv/raw` subdirectory while preserving the original. The approach correctly addresses issue #4056 (avoid repeated qcow2-to-raw conversion for VZ) and the design — preserving originals, threading driver capabilities through `SupportedImageFormats` — is sound. However, three critical regressions block merge: the new rename optimization in `EnsureDisk` skips disk expansion to the configured size, format detection runs on still-compressed data, and the digest overwrite in `getCached` bypasses both digest-based and Last-Modified cache invalidation for converted images.

## Critical Issues

1. **`EnsureDisk` skips disk expansion when format matches** — `pkg/driverutil/disk.go:104-107` (critical, regression) [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro] (pass 2)

Before this PR, the non-ISO path always called `diskUtil.Convert` with `&diskSizeInBytes`, which expanded the disk to the user-configured size (typically 100 GiB). The new code at line 104 short-circuits with `os.Rename` when the detected format matches the target, skipping expansion entirely. This is the *primary path* this PR enables: `ensureRawInCache` converts qcow2 to raw with `nil` size (line 432), the raw image at the original size (2-10 GB) is copied to the instance directory, and `EnsureDisk` detects raw == raw — so the VM boots with an undersized disk.

The existing test `TestEnsureDisk_WithNonISOImage` (disk_test.go:134) passes `"2MiB"` but `checkDisk` (disk_test.go:63-69) only asserts `fi.Size() > 0`, so the regression passes CI silently.

```diff
+        if err := diskUtil.ResizeDisk(ctx, diskPath, diskSizeInBytes); err != nil {
```

2. **Compressed images routed to raw-conversion before decompression** — `pkg/downloader/downloader.go:282-285,389-402` (critical, regression) [Codex GPT 5.4, Gemini 3.1 Pro]

```go

    imageFormat, err := nativeimgutil.DetectFormat(shadData)

// In fetch() — line 389:

    format, err := nativeimgutil.DetectFormat(shadData)
```

Both `getCached` and `fetch` call `DetectFormat` on the cached payload before `copyLocal` decompresses it. For common images like `*.qcow2.gz` or `*.img.xz`, `DetectFormat` reads compressed bytes and either fails or misidentifies the format. Even if conversion succeeds, `copyLocal` (line 335, 406) still receives `ext = path.Ext(remote)` (e.g., `.gz`), so it attempts to decompress an already-raw file.

Fix: defer format detection and conversion until after decompression. Alternatively, decompress in the cache before detecting the format, and clear the extension so `copyLocal` does not attempt decompression on the converted artifact.

3. **Cache invalidation bypassed for converted images** — `pkg/downloader/downloader.go:314-323` (critical, regression) [Gemini 3.1 Pro]

```go
} else {
    shadData = rawImgConvPath
    if currentDigestData, err := os.ReadFile(rawImgConvDigestPath); err == nil {
        currentDigest := strings.TrimSpace(string(currentDigestData))
        if d, err := digest.Parse(currentDigest); err == nil {
```

When the raw cache exists and `raw.digest` is valid, `o.expectedDigest` is overwritten with the raw digest before validation at line 332. Two consequences:

- **Template digest updates ignored:** If a user updates their template with a new digest, `getCached` replaces it with the cached raw digest and validates against itself — always passing. The stale converted image is served without re-download.
- **Last-Modified checks skipped:** For digest-less images, `shadDigest` is initially empty, which would trigger a `matchLastModified` check at line 339. The overwrite at line 320 sets `shadDigest` to a path that exists, causing `os.Stat(shadDigest)` at line 329 to succeed and bypassing the freshness check entirely.

Fix: validate the original cached image against the user's expected digest (or Last-Modified) *before* switching to the converted artifact. Only after validation passes should the code select `imgconv/raw` as the source.

## Design Observations

### Concerns

1. **Cache layer conflates transport encoding with image format conversion** (in-scope) [Codex GPT 5.4]

The old model was simple: cache the downloaded bytes, then optionally decompress on copy-out. This PR adds image-format conversion inside the cache while keying decompression off the original URL extension. That creates state combinations like "raw bytes stored under a `.qcow2.gz` URL" — the root cause of Critical Issue 2. A cleaner split: first normalize transport encoding (decompress to the canonical image), then decide whether a driver-specific converted artifact is needed. This removes a whole class of extension/content mismatches.

2. **Conversion and disk creation split across two components with no coordination** (in-scope) [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro]

The cache layer converts format with `nil` size, while `EnsureDisk` handles both format conversion and sizing. The new rename optimization in `EnsureDisk` shows the risk: skipping format conversion inadvertently skipped sizing. A safer separation: the cache owns format (converting to a driver-compatible format at the image's natural size), and `EnsureDisk` always owns sizing (calling `ResizeDisk` regardless of whether it also converted).

### Strengths

1. **Original image preservation.** Keeping the original cached download in `data` and placing the conversion in `imgconv/raw` lets different drivers share the same cache entry without conflict, directly addressing the determinism concern raised by @AkihiroSuda. [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro]

2. **Driver capability threading.** Routing format decisions through `DriverFeatures.SupportedImageFormats` is a better ownership boundary than hard-coding VZ-specific behavior in the downloader. [Codex GPT 5.4]

3. **Rename optimization for matching formats.** The idea of renaming instead of converting when formats match (driverutil/disk.go:104-109) is sound — it just needs the resize step added back. [Gemini 3.1 Pro]

This PR moves image-format compatibility decisions into the download/cache path and adds a driver-advertised `SupportedImageFormats` list so unsupported images can be converted into `imgconv/raw` while keeping the original artifact. The main new issue I found is a real provisioning regression in `driverutil.EnsureDisk`: once the downloaded image already matches the target format, Lima now skips the resize/expand step and can boot VZ/Krunkit guests with a disk smaller than the configured `disk` size.

### Findings

**Critical Issues**

None.

**Important Issues**

1. **Same-format images now bypass disk-size expansion** — `pkg/driverutil/disk.go:104-109` (important, regression)

```go
if format == string(diskImageFormat) {
	if err = os.Rename(imagePath, diskPath); err != nil {
		return err
	}
} else {
	if err := diskUtil.Convert(ctx, diskImageFormat, imagePath, diskPath, &diskSizeInBytes, false); err != nil {
```

`driverutil.EnsureDisk()` is called from `pkg/driver/vz/vz_driver_darwin.go:338` and `pkg/driver/krunkit/krunkit_driver_darwin_arm64.go:64`. Before this change, the non-ISO path always went through `diskUtil.Convert(..., &diskSizeInBytes, ...)`, so the final VM disk was expanded to the configured size. The new fast path at `pkg/driverutil/disk.go:104-107` just renames `image` to `disk` when the source format already matches the target format, which skips that expansion entirely. In practice, a raw base image cached for VZ/Krunkit that is smaller than `inst.Config.Disk` will now produce an undersized guest disk, even though the config requested a larger one.

Fix: keep the same-format short-circuit only for format conversion, not for resizing. After detecting `format == string(diskImageFormat)` at `pkg/driverutil/disk.go:104`, either:
```diff
- if format == string(diskImageFormat) {
- 	if err = os.Rename(imagePath, diskPath); err != nil {
- 		return err
- 	}
- } else {
+ if format == string(diskImageFormat) {
+ 	if err = os.Rename(imagePath, diskPath); err != nil {
+ 		return err
+ 	}
+ 	if err := diskUtil.ResizeDisk(ctx, diskPath, diskSizeInBytes); err != nil {
+ 		return fmt.Errorf("failed to resize %q: %w", diskPath, err)
+ 	}
+ } else {
  	if err := diskUtil.Convert(ctx, diskImageFormat, imagePath, diskPath, &diskSizeInBytes, false); err != nil {
```

or keep using `Convert(..., &diskSizeInBytes, ...)` for same-format images and optimize inside the image util layer, where resize semantics can stay consistent.

**Suggestions**

None.

### Design Observations

**Concerns**

- `(in-scope)` Disk normalization and disk provisioning are now split across the downloader and `EnsureDisk`, but only `EnsureDisk` knows the final size requirement. The new rename shortcut in `pkg/driverutil/disk.go:104-107` shows the risk: format optimization leaked across that boundary and skipped a provisioning invariant. A safer split is: downloader decides cached source format, while `EnsureDisk` remains the sole owner of producing the final on-disk artifact with the correct type and size.

**Strengths**

- Keeping the original cached image alongside `imgconv/raw` is a good direction. It avoids the cross-driver cache nondeterminism discussed in the PR thread and preserves a reusable source artifact for drivers that prefer non-raw formats.

### Testing Assessment

1. No test covers the new same-format fast path in `pkg/driverutil/disk.go:104-107`, especially a source raw/asif image smaller than the configured disk size.
2. No integration-style test exercises the full VZ/Krunkit path where the downloader leaves a raw image in place and `CreateDisk()` still has to honor `inst.Config.Disk`.

### Documentation Assessment

The cache-layout doc update in `website/content/en/docs/dev/internals.md` is adequate for the new `imgconv/raw` artifacts. No additional documentation gap stood out beyond the behavioral bug above.

### Acknowledged Limitations

- `"SupportedImageFormats: []string{string(qcow2.Type), string(rawImage.Type), string(vmdk.Type), string(vhdx.Type)}, // Not a comprehensive list of supported formats"` — `pkg/driver/qemu/qemu_driver.go:696`

  This comment correctly acknowledges that QEMU's advertised format list is incomplete. Because this list now influences cache conversion behavior, the incompleteness matters more than it used to, but it is already documented inline.

### [Gemini] — Pass 1

### Executive Summary

This PR introduces an image conversion cache layer that converts downloaded images (e.g., QCOW2) into formats natively supported by the current VM driver (e.g., RAW for VZ) before copying them to the instance directory. While this improves boot determinism and speeds up subsequent boots, moving domain-specific conversion logic into the core caching mechanism introduces critical regressions: it breaks caching invalidation logic by erroneously overwriting validation variables, and it completely breaks the download flow for compressed image formats (like `.gz` and `.zst`) because conversion is attempted on the raw, still-compressed stream.

### Findings

#### Critical Issues

1. **Cache invalidation is completely bypassed for converted images** — `pkg/downloader/downloader.go:297-321` (critical, regression)

```go
			} else {
				shadData = rawImgConvPath
				if currentDigestData, err := os.ReadFile(rawImgConvDigestPath); err == nil {
					currentDigest := strings.TrimSpace(string(currentDigestData))
					if d, err := digest.Parse(currentDigest); err == nil {
						o.expectedDigest = d
						shadDigest = rawImgConvDigestPath
					}
				}
			}
// ... later in the function ...
	if _, err := os.Stat(shadDigest); err == nil {
		if err := validateCachedDigest(shadDigest, o.expectedDigest); err != nil {
```

When checking the cache for an image that requires conversion, `getCached` conditionally overwrites both the `shadDigest` path and the user's `o.expectedDigest` value with the converted raw image's properties. This overwrite occurs *before* validating the original cached image. 
If a user updates their template with a new `expectedDigest`, `getCached` will overwrite that new digest with the old `raw.digest` value from the cache, compare it against itself (which always succeeds), and incorrectly return the stale converted image.
Additionally, if no `expectedDigest` is provided, `shadDigest` is initially empty, which would normally trigger a `matchLastModified` check against the server. Because it gets overwritten with the path to `raw.digest` (which exists), `os.Stat(shadDigest)` succeeds and the `matchLastModified` check is completely skipped, meaning digest-less images will be cached indefinitely without checking for upstream updates.

Fix: Perform cache validation (digest or last-modified check) on the *original* cached file (`shadData`) *before* evaluating conversion and overwriting the validation variables.

2. **Image conversion in cache fails for compressed images** — `pkg/downloader/downloader.go:389-402` (critical, regression)

```go
	// Try to convert to raw if it's an non-ISO image and not supported by the driver
	if isNonISOImage(o.description, remote, o.supportedImageFormats) {
		format, err := nativeimgutil.DetectFormat(shadData)
		if err != nil {
			return nil, err
		}
```

The PR moves image format detection and conversion into the cache download phase (`fetch` and `getCached`). It calls `nativeimgutil.DetectFormat(shadData)` directly on the bytes downloaded from the server. However, templates can specify compressed images (e.g., `.qcow2.gz` or `.zst`), and the caching layer intentionally defers decompression until `copyLocal` is called later.
As a result, `DetectFormat` will attempt to parse a compressed stream, which will either fail with an error or incorrectly identify the stream as a "raw" disk. If it proceeds to `ensureRawInCache`, `qemu-img convert` will fail because it cannot convert a compressed stream.

Fix: If conversion is to occur in the cache, the caching layer must decompress the stream *before* attempting format detection and conversion. Alternatively, keep the conversion logic at the instance preparation phase (`driverutil.EnsureDisk`) where it correctly operated on the already decompressed `imagePath`.

#### Important Issues

3. **Parsing `description` for flow control is brittle** — `pkg/downloader/downloader.go:357` (important, gap)

```go
func isNonISOImage(description, remote string, supportedImageFormats []string) bool {
	return strings.Contains(description, "image") && supportedImageFormats != nil && path.Ext(remote) != ".iso"
}
```

The code uses `strings.Contains(description, "image")` to determine if a download is a disk image that requires format conversion. Relying on a human-readable UI description string for core control flow is brittle and will break if the string is localized, slightly modified, or accidentally altered by a template change.

Fix: Since `pkg/instance/start.go` only passes `supportedImageFormats` for the main image and passes `nil` for kernels and initrds, checking `len(supportedImageFormats) > 0` is sufficient. Remove the `strings.Contains` check entirely.

#### Suggestions

4. **`ValidatedDigest` evaluates to true when no expected digest was provided** — `pkg/downloader/downloader.go:414` (suggestion, gap)

```go
	res := &Result{
		Status:          StatusDownloaded,
		CachePath:       shadData,
		LastModified:    readTime(shadTime),
		ContentType:     readFile(shadType),
		ValidatedDigest: o.expectedDigest != "",
	}
```

If a user does not provide an expected digest, `fetch` generates one for the raw converted file and sets `o.expectedDigest = rawDigest`. When returning the `Result`, `ValidatedDigest` evaluates to `true` (because the variable is no longer empty), which incorrectly signals to callers that the original downloaded file was verified against a user-provided expected digest.

Fix: Capture `validatedDigest := o.expectedDigest != ""` at the start of `fetch` or `getCached` and return that captured value.

### Design Observations

**Concerns**
- **Overloading the Cache Layer (in-scope):** Moving format conversion into `pkg/downloader` conflates HTTP downloading/caching with domain-specific virtual machine image logic. The cache layer now depends on `nativeimgutil` and `proxyimgutil` (which internally depends on `qemu-img`). A cleaner architectural approach would be having the cache only manage raw downloaded bits (and their decompression), while letting the VM drivers explicitly handle format conversions on their own time, possibly maintaining a shared pool of converted images if deduplication is desired.

**Strengths**
- Renaming the file instead of converting it when the source format matches the target format (`pkg/driverutil/disk.go` lines 105-109) is an excellent optimization that avoids unnecessary `qemu-img` processing and speeds up initialization.

### Testing Assessment

1. **Cache Invalidation:** There are no tests verifying that updating the `expectedDigest` of an already-cached and converted image triggers a cache miss and re-download.
2. **Last-Modified Checks:** There are no tests ensuring that a converted image without a digest is correctly updated when the upstream server updates the `Last-Modified` header.
3. **Compressed Images:** There are no end-to-end tests validating the download and conversion flow of a compressed image (e.g., `.qcow2.gz`).

### Documentation Assessment

The internal documentation update in `website/content/en/docs/dev/internals.md` accurately reflects the new directory structure for the `imgconv` cache feature.

  Low-priority: The PR author did not adjust the slice order in `pkg/driver/vz/vz_driver_darwin.go:431`. While `ensureRawInCache` currently hardcodes conversion to `raw` so the order doesn't functionally matter yet, the slice order remains unaddressed.

### [Gemini] — Pass 2

### Executive Summary

This PR modifies the image caching layer to pre-convert downloaded non-ISO images into formats supported by the active VM driver (falling back to RAW). While Pass 1 identified issues with decompression sequencing and cache invalidation, this second pass uncovered a critical regression in disk provisioning: when the cached image format already matches the driver's required format, the fast-path bypasses the disk resizing step entirely, causing VMs to boot with severely undersized disks.

### Findings

#### Critical Issues

1. **`EnsureDisk` bypasses disk resizing when formats match** — `pkg/driverutil/disk.go:102-106` (critical, regression)

```go
		if format == string(diskImageFormat) {
			if err = os.Rename(imagePath, diskPath); err != nil {
				return err
			}
		} else {
```

When the cached image format matches the requested `diskImageFormat` (which is highly likely now that images are proactively converted to `raw` in the cache for VZ and Krunkit drivers), the code takes a fast-path that simply renames the decompressed image to the disk path. However, it completely bypasses the `diskUtil.Convert` call, which was previously responsible for expanding the disk image to the user-requested virtual size (`diskSizeInBytes`). 

As a result, VMs using the VZ or Krunkit drivers will boot with a disk identical in size to the base OS image (typically just a few gigabytes), completely ignoring the `disk` size configured in `lima.yaml` (default 100GiB) and likely leading to immediate disk-full errors.

Fix: Explicitly invoke the resize operation after renaming the image on the fast-path:
```diff
 		if format == string(diskImageFormat) {
 			if err = os.Rename(imagePath, diskPath); err != nil {
-				return err
+				return fmt.Errorf("failed to rename image: %w", err)
+			}
+			if err := diskUtil.ResizeDisk(ctx, diskPath, diskSizeInBytes); err != nil {
+				return fmt.Errorf("failed to resize %q to %d bytes: %w", diskPath, diskSizeInBytes, err)
 			}
 		} else {
```

#### Important Issues

2. **Redundant and ineffective sparsification block in `ensureRawInCache`** — `pkg/downloader/downloader.go:440-451` (important, gap)

```go
	// Ensure the image is sparse to save cache space.
	rawTmpF, err := os.OpenFile(rawPathTmp, os.O_RDWR, 0o644)
	if err != nil { ... }
	fi, err := rawTmpF.Stat()
	if err != nil { ... }
	if err := diskUtil.MakeSparse(ctx, rawTmpF, fi.Size()); err != nil {
		logrus.WithError(err).Warnf("Failed to make %q sparse (non-fatal)", rawPathTmp)
	}
```

The code intends to sparsify the newly converted raw image by calling `diskUtil.MakeSparse` with the file's current size (`fi.Size()`). However, the implementation of `makeSparse` for native conversions (in `nativeimgutil`) simply executes `f.Truncate(offset)`. Calling `Truncate` on a file using its exact existing physical size does not "punch holes" or reclaim space from blocks of zeros; it is a literal no-op on all operating systems. 

Fix: Remove this entire block. The preceding `diskUtil.Convert` method already handles sparsification appropriately (e.g., `qemu-img` handles it naturally, and native conversion pre-truncates the file to the virtual size before writing, which relies on the OS creating sparse holes). If post-conversion hole punching were truly required, it would need OS-specific syscalls (like `fallocate` with `FALLOC_FL_PUNCH_HOLE`), not `Truncate`.

#### Suggestions

None.

### Design Observations

**Concerns**
- `(in-scope)` **Fragmented disk management responsibilities:** `diskUtil.Convert` is currently overloaded to perform both format conversion and disk resizing simultaneously. This design led directly to Finding 1 — when the fast-path bypassed the format conversion, it inadvertently bypassed the resize step. Explicitly separating `ConvertFormat` and `ResizeDisk` into independent, composable steps would make the control flow in `EnsureDisk` less fragile.

### Testing Assessment

1. **Test gap masks critical regression:** The test suite for `EnsureDisk` (`TestEnsureDisk_WithNonISOImage` and `TestEnsureDisk_ExistingDisk` in `disk_test.go`) explicitly requests a disk size of `"2MiB"` but the `checkDisk` helper only validates `assert.Assert(t, fi.Size() > 0)`. Because the test does not assert that the final disk size exactly matches the requested `diskSizeInBytes`, the regression (Finding 1) silently passed CI.

### Documentation Assessment

No new documentation gaps identified beyond those captured in the first pass.
